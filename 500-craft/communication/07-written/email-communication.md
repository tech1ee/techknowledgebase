# Email Communication для IT-специалистов

> **TL;DR:** BLUF (Bottom Line Up Front) + 5Cs (Clear, Concise, Complete, Correct, Courteous) = эффективные email. Главная идея: уважай время читателя — ключевое в первом предложении. Применяется в: code review requests, status updates, stakeholder communication.

---

## Зачем это нужно?

### Представьте ситуацию

**Sprint planning.** Вам нужно запросить у Tech Lead разрешение на рефакторинг legacy модуля. Это займёт 3 дня вместо новой фичи.

**Без навыка:**
```
Тема: Вопрос

Привет! Я тут анализировал наш PaymentProcessor и обнаружил, что там
куча проблем с coupling. Знаешь, когда мы добавляли новые методы оплаты
в прошлом квартале, мы накопили технический долг. Я посмотрел метрики —
cyclomatic complexity выросла с 15 до 47. Плюс тесты падают раз в неделю
из-за race conditions. В общем, было бы неплохо это пофиксить, если будет
время. Как думаешь?
```

**Результат:** Email проигнорирован или отложен. Непонятно что нужно решить.

**С навыком:**
```
Тема: DECISION: Рефакторинг PaymentProcessor (3 дня вместо Feature-X)

BLUF: Предлагаю взять 3 дня на рефакторинг PaymentProcessor вместо
Feature-X. Прошу твоё решение до пятницы 15:00.

Почему сейчас:
• Cyclomatic complexity: 47 (было 15) — каждое изменение рискованно
• CI падает 1x/неделю (race conditions) — блокирует другие PR
• Feature-X зависит от PaymentProcessor — лучше сначала стабилизировать

Варианты:
A) Рефакторинг сейчас → Feature-X на следующий спринт
B) Feature-X сейчас → рефакторинг позже (риск: +20% время на Feature-X)

Моя рекомендация: вариант A.
```

**Результат:** Решение принято за 2 минуты.

## Терминология

| Термин | Что это (простыми словами) | IT-аналогия |
|--------|---------------------------|-------------|
| **BLUF** | Bottom Line Up Front — суть в первом предложении | `return` в начале функции для early exit |
| **5Cs** | 5 принципов: Clear, Concise, Complete, Correct, Courteous | Code review checklist |
| **Action Required** | Явный запрос действия от получателя | Required parameter в API |
| **Subject Line Prefix** | Тег в теме: ACTION, INFO, DECISION | Git commit prefix: feat, fix, docs |
| **Async-first** | Коммуникация, не требующая моментального ответа | Event-driven architecture |

## Как это работает?

### BLUF: Bottom Line Up Front

**Происхождение:** Военный стандарт коммуникации (US Military). Цель — передать критичную информацию даже если сообщение прочитано не полностью.

```
ТРАДИЦИОННЫЙ ПОДХОД:              BLUF ПОДХОД:

┌─────────────────────┐           ┌─────────────────────┐
│     Контекст        │           │   ГЛАВНОЕ (BLUF)    │ ← Читают 100%
├─────────────────────┤           ├─────────────────────┤
│   Предыстория       │           │     Контекст        │ ← Читают 70%
├─────────────────────┤           ├─────────────────────┤
│    Анализ           │           │     Детали          │ ← Читают 40%
├─────────────────────┤           ├─────────────────────┤
│   ГЛАВНОЕ           │ ← Не      │  Call to Action     │ ← Читают 30%
│  (в конце!)         │   дочитали└─────────────────────┘
└─────────────────────┘
```

**BLUF отвечает на 5W:**
- **What:** Что нужно сделать/знать?
- **Who:** Кто должен действовать?
- **When:** К какому сроку?
- **Where:** Где (система, репозиторий)?
- **Why:** Почему это важно?

### 5Cs Framework

```
┌─────────────────────────────────────────────────────────────────┐
│                      5Cs OF EMAIL                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   CLEAR          CONCISE        COMPLETE                        │
│   ┌───┐          ┌───┐          ┌───┐                          │
│   │ ? │ ────►    │ . │ ────►    │ ✓ │                          │
│   └───┘          └───┘          └───┘                          │
│   Понятно        Кратко         Полно                          │
│   с первого      без воды       вся нужная                     │
│   прочтения                     информация                     │
│                                                                 │
│   CORRECT        COURTEOUS                                      │
│   ┌───┐          ┌───┐                                         │
│   │ ✓ │          │ ☺ │                                         │
│   └───┘          └───┘                                         │
│   Без ошибок     Вежливо                                       │
│   факты верны    уважительно                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| C | Принцип | Как применять |
|---|---------|---------------|
| **Clear** | Одна мысль — одно предложение | Избегай вложенных условий |
| **Concise** | Минимум слов, максимум смысла | Удали каждое слово без потери смысла |
| **Complete** | Вся необходимая информация | Получатель может действовать без follow-up |
| **Correct** | Проверенные факты, без опечаток | Грамматика важна — 48% судят строже |
| **Courteous** | Профессиональный тон | Благодарность, уважение времени |

### Subject Line Prefixes

Система тегов в теме письма (Harvard Business Review рекомендация):

| Prefix | Значение | Пример |
|--------|----------|--------|
| **ACTION** | Требуется действие | ACTION: Review PR #456 by Friday |
| **DECISION** | Нужно принять решение | DECISION: Choose auth provider |
| **INFO** | Информация, действия не требуется | INFO: Q4 roadmap published |
| **REQUEST** | Запрос | REQUEST: Access to production logs |
| **URGENT** | Срочно (используй редко!) | URGENT: Production down |
| **FYI** | For Your Information | FYI: Team meeting moved to 3pm |

## Пошаговый процесс

### Подготовка (до написания)

1. **Определи цель:** Что должен СДЕЛАТЬ получатель после прочтения?
2. **Выбери канал:** Email vs Slack vs Call (см. таблицу ниже)
3. **Определи аудиторию:** Технический уровень, контекст

```
ВЫБОР КАНАЛА КОММУНИКАЦИИ:

┌────────────────────────┬─────────────────────────────────────┐
│ Используй EMAIL        │ Используй SLACK                     │
├────────────────────────┼─────────────────────────────────────┤
│ • Формальные запросы   │ • Быстрые вопросы (<3 предложений) │
│ • Внешние stakeholders │ • Внутренняя команда               │
│ • Документируемые      │ • Неформальные обсуждения          │
│   решения              │ • Статус checks                    │
│ • Детальные объяснения │ • FYI без action required          │
│ • Cross-timezone       │ • Brainstorming                    │
│   communication        │ • Ответ нужен <3 часов             │
│ • Ответ нужен >24ч     │                                    │
└────────────────────────┴─────────────────────────────────────┘
```

### Шаг 1: Subject Line

**Правило:** Получатель должен понять суть НЕ открывая email.

| Плохо | Хорошо |
|-------|--------|
| "Вопрос" | "ACTION: Review PR #456 - Auth refactoring" |
| "Важно" | "DECISION: Switch from REST to GraphQL?" |
| "Насчёт митинга" | "INFO: Sprint review moved to Friday 3pm" |
| "FYI" | "FYI: CI pipeline now requires Node 20" |

### Шаг 2: BLUF (первый абзац)

**Структура:**
```
[Что нужно] + [от кого] + [к какому сроку] + [почему важно]
```

**Примеры:**

```
BLUF: Прошу одобрить PR #456 до завтра 15:00, чтобы успеть
задеплоить auth changes до выходных.
```

```
BLUF: Сообщаю о критичном баге в production — потеря данных
у ~100 пользователей. Расследование в процессе, ETA фикса: 2 часа.
```

```
BLUF: Предлагаю перенести deadline Feature-X на неделю.
Нужно твоё решение до конца дня.
```

### Шаг 3: Контекст и детали

**Используй структуру:**
- Bullet points для списков (>2 пунктов)
- Нумерацию для последовательных шагов
- Таблицы для сравнений
- Code blocks для технических деталей

**Правило 3-х абзацев:**
1. BLUF (что нужно)
2. Контекст (почему это важно)
3. Call to Action (конкретное следующее действие)

### Шаг 4: Call to Action (CTA)

**Явно укажи:**
- ЧТО нужно сделать
- К КОГДА
- КАК (если неочевидно)

```
Плохо:  "Дай знать что думаешь"
Хорошо: "Прошу ответить 'approved' или указать замечания до пятницы 15:00"
```

### Шаг 5: Проверка перед отправкой

**Чеклист:**
```
□ Subject line содержит prefix (ACTION/INFO/DECISION)
□ BLUF в первом предложении
□ Все факты проверены
□ Нет опечаток (48% судят строже!)
□ CTA явный и с дедлайном
□ Attachment приложен (если упомянут)
□ Правильные получатели в TO/CC
```

## Скрипты и Templates

### Use Case 1: Code Review Request

**Ситуация:** Нужен review PR от конкретного человека

**Скрипт:**
```
Subject: ACTION: Review PR #456 - Payment validation refactoring

BLUF: Прошу review PR #456 до завтра 12:00. Это блокирует релиз
payment feature в пятницу.

Что изменено:
• Вынес валидацию в отдельный сервис (PaymentValidator)
• Добавил 15 unit tests
• Breaking change: метод validate() теперь async

На что обратить внимание:
• src/validators/PaymentValidator.ts — основная логика
• Обработка edge cases в строках 45-67

PR: [ссылка]
Время на review: ~20 минут

Спасибо!
```

### Use Case 2: Status Update для Stakeholder

**Ситуация:** Еженедельный отчёт менеджеру/клиенту

**Скрипт:**
```
Subject: INFO: Sprint 23 Status Update - On Track

BLUF: Sprint 23 идёт по плану. Завершили 8/10 story points.
Релиз в пятницу без рисков.

Выполнено на этой неделе:
✓ User authentication (JIRA-123)
✓ Payment integration (JIRA-124)
✓ Email notifications (JIRA-125)

В работе:
→ Dashboard redesign (JIRA-126) — 70% готово
→ Performance optimization (JIRA-127) — начато

Риски: отсутствуют

Следующий update: понедельник.
```

### Use Case 3: Запрос Deadline Extension

**Ситуация:** Нужно больше времени на задачу

**Скрипт:**
```
Subject: REQUEST: Feature-X deadline extension (2 days)

BLUF: Прошу перенести deadline Feature-X с пятницы на вторник.
Нужно твоё подтверждение до завтра.

Причина:
При интеграции обнаружил legacy dependency, которая требует
рефакторинга. Без этого новый код будет нестабильным.

Варианты:
A) +2 дня → качественный код, полное покрытие тестами
B) Текущий deadline → workaround, технический долг на Q2

Моя рекомендация: вариант A.

Влияние на другие задачи: отсутствует (Feature-Y не зависит от X).
```

### Use Case 4: Bug Report

**Ситуация:** Сообщение о баге в production

**Скрипт:**
```
Subject: URGENT: Production bug - Payment failures for EU users

BLUF: Payment flow падает для EU пользователей с 14:30.
~200 affected users. Расследование ведётся, ETA фикса: 1 час.

Симптомы:
• Error 500 при оплате картой для EU region
• Stripe webhook не отвечает (timeout)

Причина: Stripe изменил endpoint URL для EU region (без notice)

Текущие действия:
1. [Done] Включён fallback на US region
2. [In Progress] Обновление endpoint в конфиге
3. [Pending] Уведомление affected users

Timeline:
• 14:30 — первые reports
• 14:45 — issue идентифицирован
• 15:00 — fallback включён
• ETA fix: 15:30

Следующий update: через 30 минут или при изменениях.
```

### Use Case 5: Meeting Decline с Alternative

**Ситуация:** Нужно отклонить meeting, но предложить альтернативу

**Скрипт:**
```
Subject: RE: Design Review Meeting - Alternative Proposal

К сожалению, не смогу присутствовать в четверг 15:00
(конфликт с release window).

Альтернативы:
1. Async review: могу оставить детальные комментарии в Figma до среды
2. Reschedule: свободен в пятницу 10:00-12:00
3. Делегировать: [Имя коллеги] в курсе контекста и может меня заменить

Какой вариант предпочтительнее?
```

## Распространённые ошибки

### Ошибка 1: Buried Lead (главное в конце)

**Неправильно:**
```
Я проанализировал production логи за последние 2 недели и нашёл
интересные паттерны. Оказывается, у нас есть проблема с memory leaks
в модуле обработки изображений. Также заметил что CPU usage растёт
по экспоненте после 1000 запросов. В связи с этим рекомендую
приостановить feature развертывание до исправления.
```

**Почему это неправильно:**
При 121 email/день получатель читает первые 12 секунд. Если главное в конце — не дочитает.

**Правильно:**
```
BLUF: Рекомендую приостановить feature deployment до исправления
memory leak (ETA: 2 дня).

Найдено при анализе production логов:
• Memory leak в image processing module
• CPU растёт экспоненциально после 1000 requests
• Риск OOM при текущем трафике: высокий
```

### Ошибка 2: Vague Subject Line

**Неправильно:**
```
Subject: Вопрос
Subject: Важно
Subject: FYI
Subject: Насчёт проекта
```

**Почему это неправильно:**
Невозможно приоритизировать в inbox из 121 email. Часто игнорируется.

**Правильно:**
```
Subject: ACTION: Review PR #456 by Friday
Subject: DECISION: Migrate to PostgreSQL?
Subject: INFO: API deprecation notice - v1 EOL March 1
```

### Ошибка 3: Wall of Text

**Неправильно:**
Один абзац на 300 слов без структуры.

**Почему это неправильно:**
- Невозможно сканировать
- Ключевые пункты теряются
- Получатель не дочитывает

**Правильно:**
- Bullet points для списков
- Короткие абзацы (2-3 предложения)
- Визуальное разделение секций
- Bold для ключевых слов

### Ошибка 4: Missing Call to Action

**Неправильно:**
```
Вот информация о баге. Дай знать если есть вопросы.
```

**Почему это неправильно:**
Неясно что делать. Email откладывается "на потом" = забыт.

**Правильно:**
```
Прошу:
1. Подтвердить что баг в backlog (reply "confirmed")
2. Назначить priority (P1/P2/P3)
3. Deadline для ответа: завтра 12:00
```

### Ошибка 5: Undermining Language

**Неправильно:**
```
"Я просто хотел спросить..."
"Не уверен, но мне кажется..."
"Извините что беспокою, но..."
"Это может быть глупый вопрос..."
```

**Почему это неправильно:**
Подрывает авторитет. Читатель воспринимает как неуверенность в своих идеях.

**Правильно:**
```
"Предлагаю рассмотреть..."
"По моему анализу..."
"Рекомендую..."
"Вопрос по..."
```

## Когда использовать

### Используй Email, когда:

| Ситуация | Причина |
|----------|---------|
| Формальные запросы | Документируется, легко найти позже |
| Внешние stakeholders | Профессиональный стандарт |
| Cross-timezone communication | Async by design |
| Детальные объяснения | Место для структуры и контекста |
| Решения требующие обдумывания | Даёт время на ответ |
| Audit trail нужен | Email сохраняется |

### НЕ используй Email, когда:

| Ситуация | Лучше использовать |
|----------|-------------------|
| Срочный вопрос (<1 час) | Slack + call |
| Быстрый FYI (<3 предложений) | Slack |
| Brainstorming | Meeting или Slack thread |
| Эмоциональный разговор | Видеозвонок |
| Конфликт | Face-to-face или call |
| Нужен real-time диалог | Slack или call |

### Email vs Slack Matrix

```
                    СРОЧНОСТЬ
                    Low         High
              ┌─────────────┬─────────────┐
         High │   EMAIL     │    CALL     │
СЛОЖНОСТЬ     │ (detailed)  │ (immediate) │
              ├─────────────┼─────────────┤
         Low  │   SLACK     │   SLACK     │
              │ (FYI)       │ (+ mention) │
              └─────────────┴─────────────┘
```

## Практика

### Упражнение 1: BLUF Rewrite

**Сценарий:** Перепиши этот email используя BLUF:

```
Привет! Я хотел обсудить с тобой вопрос по поводу нашей CI pipeline.
Вчера я заметил что билды стали занимать на 40% больше времени.
Посмотрел логи и увидел что проблема в кеширвоании npm модулей.
Можно было бы это оптимизировать если добавить правильный cache key.
Думаю это сэкономит нам около 10 минут на каждом билде.
Как думаешь, стоит ли заняться этим на этой неделе?
```

**Эталонный ответ:**
<details><summary>Показать</summary>

```
Subject: DECISION: Оптимизация CI cache (-40% build time)

BLUF: Предлагаю потратить 2 часа на оптимизацию CI cache.
Это сократит build time на 40% (~10 мин/билд).
Нужно твоё OK до завтра.

Проблема:
• Build time вырос на 40% (вчера заметил)
• Причина: неправильный cache key для npm modules

Решение:
• Добавить hash package-lock.json в cache key
• Effort: 2 часа
• Impact: -10 минут на каждом билде
```

<details><summary>Показать</summary>

**Найденные ошибки:**
1. Subject line vague — непонятно о чём речь

## Проверь себя

> [!question]- Коллега написал email с темой "Вопрос" и разместил просьбу одобрить архитектурное решение в последнем абзаце. Какие конкретные ошибки здесь допущены и как их исправить, используя BLUF и Subject Line Prefixes?
> Допущено две ключевых ошибки. Первая — Vague Subject Line: тема "Вопрос" не позволяет приоритизировать письмо среди 121+ email/день. Нужно заменить на prefix + суть, например `DECISION: Approve event-driven architecture for payments`. Вторая — Buried Lead: просьба об одобрении в конце означает, что при типичных 12 секундах внимания получатель до неё не дойдёт. Нужно вынести BLUF в первое предложение: «Прошу одобрить переход на event-driven architecture до пятницы 15:00».

> [!question]- Почему использование undermining language («Извини что беспокою», «Просто хотел спросить») в email к stakeholder вреднее, чем в Slack-сообщении команде?
> Email к stakeholder — формальный канал, который документируется и пересылается. Undermining language подрывает авторитет автора в глазах всех, кто увидит цепочку, включая людей вне исходного контекста. В Slack неформальный тон допустим, сообщения эфемерны и ограничены командой. В email каждое «просто хотел» фиксируется в audit trail и формирует долгосрочное впечатление о компетенции отправителя. Вместо этого следует использовать уверенные формулировки: «Рекомендую», «По моему анализу», «Предлагаю рассмотреть».

> [!question]- Вы — Tech Lead распределённой команды (3 часовых пояса). Нужно сообщить о критичном баге, предложить план исправления и получить одобрение от Product Owner. Как совместить принципы async-first коммуникации с требованием срочности, и какой канал (Email, Slack, Call) вы выберете для каждого действия?
> Правильный подход — разделить задачу на этапы и использовать разные каналы по матрице «Срочность / Сложность». Шаг 1: немедленное уведомление через Slack с @mention (высокая срочность, низкая сложность). Шаг 2: параллельно отправить структурированный email с BLUF, timeline, вариантами решения и CTA — это обеспечит async-first документирование для коллег из других часовых поясов, которые прочитают утром. Шаг 3: если Product Owner онлайн — короткий call для одобрения (высокая срочность + высокая сложность). Если оффлайн — email с явным дедлайном и fallback-решением. Ключевое: email остаётся единственным источником правды (audit trail), а Slack и call — ускорители.

> [!question]- Примените 5Cs framework для анализа следующего email: «Привет! Вроде бы наш сервер упал или типа того, потому что кто-то из пользователей написал. Надо бы посмотреть. И ещё, не забудь что завтра митинг по другому проекту.» Оцените каждый C.
> **Clear** — нарушен: «вроде бы», «или типа того» создают неоднозначность, неясно упал сервер или нет. **Concise** — нарушен: две несвязанные темы в одном email (баг + напоминание о митинге), нужно два отдельных сообщения. **Complete** — нарушен: нет данных о сервере, нет логов, нет информации о масштабе проблемы, получатель не может действовать без follow-up вопросов. **Correct** — нарушен: факт падения не проверен («вроде бы»), нет ссылок на мониторинг или сообщение пользователя. **Courteous** — формально соблюдён, но неуважение ко времени читателя (заставляет самого выяснять ситуацию) нарушает дух принципа.

## Ключевые карточки

BLUF — что это и откуда пришло?
?
Bottom Line Up Front — военный стандарт коммуникации (US Military). Суть информации ставится в первое предложение, чтобы сообщение было понятно даже при неполном прочтении. Отвечает на 5W: What, Who, When, Where, Why.

Какие 5Cs составляют framework качественного email?
?
Clear (понятно с первого прочтения), Concise (минимум слов, максимум смысла), Complete (вся информация для действия), Correct (факты проверены, нет опечаток), Courteous (профессиональный и уважительный тон).

Какие Subject Line Prefixes рекомендует HBR и что каждый означает?
?
ACTION — требуется действие, DECISION — нужно принять решение, INFO — информация без необходимости действий, REQUEST — запрос, URGENT — срочно (использовать редко), FYI — для сведения.

Что должен содержать эффективный Call to Action в email?
?
Три компонента: ЧТО нужно сделать (конкретное действие), К КОГДА (явный дедлайн с датой и временем), КАК ответить (если неочевидно). Пример: «Прошу ответить 'approved' или указать замечания до пятницы 15:00».

Когда использовать Email, а когда Slack — по матрице срочность/сложность?
?
Высокая сложность + низкая срочность = Email (детальное изложение). Высокая сложность + высокая срочность = Call. Низкая сложность + низкая срочность = Slack (FYI). Низкая сложность + высокая срочность = Slack с @mention.

Какие undermining-формулировки снижают авторитет автора и чем их заменить?
?
Вредные: «Я просто хотел спросить», «Не уверен, но мне кажется», «Извините что беспокою», «Это может быть глупый вопрос». Замены: «Предлагаю рассмотреть», «По моему анализу», «Рекомендую», «Вопрос по...».

Что такое правило 3-х абзацев в структуре email?
?
Абзац 1 — BLUF (что нужно: действие, решение или информация). Абзац 2 — контекст (почему это важно, данные, аргументы). Абзац 3 — Call to Action (конкретное следующее действие с дедлайном).

Почему ошибка Buried Lead критична при 121 email/день?
?
У получателя ~12 секунд на решение «читать или нет». Если главное (lead) спрятано в конце — получатель не дочитывает и email игнорируется или откладывается. BLUF решает проблему: ключевая информация в первом предложении гарантирует, что она будет прочитана.
