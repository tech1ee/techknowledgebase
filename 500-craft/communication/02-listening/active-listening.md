# Active Listening: как слышать то, что на самом деле говорят

> **TL;DR:** Active listening — техника полной концентрации на говорящем, включающая вербальные и невербальные сигналы понимания. Ключевая идея: слушание — не пассивный процесс, а активный навык, который можно тренировать. Применяется в: 1-on-1 с reports, code review discussions, requirements gathering.

---

## Зачем это нужно?

### Представьте ситуацию

Sprint planning. Product Owner описывает новую фичу 10 минут. Ты киваешь, параллельно думаешь о текущем баге. После митинга берёшь задачу и понимаешь — ничего не помнишь. Переспрашиваешь, тратишь время PO, получаешь репутацию "того, кто не слушает".

**Без active listening:**
- Постоянные переспрашивания
- Неправильно понятые requirements
- Конфликты из-за "я же говорил"
- Потеря доверия коллег

**С active listening:**
- Точное понимание с первого раза
- Меньше итераций на уточнение
- Репутация надёжного собеседника
- Коллеги охотнее делятся информацией

## Терминология

| Термин | Что это | IT-аналогия |
|--------|---------|-------------|
| **Active Listening** | Техника полной концентрации с обратной связью | Как `ACK` в TCP — подтверждаешь получение пакета |
| **Paraphrasing** | Пересказ услышанного своими словами | Как `echo` команда — отражаешь input обратно |
| **Mirroring** | Повторение ключевых слов говорящего | Как `git mirror` — точная копия |
| **Clarifying questions** | Уточняющие вопросы для устранения неясности | Как debugging — находишь корень проблемы |
| **SOLER** | Техника невербального присутствия | Как handshake в протоколе — устанавливаешь соединение |

## Как это работает?

### История: Carl Rogers и рождение техники

В 1957 году психолог **Carl Rogers** и **Richard Farson** ввели термин "active listening". Rogers, основатель клиент-центрированной терапии, обнаружил что простое "выслушивание" трансформирует людей. Чувство "меня понимают" — базовая человеческая потребность.

> "Active listening brings about changes in people. Clinical and research evidence clearly shows that sensitive listening is a most effective agent for personality change."
> — Carl Rogers

### Модель SOLER (Gerard Egan)

Gerard Egan развил идеи Rogers в практическую технику **SOLER** для невербального присутствия:

```
┌─────────────────────────────────────────────────────────────────┐
│                        SOLER TECHNIQUE                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  S — Sit SQUARELY                                               │
│      Сиди лицом к собеседнику (позиция "5 часов")               │
│      Избегай прямого противостояния — создаёт напряжение        │
│                                                                 │
│  O — OPEN posture                                               │
│      Открытая поза: руки не скрещены, ноги не переплетены       │
│      Скрещенные руки = "я закрыт для твоих идей"                │
│                                                                 │
│  L — LEAN slightly forward                                      │
│      Слегка наклонись к собеседнику                             │
│      Показывает заинтересованность                              │
│                                                                 │
│  E — EYE contact                                                │
│      Поддерживай зрительный контакт (не пялься)                 │
│      60-70% времени — оптимально                                │
│                                                                 │
│  R — RELAX                                                      │
│      Расслабься — напряжение передаётся собеседнику             │
│      Твоё спокойствие помогает ему раскрыться                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Источник:** [Counselling Tutor: Egan's SOLER Model](https://counsellingtutor.com/active-listening-in-counselling-egan-soler-model/)

### Психологический механизм

**Почему active listening работает:**

1. **Mirror neurons** — когда ты отражаешь, мозг собеседника "узнаёт" себя
2. **Cognitive load reduction** — перефразирование снимает нагрузку с памяти говорящего
3. **Validation need** — "быть услышанным" = базовая потребность по Maslow
4. **Trust building** — внимание демонстрирует уважение, формирует доверие

### Ключевые компоненты

```
┌─────────────────────────────────────────────────────────────────┐
│                    5 КОМПОНЕНТОВ ACTIVE LISTENING                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. ATTENDING (присутствие)                                     │
│     SOLER + минимизация отвлечений                              │
│     "Убрать ноутбук, телефон экраном вниз"                      │
│                                                                 │
│  2. ACKNOWLEDGING (подтверждение)                               │
│     Вербальные сигналы: "uh-huh", "понятно", "да"               │
│     Невербальные: кивки, наклон головы                          │
│                                                                 │
│  3. PARAPHRASING (перефразирование)                             │
│     "Если я правильно понял, ты говоришь что..."                │
│     "То есть проблема в том, что..."                            │
│                                                                 │
│  4. CLARIFYING (уточнение)                                      │
│     "Можешь подробнее про...?"                                  │
│     "Что ты имеешь в виду под...?"                              │
│                                                                 │
│  5. SUMMARIZING (подведение итогов)                             │
│     "Итак, основные пункты: 1)... 2)... 3)..."                  │
│     "Давай убедимся что я всё понял..."                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Пошаговый процесс

### Подготовка

1. **Уберите отвлечения** — телефон экраном вниз, закрыть Slack
2. **Настройтесь** — сделайте 2-3 глубоких вдоха
3. **Примите позу SOLER** — открытая поза, лёгкий наклон
4. **Очистите голову** — отложите свои мысли на потом

### Шаг 1: Присутствие (первые 30 секунд)

**Что делать:**
- Установить зрительный контакт
- Принять открытую позу
- Показать что готовы слушать

**Пример (1-on-1):**
```
Ты: "Привет! Как дела? Рассказывай, что у тебя на уме."
    [Откидываешься в кресле, открытая поза, смотришь на собеседника]
```

**Скрипт:**
```
"У нас 30 минут, и это время полностью твоё.
Расскажи, что сейчас важно?"
```

### Шаг 2: Подтверждение (в процессе)

**Что делать:**
- Кивать в такт речи
- Использовать короткие вербальные сигналы
- НЕ прерывать

**Пример:**
```
Коллега: "Мы уже неделю бьёмся над этим багом, и кажется
         никто не понимает откуда он..."

Ты: [кивок] "Ммм..."

Коллега: "...и тесты проходят, но на проде падает..."

Ты: [кивок] "Понятно..."
```

**Важно:** Не заполняй паузы. Дай человеку время собраться с мыслями.

### Шаг 3: Перефразирование (каждые 2-3 минуты)

**Что делать:**
- Пересказать услышанное своими словами
- Проверить правильность понимания
- Дать возможность уточнить

**Пример (requirements gathering):**
```
PO: "Нам нужна фича экспорта отчётов, но только для enterprise
    клиентов, и желательно с гибкими настройками..."

Ты: "Если я правильно понял — экспорт отчётов для enterprise tier,
    с возможностью настройки формата или полей. Верно?"

PO: "Да, именно! И ещё важно чтобы был scheduling..."
```

**Скрипты для перефразирования:**
```
"Дай убедиться что я правильно понял..."
"То есть ты говоришь, что..."
"Если я правильно услышал..."
"Звучит как..."
```

### Шаг 4: Уточнение (когда что-то неясно)

**Что делать:**
- Задавать открытые вопросы
- Копать глубже в непонятные места
- Не бояться переспрашивать

**Пример (debugging сессия):**
```
Коллега: "Этот код просто не работает как надо."

Ты: "Что именно происходит когда запускаешь?
    Какой output ожидаешь vs получаешь?"

Коллега: "Ну, должен возвращать список, а возвращает null."

Ты: "В каких условиях null? Всегда или при определённом input?"
```

**Скрипты для уточнения:**
```
"Можешь подробнее про...?"
"Что ты имеешь в виду под...?"
"Когда именно это происходит?"
"Приведи пример?"
```

### Шаг 5: Подведение итогов (в конце)

**Что делать:**
- Суммировать ключевые пункты
- Подтвердить следующие шаги
- Убедиться в взаимном понимании

**Пример (после 1-on-1):**
```
Ты: "Окей, давай я подытожу что мы обсудили:
    1. Ты чувствуешь что застрял на текущем проекте
    2. Хочешь попробовать новую технологию
    3. Следующий шаг — я поговорю с PM о возможности
       подключить тебя к проекту X.

    Всё верно? Что-то упустил?"
```

## Скрипты и Templates

### Use Case 1: 1-on-1 с прямым отчётом

**Ситуация:** Еженедельная встреча с junior разработчиком

**Открытие:**
```
"Привет! Как прошла неделя? Что хочешь обсудить?"
```

**После того как выслушал:**
```
"Понял. Звучит как [перефразирование].
Что было самым сложным?"
```

**При эмоциональной нагрузке:**
```
"Слышу, что это тебя беспокоит.
Расскажи подробнее, что именно напрягает?"
```

**Чеклист:**
□ Убрал телефон и закрыл ноутбук
□ Задал открывающий вопрос
□ Не прерывал первые 2 минуты
□ Перефразировал минимум 2 раза
□ Подвёл итоги в конце

### Use Case 2: Code Review Discussion

**Ситуация:** Обсуждение комментариев к PR

**Когда автор объясняет решение:**
```
"Понял, то есть ты выбрал этот подход потому что [перефразирование].
А рассматривал ли вариант с...?"
```

**Когда автор расстроен:**
```
"Я слышу что комментарии вызвали frustration.
Давай разберём — какой пункт кажется несправедливым?"
```

**При разногласии:**
```
"Дай убедиться что понимаю твою позицию:
ты считаешь что [X] лучше потому что [Y]. Верно?"
```

### Use Case 3: Requirements Gathering

**Ситуация:** Встреча с Product Owner по новой фиче

**Открытие:**
```
"Расскажи про фичу как видишь её в идеале.
Я буду уточнять по ходу, окей?"
```

**Техника "EARS":**
```
E — Events: "Какие действия пользователь делает?"
A — Attributes: "Какие данные нужны на экране?"
R — Relationships: "Как это связано с существующими фичами?"
S — Stories: "Расскажи user story — что делает пользователь шаг за шагом?"
```

**Закрытие:**
```
"Итого фича включает:
1. [функционал 1]
2. [функционал 2]
3. [функционал 3]

Что-то упустил? Какие edge cases обсудить?"
```

### Use Case 4: Debugging with Colleague

**Ситуация:** Коллега просит помочь с багом

**НЕ делай:**
```
❌ "Дай посмотрю" [хватаешь клавиатуру]
❌ "Ты пробовал [решение]?" [прерываешь]
```

**Делай:**
```
✅ "Расскажи что происходит. Какой output ожидаешь?"
✅ "Что уже пробовал?"
✅ "Покажи последний working state?"
```

**Rubber Duck с человеком:**
```
"Объясни мне логику этого метода как будто
я ничего не знаю про систему."
```

### Use Case 5: Remote/Video Call

**Адаптация SOLER для видео:**

```
S — Смотри в камеру, не на экран
O — Видимая часть тела — открытая поза
L — Наклон к камере невозможен — компенсируй вербально
E — Eye contact = смотреть в камеру при говорении
R — Убедись что лицо расслаблено (видеозвонки утомляют)
```

**Компенсация отсутствия body language:**
```
"Понял, это звучит как [X]" — чаще проговаривай
"Ты сказал [Y], верно?" — больше вербальных подтверждений
```

## Распространённые ошибки

### Ошибка 1: Ждёшь своей очереди говорить

**Неправильно:**
```
Коллега: "Мы столкнулись с проблемой производительности..."
Ты: [в голове: "О, я знаю как решить! Сейчас скажу..."]
    [Не слышишь остальное]
```

**Почему это неправильно:**
Ты перестаёшь слушать когда формулируешь ответ. Пропускаешь важные детали.

**Правильно:**
```
Коллега: "Мы столкнулись с проблемой производительности..."
Ты: [Полностью слушаешь]
    [После паузы]: "Расскажи подробнее, где именно тормозит?"
```

### Ошибка 2: Прыжок к решению

**Неправильно:**
```
Коллега: "У меня проблема с..."
Ты: "Попробуй [решение]!"
```

**Почему это неправильно:**
- Не понял реальную проблему
- Демонстрируешь что не заинтересован выслушать
- Часто предлагаешь не то решение

**Правильно:**
```
Коллега: "У меня проблема с..."
Ты: "Расскажи подробнее. Что уже пробовал?"
    [Слушаешь 3-5 минут]
    "Если я правильно понял, проблема в [X]?"
    [Только после подтверждения]: "Одна идея — попробовать [решение]"
```

### Ошибка 3: Режим допроса

**Неправильно:**
```
Ты: "Когда это началось?"
Коллега: "Вчера."
Ты: "Что изменилось вчера?"
Коллега: "Деплой был."
Ты: "Какие изменения в деплое?"
```

**Почему это неправильно:**
- Слишком много вопросов подряд
- Ощущение допроса, а не разговора
- Собеседник закрывается

**Правильно:**
```
Ту: "Расскажи что случилось, когда заметил проблему?"
Коллега: "Вчера после деплоя начало падать..."
Ты: "Понял — после вчерашнего деплоя. Что именно падает?"
```

### Ошибка 4: Заполнение пауз

**Неправильно:**
```
Коллега: "Ну..." [пауза 2 сек]
Ты: "Может проблема в кэше? Или в конфиге?"
```

**Почему это неправильно:**
- Паузы нормальны — человек думает
- Перебивая, ломаешь мыслительный процесс
- Навязываешь свою повестку

**Правильно:**
```
Коллега: "Ну..." [пауза 5 сек]
Ты: [Молчишь, смотришь]
Коллега: "...на самом деле, я думаю проблема глубже.
         Это не просто баг, это архитектурная проблема."
```

**Правило:** Выжди 3-5 секунд после того как собеседник замолчал.

### Ошибка 5: Слушание для опровержения

**Неправильно:**
```
Коллега: "Я думаю мы должны переписать этот модуль..."
Ты: [в голове: "Ищу аргументы против..."]
```

**Почему это неправильно:**
- Слушаешь не для понимания, а для атаки
- Пропускаешь валидные аргументы
- Создаёшь adversarial динамику

**Правильно:**
```
Коллега: "Я думаю мы должны переписать этот модуль..."
Ту: [в голове: "Пойму его позицию полностью"]
    "Расскажи, почему считаешь что переписывание лучше рефакторинга?"
```

## Когда использовать / НЕ использовать

### Используй active listening, когда:

| Ситуация | Почему важно |
|----------|--------------|
| **1-on-1 meetings** | Человек должен чувствовать что его слышат |
| **Requirements gathering** | Точность понимания = меньше rework |
| **Conflict de-escalation** | Слушание снижает напряжение |
| **Mentoring/coaching** | Помогает человеку найти ответ самому |
| **Интервью (как interviewer)** | Получаешь больше информации |
| **Получение feedback** | Не уходишь в defensive mode |

### НЕ используй active listening, когда:

| Ситуация | Что делать вместо |
|----------|-------------------|
| **Production incident** | Быстрые директивные решения |
| **Время критично** | "У нас 5 минут, давай ключевые пункты" |
| **Человек хочет быстрый ответ** | Отвечай, не копай |
| **Small talk** | Обычное общение, не нужна техника |

### Сравнение с альтернативами

| Подход | Когда использовать | Плюсы | Минусы |
|--------|-------------------|-------|--------|
| **Active Listening** | Понять позицию, получить информацию | Глубина понимания, доверие | Требует времени |
| **Directive Listening** | Срочность, чёткая задача | Быстро, эффективно | Можно упустить контекст |
| **Critical Listening** | Оценка аргументов, принятие решений | Находишь слабые места | Может создать adversarial tone |

## Практика

### Упражнение 1: 3-минутное слушание

**Сценарий:**
Попроси коллегу рассказать о последней сложной задаче. Твоя цель — 3 минуты только слушать (acknowledging), без вопросов и комментариев.

**Задача:**
- Не прерывать
- Использовать только невербальные сигналы
- Заметить, сколько раз хотелось перебить

**Эталонный ответ:**
<details><summary>Показать</summary>
После 3 минут перефразируй услышанное:
"Если я правильно понял, ты работал над [X] и столкнулся с [Y].
Самым сложным было [Z]. Верно?"

Обрати внимание на реакцию — если человек говорит "Да, именно!" — ты слушал хорошо.
</details>

### Упражнение 2: Paraphrasing drill

**Сценарий:**
На следующем 1-on-1 или митинге после каждых 2-3 минут речи собеседника делай паузу и перефразируй.

**Задача:**
- Минимум 3 перефразирования за разговор
- Использовать разные формулировки
- Получить подтверждение "да" или уточнение

**Фразы для практики:**
```
"Дай проверю что понял..."
"То есть ты говоришь что..."
"Звучит как..."
"Если я правильно услышал..."
```

### Упражнение 3: Silence tolerance

**Сценарий:**
После того как собеседник закончил говорить, выжди 5 секунд перед ответом.

**Задача:**
- Отследить желание заполнить паузу
- Заметить, продолжает ли собеседник после паузы
- Практиковать неделю

**Эталонный результат:**
<details><summary>Показать</summary>
В 30-40% случаев собеседник добавит важную информацию после паузы.
Это информация, которую ты бы упустил прерыванием.
</details>

## Проверь себя

> [!question]- Коллега на 1-on-1 говорит: "Мне кажется, я не расту..." и замолкает. Ты хочешь помочь. Какие три конкретных действия из active listening ты предпримешь в следующие 30 секунд и почему именно в таком порядке?
> 1. **Выдержать паузу 5 секунд** (silence tolerance) — в 30-40% случаев человек сам продолжит и скажет то, что действительно важно; прерывание паузы сломает его мыслительный процесс. 2. **Использовать acknowledging** — кивок, "я слушаю" — показать что ты присутствуешь, без давления. 3. **Перефразировать**: "Звучит как будто ты чувствуешь что застрял на месте. Расскажи, в чём именно?" — проверить понимание и открыть путь для уточнения. Порядок важен: сначала дать пространство, потом подтвердить присутствие, потом копнуть глубже.

> [!question]- Почему техника "прыжок к решению" (Jump to Solution) разрушительна, даже если предложенное решение технически верное?
> Потому что прыжок к решению: (1) демонстрирует незаинтересованность в позиции собеседника — ты не дослушал; (2) ты мог не понять реальную проблему — часто за первым описанием бага скрывается архитектурная проблема; (3) это убивает доверие — человек перестаёт делиться, потому что знает что будет перебит; (4) нарушает цикл validation need по Maslow — базовая потребность "быть услышанным" не удовлетворена. Даже верное решение, предложенное без выслушивания, воспринимается как обесценивание.

> [!question]- Как принцип active listening ("подтверждение получения" через paraphrasing) аналогичен механизму TCP ACK в сетевых протоколах? В чём сходство и в чём отличие?
> **Сходство:** и TCP ACK, и paraphrasing выполняют одну функцию — подтверждают что сообщение получено и понято корректно. Без ACK sender не знает, дошёл ли пакет; без paraphrasing говорящий не знает, поняли ли его. Оба механизма создают reliable communication channel. **Отличие:** TCP ACK — точное подтверждение бит-в-бит (checksum), а paraphrasing — пересказ своими словами. Это даёт дополнительную ценность: если пересказ неточен, отправитель может уточнить (коррекция ошибок на уровне смысла, а не данных). Кроме того, paraphrasing выполняет эмоциональную функцию (validation), которой у TCP нет.

> [!question]- Ты проводишь requirements gathering по видеозвонку. PO описывает сложную фичу уже 8 минут, и ты замечаешь что теряешь нить. Проанализируй: какие компоненты active listening ты упустил и как вернуть контроль, не обидев PO?
> Скорее всего упущены: (1) **Paraphrasing каждые 2-3 минуты** — 8 минут без перефразирования = перегрузка рабочей памяти; (2) **Адаптация SOLER для видео** — возможно, взгляд ушёл на заметки или другой экран, и PO не получает невербальной обратной связи. Чтобы вернуть контроль: мягко прервать с помощью summarizing — "Подожди, дай зафиксирую что понял на этот момент: 1)... 2)... Верно? Продолжай." Это не обидит PO, потому что: (а) ты демонстрируешь что слушал, (б) используешь EARS-технику (Events, Attributes, Relationships, Stories) для структурирования, (в) даёшь PO возможность скорректировать понимание.

## Ключевые карточки

Active Listening — определение?
?
Техника полной концентрации на говорящем с вербальными и невербальными сигналами понимания. Ключевая идея: слушание — не пассивный процесс, а активный навык. Введена Carl Rogers и Richard Farson в 1957 году.

Что означает акроним SOLER?
?
**S** — Sit Squarely (сиди лицом к собеседнику), **O** — Open posture (открытая поза), **L** — Lean slightly forward (лёгкий наклон к собеседнику), **E** — Eye contact (зрительный контакт 60-70% времени), **R** — Relax (расслабься — напряжение передаётся).

Назови 5 компонентов active listening в правильном порядке.
?
1. **Attending** (присутствие) — SOLER + убрать отвлечения. 2. **Acknowledging** (подтверждение) — "uh-huh", кивки. 3. **Paraphrasing** (перефразирование) — "Если я правильно понял...". 4. **Clarifying** (уточнение) — "Что ты имеешь в виду под...?". 5. **Summarizing** (подведение итогов) — "Итак, основные пункты:...".

Почему нельзя заполнять паузы в разговоре?
?
В 30-40% случаев собеседник после паузы добавит важную информацию, которую ты бы упустил, перебив. Паузы — нормальная часть мышления. Прерывая паузу, ты: ломаешь мыслительный процесс собеседника, навязываешь свою повестку, лишаешь себя ценной информации. Правило: выжди 3-5 секунд после молчания.

Что такое техника EARS в requirements gathering?
?
**E** — Events (какие действия делает пользователь), **A** — Attributes (какие данные нужны на экране), **R** — Relationships (как связано с существующими фичами), **S** — Stories (user story — что делает пользователь шаг за шагом).

Как перефразировать — четыре базовых скрипта?
?
1. "Дай убедиться что я правильно понял..." 2. "То есть ты говоришь, что..." 3. "Если я правильно услышал..." 4. "Звучит как..." — после каждого ждёшь подтверждение ("да") или уточнение от собеседника.

В каких ситуациях НЕ следует применять active listening?
?
1. **Production incident** — нужны быстрые директивные решения. 2. **Критичное по времени** — "У нас 5 минут, давай ключевые пункты". 3. **Человек хочет быстрый ответ** — отвечай, не копай. 4. **Small talk** — обычное общение, техника не нужна. В этих случаях эффективнее directive listening.

Почему active listening работает? Четыре психологических механизма.
?
1. **Mirror neurons** — когда ты отражаешь, мозг собеседника "узнаёт" себя. 2. **Cognitive load reduction** — перефразирование снимает нагрузку с памяти говорящего. 3. **Validation need** — "быть услышанным" = базовая потребность по Maslow. 4. **Trust building** — внимание демонстрирует уважение и формирует доверие.
