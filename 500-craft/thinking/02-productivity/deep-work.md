# Deep Work: фокус в мире отвлечений

3-4 часа глубокой работы > 8 часов многозадачности. Но после каждого прерывания нужно 23 минуты на возврат концентрации. 15 прерываний в день = 345 минут потерь. Из 8 часов эффективны только 2.

---

## Проблема, которую решает Deep Work

Исследование UC Irvine обнаружило:

> **Мы работаем над задачей в среднем 12 минут, прежде чем нас прерывают.**
> После прерывания требуется **23 минуты**, чтобы вернуться к полной концентрации.

Посчитаем:
- 8-часовой рабочий день = 480 минут
- Типичные прерывания: 15-20 раз в день
- Потеря на переключение: 15 × 23 = 345 минут
- **Эффективная работа: ~2 часа из 8**

Это не лень. Это архитектура современной работы.

```
Типичный день программиста:

09:00 ┬─ Открыл IDE
      │  Slack: "Привет! Есть минутка?"
09:05 ┼─ Ответил в Slack
      │  Вернулся к коду
09:08 ┼─ Email notification
      │  "Нужно срочно глянуть"
09:20 ┼─ Закрыл email
      │  Вспоминает: "На чём остановился?"
09:25 ┼─ Начал понимать код
      │  Stand-up через 5 минут!
09:30 ┼─ Stand-up (30 мин)
10:00 ┼─ Вернулся. Код кажется незнакомым.
      │  ...и так весь день
```

---

## Что такое Deep Work

Cal Newport (профессор computer science, автор концепции) определяет:

> **Deep Work** — профессиональная деятельность в состоянии концентрации без отвлечений, которая push'ит твои когнитивные способности на предел. Такая работа создаёт новую ценность, улучшает навыки и трудно воспроизводима.

**Shallow Work** — противоположность:
- Логистические задачи
- Email, Slack, митинги
- Работа "на автопилоте"
- Легко делегируется или автоматизируется

```
Deep Work:                          Shallow Work:
─────────────────────               ────────────────────
• Архитектура системы               • Ответы в Slack
• Написание алгоритма               • Status updates
• Изучение новой технологии         • Рутинные code reviews
• Сложный debugging                 • Email-переписка
• Написание документации            • Митинги-статусы
```

---

## Почему Deep Work работает (нейронаука)

### Attention Residue

Когда ты переключаешься между задачами, часть внимания остаётся на предыдущей задаче. Это "attention residue" — осадок внимания.

```
Задача A → Переключение → Задача B

          ┌─────────────────────────────┐
          │ Твой мозг думает о B...     │
          │ ...но часть ещё на A        │
          │                             │
          │   A ░░░░░░░░                │
          │   B ████████████            │
          │                             │
          │   Эффективность: ~60%       │
          └─────────────────────────────┘

Через 23 минуты:
          ┌─────────────────────────────┐
          │   A                         │
          │   B ████████████████████    │
          │                             │
          │   Эффективность: ~100%      │
          └─────────────────────────────┘
```

### Flow State

Deep Work — это форма flow state (потока). Нейроисследования показывают:
- Снижается активность prefrontal cortex (внутренний критик)
- Повышается выработка дофамина и эндорфинов
- Субъективное восприятие времени меняется
- Работа приносит удовольствие сама по себе

## Конкретная методика

### Формула Newport'а

> **3-4 часа глубокой работы в день, 5 дней в неделю = огромная продуктивность.**

Не 8 часов. Не "весь день". 3-4 часа.

### Четыре стратегии планирования Deep Work

**1️⃣ Monastic (Монашеский)**
- Полная изоляция на длительные периоды
- Для: писателей, учёных, исследователей
- Пример: Neal Stephenson не отвечает на email вообще

*Реалистично для IT: почти никогда. Требует полной автономии.*

**2️⃣ Bimodal (Двухрежимный)**
- Чередование периодов изоляции и доступности
- Неделя deep work → неделя обычной работы
- Или: 4 часа утром изоляция, после обеда — доступен

*Реалистично для IT: возможно для сеньоров, тимлидов с автономией.*

**3️⃣ Rhythmic (Ритмичный)**
- Фиксированные блоки каждый день
- Например: 6:00-9:00 — deep work, до начала рабочего дня
- Или: блок после обеда, когда меньше митингов

*Реалистично для IT: самый практичный вариант для большинства.*

**4️⃣ Journalistic (Журналистский)**
- Deep work в любое свободное окно
- Требует способности мгновенно переключаться
- Для людей с большим опытом концентрации

*Реалистично для IT: сложно, требует тренировки.*

## Практическая реализация для разработчиков

### Утренний блок (рекомендую)

```
05:30 - 06:00   Подъём, кофе, без телефона
06:00 - 09:00   DEEP WORK БЛОК
                - Телефон в другой комнате
                - Slack/Email закрыты
                - Notifications отключены
                - Одна задача

09:00           Рабочий день начинается
                - Stand-ups, митинги, Slack
                - Shallow work

Результат: 3 часа чистой работы ДО того,
           как мир начнёт тебя отвлекать
```

### Блок "после митингов"

```
Если утро не вариант:

13:00 - 14:00   Обед
14:00 - 16:30   DEEP WORK БЛОК
                - "В фокусе, доступен через 2.5 часа"
                - Наушники как сигнал "не беспокоить"

16:30 - 18:00   Shallow work, завершение дня
```

### Правила для блока

```
ОБЯЗАТЕЛЬНО:
✓ Закрыть ВСЕ чаты (не "поставить на mute", а закрыть)
✓ Убрать телефон из поля зрения
✓ Одна задача на весь блок
✓ Таймер (Pomodoro или просто время окончания)

ОПЦИОНАЛЬНО, НО ПОМОГАЕТ:
○ Музыка без слов / белый шум
○ Полный экран IDE
○ Отдельный browser profile без соцсетей
○ "Ритуал начала" (кофе, конкретное место)
```

## Подводные камни и честная правда

### Проблема 1: Культура компании может быть против

```
Что говорит Deep Work:              Что говорит твой PM:
───────────────────────             ─────────────────────
"Отключи Slack на 3 часа"    vs    "Ответь в течение 15 минут"
"Никаких митингов утром"     vs    "Stand-up в 9:30 обязателен"
"Фокус на одной задаче"      vs    "Многозадачность = эффективность"
```

**Реальность:** В большинстве компаний Deep Work требует:
- Явного разрешения от руководства
- Изменения культуры команды
- Или... делать это за пределами рабочих часов (5-8 утра)

### Проблема 2: Не всем подходит

Deep Work предполагает, что твоя работа требует глубокой концентрации. Но:

- **Support-инженер** — должен быть доступен
- **Junior в первые месяцы** — нужны частые консультации
- **Менеджер** — работа = коммуникация
- **DevOps на oncall** — реагирование > фокус

*Для этих ролей другие стратегии работают лучше.*

### Проблема 3: Начать сложнее, чем кажется

```
День 1: "Сяду и сфокусируюсь на 3 часа"

Минута 5:  "Интересно, что в Slack..."
Минута 12: "Может, проверю email быстро..."
Минута 20: "Скучно. Может, другую задачу?"
Минута 30: "Сдаюсь. Не моё это."
```

**Правда:** Концентрация — это навык, который тренируется. Начни с 25 минут (Pomodoro), увеличивай постепенно.

### Проблема 4: Иллюзия продуктивности

Быть занятым ≠ быть продуктивным.

Многим **кажется**, что они работают, когда:
- Отвечают в чатах
- Сидят на митингах
- Переключаются между задачами

Это создаёт ощущение занятости. Deep Work требует **отказа от этого ощущения** — что психологически некомфортно.

---

## Что говорит Cal Newport в 2024-2025

В книге **"Slow Productivity" (2024)** Newport развивает идеи:

1. **Меньше задач** — сокращай список до ключевых
2. **Устойчивый темп** — баланс продуктивности и отдыха
3. **Качество > количество** — лучше сделать одну вещь отлично

> "Псевдо-продуктивность — использование видимой активности как proxy для реальной работы — это ловушка."

## Метрики успеха

Как понять, что Deep Work работает:

| Метрика | До | После |
|---------|----|----|
| Часов на сложную задачу | 3-4 дня (с прерываниями) | 1 день (блоком) |
| Ощущение в конце дня | "Ничего не успел" | "Сделал главное" |
| Качество кода/работы | Много мелких багов | Меньше переделок |
| Время входа в "поток" | 30-40 минут | 10-15 минут |

## Actionable: что делать

**Неделя 1 (эксперимент):**
- Выбери один 90-минутный блок в день
- Закрой ВСЁ, кроме одной задачи
- Отслеживай: сколько реально удалось сфокусироваться?

**Неделя 2-4 (привычка):**
- Постепенно увеличивай до 2-3 часов
- Найди "своё" время (утро/после обеда)
- Сообщи команде о своих "блоках фокуса"

**Долгосрок:**
- Добейся системной поддержки (no-meeting блоки в календаре)
- Защищай время от "срочных" задач
- Периодически пересматривай: работает ли система?

**Если не получается:**
- Это нормально для некоторых ролей
- Попробуй другие стратегии (batching, time-boxing)
- Возможно, проблема в культуре компании, а не в тебе

## Проверь себя

> [!question]- Почему ритмичная (rhythmic) стратегия Deep Work считается самой реалистичной для большинства разработчиков?
> Rhythmic стратегия предполагает фиксированные блоки каждый день (например, 6:00-9:00). Она реалистична потому, что не требует полной автономии (как monastic), не зависит от длинных периодов изоляции (как bimodal) и не требует способности мгновенно входить в концентрацию (как journalistic). Фиксированный ритм формирует привычку, снижая когнитивные затраты на принятие решения "когда начать".

> [!question]- Тимлид говорит: "У нас agile, все должны быть доступны в Slack постоянно". Как аргументировать необходимость блоков Deep Work?
> Аргументация через метрики: 15 прерываний в день = 345 минут потерь на восстановление контекста (UC Irvine). Из 8 часов эффективны только 2. Предложение: 2-3 часа "no-disturb" блока утром с SLA ответа после блока. Это увеличит output команды без потери доступности. Можно провести эксперимент на 2 недели и измерить количество закрытых задач.

> [!question]- Почему совет "просто отключи уведомления" -- необходимое, но недостаточное условие для Deep Work?
> Отключение уведомлений убирает внешние (external) прерывания, но не решает проблему внутренних переключений: привычка проверять Slack, прокрастинация через "быстро гляну", attention residue от предыдущих задач. Deep Work требует ещё: одну конкретную задачу, ритуал начала, тренировку концентрации (начиная с 25 минут), и среду, исключающую соблазны.

> [!question]- Cal Newport утверждает, что 3-4 часа Deep Work > 8 часов обычной работы. При каких условиях это утверждение НЕ верно?
> Утверждение не верно для ролей, где основная ценность -- коммуникация и реагирование: support-инженеры, менеджеры, DevOps на oncall, junior-разработчики в первые месяцы. Также не верно, когда задача не требует глубокой когнитивной нагрузки (рутинные операции, shallow work). Deep Work даёт преимущество только для работы, требующей концентрации на пределе когнитивных способностей.

## Ключевые карточки

Что такое Deep Work по определению Cal Newport?
?
Профессиональная деятельность в состоянии концентрации без отвлечений, которая push'ит когнитивные способности на предел. Создаёт новую ценность, улучшает навыки и трудно воспроизводима.

Что такое Shallow Work?
?
Логистические задачи, не требующие глубокой концентрации: email, Slack, митинги, работа "на автопилоте". Легко делегируется или автоматизируется.

Сколько часов Deep Work в день рекомендует Newport?
?
3-4 часа в день, 5 дней в неделю. Не 8 часов. Концентрация -- ресурс, который истощается.

Назови четыре стратегии планирования Deep Work.
?
1) Monastic -- полная изоляция. 2) Bimodal -- чередование периодов. 3) Rhythmic -- фиксированные блоки каждый день. 4) Journalistic -- deep work в любое свободное окно. Для IT самая практичная -- rhythmic.

Что такое attention residue?
?
Часть внимания, которая остаётся на предыдущей задаче после переключения. Снижает эффективность на новой задаче до ~60%. Полное восстановление занимает ~23 минуты.

Почему "иллюзия продуктивности" -- ловушка?
?
Быть занятым (отвечать в чатах, сидеть на митингах) создаёт ощущение работы, но не создаёт ценность. Deep Work требует отказа от этого комфортного ощущения занятости в пользу реального результата.
