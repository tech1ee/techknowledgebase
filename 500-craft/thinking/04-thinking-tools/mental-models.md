# Ментальные модели: инструментарий мышления

Эксперты не умнее — они используют правильные модели для конкретных ситуаций. Ментальная модель — это упрощённая карта реальности, и чем больше таких карт, тем точнее ориентируешься.

---

## Почему это важно

В 2002 году Elon Musk хотел купить ракету для Mars Oasis. Цена: $65 миллионов за одну б/у ракету. Он спросил: "Из чего состоит ракета?"

Ответ: алюминий, титан, медь, углеволокно. Стоимость материалов: ~2% от цены ракеты.

Вывод Маска: проблема не в физике, а в бизнес-моделях индустрии. Можно строить ракеты в 10 раз дешевле. SpaceX сделал именно это.

1️⃣ Стандартное мышление: "Сколько стоит ракета на рынке?"

2️⃣ Мышление первых принципов: "Из чего состоит ракета? Какова фундаментальная стоимость?"

*Разница между этими вопросами — миллиарды долларов.*

---

## Что такое ментальные модели

Ментальная модель — это упрощённое представление о том, как что-то работает.

```
Реальность слишком сложна для прямого понимания.
Модели — это карты, которые помогают навигировать.

Карта ≠ территория, но без карты — заблудишься.
```

### Идея latticework (Чарли Мангер)

Чарли Мангер (партнёр Уоррена Баффета, $2.6 млрд состояние) говорит:

> "You've got to have models in your head. And you've got to array your experience — both vicarious and direct — on this latticework of models."

**Latticework** — это решётка, где модели переплетаются и усиливают друг друга.

Пример: понимание бизнеса требует моделей из:
- Психологии (поведение потребителей)
- Экономики (спрос, предложение)
- Биологии (эволюция, конкуренция)
- Физики (точки невозврата, инерция)

*Мангер читает 500+ страниц в день. 90-летний миллиардер. Совпадение?*

## Три главные модели для инженеров

### 1️⃣ First Principles Thinking (Мышление первых принципов)

**Что это:** Разбор проблемы до базовых, неоспоримых истин. Построение решения с нуля.

**Почему работает:** Большинство решений — это аналогии с прошлым. Но прошлое не всегда применимо к новым ситуациям.

**Процесс:**

```
Шаг 1: Определи проблему
Шаг 2: Разбей на базовые компоненты
Шаг 3: Задай вопрос: "Что я знаю точно?" (факты, не мнения)
Шаг 4: Построй решение из этих фактов
```

**Пример — батареи для Tesla:**

```
❌ Аналогия: "Батареи стоят $600/кВтч. Это дорого. Электромобили невыгодны."

✅ First Principles:
- Из чего состоит батарея? Кобальт, никель, литий, полимеры, сталь.
- Какова стоимость этих материалов на бирже? ~$80/кВтч.
- Почему разница? Производственные процессы, маржа, логистика.
- Вывод: Можно производить батареи значительно дешевле.
```

Tesla снизил стоимость батарей на 89% за 10 лет.

**Когда использовать:**
- Индустрия застряла в устаревших практиках
- Нужно радикально новое решение
- Все говорят "это невозможно"

**Когда НЕ использовать:**
- Проблема хорошо изучена
- Времени мало, нужно быстрое решение
- Риски низкие

### 2️⃣ Inversion (Инверсия)

**Что это:** Вместо "как достичь успеха?" спросить "как гарантированно провалиться?" и избежать этого.

Карл Якоби (математик XIX века): "Invert, always invert."

**Почему работает:** Легче увидеть, что точно не работает, чем что точно работает.

**Примеры:**

```
Прямой вопрос: Как сделать успешный продукт?
Инверсия: Как гарантированно провалить продукт?
- Игнорировать фидбек пользователей
- Не тестировать перед релизом
- Добавлять фичи без стратегии
→ Избегай этого — шансы на успех растут.
```

```
Прямой вопрос: Как стать хорошим программистом?
Инверсия: Как гарантированно остаться плохим?
- Не читать чужой код
- Не принимать критику
- Не изучать новое после работы
→ Делай противоположное.
```

**Применение в архитектуре:**

```
Вместо: "Как сделать систему надёжной?"
Спроси: "Какие ошибки обязательно положат систему?"
- Single point of failure
- Нет мониторинга
- Нет бэкапов
- Нет rate limiting
→ Устрани каждый пункт.
```

*Амазон использует инверсию систематически: команды должны описать, как их сервис может упасть, прежде чем запускать его.*

### 3️⃣ Second-Order Thinking (Мышление второго порядка)

**Что это:** Думать не только о прямых последствиях, но и о последствиях последствий.

**Почему работает:** Большинство людей думают только о первом порядке. Конкурентное преимущество — в глубине. Шахматист-новичок видит один ход. Гроссмейстер — десять ходов вперёд.

**Структура:**

```
Действие → Последствие 1-го порядка → Последствие 2-го порядка → ...
```

**Пример 1: "Добавим кэш"**

```
Решение: Добавить Redis для кэширования

1-й порядок: Ответы быстрее ✅ Отлично!
     │
     ▼
2-й порядок: Сложнее инвалидация ⚠️
     │       Когда данные в БД обновились, кэш устарел
     ▼
3-й порядок: Нужен мониторинг кэша
     │       Добавляем Grafana, алерты, дашборды
     ▼
4-й порядок: Больше инфраструктуры = больше точек отказа
     │       Redis упал → всё упало (если не продумали fallback)
     ▼
5-й порядок: Нужен кластер Redis + репликация
             Это уже не "простой кэш", это целая система
```

*Это не значит "не добавляй кэш". Это значит "думай на 5 шагов вперёд, прежде чем добавлять".*

**Пример 2: "Перепишем на микросервисы"**

```
Решение: Разбить монолит на микросервисы

1-й порядок: Команды работают независимо ✅
     │
     ▼
2-й порядок: Нужен API Gateway, Service Discovery
     │       (+3 новых технологии в стек)
     ▼
3-й порядок: Дебаг сложнее — запрос проходит через 5 сервисов
     │       Нужен distributed tracing (Jaeger, Zipkin)
     ▼
4-й порядок: Latency выросла (network calls вместо function calls)
     │       Пользователи жалуются на скорость
     ▼
5-й порядок: Нужен кэш между сервисами + оптимизация
             Мы вернулись к проблемам кэширования...
```

**Пример 3: "Уволим медленного разработчика"**

```
Решение: Уволить разработчика за низкую производительность

1-й порядок: Экономия на зарплате ✅
     │
     ▼
2-й порядок: Он знал легаси-систему лучше всех ⚠️
     │       Теперь никто не понимает модуль X
     ▼
3-й порядок: Баги в модуле X — некому чинить
     │       Клиенты уходят
     ▼
4-й порядок: Нанимаем нового за 2x зарплату
     │       Онбординг 3 месяца
     ▼
5-й порядок: Потеряли 6 месяцев + репутацию
             "Сэкономили" на зарплате, потеряли миллионы
```

**Практический вопрос:** Перед любым решением спроси: "А что будет потом? А потом? А потом?"

*Jeff Bezos строил Amazon с горизонтом 7+ лет. Большинство компаний думают на квартал вперёд. Угадай, кто выиграл.*

## Примеры для IT-специалистов

### First Principles для выбора технологий

**Ситуация:** Команда выбирает между микросервисами и монолитом.

**Обычный подход:** "Netflix использует микросервисы → нам тоже нужно"

**First Principles:**
1. Какие проблемы мы решаем?
2. Каков размер команды? (< 10 человек → монолит эффективнее)
3. Какова частота деплоев? (1 раз в месяц → микросервисы overkill)
4. Есть ли инфра-команда?

**Вывод:** Для стартапа с 5 разработчиками монолит = правильный выбор.

---

### Inversion для Debugging

**Ситуация:** Приложение тормозит, неясно почему.

**Обычный подход:** "Где может быть bottleneck?"

**Inversion:** "Что должно работать быстро, чтобы приложение не тормозило?"
- Database queries < 50ms
- API responses < 200ms
- Rendering < 16ms

**Действие:** Измеряем каждый компонент → находим тот, что нарушает правило.

---

### Second-Order Thinking для архитектурных решений

**Ситуация:** Добавить кеширование Redis.

**First-order:** "Кеш ускорит запросы" ✓

**Second-order:**
- Инвалидация кеша → сложность кода → баги
- Ещё один сервис → точка отказа → мониторинг
- Stale data → проблемы с consistency → жалобы юзеров

**Решение:** Кеш нужен? Да. Но сначала оптимизировать SQL.

---

### Map vs Territory для документации

**Ситуация:** Документация говорит одно, код делает другое.

**Map (документация):** "API возвращает 200 при успехе"
**Territory (реальность):** API иногда возвращает 201, 204

**Принцип:** Код = territory. Документация = map. Когда они расходятся — верь коду.

---

### Circle of Competence для специализации

**Ситуация:** Разработчик хочет стать fullstack.

**Вопрос:** Где твой Circle of Competence?
- Backend: 80% понимания → внутри круга
- Frontend: 30% понимания → на границе
- DevOps: 10% понимания → за пределами

**Стратегия:**
- Углублять backend (comparative advantage)
- Изучать frontend для коммуникации (расширять круг)
- DevOps: достаточно знать кого спросить

---

### Pareto для оптимизации производительности

**Ситуация:** Приложение медленное, нужна оптимизация.

**Pareto 80/20:** 80% времени выполнения — в 20% кода.

**Действие:**
1. Профилирование → находишь 3 функции, которые жрут 75% CPU
2. Оптимизируешь только их
3. Производительность выросла в 4 раза

**Антипаттерн:** Оптимизировать весь код равномерно → потратить месяц → результат +5%.

---

### Occam's Razor для troubleshooting

**Ситуация:** Production-инцидент, пользователи не могут залогиниться.

**Сложная гипотеза:**
"Race condition в distributed lock manager при одновременном входе с разных устройств..."

**Простая гипотеза:**
- База недоступна?
- Redis упал?
- SSL-сертификат истёк?

**Статистика:** 90% инцидентов — простые причины (диск переполнен, память кончилась, сетевые проблемы).

**Правило:** Проверяй простое первым. Сложное — если простое не объясняет.

## Дополнительные модели

### Circle of Competence (Круг компетенции)

**Суть:** Знай, что ты знаешь. Знай, что ты НЕ знаешь.

```
      Зона: "Знаю, что знаю"
           ┌─────────────────┐
           │  Принимай       │
           │  решения здесь  │
           └─────────────────┘
                  │
    ┌─────────────┴─────────────┐
    │ Зона: "Знаю, что не знаю" │
    │ Консультируйся, делегируй │
    └───────────────────────────┘
                  │
    ┌─────────────┴──────────────┐
    │ Зона: "Не знаю, что не знаю"│
    │       Опасная зона          │
    └─────────────────────────────┘
```

**Пример:** Backend-разработчик пишет CSS. Результат: "работает, но что-то не то". Лучше спросить фронтендера или изучить тему до принятия решений.

*Баффет и Мангер отказались от инвестиций в технологические компании на десятилетия, потому что это было вне их круга компетенции. Пропустили Amazon, Google... но и не потеряли деньги на доткомах.*

### Occam's Razor (Бритва Оккама)

**Суть:** Из двух объяснений выбирай более простое (при прочих равных).

```
Баг: API возвращает 500 ошибку

Сложное объяснение:
"Наверное, race condition в многопоточности
при специфической комбинации запросов
с определённым состоянием кэша..."

Простое объяснение:
"База данных недоступна."

→ Проверь простое сначала.
```

*80% production-инцидентов — простые причины: диск переполнен, память кончилась, сертификат истёк.*

### Pareto Principle (Принцип Парето)

**Суть:** 80% результатов от 20% усилий.

```
В коде: 80% багов — в 20% модулей
В продукте: 80% использования — 20% фич
В команде: 80% работы — от 20% людей (не всегда хорошо)
```

**Применение:**
- Найди 20%, которые дают 80% результата
- Оптимизируй их в первую очередь
- Остальное — по возможности

### Hanlon's Razor (Бритва Хэнлона)

**Суть:** Не приписывай злому умыслу то, что можно объяснить некомпетентностью (или недопониманием).

```
❌ "Заказчик специально даёт противоречивые требования,
    чтобы нас подставить."

✅ "Заказчик сам не до конца понимает, что ему нужно.
    Нужно помочь уточнить требования."
```

*Большинство конфликтов в командах — не из-за плохих намерений, а из-за разного понимания контекста.*

## Как использовать модели

### Шаг 1: Изучай модели из разных областей

Чем больше разных моделей — тем гибче мышление.

| Область | Полезные модели |
|---------|-----------------|
| Физика | Критическая масса, точки невозврата |
| Биология | Эволюция, симбиоз, паразитизм |
| Психология | Когнитивные искажения, социальное доказательство |
| Экономика | Спрос/предложение, альтернативные издержки |
| Математика | Вероятность, степенные законы |

### Шаг 2: Практикуй применение

При каждой проблеме спрашивай:
- Какая модель здесь применима?
- Какую модель я ещё не пробовал?
- Что даст инверсия?
- Какие последствия второго порядка?

### Шаг 3: Записывай

Веди журнал:
```
Ситуация: [описание]
Модель: [какую применил]
Результат: [что получилось]
Вывод: [что понял]
```

## Типичные ошибки

### "Молоток и гвозди"

> "Если единственный инструмент — молоток, всё кажется гвоздём."

**Проблема:** Использование одной любимой модели для всех ситуаций.

**Решение:** Намеренно пробовать разные модели. Спрашивать: "Какую модель я НЕ использовал?"

### Переусложнение

**Проблема:** Применение сложной модели там, где достаточно простой.

**Решение:** Начинай с простого. Усложняй только если простое не работает.

### Игнорирование контекста

**Проблема:** Модель работает в одном контексте, но не в другом.

**Пример:** First principles отлично работает для SpaceX с миллиардами инвестиций. Для стартапа с $50K — иногда лучше скопировать работающее решение.

## Практическое упражнение

**На этой неделе:**

1. Выбери одну проблему из работы
2. Примени к ней три разные модели:
   - First Principles: разбери на базовые компоненты
   - Inversion: как гарантированно провалиться?
   - Second-Order: какие последствия последствий?
3. Сравни выводы. Какие инсайты появились?

## Проверь себя

> [!question]- Как мышление первых принципов (first principles) применяется к решению "писать свой фреймворк vs использовать готовый"?
> Мышление по аналогии: "все используют React/Spring, значит и мы должны". First principles: (1) Какую проблему решаем? (2) Какие базовые требования? (3) Какова реальная стоимость создания vs использования готового? (4) Какие факты (не мнения) у нас есть? Если требования уникальны и готовые решения не покрывают 80% -- свой может быть оправдан. Если стандартная задача -- first principles покажет, что готовое решение дешевле.

> [!question]- Почему Чарли Мангер настаивает на latticework (решётке) моделей из разных дисциплин, а не на глубокой специализации?
> Каждая модель -- упрощённая карта реальности. Одна модель видит одну грань проблемы. Latticework -- переплетение моделей из разных областей (психология, экономика, биология, физика), которые усиливают друг друга. Пример: для понимания бизнеса нужны модели спроса/предложения (экономика), поведения потребителей (психология), конкуренции (биология), инерции (физика). Специалист с одной моделью -- "молотком всё похоже на гвоздь".

> [!question]- Приведи пример, когда аналогичное мышление ("все так делают") привело бы к плохому архитектурному решению, а first principles -- к хорошему.
> Аналогия: "Крупные компании используют Kubernetes, значит и нам нужен Kubernetes для 3 микросервисов с 100 RPS". First principles: (1) Какую проблему решаем? Оркестрация контейнеров. (2) Какова наша реальная нагрузка? 100 RPS. (3) Факты: K8s создан для масштаба тысяч сервисов, его операционная стоимость -- 1-2 инженера. (4) Вывод: Docker Compose или простой сервис достаточен. K8s = overengineering по аналогии.

## Ключевые карточки

Что такое ментальная модель?
?
Упрощённое представление о том, как что-то работает. "Карта территории" -- не сама реальность, но инструмент навигации. Чем больше моделей в арсенале, тем точнее ориентируешься. "Карта != территория, но без карты заблудишься."

Что такое first principles thinking?
?
Разбор проблемы до базовых, неоспоримых истин и построение решения с нуля. Противоположность мышлению по аналогии ("все так делают"). Процесс: определи проблему -> разбей на компоненты -> что я знаю точно (факты, не мнения)? -> построй решение из фактов.

Что такое latticework моделей (Чарли Мангер)?
?
"Решётка" -- переплетение моделей из разных дисциплин, усиливающих друг друга. Понимание бизнеса требует моделей из психологии, экономики, биологии, физики. Специализация на одной модели создаёт "молоток, которому всё кажется гвоздём".

Почему "карта не равна территории"?
?
Любая модель -- упрощение реальности. Она полезна для навигации, но не является самой реальностью. Опасность: принять модель за реальность и игнорировать то, что в модель не вписывается. Нужно помнить ограничения каждой модели и использовать несколько.

Когда аналогичное мышление опасно?
?
Когда условия изменились, а прошлый опыт применяется некритично. "Все используют X" -- не аргумент, если контекст другой. Пример: "Netflix использует микросервисы" != "нам нужны микросервисы" (другой масштаб, другая команда, другие требования). First principles помогает избежать слепого копирования.
