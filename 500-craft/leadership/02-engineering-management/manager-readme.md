# Manager README: Как работать со мной

> **TL;DR:** Manager README — документ, объясняющий твой стиль работы, values, и expectations. Ускоряет onboarding, снижает friction, строит trust через прозрачность. Это не самореклама и не замена relationship building — это starting point для conversations.

---

## Зачем это нужно?

### Типичная ситуация

Новый член команды несколько месяцев пытается понять: "Что менеджер ценит? Как предпочитает получать информацию? Когда можно беспокоить?" Тратит энергию на угадывание. Первые conflicts возникают из-за misalignment, которого могло не быть.

**Без Manager README:**
- Долгий период "притирки"
- Unnecessary friction
- Assumptions вместо facts
- Trial and error дорого стоит

**С Manager README:**
- Faster onboarding
- Transparency с дня 1
- Clear expectations
- Foundation для trust

### Статистика

| Метрика | Значение | Источник |
|---------|----------|----------|
| Время на "понять менеджера" без README | 2-4 месяца | First Round |
| С README | 2-4 недели | First Round |
| Teams с documentation culture | 23% higher satisfaction | Atlassian |
| Trust correlation с transparency | +0.7 | Edelman |

### Что README НЕ является

| Это НЕ README | Почему |
|---------------|--------|
| Замена conversations | Документ — start, не end |
| Excuse for behavior | "Я предупредил" — не оправдание |
| Static document | Update регулярно |
| Self-promotion | Честность > хороший образ |
| Rules for team | Это про тебя, не про них |

### Principles при написании

```
ЧЕСТНОСТЬ > ХОРОШИЙ ОБРАЗ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Пиши про реального себя, не идеального.
Включай weaknesses и quirks.
Если все говорят что ты X, напиши X.

КОНКРЕТИКА > АБСТРАКЦИЯ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Плохо: "Я ценю коммуникацию"
Хорошо: "Предпочитаю Slack для quick Qs,
email для async, meeting для complex."

ACTIONABLE > PHILOSOPHICAL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Плохо: "Верю в прозрачность"
Хорошо: "Спрашивай меня anything.
Если не могу ответить, скажу почему."

LIVING DOCUMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Update после feedback от команды.
Review каждые 6 месяцев.
Add new learnings о себе.
```

## Пошаговый процесс создания

### Шаг 1: Self-Reflection

```
ВОПРОСЫ К СЕБЕ:

COMMUNICATION:
□ Как предпочитаю узнавать о проблемах?
□ Какие каналы для чего использую?
□ Когда я НЕ доступен?
□ Как быстро отвечаю обычно?

FEEDBACK:
□ Как даю критический feedback?
□ Как реагирую на feedback о себе?
□ Что делать если со мной не согласен?

VALUES:
□ Что ценю больше всего в работе?
□ Что точно НЕ приемлю?
□ Как принимаю решения?

QUIRKS:
□ Что меня бесит (пусть и иррационально)?
□ Какие у меня привычки?
□ Когда я работаю лучше/хуже?

WEAKNESSES:
□ Где я not great?
□ Что пытаюсь улучшить?
□ Как это влияет на команду?
```

### Шаг 2: Собери feedback

```
СПРОСИ У ТЕКУЩЕЙ КОМАНДЫ:
"Что бы ты сказал новому члену команды
о работе со мной?"

СПРОСИ У ПРОШЛЫХ КОЛЛЕГ:
"Что было неожиданно в работе со мной?"
"Что стоило бы знать с самого начала?"

СПРОСИ У СВОЕГО МЕНЕДЖЕРА:
"Как бы ты описал мой стиль?"
"Что я могу не осознавать о себе?"
```

### Шаг 3: Draft

```
НАПИСАТЬ DRAFT:
□ Следуй структуре (8 sections)
□ 1-2 страницы максимум
□ Bullet points > paragraphs
□ Honest and specific
□ Include examples
```

### Шаг 4: Review

```
ПОПРОСИ FEEDBACK НА DRAFT:
□ От trusted colleague
□ От текущего report (если есть)
□ От менеджера

QUESTIONS:
"Это похоже на меня?"
"Что пропустил?"
"Что звучит BS?"
```

### Шаг 5: Share and Iterate

```
SHARING:
□ В shared doc (Notion, Confluence)
□ Дай новым members в первую неделю
□ Обсуди на первом 1-on-1

ITERATE:
□ Спрашивай "Что оказалось неточным?"
□ Update после significant feedback
□ Review каждые 6 месяцев
```

## Полный Template

```markdown
# Manager README: [Your Name]

*Last updated: [Date]*
*Reading time: ~5 minutes*

## Who I Am

[2-3 sentences about your background and role]

My role is to:
- [Responsibility 1 — e.g., help you grow professionally]
- [Responsibility 2 — e.g., provide context from leadership]
- [Responsibility 3 — e.g., remove blockers]

## How I Communicate

**Channels:**
- **Slack:** Quick questions, casual chat. Response: same day.
- **Email:** Async, non-urgent. Response: within 24h.
- **Meeting:** Complex discussions, 1-on-1s, decisions.

**Availability:**
- Working hours: [e.g., 9am-6pm CET]
- I try not to message evenings/weekends. If I do, no response expected.
- **Urgent after hours:** [How to reach — e.g., SMS, phone]

**Meeting preferences:**
- [e.g., "I prefer meetings with agendas"]
- [e.g., "Feel free to decline meetings if unclear purpose"]

## How I Give Feedback

- I try to give feedback soon after observation (not months later).
- Constructive feedback is always private, never public.
- I use [SBI/other framework] — situation, behavior, impact.
- Positive feedback: I try to do publicly when appropriate.

**If I don't give feedback:**
It doesn't mean everything is perfect. Please ask!
"Do you have any feedback for me?" is always welcome.

## How I Receive Feedback

- I genuinely want feedback. It helps me improve.
- Best way: direct, specific, in 1-on-1 or Slack DM.
- I may need a moment to process, but I won't react defensively.
- Anonymous is okay if you prefer (via [channel]).

**What to do if you disagree with me:**
Tell me. I'm often wrong. If I'm not convinced, I'll explain why.
Disagree and commit is fine, but I'd rather find shared understanding.

## What I Value

- **[Value 1]:** [Explanation + what it looks like]
  - Example: Transparency — I share context even when uncomfortable
- **[Value 2]:** [Explanation]
- **[Value 3]:** [Explanation]

## What I Don't Tolerate

- [Behavior 1 — e.g., "Disrespecting teammates"]
- [Behavior 2 — e.g., "Hiding problems until they explode"]
- [Behavior 3 — e.g., "Blame without ownership"]

## My Quirks and Tendencies

**Things to know:**
- [e.g., "I think out loud. First reaction isn't final."]
- [e.g., "I prefer written async over meetings for info sharing."]
- [e.g., "I'm not a morning person. Better conversations after 10am."]

**What makes me grumpy:**
- [e.g., "Surprises in public (tell me bad news privately first)"]
- [e.g., "Meetings without agendas"]
- [e.g., "Repeating same info multiple times"]

**What energizes me:**
- [e.g., "Seeing people grow"]
- [e.g., "Solving hard problems together"]

## What I Expect From You

- **[Expectation 1]:** [e.g., "Communicate proactively — no surprises"]
- **[Expectation 2]:** [e.g., "Own your work end-to-end"]
- **[Expectation 3]:** [e.g., "Ask for help when stuck"]

## What You Can Expect From Me

- **[Commitment 1]:** [e.g., "Weekly 1-on-1s, protected time"]
- **[Commitment 2]:** [e.g., "Honest feedback, even when hard"]
- **[Commitment 3]:** [e.g., "Advocate for you in room where you're not"]

## 1-on-1s

- **Frequency:** [e.g., Weekly, 30 min]
- **Purpose:** Your time, not status update. Topics you want to discuss.
- **What I typically cover:** Career, blockers, feedback both ways

**I find it helpful when you:**
- Come with topics (even one is fine)
- Tell me about wins AND struggles
- Give me feedback on how I can help better

## Known Bugs in My Operating System

Nobody's perfect. Things I'm actively working on:

- **[Bug 1]:** [e.g., "I sometimes jump to solutions too fast"]
  - *How to help:* [e.g., "Tell me 'let me finish thinking'"]

- **[Bug 2]:** [e.g., "I can over-commit and then miss things"]
  - *How to help:* [e.g., "Remind me if I forget something"]

- **[Bug 3]:** [e.g., "I'm not great at small talk"]
  - *How to help:* [e.g., "Just dive into substance, I won't be offended"]

## Questions for You

After reading this, I'd love to know:
1. What's your preferred communication style?
2. How do you like to receive feedback?
3. What should I know about working with you?

---

*This is a living document. Feedback always welcome.*
```

## Распространённые ошибки

### Ошибка 1: PR Document

**Как выглядит:**
Только хорошее. "Я ценю людей, прозрачность, excellence..."
Никаких слабостей, quirks, real talk.

**Почему это проблема:**
- Неаутентично
- Team не доверяет
- Бесполезно для onboarding

**Как исправить:**
```
ДОБАВЬ "KNOWN BUGS":
• "Я иногда перебиваю когда excited"
• "Склонен к micromanagement под давлением"
• "Не great at remembering personal details"

If your README has no weaknesses,
it's not honest.
```

### Ошибка 2: Substitute for Conversation

**Как выглядит:**
"Прочитай мой README" вместо разговора.
README как excuse для не relationship building.

**Почему это проблема:**
- README — starting point, не end
- Trust через conversations
- README не заменяет 1-on-1

**Как исправить:**
```
README + CONVERSATION:
"Прочитай README как background.
В первый 1-on-1 обсудим что resonates,
что surprises, что хочешь clarify."
```

### Ошибка 3: Never Updated

**Как выглядит:**
Написал 2 года назад. С тех пор вырос, изменился. README не отражает текущего тебя.

**Почему это проблема:**
- Misleads новых members
- Outdated expectations
- Shows neglect

**Как исправить:**
```
REVIEW SCHEDULE:
□ После каждого significant feedback
□ После смены роли/команды
□ Минимум каждые 6 месяцев
□ При онбординге нового человека
```

### Ошибка 4: Too Long

**Как выглядит:**
10 страниц философии. Никто не читает целиком.

**Почему это проблема:**
- Information overload
- Key points теряются
- Nobody reads

**Как исправить:**
```
CONSTRAINTS:
• 1-2 pages maximum
• Bullet points > paragraphs
• Most important first
• "Read in 5 minutes"

Edit ruthlessly:
"Is this ACTUALLY useful to know?"
```

## Когда применять

### Создай README когда:
- Только стал менеджером (first 30 days)
- Пришёл в новую команду
- Team grows (новые members)
- После feedback что тебя "сложно понять"

### Update README когда:
- Получил feedback что README неточный
- Значительно изменился стиль
- Сменил роль/level
- Каждые 6 месяцев (minimum)

### README особенно ценен:
- Remote/distributed teams
- Large teams (8+ reports)
- Fast onboarding needed
- Cross-cultural teams

## Проверь себя

> [!question]- Почему секция "Known Bugs" в Manager README критически важна?
> Без weaknesses и quirks README превращается в PR-документ, которому никто не доверяет. Честное признание limitations ("иногда перебиваю", "склонен к micromanagement под давлением") строит trust через vulnerability и показывает self-awareness. Если README не содержит weaknesses -- он нечестный.

> [!question]- Менеджер написал README 2 года назад и с тех пор не обновлял. Какие проблемы это создаёт?
> README не отражает текущего менеджера -- misleads новых members, outdated expectations, показывает neglect. Решение: review после каждого significant feedback, при смене роли/команды, минимум каждые 6 месяцев, при онбординге нового человека.

> [!question]- Почему Manager README не заменяет relationship building?
> README -- starting point для conversations, не end. Trust строится через relationships, не документы. Правильный подход: "Прочитай README как background. В первый 1-on-1 обсудим что resonates, что surprises, что хочешь clarify." Документ ускоряет process, но не заменяет его.

## Ключевые карточки

Какие 8 секций должен содержать Manager README?
?
1) Intro & Role, 2) Communication (каналы, availability), 3) Feedback (как даю, как получаю), 4) Values & Principles, 5) Working Style (quirks, что energizes/grumpy), 6) Expectations (от тебя/от меня), 7) Growth & 1-on-1s, 8) Known Bugs & Limitations.

Какие три принципа при написании README?
?
1) Честность > хороший образ (реальный себя, не идеальный). 2) Конкретика > абстракция ("Slack для quick Qs" вместо "ценю коммуникацию"). 3) Actionable > philosophical ("спрашивай anything" вместо "верю в прозрачность"). Плюс Living Document -- update регулярно.

На сколько README ускоряет onboarding?
?
Без README "понять менеджера" занимает 2-4 месяца, с README -- 2-4 недели (First Round). Teams с documentation culture имеют 23% higher satisfaction (Atlassian). Trust correlation с transparency: +0.7 (Edelman).
