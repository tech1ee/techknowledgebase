# RAG и Prompt Engineering: основы

LLM не знает что было вчера (knowledge cutoff) и не видит твои документы. RAG решает обе проблемы: ищем релевантные куски → добавляем в prompt. Prompt Engineering — как задать вопрос, чтобы получить нужный ответ.

---

## Терминология

| Термин | Значение |
|--------|----------|
| **RAG** | Retrieval-Augmented Generation |
| **Knowledge cutoff** | Дата, до которой модель "знает" мир |
| **Zero-shot** | Prompt без примеров |
| **Few-shot** | Prompt с несколькими примерами |
| **Chain-of-thought** | "Думай пошагово" — улучшает reasoning |
| **Vector database** | БД для хранения embeddings |
| **Semantic search** | Поиск по смыслу, не по ключевым словам |
| **Chunking** | Разбиение документа на куски для индексации |

---

## Теоретические основы

> **RAG** (Retrieval-Augmented Generation) — архитектурный паттерн, формализованный Lewis et al. (2020), при котором генеративная модель дополняется внешним retrieval-компонентом: $P(y|x) = \sum_{z \in \text{top-k}} P(z|x) \cdot P(y|x, z)$, где $z$ — найденные документы, $x$ — запрос, $y$ — ответ.

### Теоретические корни

| Концепция | Автор/Год | Вклад в RAG/Prompt Engineering |
|-----------|-----------|-------------------------------|
| Information Retrieval | Salton, 1971 | TF-IDF, векторная модель поиска |
| Semantic similarity | Deerwester et al., 1990 (LSA) | Латентное семантическое пространство |
| Attention mechanism | Bahdanau et al., 2014 | Избирательное внимание к контексту |
| In-context learning | Brown et al., 2020 (GPT-3) | Few-shot через примеры в промпте |
| **RAG** | Lewis et al., 2020 | Формализация retrieval + generation |
| Chain-of-Thought | Wei et al., 2022 | Пошаговое рассуждение в промпте |

### Prompt Engineering как дисциплина

> **In-context learning** — способность LLM решать задачи на основе примеров, предоставленных в промпте, без обновления весов. Теоретическое объяснение: transformer неявно реализует gradient descent в forward pass (Akyürek et al., 2022).

Иерархия техник промптинга по сложности:

```
Zero-shot → Few-shot → Chain-of-Thought → Tree-of-Thought → ReAct
    │           │              │                  │             │
 Простой     2-5 примеров   "Думай         Ветвление      Действия
 запрос      в промпте      пошагово"      рассуждений    + наблюдение
```

> **Lost in the Middle** (Liu et al., 2023) — эмпирически показано, что LLM хуже обрабатывают информацию в середине длинного контекста, что определяет стратегию размещения чанков в RAG-промпте.

## Зачем это нужно

У LLM (Large Language Models — больших языковых моделей) есть две фундаментальные проблемы:

1️⃣ **Знания устаревают**

GPT-4 обучен на данных до определённой даты (knowledge cutoff). Он не знает, что произошло вчера. Не знает о новых библиотеках, релизах, событиях.

**Пример:** Спросишь про "Claude 3.5 Sonnet" у модели с cutoff до июня 2024 — она не знает, что это такое.

2️⃣ **Нет доступа к твоим данным**

Модель не видит твою базу знаний, документы компании, историю переписок, внутренние wiki.

**Пример:** "Какая у нас политика отпусков?" — модель не может ответить, потому что не видела твой HR-документ.

RAG и Prompt Engineering — два главных инструмента для решения этих проблем.

## Prompt Engineering

### Что это

**Prompt Engineering** — это искусство и наука формулирования запросов к модели так, чтобы получить нужный результат.

Почему это важно: один и тот же вопрос, заданный по-разному, даёт кардинально разные ответы.

```
❌ "Напиши код"
   → Модель не знает какой код, на каком языке, для чего

✅ "Напиши Python-функцию, которая принимает список чисел
    и возвращает их среднее значение. Добавь docstring
    и обработку пустого списка."
   → Чёткий, предсказуемый результат
```

### Базовые техники

#### 1️⃣ Zero-shot prompting

Просто задаёшь вопрос без примеров. Модель опирается только на свои знания.

```
Переведи на английский: "Привет, мир"
```

**Когда работает:** Простые, однозначные задачи. Переводы, саммаризация, ответы на фактические вопросы.

**Когда не работает:** Сложные задачи с нестандартным форматом вывода.

#### 2️⃣ Few-shot prompting

Даёшь несколько примеров перед запросом. Модель учится на примерах "на лету".

```
Классифицируй отзыв как позитивный или негативный:

"Отличный товар, доставили быстро" -> позитивный
"Ужасное качество, не рекомендую" -> негативный
"Товар нормальный, но доставка затянулась" -> ?
```

**Почему работает:** Модель видит паттерн и применяет его. Это называется in-context learning — обучение в контексте, без изменения весов модели.

**Сколько примеров:** Обычно 2-5 достаточно. Больше — не всегда лучше, плюс тратишь токены.

#### 3️⃣ Chain-of-Thought (CoT)

Просишь модель рассуждать пошагово, показывая ход мыслей.

```
Реши задачу. Думай шаг за шагом.

У Маши было 5 яблок. Она отдала Пете 2 яблока, а потом
купила ещё 3. Сколько яблок у Маши?
```

**Ответ модели с CoT:**
```
Шаг 1: У Маши было 5 яблок
Шаг 2: Она отдала 2, осталось 5 - 2 = 3
Шаг 3: Купила ещё 3, стало 3 + 3 = 6
Ответ: 6 яблок
```

**Почему важно:** Без CoT модели часто ошибаются в математике и логике. С CoT точность резко возрастает. Исследования Google показали улучшение на 50%+ в арифметических задачах.

**Вариации:**
- **Zero-shot CoT:** Просто добавь "Let's think step by step" или "Думай шаг за шагом"
- **Few-shot CoT:** Покажи примеры с рассуждениями

#### 4️⃣ System prompts

Задаёшь роль, контекст и правила в начале разговора. System prompt — это инструкция, которая действует на всю сессию.

```
System: Ты — опытный Python-разработчик с 10-летним опытом.
Правила:
- Отвечай кратко и по делу
- Всегда давай примеры кода
- Если не уверен — говори об этом
- Используй type hints в Python-коде
```

**Почему важно:** System prompt формирует "личность" и поведение модели. Без него — получаешь дефолтное поведение.

#### 5️⃣ Structured output prompting

Просишь модель выдать ответ в конкретном формате (JSON, XML, Markdown).

```
Извлеки информацию из текста и верни в JSON-формате:

Текст: "Иван Петров, 35 лет, работает программистом в Яндексе"

Формат:
{
  "name": "...",
  "age": ...,
  "job": "...",
  "company": "..."
}
```

**Почему важно:** Структурированный вывод можно парсить программно. Без этого — получаешь свободный текст, который сложно обрабатывать.

### Параметры генерации

При работе с LLM API есть параметры, влияющие на результат:

**Temperature** (0.0 - 2.0)
- **0.0** — детерминированный вывод, модель выбирает наиболее вероятные токены
- **0.7** — баланс между креативностью и предсказуемостью
- **1.0+** — более "креативный", но менее предсказуемый вывод

```python
# Для кода и фактических ответов
response = client.chat(temperature=0.0)

# Для креативных задач
response = client.chat(temperature=0.8)
```

**Max tokens** — ограничение длины ответа. Не путать с context window (максимальный контекст, который модель может обработать).

**Top_p** (nucleus sampling) — альтернатива temperature. Обычно используют что-то одно.

### Частые ошибки в промптах

| Ошибка | Пример | Решение |
|--------|--------|---------|
| Слишком общий промпт | "Напиши текст" | Добавить конкретику: тема, длина, стиль, аудитория |
| Нет примеров | "Классифицируй отзывы" | Добавить 2-3 примера с ответами |
| Нет ограничений | "Объясни ML" | "Объясни ML в 3 предложениях для новичка" |
| Противоречия | "Будь кратким. Дай подробный ответ" | Убрать противоречия |
| Негативные инструкции | "Не используй слово X" | Модели плохо обрабатывают отрицания |

### Про "магические промпты"

*Честно говоря, большинство "секретных техник" из интернета — переоценённый хайп. Работает простое правило: чем точнее ты описал задачу, тем лучше результат. Никакой магии.*

## RAG (Retrieval-Augmented Generation)

### Что это

**RAG** — архитектурный паттерн, когда перед генерацией ответа модель получает релевантные данные из внешнего источника.

Простыми словами: вместо "ответь на вопрос" мы делаем "вот контекст из нашей базы, теперь ответь на вопрос на его основе".

**Retrieval** = поиск, **Augmented** = дополненный, **Generation** = генерация.

### Как работает RAG

**Этап 1: Подготовка базы знаний (делается один раз)**

```
┌──────────────────────────────────────────────────────────────┐
│                    ТВОИ ДОКУМЕНТЫ                            │
│  📄 FAQ.pdf  📄 Политика.docx  📄 Инструкции.md              │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                   РАЗБИЕНИЕ НА CHUNKS                        │
│  "Отпуск 28 дней..."  "Больничный оплачивается..."  "..."    │
│       chunk 1              chunk 2              chunk N      │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                ПРЕВРАЩЕНИЕ В EMBEDDINGS                      │
│  [0.2, 0.8, -0.1...]  [0.5, 0.3, 0.7...]  [...]             │
│     вектор 1              вектор 2           вектор N        │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                   ВЕКТОРНАЯ БАЗА ДАННЫХ                      │
│  🗄️ Chroma / Pinecone / Weaviate                            │
│  Хранит: вектор + оригинальный текст + метаданные           │
└──────────────────────────────────────────────────────────────┘
```

**Этап 2: Ответ на вопрос (каждый запрос)**

```
┌─────────────────┐
│  "Сколько дней  │
│   отпуска?"     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐     ┌─────────────────────────────────┐
│  Вопрос →       │     │                                 │
│  Embedding      │────▶│   🔍 Поиск похожих векторов    │
│  [0.3, 0.7...]  │     │      в базе данных             │
└─────────────────┘     └────────────┬────────────────────┘
                                     │
         ┌───────────────────────────┘
         │ Найдено: "Отпуск 28 дней...", "Дополнительные дни..."
         ▼
┌─────────────────────────────────────────────────────────────┐
│                    ФОРМИРОВАНИЕ ПРОМПТА                     │
│                                                             │
│  "Ответь на вопрос, используя ТОЛЬКО эту информацию:       │
│                                                             │
│   Контекст:                                                 │
│   - Отпуск составляет 28 календарных дней...               │
│   - Дополнительные дни за выслугу лет...                   │
│                                                             │
│   Вопрос: Сколько дней отпуска?"                           │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│                         LLM (GPT-4)                         │
│  "Сотрудники имеют право на 28 календарных дней отпуска.   │
│   При выслуге более 5 лет добавляется 3 дополнительных."   │
└─────────────────────────────────────────────────────────────┘
```

**Ключевая магия:** LLM отвечает НЕ из своих знаний, а из твоих документов. Поэтому не галлюцинирует (ну, почти).

### Ключевые термины

**Embedding (эмбеддинг)** — числовое представление текста в виде вектора. Модель превращает текст в массив чисел (например, 1536 чисел для OpenAI text-embedding-ada-002).

**Почему это работает:** Похожие по смыслу тексты имеют близкие векторы. "Кошка" и "котёнок" будут ближе друг к другу, чем "кошка" и "автомобиль".

```python
from openai import OpenAI
client = OpenAI()

# Получаем embedding для текста
response = client.embeddings.create(
    input="Как работает garbage collector?",
    model="text-embedding-ada-002"
)
vector = response.data[0].embedding  # [0.023, -0.041, 0.089, ...]
```

**Chunk (чанк)** — кусок документа. Большой документ разбивается на части, каждая индексируется отдельно.

**Векторная база данных** — специализированная БД для хранения и поиска по векторам. Примеры: Pinecone, Chroma, Weaviate, Qdrant.

**Similarity search** — поиск по похожести. Находим chunks, чьи векторы ближе всего к вектору вопроса.

### Пример RAG (подробный код)

```python
from openai import OpenAI
import chromadb

client = OpenAI()
chroma = chromadb.Client()
collection = chroma.create_collection("knowledge_base")

# === Этап 1: Индексация документов ===

documents = [
    "Политика отпусков: сотрудники имеют право на 28 дней отпуска в год.",
    "Больничный оплачивается с первого дня. Нужна справка от врача.",
    "Удалённая работа разрешена 2 дня в неделю по согласованию."
]

# Получаем embeddings для всех документов
for i, doc in enumerate(documents):
    response = client.embeddings.create(
        input=doc,
        model="text-embedding-ada-002"
    )
    embedding = response.data[0].embedding

    # Сохраняем в векторную БД
    collection.add(
        ids=[f"doc_{i}"],
        embeddings=[embedding],
        documents=[doc]
    )

# === Этап 2: Поиск и генерация ===

question = "Сколько дней отпуска у сотрудников?"

# Получаем embedding вопроса
q_response = client.embeddings.create(
    input=question,
    model="text-embedding-ada-002"
)
q_embedding = q_response.data[0].embedding

# Ищем похожие документы (top_k=2 — вернуть 2 самых похожих)
results = collection.query(
    query_embeddings=[q_embedding],
    n_results=2
)

# Найденный контекст
context = "\n".join(results["documents"][0])

# Формируем промпт
prompt = f"""Ответь на вопрос, используя ТОЛЬКО информацию из контекста.
Если в контексте нет ответа — скажи "Информация не найдена".

Контекст:
{context}

Вопрос: {question}
"""

# Генерируем ответ
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": prompt}],
    temperature=0.0
)

print(response.choices[0].message.content)
# "Сотрудники имеют право на 28 дней отпуска в год."
```

### Зачем RAG, если можно всё засунуть в промпт?

1️⃣ **Ограничение контекста (context window)**

У моделей есть лимит токенов:
- GPT-3.5: 16k токенов
- GPT-4: 128k токенов
- Claude 3: 200k токенов

Вся корпоративная документация туда не влезет. 100 страниц текста — это примерно 75k токенов.

2️⃣ **Стоимость**

Больше токенов = больше денег. GPT-4 стоит $30-60 за 1M токенов. RAG достаёт только релевантные куски.

3️⃣ **Актуальность**

Векторную базу можно обновлять мгновенно. Переобучать модель — дорого (fine-tuning) или невозможно (OpenAI GPT).

4️⃣ **Качество ответов**

Парадоксально, но меньше контекста = лучше ответы. Модели "теряются" в слишком большом контексте (эффект "lost in the middle").

### RAG vs Fine-tuning vs Prompt Engineering

| Метод | Когда использовать | Примеры |
|-------|-------------------|---------|
| **Prompt Engineering** | Простые задачи, форматирование | Переводы, саммаризация, классификация |
| **RAG** | Доступ к внешним данным, актуальность | Q&A по документам, корп. чат-боты |
| **Fine-tuning** | Специфический стиль, узкая область | Юридический язык, медицинские термины |

*На практике часто комбинируют все три. Например: fine-tuned модель + RAG + хороший system prompt.*

## Продвинутые техники RAG

### HyDE (Hypothetical Document Embeddings)

**Проблема:** Вопрос пользователя короткий, а документы длинные. Их embeddings могут не совпадать.

**Решение:** Генерируем гипотетический ответ и ищем по нему.

```
Вопрос: "Как работает GC в Python?"
         ↓
LLM генерирует гипотетический ответ:
"Python использует reference counting как основной механизм.
Когда счётчик ссылок объекта падает до нуля, память освобождается.
Для циклических ссылок используется generational garbage collector..."
         ↓
Поиск по этому тексту (embedding ближе к реальным документам)
```

### Re-ranking

**Проблема:** Первичный поиск (similarity search) не всегда точен.

**Решение:** После поиска пропускаем результаты через модель-ранкер.

```python
# 1. Получаем top-20 результатов
initial_results = vector_db.search(query, top_k=20)

# 2. Re-rank через специализированную модель
reranked = reranker.rank(query, initial_results)

# 3. Берём top-5 после re-ranking
final_results = reranked[:5]
```

**Инструменты:** Cohere Rerank, BGE Reranker, cross-encoders из sentence-transformers.

### Стратегии Chunking

Как разбивать документы:

| Стратегия | Описание | Плюсы | Минусы |
|-----------|----------|-------|--------|
| **Фиксированный размер** | 500 символов | Простота | Режет предложения |
| **По предложениям** | Каждое предложение отдельно | Целостность | Теряется контекст |
| **По параграфам** | Естественные границы | Баланс | Разный размер |
| **С перекрытием** | Chunks пересекаются на 20% | Сохраняет контекст | Больше данных |
| **Семантический** | Разбиение по смыслу | Лучшее качество | Сложнее реализовать |

*Выбор стратегии chunking влияет на качество больше, чем кажется. Это одно из первых мест для оптимизации.*

### Metadata filtering

Добавляем метаданные к chunks и фильтруем при поиске.

```python
# При индексации
collection.add(
    documents=["..."],
    metadatas=[{
        "source": "hr_policy.pdf",
        "department": "HR",
        "date": "2025-01-15"
    }]
)

# При поиске — фильтруем по отделу
results = collection.query(
    query_embeddings=[...],
    where={"department": "HR"}  # Только HR-документы
)
```

## Типичные проблемы и решения

### "Модель игнорирует контекст"

**Причины:**
- Контекст слишком большой (модель "теряется")
- Нерелевантные документы в контексте
- Плохой промпт без явных инструкций

**Решения:**
- Уменьшить top_k (меньше документов)
- Добавить re-ranking
- Явно сказать: "Отвечай ТОЛЬКО на основе документов ниже"

### "Модель галлюцинирует"

**Галлюцинации** — модель выдумывает информацию, которой нет в контексте. Подробнее: [[ai-engineering-intro]]

**Решения:**
- Инструкция: "Если в документах нет ответа — скажи 'Информация не найдена'"
- Требовать цитаты: "Цитируй источник для каждого утверждения"
- Temperature = 0.0 (детерминированный вывод)
- Добавить self-consistency checking (несколько запросов, сравнение ответов)

### "Поиск находит не то"

**Решения:**
- Попробовать другую модель эмбеддингов (например, BGE вместо Ada)
- Изменить стратегию chunking (меньше/больше размер)
- Добавить metadata filtering
- Использовать hybrid search (vector + keyword)

## Практическое применение

### Где RAG работает хорошо

- Q&A по документации (техническая документация, wiki)
- Корпоративные чат-боты (HR, IT support)
- Поиск по базе знаний
- Customer support (ответы на типовые вопросы)
- Legal/compliance (поиск по нормативным документам)

### Где RAG работает плохо

- Задачи, требующие рассуждений (лучше Chain-of-Thought)
- Генерация креативного контента (нет базы для retrieval)
- Задачи без чёткой базы знаний
- Real-time данные (лучше прямые API-вызовы)

## Проверь себя

> [!question]- Почему RAG даёт более точные ответы, чем простое увеличение context window и вставка всех документов в промпт?
> Дело не только в лимите токенов. Даже если контекстное окно позволяет вместить все документы, модель страдает от эффекта "lost in the middle" — она хуже обрабатывает информацию в середине длинного контекста. RAG решает это, подавая только релевантные фрагменты (обычно 2-5 chunks), что повышает точность. Кроме того, RAG экономит деньги: меньше токенов = меньше стоимость каждого запроса.

> [!question]- У тебя RAG-система для корпоративной документации. Пользователи жалуются, что на вопрос "какие бонусы у сотрудников?" система отвечает про отпуска. Какие три шага ты предпримешь для диагностики и исправления?
> 1) **Проверить chunking** — возможно, информация о бонусах и отпусках попала в один chunk, и при поиске возвращается нерелевантный фрагмент. Решение: уменьшить размер chunks или использовать семантическое разбиение. 2) **Добавить re-ranking** — первичный similarity search нашёл "близкие" по теме HR документы, но не самые релевантные. Cross-encoder reranker переоценит результаты. 3) **Добавить metadata filtering** — разметить документы по категориям (compensation, leave, benefits) и фильтровать при поиске.

> [!question]- Чем Few-shot prompting принципиально отличается от fine-tuning, если оба подхода используют примеры для обучения модели?
> Few-shot — это in-context learning: модель видит примеры в промпте и применяет паттерн "на лету", но веса модели не меняются. После завершения сессии модель "забывает" примеры. Fine-tuning физически изменяет веса модели на специализированных данных — результат сохраняется навсегда. Few-shot быстрый и дешёвый (2-5 примеров), но ограничен контекстным окном. Fine-tuning дорогой, но даёт глубокую специализацию (стиль, терминология, доменное знание).

> [!question]- Разработчик установил temperature=0.0, но получает разные ответы на один и тот же промпт. Почему это возможно?
> Temperature=0.0 делает выбор наиболее вероятного токена детерминированным, но не гарантирует полную идентичность ответов. Причины: 1) API-провайдеры могут использовать разные GPU с разной точностью вычислений (floating point). 2) Batch processing на стороне сервера может влиять на вычисления. 3) Модель может обновляться (versioning) — OpenAI периодически обновляет модели. Для критичных задач нужно фиксировать версию модели (например, `gpt-4-0613`) и использовать seed-параметр.

## Ключевые карточки

Что такое RAG и какую проблему LLM он решает?
?
RAG (Retrieval-Augmented Generation) — паттерн, при котором перед генерацией ответа модель получает релевантные данные из внешнего источника. Решает две проблемы: knowledge cutoff (устаревшие знания) и отсутствие доступа к пользовательским данным.

Назови 5 базовых техник Prompt Engineering
?
1) Zero-shot — запрос без примеров. 2) Few-shot — запрос с 2-5 примерами. 3) Chain-of-Thought — пошаговое рассуждение. 4) System prompts — задание роли и правил. 5) Structured output — запрос ответа в конкретном формате (JSON, XML).

Что такое embedding и почему semantic search работает?
?
Embedding — числовое представление текста в виде вектора (например, 1536 чисел). Работает потому, что похожие по смыслу тексты получают близкие векторы в многомерном пространстве. "Кошка" и "котёнок" будут ближе друг к другу, чем "кошка" и "автомобиль".

Какие 5 стратегий chunking существуют и какая лучше для старта?
?
1) Фиксированный размер — простой, но режет предложения. 2) По предложениям — целостный, но теряет контекст. 3) По параграфам — естественные границы, разный размер. 4) С перекрытием (overlap 20%) — сохраняет контекст между chunks. 5) Семантический — лучшее качество, сложнее реализовать. Для старта лучше всего: фиксированный размер с перекрытием (~500 символов, overlap 20%).

Что такое HyDE и какую проблему решает?
?
HyDE (Hypothetical Document Embeddings) — техника, при которой сначала LLM генерирует гипотетический ответ на вопрос, а затем ищет похожие документы по embedding этого ответа. Решает проблему несовпадения коротких вопросов и длинных документов в векторном пространстве.

Когда использовать RAG, а когда fine-tuning?
?
RAG — когда нужен доступ к внешним данным и актуальность (Q&A по документам, корпоративные чат-боты). Fine-tuning — когда нужен специфический стиль или узкая область (юридический язык, медицинская терминология). На практике часто комбинируют оба подхода.

Что такое эффект "lost in the middle" и как он влияет на RAG?
?
Модели хуже обрабатывают информацию, расположенную в середине длинного контекста — они лучше "помнят" начало и конец. Поэтому в RAG важно: 1) подавать только релевантные chunks (меньше контекста = лучше ответ), 2) размещать самые важные фрагменты в начале промпта, 3) использовать re-ranking для отбора лучших результатов.
