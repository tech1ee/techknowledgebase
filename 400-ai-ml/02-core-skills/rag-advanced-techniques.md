## Терминология

| Термин | Значение | Аналогия |
|--------|----------|----------|
| **RAG** | Retrieval-Augmented Generation — дополнение промпта релевантными документами | Дать эксперту справочные материалы перед ответом |
| **Chunk** | Фрагмент документа для индексации (обычно 500-1000 токенов) | Страница из книги |
| **Retrieval** | Поиск релевантных чанков по запросу | Поиск в библиотеке |
| **Vector Search** | Поиск по semantic similarity через embeddings | Найти "похожее по смыслу" |
| **BM25** | Keyword-based поиск (как Google до neural search) | Найти "точное совпадение слов" |
| **Hybrid Search** | Комбинация Vector + BM25 | И по смыслу, и по ключевым словам |
| **Reranking** | Переоценка релевантности после retrieval | Проверить и отсортировать результаты |
| **Query Decomposition** | Разбиение сложного запроса на подзапросы | Разбить задачу на подзадачи |
| **Agentic RAG** | RAG с автономным агентом (planning, self-correction) | Исследователь, который сам решает как искать |
| **Self-RAG** | RAG с self-reflection токенами для самопроверки | Студент, который проверяет свои ответы |
| **CRAG** | Corrective RAG — fallback на web search при плохом retrieval | План Б, если не нашёл в базе |
| **GraphRAG** | RAG с knowledge graph для thematic вопросов | Карта связей между концептами |
| **Contextual Retrieval** | Добавление контекста к чанкам перед embedding | Подписать страницу: "это из отчёта Apple за Q2" |
| **Faithfulness** | Метрика: ответ основан на контексте, нет галлюцинаций | "Говорю только то, что прочитал" |
| **RAGAS** | Фреймворк для оценки качества RAG | Экзамен для RAG-системы |

## TL;DR

> **Advanced RAG** выходит за рамки простого "retrieve + generate". Включает: Query Decomposition (разбиение сложных запросов), Agentic RAG (агенты с самокоррекцией), Self-RAG (reflection tokens), GraphRAG (knowledge graphs), Contextual Retrieval (Anthropic). Оценка через RAGAS metrics.

## Зачем это нужно

### Проблема: Naive RAG — это только начало

| Симптом | Причина | Последствия |
|---------|---------|-------------|
| **"Ответ не по теме"** | Vector search находит "похожий" текст, а не релевантный | Галлюцинации, потеря доверия |
| **"Не отвечает на сложные вопросы"** | Multi-aspect запросы требуют разных источников | Неполные ответы |
| **"Работает на тестах, ломается в production"** | Нет self-correction и fallback | 30-50% ошибок на реальных данных |
| **"Не видит картину целиком"** | Vector search — needle, а не haystack | Невозможно отвечать на "Какие темы?" |

### Эволюция RAG: 2020 → 2025

```
2020: Naive RAG       → embed + retrieve + generate
2023: Advanced RAG    → + reranking, hybrid search, better chunking
2024: Agentic RAG     → + planning, self-correction, multi-source
2025: Специализация   → GraphRAG, Self-RAG, Contextual Retrieval
```

## Теоретические основы

### Формальное определение RAG

> **Retrieval-Augmented Generation (RAG)** — архитектурный паттерн, в котором генеративная модель дополняется механизмом извлечения релевантной информации из внешнего корпуса документов. Формально: P(y|x) = Σ P(y|x,z) · P(z|x), где z — извлечённые документы, x — запрос, y — ответ (Lewis et al., 2020, *"Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"*, Facebook AI).

### Ключевые работы

| Год | Работа | Авторы | Вклад |
|-----|--------|--------|-------|
| 2020 | RAG | Lewis et al. (Facebook AI) | Формализация retrieve-then-generate |
| 2020 | REALM | Guu et al. (Google) | Pre-training с retrieval |
| 2023 | Self-RAG | Asai et al. (U. Washington) | Reflection tokens для самопроверки |
| 2023 | CRAG | Yan et al. | Corrective RAG с web search fallback |
| 2024 | GraphRAG | Microsoft Research | Knowledge graph для тематических запросов |
| 2024 | Contextual Retrieval | Anthropic | Добавление контекста к чанкам, -49% failed retrieval |

### Эволюция архитектур RAG

> **Naive RAG** → фиксированный pipeline: embed → retrieve → generate. Проблемы: нет самокоррекции, единственный запрос, нет оценки качества.
>
> **Advanced RAG** → pre-retrieval (query rewriting, decomposition) + post-retrieval (reranking, compression). Качество retrieval улучшается, но pipeline остаётся фиксированным.
>
> **Modular RAG** → компонентная архитектура с взаимозаменяемыми модулями (Gao et al., 2024, *"Retrieval-Augmented Generation for Large Language Models: A Survey"*).
>
> **[[agentic-rag|Agentic RAG]]** → агент контролирует retrieval: решает нужен ли поиск, формулирует query, оценивает результаты, итерирует.

### Метрики качества RAG (RAGAS Framework)

| Метрика | Формула (упрощённо) | Что измеряет |
|---------|---------------------|--------------|
| **Faithfulness** | claims_supported / total_claims | Ответ основан на контексте |
| **Answer Relevance** | cosine_sim(answer, question) | Ответ отвечает на вопрос |
| **Context Precision** | relevant_chunks_ranked_high / total | Релевантные чанки вверху |
| **Context Recall** | gt_claims_in_context / total_gt_claims | Нужная информация найдена |

Фреймворк RAGAS (Es et al., 2023) стал стандартом де-факто для оценки RAG-систем.

**Результаты продвинутых техник:**
- Self-RAG: **+40% accuracy** за счёт self-reflection
- GraphRAG: **+76%** на сложных reasoning задачах
- Contextual Retrieval: **-49% ошибок** при поиске
- Agentic RAG: **60-75% снижение затрат** + 90% cache hit rates

## История: Как мы дошли до RAG

### Предыстория: мир до RAG

Представь себе 2022 год. ChatGPT только появился и поразил мир своими способностями. Но у него была одна огромная проблема: **он не знал ничего после своей даты обучения**. Спроси его про вчерашние новости - и он выдаст уверенную, но полностью выдуманную историю.

Компании смотрели на эту технологию и думали: "Как мы можем использовать это для наших данных? У нас терабайты документации, политик, FAQ - и мы хотим, чтобы AI отвечал на вопросы по ним".

Было несколько очевидных подходов, и все они имели критические недостатки:

**Подход 1: Fine-tuning** - дообучить модель на своих данных.
- Проблема: дорого, долго, при обновлении данных нужно переобучать заново. И модель все равно может галлюцинировать "своими" фактами.

**Подход 2: Запихнуть все в контекст** - просто отправлять все документы вместе с вопросом.
- Проблема: context window ограничен (тогда - 4K токенов), стоимость растет линейно, модель теряется в длинном тексте.

**Подход 3: Классический поиск** - найти релевантные документы и показать пользователю.
- Проблема: это не AI, пользователь сам должен читать и находить ответ.

### Рождение RAG: сентябрь 2020

Революция произошла в статье "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" от Facebook AI Research (теперь Meta AI). Авторы предложили элегантную идею: **объединить retrieval и generation**.

Суть была гениально простой:

```
Вопрос пользователя
       |
       v
[Поиск похожих документов в базе]
       |
       v
[Объединение: вопрос + найденные документы]
       |
       v
[LLM генерирует ответ]
       |
       v
Ответ с источниками
```

Почему это было революцией:
1. **Актуальность**: данные можно обновлять без переобучения модели
2. **Верифицируемость**: ответ основан на конкретных документах, которые можно показать
3. **Масштабируемость**: можно работать с миллионами документов
4. **Экономичность**: не нужен дорогой fine-tuning

### Эволюция: от Naive к Agentic

С 2020 по 2025 год RAG прошел через три эпохи:

**2020-2023: Naive RAG**
Простейшая схема: embed вопрос -> найди top-k похожих чанков -> отправь в LLM.
Работало, но часто плохо. Сложные вопросы ставили систему в тупик.

**2023-2024: Advanced RAG**
Инженеры начали добавлять оптимизации: query rewriting, hybrid search (vector + BM25), reranking. Качество выросло, но система все еще была "тупой" - она просто выполняла заданный pipeline.

**2024-2025: Agentic RAG**
Настоящий прорыв: RAG стал умным агентом. Теперь система сама решает, как лучше ответить на вопрос. Разбивает сложные запросы на простые. Проверяет качество найденных документов. Если результаты плохие - переформулирует запрос и ищет снова. Может использовать разные источники данных.

Результаты улучшений впечатляют:
- **Self-RAG**: +40% accuracy за счет самопроверки
- **GraphRAG**: +76% на сложных reasoning задачах
- **Contextual Retrieval**: -49% ошибок при поиске
- **Production Agentic RAG**: 60-75% снижение затрат + 90%+ cache hit rates

## Проблема простым языком

### Аналогия: эксперт с устаревшей энциклопедией

Представь, что ты нанял гениального эксперта. Он знает все в мире - историю, науку, бизнес. Но есть проблема: **его энциклопедия датирована январем 2024 года**. Все, что произошло после - для него не существует.

Ты спрашиваешь: "Какие изменения в нашей корпоративной политике отпусков произошли в этом году?"

Эксперт уверенно отвечает: "Согласно стандартным практикам, обычно компании предоставляют..." - и несет какую-то общую информацию, не имеющую отношения к ТВОЕЙ компании.

**RAG решает эту проблему элегантно**: перед тем как эксперт начнет отвечать, мы даем ему прочитать актуальные документы. Теперь он отвечает не из своей устаревшей памяти, а на основе свежих данных.

### Почему Naive RAG недостаточно

Представь более сложный сценарий. Клиент спрашивает:

> "Сравните условия кредитования для малого бизнеса в ваших продуктах Premium, Standard и Basic, и объясните, какой лучше подходит для сезонного бизнеса с неравномерным cash flow"

Naive RAG сделает простой vector search по этому запросу. Проблема: **нет одного документа, который отвечает на этот вопрос**. Информация о Premium - в одном документе. О Standard - в другом. О сезонном бизнесе - в третьем. И они все не похожи семантически на исходный запрос.

Результат: система найдет что-то отдаленно похожее и сгенерирует неполный или неточный ответ.

Продвинутые техники RAG были созданы именно для таких случаев.

## Query Decomposition

### Реальный сценарий

Допустим, ты работаешь в техподдержке банка. Приходит запрос от клиента:

> "У меня бизнес-карта Premium. В прошлом месяце была транзакция на 50000, которую я не совершал. При этом у меня настроен SMS-оповещения, но я ничего не получил. Хочу понять: как такое могло произойти, какая у вас политика по спорным транзакциям, и могу ли я получить возврат?"

Этот вопрос на самом деле содержит **три разных подвопроса**:
1. Как работают SMS-оповещения и почему они могли не прийти?
2. Какая политика банка по спорным транзакциям?
3. Каков процесс возврата средств?

Naive RAG попытается найти один документ, который отвечает на все сразу. Такого документа не существует.

### Как работает Query Decomposition

Техника проста и гениальна: **попросить LLM разбить сложный вопрос на простые, ответить на каждый отдельно, а потом синтезировать финальный ответ**.

```
┌────────────────────────────────────────────────────────────────────┐
│                    QUERY DECOMPOSITION                              │
├────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ИСХОДНЫЙ ЗАПРОС:                                                  │
│   "Сравните AI-стратегии Google, Microsoft и OpenAI,               │
│    и как каждая влияет на их облачный бизнес"                      │
│                                                                     │
│                           |                                         │
│                           v                                         │
│                    [LLM: DECOMPOSE]                                 │
│                           |                                         │
│          +----------------+----------------+                        │
│          |                |                |                        │
│          v                v                v                        │
│   ┌────────────┐   ┌────────────┐   ┌────────────┐                 │
│   │ Подвопрос 1│   │ Подвопрос 2│   │ Подвопрос 3│                 │
│   │ "AI-       │   │ "AI-       │   │ "AI-       │                 │
│   │  стратегия │   │  стратегия │   │  стратегия │                 │
│   │  Google?"  │   │  Microsoft?│   │  OpenAI?"  │                 │
│   └─────┬──────┘   └─────┬──────┘   └─────┬──────┘                 │
│         |                |                |                         │
│         v                v                v                         │
│   [RETRIEVE]       [RETRIEVE]       [RETRIEVE]                     │
│         |                |                |                         │
│         v                v                v                         │
│   [ANSWER]         [ANSWER]         [ANSWER]                       │
│         |                |                |                         │
│         +----------------+----------------+                         │
│                           |                                         │
│                           v                                         │
│                    [LLM: SYNTHESIZE]                                │
│                           |                                         │
│                           v                                         │
│                   ФИНАЛЬНЫЙ ОТВЕТ                                   │
│                                                                     │
└────────────────────────────────────────────────────────────────────┘
```

### Почему это работает

1. **Фокусировка поиска**: Вместо одного размытого запроса - несколько точных. Каждый подзапрос имеет больше шансов найти релевантные документы.

2. **Параллелизация**: Подзапросы можно обрабатывать параллельно, что ускоряет общее время ответа.

3. **Качество синтеза**: LLM отлично справляется с объединением нескольких четких ответов в один связный.

### Код с подробными пояснениями

```python
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
import asyncio

# Инициализируем модель. gpt-4o-mini - хороший баланс цены и качества
# для задач decomposition и synthesis
llm = ChatOpenAI(model="gpt-4o-mini")

# ШАГ 1: Промпт для разбиения вопроса
# Ключевой момент: просим вернуть ТОЛЬКО подвопросы, без лишнего текста.
# Это важно для парсинга - мы будем split по newline.
decomposition_prompt = ChatPromptTemplate.from_template("""
Ты эксперт в анализе сложных вопросов. Твоя задача - разбить
комплексный вопрос на 2-4 простых подвопроса.

Исходный вопрос: {question}

Правила:
1. Каждый подвопрос должен быть самодостаточным
2. Подвопросы вместе должны полностью покрывать исходный вопрос
3. Избегай дублирования информации между подвопросами

Верни ТОЛЬКО подвопросы, каждый на новой строке, без нумерации.
""")

# ШАГ 2: Промпт для ответа на подвопрос
# Важно: просим краткий ответ, чтобы не раздувать контекст для синтеза
answer_prompt = ChatPromptTemplate.from_template("""
Ответь на вопрос, основываясь ТОЛЬКО на предоставленном контексте.
Если в контексте нет информации - честно скажи об этом.

Вопрос: {sub_question}

Контекст из базы знаний:
{context}

Дай краткий, фактологический ответ (2-3 предложения).
""")

# ШАГ 3: Промпт для синтеза
# Здесь LLM получает все ответы и создает связный итоговый текст
synthesis_prompt = ChatPromptTemplate.from_template("""
Исходный вопрос клиента: {original_question}

Результаты исследования:
{sub_answers}

На основе этих данных сформируй полный, связный ответ на исходный вопрос.
Структурируй ответ логически. Если какие-то аспекты не были найдены в
базе знаний - укажи это.
""")

async def decompose_and_answer(question: str, retriever):
    """
    Полный pipeline Query Decomposition.

    Args:
        question: Сложный вопрос пользователя
        retriever: LangChain retriever для поиска документов

    Returns:
        Синтезированный ответ на основе подвопросов
    """

    # ШАГ 1: Разбиваем вопрос
    # chain = prompt | llm | parser - стандартный паттерн LangChain
    decomposition_chain = decomposition_prompt | llm | StrOutputParser()
    sub_questions_text = await decomposition_chain.ainvoke({"question": question})

    # Парсим результат: убираем пустые строки и пробелы
    sub_questions = [q.strip() for q in sub_questions_text.split("\n") if q.strip()]

    print(f"Разбили на {len(sub_questions)} подвопросов:")
    for i, q in enumerate(sub_questions, 1):
        print(f"  {i}. {q}")

    # ШАГ 2: Отвечаем на каждый подвопрос ПАРАЛЛЕЛЬНО
    # asyncio.gather позволяет запустить все запросы одновременно
    async def answer_subq(sub_q: str) -> str:
        # Ищем релевантные документы для этого конкретного подвопроса
        docs = await retriever.aget_relevant_documents(sub_q)
        context = "\n".join([doc.page_content for doc in docs])

        # Генерируем ответ
        answer_chain = answer_prompt | llm | StrOutputParser()
        answer = await answer_chain.ainvoke({
            "sub_question": sub_q,
            "context": context
        })
        return f"Вопрос: {sub_q}\nОтвет: {answer}"

    # Запускаем все подзапросы параллельно - это критично для latency
    sub_answers = await asyncio.gather(*[answer_subq(q) for q in sub_questions])

    # ШАГ 3: Синтезируем финальный ответ
    synthesis_chain = synthesis_prompt | llm | StrOutputParser()
    final_answer = await synthesis_chain.ainvoke({
        "original_question": question,
        "sub_answers": "\n\n---\n\n".join(sub_answers)
    })

    return final_answer
```

### Multi-Hop Reasoning: особый случай

Иногда подвопросы зависят друг от друга. Пример:

> "Кто CEO компании, которая приобрела Instagram?"

Здесь нельзя параллельно искать ответы. Нужна **цепочка**:
1. Сначала: "Какая компания приобрела Instagram?" -> Facebook (Meta)
2. Потом: "Кто CEO Meta?" -> Mark Zuckerberg

Naive RAG провалится: нет документа, где написано "CEO компании, купившей Instagram - Цукерберг".

```
Multi-Hop пример:

Вопрос: "Кто CEO компании, которая приобрела Instagram?"

HOP 1: "Какая компания приобрела Instagram?"
       -> Retrieve -> Answer: "Facebook (Meta)"

HOP 2: "Кто CEO Facebook/Meta?"  <- используем результат HOP 1
       -> Retrieve -> Answer: "Mark Zuckerberg"

ФИНАЛ: "Mark Zuckerberg является CEO Meta,
        которая приобрела Instagram в 2012 году."
```

### Типичные ошибки в Query Decomposition

| Ошибка | Почему плохо | Как исправить |
|--------|--------------|---------------|
| Слишком много подвопросов (>5) | Увеличивает latency и стоимость, ухудшает синтез | Ограничь до 2-4 подвопросов |
| Дублирующие подвопросы | Избыточная работа, запутанный синтез | Явно укажи в промпте избегать дублирования |
| Слишком абстрактные подвопросы | Retrieval не находит релевантных документов | Проси конкретные, поисковые формулировки |
| Нет обработки пустых результатов | Галлюцинации в ответе | Добавь fallback: "информация не найдена" |

### Когда использовать vs когда нет

**Используй Query Decomposition когда:**
- Вопрос содержит несколько аспектов ("сравни X, Y и Z")
- Нужно собрать информацию из разных источников
- Multi-hop reasoning (ответ на один вопрос нужен для другого)

**Не используй когда:**
- Простой factual вопрос ("Какой у нас адрес офиса?")
- Время критично (добавляет 1-2 LLM вызова)
- Вопрос уже атомарный

## Agentic RAG

### Проблема: когда простого retrieval недостаточно

Представь службу поддержки авиакомпании. Клиент спрашивает:

> "Мой рейс SU1234 завтра, но я вижу на сайте, что он задержан. Какие у меня варианты? Могу ли я перебронировать на сегодня или получить компенсацию?"

Чтобы ответить на этот вопрос, система должна:
1. Проверить статус рейса SU1234 в real-time API
2. Найти политику компании по задержкам
3. Проверить доступность альтернативных рейсов
4. Найти информацию о компенсациях для данного класса билета

Это не просто "найди документ и ответь". Нужен **агент**, который может:
- Планировать последовательность действий
- Использовать разные инструменты (API, базы данных, vector search)
- Проверять качество своих результатов
- Корректировать стратегию на ходу

### Архитектура Agentic RAG

```
┌─────────────────────────────────────────────────────────────────────┐
│                     AGENTIC RAG ARCHITECTURE                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│                         [QUERY]                                     │
│                            |                                        │
│                            v                                        │
│                     ┌───────────────┐                               │
│                     │    PLANNER    │                               │
│                     │  "Как лучше   │                               │
│                     │   ответить?"  │                               │
│                     └───────┬───────┘                               │
│                             |                                       │
│              +--------------+--------------+                        │
│              |              |              |                        │
│              v              v              v                        │
│        ┌──────────┐   ┌──────────┐   ┌──────────┐                  │
│        │ VectorDB │   │  SQL DB  │   │ Web/API  │                  │
│        │ (docs)   │   │ (struct) │   │ (live)   │                  │
│        └────┬─────┘   └────┬─────┘   └────┬─────┘                  │
│             |              |              |                         │
│             +-------+------+------+-------+                         │
│                     |                                               │
│                     v                                               │
│              ┌─────────────────┐                                    │
│              │    EVALUATOR    │<----------+                        │
│              │ "Достаточно ли  │           |                        │
│              │  информации?"   │           |                        │
│              └────────┬────────┘           |                        │
│                       |                    |                        │
│              +--------+--------+           |                        │
│              |                 |           |                        │
│           YES: OK           NO: retry     |                        │
│              |                 |           |                        │
│              v                 v           |                        │
│        ┌──────────┐     ┌───────────┐     |                        │
│        │ GENERATE │     │  REFINER  │-----+                        │
│        │  Answer  │     │ New query │                               │
│        └──────────┘     └───────────┘                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Ключевые паттерны Agentic RAG

**1. Conditional Retrieval: нужен ли вообще поиск?**

Не каждый вопрос требует retrieval:
- "Сколько будет 2+2?" -> Не нужен retrieval
- "Какая погода в Москве?" -> Нужен web search
- "Какая наша политика отпусков?" -> Нужен vector search

Агент сначала классифицирует вопрос и выбирает стратегию.

**2. Router: выбор источника данных**

Разные вопросы требуют разных источников:
- Семантический поиск -> VectorDB
- Структурированные данные ("выручка за Q3") -> SQL
- Актуальные события -> Web search
- Real-time данные -> API

**3. Self-Reflection Loop: самопроверка**

Агент постоянно спрашивает себя:
1. "Нашел ли я релевантные документы?"
2. "Если нет - как переформулировать запрос?"
3. "Ответ получился хорошим?"
4. "Нужно ли искать дополнительную информацию?"

### Почему это работает

1. **Адаптивность**: система подстраивается под конкретный вопрос, а не слепо следует pipeline

2. **Самоисправление**: если первая попытка неудачна, агент пробует другой подход

3. **Мультимодальность**: может комбинировать разные источники данных

4. **Ограниченные циклы**: с max retries система не зависнет в бесконечном поиске

### Реализация с LangGraph

LangGraph - это библиотека для построения stateful агентов. Она позволяет описать граф состояний и переходов.

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List, Annotated
import operator

# ШАГ 1: Определяем структуру состояния агента
# TypedDict дает нам type hints и проверку структуры
class AgentState(TypedDict):
    question: str                                    # Исходный вопрос (может меняться при rewrite)
    documents: Annotated[List[str], operator.add]    # Найденные документы (накапливаются)
    generation: str                                  # Сгенерированный ответ
    retrieval_grade: str                             # Оценка качества retrieval: "good" или "poor"
    answer_grade: str                                # Оценка качества ответа
    retries: int                                     # Счетчик попыток (защита от бесконечного цикла)


# ШАГ 2: Определяем ноды графа
# Каждая нода - это функция, которая принимает состояние и возвращает обновления

def retrieve(state: AgentState) -> AgentState:
    """
    Нода retrieval: ищет документы по текущему вопросу.

    Почему это отдельная нода: позволяет переиспользовать
    после query rewriting без дублирования кода.
    """
    question = state["question"]

    # Используем similarity_search для semantic поиска
    # k=5 - разумный баланс между полнотой и фокусом
    documents = vectorstore.similarity_search(question, k=5)

    return {"documents": [doc.page_content for doc in documents]}


def grade_documents(state: AgentState) -> AgentState:
    """
    Нода оценки: проверяет релевантность каждого документа.

    Это критически важно! Без этой проверки мы можем получить
    красивый, но галлюцинированный ответ на основе нерелевантных документов.
    """
    question = state["question"]
    documents = state["documents"]

    # Промпт для LLM-as-a-judge
    # Просим бинарный ответ для простоты парсинга
    grader_prompt = """
    Вопрос пользователя: {question}

    Документ из базы знаний:
    {document}

    Содержит ли этот документ информацию, полезную для ответа на вопрос?
    Ответь ТОЛЬКО 'yes' или 'no'.
    """

    relevant_docs = []
    for doc in documents:
        response = llm.invoke(grader_prompt.format(
            question=question,
            document=doc
        ))
        if "yes" in response.content.lower():
            relevant_docs.append(doc)

    # Порог: нужно минимум 2 релевантных документа
    # Это эмпирический выбор - можно настраивать
    grade = "good" if len(relevant_docs) >= 2 else "poor"

    print(f"Найдено релевантных документов: {len(relevant_docs)}/{len(documents)}")

    return {"documents": relevant_docs, "retrieval_grade": grade}


def decide_to_generate(state: AgentState) -> str:
    """
    Conditional edge: решает, генерировать ответ или переформулировать запрос.

    Возвращает имя следующей ноды.
    """
    if state["retrieval_grade"] == "good":
        return "generate"
    elif state["retries"] < 3:
        # Не сдаемся сразу - пробуем переформулировать
        return "transform_query"
    else:
        # После 3 попыток генерируем с тем что есть
        # Лучше частичный ответ, чем вечный цикл
        print("Достигнут лимит попыток, генерируем с доступными данными")
        return "generate"


def transform_query(state: AgentState) -> AgentState:
    """
    Нода переформулирования: пытается улучшить запрос для лучшего retrieval.

    Типичные улучшения:
    - Более конкретные термины
    - Синонимы
    - Разбиение на части
    - Удаление "шума"
    """
    question = state["question"]

    rewrite_prompt = """
    Исходный вопрос не дал хороших результатов поиска.

    Исходный вопрос: {question}

    Переформулируй его так, чтобы он лучше подходил для поиска в базе знаний.
    Используй более конкретные термины, возможно синонимы.

    Переформулированный вопрос:
    """

    response = llm.invoke(rewrite_prompt.format(question=question))
    new_question = response.content.strip()

    print(f"Переформулировка: '{question}' -> '{new_question}'")

    return {
        "question": new_question,
        "retries": state["retries"] + 1
    }


def generate(state: AgentState) -> AgentState:
    """
    Нода генерации: создает финальный ответ на основе найденных документов.
    """
    question = state["question"]
    documents = state["documents"]

    # Объединяем документы в контекст
    context = "\n\n---\n\n".join(documents)

    generation_prompt = f"""
    Ответь на вопрос пользователя, основываясь ТОЛЬКО на предоставленном контексте.
    Если информации недостаточно - честно скажи об этом.

    Контекст из базы знаний:
    {context}

    Вопрос: {question}

    Ответ:
    """

    response = llm.invoke(generation_prompt)
    return {"generation": response.content}


# ШАГ 3: Собираем граф
workflow = StateGraph(AgentState)

# Добавляем ноды
workflow.add_node("retrieve", retrieve)
workflow.add_node("grade_documents", grade_documents)
workflow.add_node("transform_query", transform_query)
workflow.add_node("generate", generate)

# Определяем переходы
workflow.set_entry_point("retrieve")                    # Начинаем с retrieval
workflow.add_edge("retrieve", "grade_documents")        # После retrieval - оценка
workflow.add_conditional_edges(                          # После оценки - решение
    "grade_documents",
    decide_to_generate,
    {
        "generate": "generate",
        "transform_query": "transform_query"
    }
)
workflow.add_edge("transform_query", "retrieve")        # После rewrite - снова retrieval
workflow.add_edge("generate", END)                      # После генерации - конец

# Компилируем граф в исполняемое приложение
app = workflow.compile()

# Использование
result = app.invoke({
    "question": "Какие изменения в AI-регулировании в ЕС произошли в 2024?",
    "documents": [],
    "retries": 0
})

print(result["generation"])
```

### Типичные ошибки в Agentic RAG

| Ошибка | Последствия | Решение |
|--------|-------------|---------|
| Нет лимита попыток | Бесконечный цикл, огромные затраты | `retries < 3` с fallback |
| Слишком строгий grading | Отбрасывает полезные документы | Калибровать threshold, добавить partial relevance |
| Нет логирования решений | Невозможно дебажить | Логировать каждое решение агента |
| Жадный retrieval | Слишком много документов, теряется фокус | k=5-10, reranking |
| Нет timeout | Зависание на внешних API | asyncio.timeout или deadline |

### Когда использовать Agentic RAG

**Используй когда:**
- Вопросы требуют multi-hop reasoning
- Нужно комбинировать разные источники данных
- Качество ответа критично (лучше дольше, но точнее)
- Есть ресурсы на дополнительные LLM вызовы

**Не используй когда:**
- Простые FAQ-вопросы
- Latency критична (каждый retry = +2-5 секунд)
- Ограниченный бюджет на LLM
- Данные и так высокого качества

## Self-RAG и Corrective RAG

### Проблема: агент, который не сомневается в себе

Представь студента, который отвечает на экзамене. Обычный студент: прочитал вопрос -> вспомнил что-то похожее -> написал ответ. Если повезло - ответ правильный. Если нет - уверенно написал чушь.

Отличный студент действует иначе:
1. "Нужно ли мне вообще искать информацию, или я точно это знаю?"
2. "То, что я нашел - это действительно релевантно?"
3. "Мой ответ реально поддерживается тем, что я прочитал?"
4. "Насколько мой ответ полезен для спрашивающего?"

Self-RAG и CRAG - это техники, которые заставляют RAG-систему задавать себе эти вопросы.

### Self-RAG: Reflection Tokens

Self-RAG использует специальные "токены рефлексии" - это как внутренний голос, который комментирует каждый шаг.

```
┌─────────────────────────────────────────────────────────────────────┐
│                         SELF-RAG                                    │
│                 Reflection Tokens                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [Retrieve] - Решение о необходимости retrieval:                   │
│   ────────────────────────────────────────────────                  │
│   • "yes"      → нужен retrieval, ищем                              │
│   • "no"       → можем ответить из знаний модели                    │
│   • "continue" → нужно больше контекста                             │
│                                                                     │
│   Пример:                                                           │
│   "Сколько будет 2+2?" → [Retrieve: no] → сразу отвечаем           │
│   "Когда основана ваша компания?" → [Retrieve: yes] → ищем         │
│                                                                     │
│   ─────────────────────────────────────────────────────────────────│
│                                                                     │
│   [ISREL] - Релевантен ли документ?                                 │
│   ────────────────────────────────                                  │
│   • "relevant"   → документ содержит полезную информацию           │
│   • "irrelevant" → не относится к вопросу                           │
│                                                                     │
│   Это фильтр: нерелевантные документы отбрасываются ПЕРЕД генерацией│
│                                                                     │
│   ─────────────────────────────────────────────────────────────────│
│                                                                     │
│   [ISSUP] - Поддерживается ли ответ документами?                    │
│   ────────────────────────────────────────────────                  │
│   • "fully supported"     → все claims подтверждены                 │
│   • "partially supported" → часть информации из документов          │
│   • "no support"          → ответ выдуман, галлюцинация!           │
│                                                                     │
│   Критический момент: если "no support" → перегенерируем           │
│                                                                     │
│   ─────────────────────────────────────────────────────────────────│
│                                                                     │
│   [ISUSE] - Насколько полезен ответ?                                │
│   ────────────────────────────────                                  │
│   • 5 → очень полезен, полностью отвечает                           │
│   • 4 → хорошо, но можно лучше                                      │
│   • 3 → средне, частичный ответ                                     │
│   • 2 → плохо, мало полезной информации                             │
│   • 1 → бесполезен                                                  │
│                                                                     │
│   При низкой оценке → ищем дополнительные документы                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Как работает Self-RAG Flow

```
Вопрос пользователя
        |
        v
+---------------+
| [Retrieve?]   |-------- "no" --------> Генерируем напрямую
+-------+-------+                               |
        | "yes"                                 |
        v                                       |
+---------------+                               |
|   Retrieve    |                               |
|   Documents   |                               |
+-------+-------+                               |
        |                                       |
        v                                       |
+---------------+                               |
|   [ISREL]     |---- "irrelevant" ---> Retrieve more
+-------+-------+                               |
        | "relevant"                            |
        v                                       |
+---------------+<------------------------------+
|   Generate    |
|    Answer     |
+-------+-------+
        |
        v
+---------------+
|   [ISSUP]     |---- "no support" ---> Revise answer
+-------+-------+
        | "supported"
        v
+---------------+
|   [ISUSE]     |---- score < 3 ---> Retrieve more, retry
+-------+-------+
        | score >= 3
        v
   FINAL OUTPUT
```

### Почему Self-RAG работает

1. **On-demand retrieval**: не тратит ресурсы на retrieval для тривиальных вопросов

2. **Фильтрация шума**: нерелевантные документы отсеиваются до генерации, не загрязняют контекст

3. **Антигаллюцинация**: проверка [ISSUP] ловит выдуманные факты

4. **Quality gate**: [ISUSE] не пропускает плохие ответы к пользователю

**Результат**: +40% accuracy на бенчмарках по сравнению с обычным RAG.

### Corrective RAG (CRAG): фокус на качестве retrieval

CRAG - это более простая техника, которая фокусируется на одной проблеме: **что делать, если retrieval вернул плохие результаты?**

```
┌─────────────────────────────────────────────────────────────────────┐
│                       CORRECTIVE RAG (CRAG)                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Query                                                             │
│     |                                                               │
│     v                                                               │
│   +------------------+                                              │
│   |     Retrieve     |                                              │
│   |    Documents     |                                              │
│   +---------+--------+                                              │
│             |                                                       │
│             v                                                       │
│   +--------------------+                                            │
│   |     EVALUATOR      |                                            │
│   |  (Relevance Score) |                                            │
│   +----------+---------+                                            │
│              |                                                      │
│     +--------+--------+---------+                                   │
│     |                 |         |                                   │
│     v                 v         v                                   │
│   CORRECT          AMBIGUOUS   INCORRECT                            │
│  (score > 0.7)    (0.3-0.7)   (score < 0.3)                        │
│     |                 |         |                                   │
│     |                 |         v                                   │
│     |                 |   +-----------+                             │
│     |                 |   | Web Search|                             │
│     |                 |   | for more  |                             │
│     |                 |   | context   |                             │
│     |                 |   +-----+-----+                             │
│     |                 |         |                                   │
│     |                 v         |                                   │
│     |          +-------------+  |                                   │
│     |          | Knowledge   |<-+                                   │
│     |          | Refinement  |                                      │
│     |          +------+------+                                      │
│     |                 |                                             │
│     +--------+--------+                                             │
│              |                                                      │
│              v                                                      │
│        +-----------+                                                │
│        |  GENERATE |                                                │
│        |   Answer  |                                                │
│        +-----------+                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Логика CRAG

1. **Retrieval**: стандартный поиск документов

2. **Evaluation**: LLM оценивает релевантность каждого документа по шкале 0-1

3. **Три пути в зависимости от оценки**:
   - **CORRECT (>0.7)**: документы хорошие, генерируем ответ
   - **INCORRECT (<0.3)**: документы плохие, идем в web search
   - **AMBIGUOUS (0.3-0.7)**: частично релевантно, комбинируем с web search

4. **Knowledge Refinement**: очищаем и структурируем найденную информацию

5. **Generation**: генерируем ответ на основе отфильтрованных данных

### Сравнение Self-RAG и CRAG

| Аспект | Self-RAG | CRAG |
|--------|----------|------|
| **Когда оценивает** | Во время генерации | До генерации |
| **Требования** | Fine-tuned модель с токенами | Любая LLM + отдельный evaluator |
| **Retrieval** | On-demand (может пропустить) | Всегда, с fallback на web |
| **Сложность** | Высокая (спец. модель) | Средняя (проще интегрировать) |
| **Когда использовать** | Разная retrieval need | Критичные к качеству системы |
| **Latency** | Может быть ниже (skip retrieval) | Стабильная |

### Практический выбор

**Self-RAG подходит когда:**
- У вас есть возможность fine-tune модель
- Много вопросов не требуют retrieval
- Важна минимизация затрат (меньше retrieval)

**CRAG подходит когда:**
- Нужно быстро внедрить в существующую систему
- Есть web search как fallback
- Все вопросы требуют retrieval из базы

## Contextual Retrieval

### Проблема: потеря контекста при chunking

Представь, что ты читаешь финансовый отчет компании. На странице 47 написано:

> "Выручка выросла на 3% по сравнению с предыдущим кварталом."

Эта фраза понятна в контексте документа: ты знаешь, что это отчет Apple за Q2 2024, и что предыдущий квартал имел выручку $94.8 млрд.

Но когда мы режем документ на чанки для RAG, эта фраза становится **сиротой**: какая компания? какой квартал? какая база для сравнения?

### Как работает Contextual Retrieval (Anthropic)

Идея проста и элегантна: **перед embed добавить к каждому чанку объяснение его контекста**.

```
┌─────────────────────────────────────────────────────────────────────┐
│                   CONTEXTUAL RETRIEVAL                              │
│                     (Anthropic, 2024)                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ПРОБЛЕМА:                                                         │
│   ─────────                                                         │
│   Chunk: "Выручка выросла на 3% по сравнению с                     │
│           предыдущим кварталом."                                    │
│                                                                     │
│   Вопрос: Какая компания? Какой период? Это хорошо или плохо?      │
│   Контекст потерян при chunking!                                    │
│                                                                     │
│   ─────────────────────────────────────────────────────────────────│
│                                                                     │
│   РЕШЕНИЕ:                                                          │
│   ────────                                                          │
│                                                                     │
│   Документ + Chunk  ──►  Claude  ──►  Contextual Chunk             │
│                                                                     │
│   Промпт для Claude:                                               │
│   "Вот полный документ: {document}                                 │
│    Вот отдельный чанк из него: {chunk}                             │
│    Дай краткий контекст (1-2 предложения), который поможет         │
│    понять этот чанк в отрыве от документа."                        │
│                                                                     │
│   РЕЗУЛЬТАТ:                                                        │
│   ──────────                                                        │
│   "Этот чанк из SEC-отчета Apple за Q2 2024. В предыдущем          │
│   квартале (Q1 2024) выручка составила $94.8 млрд.                 │
│   Выручка выросла на 3% по сравнению с предыдущим кварталом."      │
│                                                                     │
│   Теперь поиск по "рост выручки Apple" найдет этот чанк!           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Почему это работает

1. **Semantic enrichment**: чанк получает ключевые слова и контекст, которые увеличивают вероятность совпадения при поиске

2. **Disambiguation**: если в корпусе 10 компаний с фразой "выручка выросла", контекст помогает найти нужную

3. **Self-contained chunks**: каждый чанк становится полноценным источником информации

### Результаты (из статьи Anthropic)

```
Failure Rate (Top-20 Chunk Retrieval):

Baseline (только embeddings):      5.7%
+ Contextual Embeddings:           3.7%  (-35%)
+ Contextual Embeddings + BM25:    2.9%  (-49%)
+ Reranking:                       1.9%  (-67%)
```

### Реализация с Prompt Caching

Наивная реализация была бы очень дорогой: для каждого чанка отправлять весь документ. При документе в 50K токенов и 100 чанках это 5M токенов!

Решение - **Prompt Caching**: документ кэшируется при первом запросе, последующие запросы с тем же документом стоят в 10 раз дешевле.

```python
import anthropic

client = anthropic.Anthropic()

def add_context_to_chunks(document: str, chunks: list[str]) -> list[str]:
    """
    Добавляет контекстную информацию к каждому чанку.

    Использует prompt caching для оптимизации стоимости:
    - Первый чанк: полная стоимость (документ кэшируется)
    - Последующие чанки: ~90% экономия (документ из кэша)

    Args:
        document: Полный текст документа
        chunks: Список чанков документа

    Returns:
        Список чанков с добавленным контекстом
    """

    contextualized = []

    for i, chunk in enumerate(chunks):
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=200,  # Контекст должен быть кратким
            messages=[
                {
                    "role": "user",
                    "content": [
                        {
                            # ВАЖНО: документ идет первым и кэшируется
                            "type": "text",
                            "text": f"<document>\n{document}\n</document>",
                            "cache_control": {"type": "ephemeral"}  # Включаем кэширование
                        },
                        {
                            # Чанк идет вторым - он меняется для каждого запроса
                            "type": "text",
                            "text": f"""
<chunk>
{chunk}
</chunk>

Дай краткий контекст (1-2 предложения), который объясняет место этого
чанка в общем документе. Включи: название документа/источника,
временной период (если есть), ключевые сущности (компании, люди, продукты).

Ответь ТОЛЬКО контекстом, без пояснений.
"""
                        }
                    ]
                }
            ]
        )

        context = response.content[0].text

        # Объединяем контекст и оригинальный чанк
        contextualized_chunk = f"{context}\n\n{chunk}"
        contextualized.append(contextualized_chunk)

        # Опционально: логируем прогресс
        if (i + 1) % 10 == 0:
            print(f"Обработано чанков: {i + 1}/{len(chunks)}")

    return contextualized


# Пример использования
document = """
Apple Inc. Q2 2024 Financial Report
...
[текст документа]
...
"""

chunks = [
    "Выручка выросла на 3% по сравнению с предыдущим кварталом.",
    "Расходы на R&D увеличились на 15%.",
    # ... остальные чанки
]

contextualized_chunks = add_context_to_chunks(document, chunks)

# Теперь можно embed и загрузить в vector store
for chunk in contextualized_chunks:
    embedding = embed(chunk)
    vector_store.add(embedding, chunk)
```

### Late Chunking vs Contextual Retrieval

Есть альтернативный подход от Jina AI - **Late Chunking**:

```
┌────────────────────────────────────────────────────────────────────┐
│          LATE CHUNKING VS CONTEXTUAL RETRIEVAL                     │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│   LATE CHUNKING (Jina AI):                                         │
│   ──────────────────────                                           │
│   1. Весь документ → Embedding Model (long-context)                │
│   2. Transformer выдает токен embeddings для всего документа       │
│   3. Chunking ПОСЛЕ transformer layers                              │
│   4. Mean pooling на уровне каждого чанка                          │
│                                                                    │
│   Плюсы:                                                           │
│   + Вычислительно эффективнее (1 проход)                           │
│   + Нет дополнительных LLM вызовов                                  │
│                                                                    │
│   Минусы:                                                          │
│   - Требует long-context embedding model                            │
│   - Менее интерпретируемо                                          │
│                                                                    │
│   ─────────────────────────────────────────────────────────────────│
│                                                                    │
│   CONTEXTUAL RETRIEVAL (Anthropic):                                │
│   ─────────────────────────────────                                │
│   1. Chunking документа                                            │
│   2. LLM генерирует контекст для каждого чанка                     │
│   3. Контекст + чанк → любая embedding model                       │
│                                                                    │
│   Плюсы:                                                           │
│   + Работает с любой embedding model                                │
│   + Контекст читаемый, можно дебажить                               │
│   + С prompt caching - экономично                                   │
│                                                                    │
│   Минусы:                                                          │
│   - Требует LLM вызовы при индексации                               │
│   - Дороже без caching                                              │
│                                                                    │
│   ─────────────────────────────────────────────────────────────────│
│                                                                    │
│   РЕКОМЕНДАЦИЯ:                                                    │
│   ─────────────                                                    │
│   • < 200K токенов → Включи весь документ в prompt                 │
│   • Production + бюджет → Contextual Retrieval + caching           │
│   • Self-hosted → Late Chunking с long-context model               │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### Типичные ошибки в Contextual Retrieval

| Ошибка | Последствия | Решение |
|--------|-------------|---------|
| Слишком длинный контекст | Увеличивает размер embeddings, размывает семантику | Ограничь до 1-2 предложений |
| Нет prompt caching | Огромные затраты на индексацию | Всегда включай cache_control |
| Контекст повторяет чанк | Не добавляет ценности | Явно проси "дополнительную" информацию |
| Нет пакетной обработки | Медленная индексация | Параллельные запросы (осторожно с rate limits) |

## GraphRAG

### Проблема: вопросы о "лесе", а не о "деревьях"

Представь, что у тебя есть 10,000 новостных статей о технологической индустрии. Пользователь спрашивает:

> "Какие основные темы и тренды обсуждаются в этих статьях?"

Naive RAG полностью провалится:
- Vector search ищет похожие чанки к запросу
- Нет конкретного чанка про "основные темы"
- Система вернет случайные статьи

Это как спросить "какие темы в библиотеке" и получить в ответ случайную страницу из случайной книги.

### Как работает GraphRAG (Microsoft)

GraphRAG строит **граф знаний** из всего корпуса документов:

1. **Entity Extraction**: LLM извлекает сущности и их отношения из каждого документа

2. **Knowledge Graph**: сущности становятся нодами, отношения - ребрами графа

3. **Community Detection**: алгоритм Leiden находит "сообщества" связанных сущностей

4. **Community Summaries**: LLM создает описание каждого сообщества

### Почему GraphRAG работает для thematic вопросов

1. **Pre-computed aggregation**: Community summaries уже содержат агрегированную информацию. Вопрос о "темах" находит готовый ответ в summaries.

2. **Структура вместо семантики**: вместо поиска "похожего текста" мы используем реальные связи между концептами.

3. **Иерархия**: communities организованы иерархически - от крупных тем до мелких подтем.

4. **Completeness**: Map-Reduce по всем communities гарантирует, что мы не пропустим важную тему.

### Когда использовать GraphRAG

| Сценарий | GraphRAG | Обычный RAG |
|----------|----------|-------------|
| Поиск конкретного факта | O | X (лучше) |
| "Какие темы в документах?" | X (лучше) | O |
| Связи между сущностями | X (лучше) | O |
| Multi-hop reasoning | X (лучше) | O |
| Simple QA | O | X (лучше) |
| Real-time обновления | O (дорого) | X (лучше) |

### Цена GraphRAG

GraphRAG требует значительных ресурсов:

- **Индексация**: ~$5-10 на 1000 документов (entity extraction + summarization)
- **Время**: индексация может занять часы для больших корпусов
- **Обновления**: при изменении документов нужно перестраивать граф

Это оправдано когда:
- Корпус относительно статичен
- Много вопросов о темах и трендах
- Multi-hop reasoning критичен
- Есть бюджет на индексацию

## Оценка RAG (RAGAS)

### Проблема: как понять, что RAG работает хорошо?

Ты построил RAG-систему. Она отвечает на вопросы. Но как понять, хорошо ли она работает?

Нельзя же проверять каждый ответ вручную. А традиционные метрики NLP (BLEU, ROUGE) плохо работают для генеративных задач - два разных текста могут быть одинаково правильными.

### RAGAS: LLM-as-a-Judge

RAGAS (Retrieval Augmented Generation Assessment) использует LLM для оценки качества RAG. Это работает потому что LLM хорошо понимает семантику и может судить о качестве ответов.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        RAGAS METRICS                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   RETRIEVAL METRICS (качество поиска):                              │
│   ────────────────────────────────────                              │
│                                                                     │
│   Context Precision                                                │
│   • Вопрос: Какая доля найденных документов реально релевантна?    │
│   • High score = мало "мусора" в контексте                         │
│   • Формула: relevant_docs / total_retrieved_docs                   │
│                                                                     │
│   Context Recall                                                   │
│   • Вопрос: Нашли ли мы ВСЮ нужную информацию?                     │
│   • High score = не пропустили важные документы                    │
│   • Требует ground truth (какие документы ДОЛЖНЫ быть найдены)     │
│                                                                     │
│   ─────────────────────────────────────────────────────────────────│
│                                                                     │
│   GENERATION METRICS (качество ответа):                             │
│   ──────────────────────────────────────                            │
│                                                                     │
│   Faithfulness (Верность)                                          │
│   • Вопрос: Все ли утверждения в ответе подтверждены контекстом?   │
│   • High score = нет галлюцинаций                                  │
│   • Это ГЛАВНАЯ метрика для production RAG                          │
│   • LLM разбивает ответ на claims и проверяет каждый               │
│                                                                     │
│   Answer Relevancy (Релевантность ответа)                          │
│   • Вопрос: Ответ реально отвечает на заданный вопрос?             │
│   • High score = ответ по теме, не уходит в сторону                │
│   • LLM генерирует вопросы из ответа и сравнивает с исходным       │
│                                                                     │
│   Answer Correctness (Правильность)                                │
│   • Вопрос: Ответ фактически правильный?                           │
│   • Требует ground truth ответа                                    │
│   • Комбинация semantic similarity + factual overlap               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### RAG Triad: три измерения качества

```
                        ┌───────────┐
                        │  ВОПРОС   │
                        └─────┬─────┘
                              │
             Context          │         Answer
             Relevance        │         Relevancy
                ↓             │            ↓
         ┌───────────┐        │      ┌───────────┐
         │  КОНТЕКСТ │◄───────┴─────►│   ОТВЕТ   │
         └─────┬─────┘               └─────┬─────┘
               │                           │
               └───────────┬───────────────┘
                           │
                     Groundedness
                     (Faithfulness)
```

Три ключевых измерения:

1. **Context Relevance**: Контекст релевантен вопросу?
2. **Answer Relevancy**: Ответ релевантен вопросу?
3. **Groundedness**: Ответ основан на контексте (не выдуман)?

Хороший RAG оптимизирует все три.

### Практическое использование RAGAS

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset

# ШАГ 1: Подготовка данных для оценки
# Нужны: вопросы, ответы RAG, контексты, и (опционально) ground truth

data = {
    "question": [
        "Что такое machine learning?",
        "Кто изобрел трансформеры?"
    ],
    "answer": [
        "Machine learning - это подраздел AI, который позволяет системам учиться на данных.",
        "Архитектура трансформеров была представлена в статье 'Attention is All You Need'."
    ],
    "contexts": [
        ["Machine learning (ML) - это тип искусственного интеллекта, который позволяет..."],
        ["Трансформер был представлен в 2017 году Vaswani et al. в статье..."]
    ],
    # Ground truth нужен только для некоторых метрик (recall, correctness)
    "ground_truth": [
        "Machine learning - это область AI, где системы учатся на данных.",
        "Трансформеры были изобретены Vaswani et al. в Google."
    ]
}

dataset = Dataset.from_dict(data)

# ШАГ 2: Запуск оценки
# RAGAS использует LLM (по умолчанию OpenAI) для оценки

result = evaluate(
    dataset,
    metrics=[
        faithfulness,       # Главная метрика: нет ли галлюцинаций?
        answer_relevancy,   # Ответ по теме?
        context_precision,  # Контекст без мусора?
        context_recall,     # Весь нужный контекст найден?
    ],
)

# ШАГ 3: Анализ результатов
print(result)
# {'faithfulness': 0.92, 'answer_relevancy': 0.88,
#  'context_precision': 0.85, 'context_recall': 0.90}

# Интерпретация:
# - faithfulness 0.92: 8% ответов содержат непроверенные утверждения
# - answer_relevancy 0.88: ответы в целом по теме, но есть уходы в сторону
# - context_precision 0.85: 15% найденных документов - нерелевантный шум
# - context_recall 0.90: 10% нужной информации не находится
```

### Целевые метрики для production

| Метрика | Target | Почему важно |
|---------|--------|--------------|
| Faithfulness | > 0.9 | Главное: нет галлюцинаций |
| Context Precision | > 0.8 | Меньше шума в контексте |
| Answer Relevancy | > 0.85 | Ответы по теме |
| Latency (p95) | < 3s | UX |
| Hallucination rate | < 5% | Доверие пользователей |

### Как улучшать метрики

```
Проблема                    Метрика                Решение
──────────────────────────────────────────────────────────────
Низкая faithfulness      → Reranking, фильтрация контекста
                         → Улучшить prompt ("only use context")
                         → Self-RAG / CRAG

Низкая context precision → Reranking
                         → Уменьшить k в retrieval
                         → Улучшить chunking

Низкая context recall    → Увеличить k
                         → Hybrid search (vector + BM25)
                         → Query expansion

Низкая answer relevancy  → Улучшить system prompt
                         → Chain-of-thought
                         → Query decomposition
```

## Типичные ошибки и как их избежать

### Ошибка 1: "Больше документов = лучше"

**Проблема**: Разработчик ставит k=50 для retrieval, надеясь "точно найти нужное".

**Почему плохо**:
- LLM теряется в длинном контексте ("lost in the middle")
- Нерелевантные документы вносят шум
- Увеличивается latency и стоимость
- Context precision падает

**Решение**: k=5-10 с quality > quantity. Лучше reranking, чем много документов.

### Ошибка 2: Игнорирование hybrid search

**Проблема**: Используют только vector search.

**Почему плохо**:
- Vector search плохо работает с точными терминами (product codes, IDs)
- Пропускает документы с точным keyword match
- Теряет edge cases

**Решение**: Всегда комбинируй vector + BM25 (keyword). Большинство production систем используют hybrid.

### Ошибка 3: Chunking без overlap

**Проблема**: Режут документы на чанки без перекрытия.

**Почему плохо**:
- Информация на границе чанков теряется
- "Выручка компании составила" в одном чанке, "500 миллионов" в следующем
- Retrieval может найти только половину ответа

**Решение**: 10-20% overlap между чанками. Или semantic chunking.

### Ошибка 4: Нет fallback стратегии

**Проблема**: Если retrieval не нашел ничего релевантного, система все равно генерирует ответ.

**Почему плохо**:
- Гарантированные галлюцинации
- Уверенно неправильные ответы хуже, чем честное "не знаю"
- Подрывает доверие пользователей

**Решение**:
- Проверять relevance score найденных документов
- Если ниже threshold: "Я не нашел информации по этому вопросу в базе знаний"
- Опционально: fallback на web search или escalation

### Ошибка 5: Игнорирование evaluation

**Проблема**: Построили RAG, вручную проверили 10 вопросов, задеплоили.

**Почему плохо**:
- Нет baseline для сравнения изменений
- Регрессии остаются незамеченными
- Нет понимания реального качества

**Решение**:
- RAGAS evaluation в CI/CD
- Тестовый набор 50-100 вопросов с ground truth
- Мониторинг метрик в production

### Ошибка 6: Один prompt для всех вопросов

**Проблема**: Используют один и тот же prompt независимо от типа вопроса.

**Почему плохо**:
- Factual вопросы и conversational требуют разного подхода
- Comparison вопросы нужно обрабатывать иначе
- Теряется качество на edge cases

**Решение**:
- Классификация типа вопроса
- Разные prompt templates для разных типов
- Routing на разные pipelines

### Ошибка 7: Нет обработки длинного контекста

**Проблема**: Найденные документы превышают context window модели.

**Почему плохо**:
- Truncation теряет информацию
- Ошибки API
- Непредсказуемое поведение

**Решение**:
- Лимит на суммарный размер контекста
- Reranking + take top-N что влезает
- Summarization если контекст слишком большой

### Ошибка 8: Sync вместо async

**Проблема**: Sequential обработка запросов.

**Почему плохо**:
- Latency = sum всех шагов
- Плохо масштабируется
- Пользователи ждут

**Решение**:
- asyncio для I/O-bound операций
- Parallel retrieval из разных источников
- Streaming ответов

Продвинутые RAG-техники, такие как multi-index routing или hierarchical retrieval, требуют понимания индексов (HNSW, IVF) и особенностей конкретных векторных БД.

## Проверь себя

> [!question]- Почему Naive RAG (embed-retrieve-generate) недостаточен для production и какие три проблемы он не решает?
> 1) Не обрабатывает multi-aspect запросы (нужен query decomposition). 2) Не проверяет качество retrieved документов (нет self-correction). 3) Не отвечает на thematic вопросы ("Какие темы обсуждаются?") -- vector search ищет needle, а не haystack. Решения: Agentic RAG, Self-RAG, GraphRAG.

> [!question]- Вы строите RAG-систему для юридических документов, где точность критична. Какие продвинутые техники примените?
> Contextual Retrieval (Anthropic) для добавления контекста к chunks (-49% ошибок), hybrid search (vector + BM25), reranking для финального ранжирования (-67% ошибок с reranking), Self-RAG с reflection tokens для самопроверки ответа на faithfulness, и evaluation через RAGAS metrics.

> [!question]- В чем принципиальное отличие GraphRAG от стандартного vector-based RAG?
> Vector RAG ищет конкретные документы по similarity (needle in haystack). GraphRAG строит knowledge graph (entities + relationships), выполняет community detection и создает community summaries. Это позволяет отвечать на thematic вопросы ("Какие основные темы?") с +76% accuracy на reasoning задачах.

> [!question]- Как Self-RAG достигает +40% accuracy по сравнению с обычным RAG?
> Self-RAG обучает модель генерировать специальные reflection tokens: [Retrieve] (нужен ли поиск?), [ISREL] (релевантен ли документ?), [ISSUP] (подтвержден ли ответ?), [ISUSE] (полезен ли ответ?). Модель сама решает когда искать и проверяет качество каждого шага, отфильтровывая нерелевантные документы.

## Ключевые карточки

Какие три поколения RAG существуют?
?
Naive RAG (2020): embed + retrieve + generate. Advanced RAG (2023): + reranking, hybrid search, better chunking. Agentic RAG (2024-2025): + planning, self-correction, multi-source retrieval, adaptive strategies.

Что такое RAGAS и какие метрики оценивают RAG?
?
RAGAS -- фреймворк оценки RAG. Основные метрики: Faithfulness (ответ основан на контексте), Context Relevance (найденные документы релевантны), Answer Relevance (ответ отвечает на вопрос), Context Precision (доля полезных документов среди найденных).

Что такое Contextual Retrieval (Anthropic)?
?
Техника добавления контекста к каждому chunk перед embedding. LLM генерирует краткое описание: "Этот chunk из отчета Apple Q2 2024, раздел о доходах". Уменьшает failed retrievals на 49%, с reranking -- на 67%.

Что такое Query Decomposition?
?
Разбиение сложного multi-aspect запроса на простые подзапросы. Пример: "Сравните производительность React и Vue для enterprise" -> подзапросы по performance, ecosystem, scalability. Каждый подзапрос обрабатывается отдельно, результаты агрегируются.

Agentic RAG vs обычный RAG?
?
Agentic RAG использует AI-агента с planning, self-correction и multi-source retrieval. Агент решает: какие источники искать, нужен ли дополнительный поиск, корректен ли ответ. Результат: 60-75% снижение затрат, 90% cache hit rates, адаптивность к сложным запросам.
