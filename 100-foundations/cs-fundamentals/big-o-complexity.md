# Big O Notation: как понять скорость алгоритмов

> **В одном предложении:** Big O — это способ предсказать, как сильно замедлится программа, когда данных станет больше.

---

## Теоретические основы: формальный базис анализа сложности

### Big O: формальное определение

> **Определение (Bachmann, 1894; Landau, 1909):** f(n) = O(g(n)) ⟺ ∃ c > 0, n₀ > 0: ∀ n ≥ n₀: f(n) ≤ c · g(n)

Говоря простым языком: f растёт **не быстрее** g с точностью до константы. Нотация введена Паулем Бахманом (1894) и популяризирована Эдмундом Ландау (1909), поэтому называется "символ Ландау".

### Семейство асимптотических нотаций

| Нотация | Определение | Аналогия |
|---------|-------------|----------|
| **O(g)** | f ≤ c·g (верхняя граница) | "не хуже чем" |
| **Ω(g)** | f ≥ c·g (нижняя граница) | "не лучше чем" |
| **Θ(g)** | c₁·g ≤ f ≤ c₂·g (точная) | "ровно" |
| **o(g)** | f/g → 0 (строго меньше) | "строго лучше" |
| **ω(g)** | f/g → ∞ (строго больше) | "строго хуже" |

**Распространённая ошибка:** Говорить "O(n²)" когда имеют в виду "Θ(n²)". O — верхняя граница: алгоритм O(n²) может быть и O(n) в лучшем случае.

### Применение в CS: Donald Knuth

> **Атрибуция:** В computer science асимптотическая нотация популяризирована **Donald Knuth** (1976, "Big Omicron and Big Omega and Big Theta") для анализа алгоритмов. Knuth предложил использовать все три нотации (O, Ω, Θ) для полного описания поведения алгоритма.

### Классы сложности

```
O(1) ⊂ O(log n) ⊂ O(√n) ⊂ O(n) ⊂ O(n log n) ⊂ O(n²) ⊂ O(n³) ⊂ O(2ⁿ) ⊂ O(n!)
const  logarithm  square    linear  linearithmic  quadratic cubic    exponential factorial
                   root
```

## Зачем это нужно (Проблема → Решение)

**Представь ситуацию:** У тебя есть два алгоритма поиска. Оба работают за 1 миллисекунду на 100 элементах. Какой выбрать?

Без Big O ты скажешь: "Оба одинаковые". И ошибёшься.

С Big O ты увидишь:
- Алгоритм A (O(n)): на миллионе элементов = 10 секунд
- Алгоритм B (O(log n)): на миллионе элементов = 0.003 секунды

**Разница в 3333 раза.** Big O помогает предсказать это заранее.

**На интервью Big O спрашивают всегда.** После решения задачи интервьюер скажет: "Какая сложность по времени? А по памяти?" Если не знаешь — это минус.

## УРОВЕНЬ 1: Объяснение для бабушки (без кода)

### Что такое Big O простыми словами

**Big O отвечает на вопрос:** "Если данных станет в 10 раз больше, во сколько раз замедлится программа?"

Это как спросить: "Если гостей на вечеринке станет в 10 раз больше, сколько времени потребуется, чтобы всех обслужить?"

### Аналогия 1: Драйв-тру vs Доставка пиццы

**O(1) — Константное время (Почтовый ящик)**
```
Представь, что ты проверяешь почтовый ящик.
Неважно, сколько домов в городе —
ты идёшь к СВОЕМУ ящику и проверяешь его.

Время не зависит от количества домов.
Это O(1) — константа.
```

**O(n) — Линейное время (Драйв-тру)**
```
Ты в очереди драйв-тру McDonald's.
Перед тобой 5 машин — ждёшь 5 минут.
Перед тобой 50 машин — ждёшь 50 минут.

Время ожидания растёт ПРОПОРЦИОНАЛЬНО количеству машин.
Это O(n) — линейная зависимость.
```

**O(log n) — Логарифмическое время (Телефонная книга)**
```
Ты ищешь номер "Иванов" в телефонной книге.
НЕ листаешь с первой страницы!

1. Открываешь посередине → видишь "М" → слишком далеко
2. Открываешь левую половину → видишь "Е" → близко!
3. Открываешь правую часть от "Е" → нашёл "И"!

В книге 1000 страниц → ~10 открытий
В книге 1,000,000 страниц → ~20 открытий

Каждый раз отбрасываешь ПОЛОВИНУ книги.
Это O(log n) — логарифм.
```

**O(n²) — Квадратичное время (Рукопожатия)**
```
На вечеринке 10 человек.
Каждый должен пожать руку каждому.

10 человек = 45 рукопожатий
100 человек = 4,950 рукопожатий
1000 человек = 499,500 рукопожатий

Удвоил людей → в 4 раза больше рукопожатий!
Это O(n²) — квадратичная зависимость.
```

### Аналогия 2: Список покупок

```
ЗАДАЧА: Найти "молоко" в списке покупок.

СПОСОБ 1 — O(n):
Читаешь весь список сверху вниз:
"хлеб... яйца... сыр... молоко!" — нашёл!
В списке 100 продуктов → проверишь до 100.

СПОСОБ 2 — O(1):
Список отсортирован по категориям.
Ты точно знаешь: молочные — третья полка.
Идёшь сразу туда.
В списке 100 или 10,000 — время одинаковое.
```

### Аналогия 3: Бейджики на конференции

```
ЗАДАЧА: Найти бейджик с именем "Анна".

СПОСОБ 1 — O(n) (плохой):
Все бейджики в одной куче.
Перебираешь по одному.
1000 бейджиков = 1000 проверок (в худшем случае).

СПОСОБ 2 — O(1) (хороший):
Бейджики разложены по коробкам: А, Б, В...
Идёшь к коробке "А" → ищешь "Анна".
Это как хэш-таблица!
1000 или 1,000,000 бейджиков — идёшь к нужной коробке сразу.
```

### Визуальное сравнение скоростей

```
ЕСЛИ У ТЕБЯ 1000 ЭЛЕМЕНТОВ:

O(1)      →  1 операция         🟢 мгновенно
O(log n)  →  10 операций        🟢 мгновенно
O(n)      →  1,000 операций     🟢 быстро
O(n log n)→  10,000 операций    🟡 нормально
O(n²)     →  1,000,000 операций 🔴 медленно (1 сек)
O(2^n)    →  БОЛЬШЕ АТОМОВ ВО ВСЕЛЕННОЙ 💀

ЕСЛИ У ТЕБЯ 1,000,000 ЭЛЕМЕНТОВ:

O(1)      →  1 операция         🟢 мгновенно
O(log n)  →  20 операций        🟢 мгновенно
O(n)      →  1,000,000 операций 🟢 ~1 секунда
O(n log n)→  20,000,000 операций 🟡 ~2-3 секунды
O(n²)     →  10^12 операций     🔴 ~11 ДНЕЙ
O(2^n)    →  невозможно         💀 конец вселенной
```

## УРОВЕНЬ 2: Объяснение для студента (формально, но понятно)

### Формальное определение Big O

**Big O — это математическая нотация** для описания верхней границы роста функции.

> **f(n) = O(g(n))** означает:
> Существуют константы **c > 0** и **k ≥ 0** такие, что:
> **|f(n)| ≤ c × |g(n)|** для всех **n ≥ k**

**Переведём на человеческий:**
- f(n) — реальное время работы алгоритма
- g(n) — функция, с которой сравниваем (n, n², log n, etc.)
- c — какой-то множитель (константа, нас не интересует)
- k — начиная с какого размера данных правило работает

**Интуиция:** "Алгоритм работает НЕ ДОЛЬШЕ, чем g(n)"

### Три нотации: O, Ω, Θ

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   c₂·g(n) ─────────────────────────────────  верхняя граница   │
│                     ╱╲                                          │
│                    ╱  ╲                                         │
│   f(n) ───────────╱────╲───────────────────  наша функция      │
│                  ╱      ╲                                       │
│   c₁·g(n) ─────╱────────╲──────────────────  нижняя граница    │
│               ╱          ╲                                      │
│              k            n                                     │
│                                                                 │
│   Big O (O):     f(n) ≤ c₂·g(n)  — верхняя граница (худший)    │
│   Big Omega (Ω): f(n) ≥ c₁·g(n)  — нижняя граница (лучший)     │
│   Big Theta (Θ): обе границы     — точная оценка               │
└─────────────────────────────────────────────────────────────────┘
```

| Нотация | Что означает | Когда используем | Пример |
|---------|--------------|------------------|--------|
| **O (Big O)** | Не хуже, чем... | Интервью, практика | "Алгоритм O(n²) — в худшем случае" |
| **Ω (Big Omega)** | Не лучше, чем... | Теоретические доказательства | "Сортировка сравнениями минимум Ω(n log n)" |
| **Θ (Big Theta)** | Точно равно... | Точный анализ | "Merge Sort всегда Θ(n log n)" |

**На интервью используют только Big O** — потому что важен worst case.

### Как вычислять Big O: пошаговый алгоритм

```
ШАГ 1: Определи размер входных данных (n)
       Что растёт? Массив? Строка? Граф?

ШАГ 2: Подсчитай операции как функцию от n
       Цикл по массиву = n операций
       Вложенный цикл = n × n = n² операций

ШАГ 3: Найди доминирующий член
       n² + n + 1 → n² доминирует

ШАГ 4: Отбрось константы
       2n → n
       100n → n
       n/2 → n

ШАГ 5: Запиши в Big O нотации
       O(n²)
```

### Правило 1: Отбрасываем константы

```python
# O(2n) → O(n)
def print_twice(arr):
    for x in arr:      # n операций
        print(x)
    for x in arr:      # n операций
        print(x)
# Итого: 2n операций = O(n)

# O(n/2) → O(n)
def print_half(arr):
    for i in range(len(arr) // 2):  # n/2 операций
        print(arr[i])
# Итого: n/2 операций = O(n)
```

**Почему?** При n → ∞ константы становятся несущественными.
- O(2n) при n=1,000,000: 2,000,000 операций
- O(n) при n=1,000,000: 1,000,000 операций
- Разница всего в 2 раза — это ничто по сравнению с O(n²) vs O(n)

### Правило 2: Отбрасываем меньшие члены

```python
# O(n² + n) → O(n²)
def mixed_complexity(arr):
    # Часть 1: O(n²)
    for i in arr:
        for j in arr:
            print(i, j)

    # Часть 2: O(n)
    for x in arr:
        print(x)

# Итого: n² + n операций
# При n = 1000: 1,000,000 + 1,000 = ~1,000,000
# Член n² доминирует → O(n²)
```

### Правило 3: Разные входы — разные переменные

```python
# O(a + b), НЕ O(n)
def print_both(arr1, arr2):
    for x in arr1:    # a операций
        print(x)
    for y in arr2:    # b операций
        print(y)
# Итого: O(a + b)

# O(a × b), НЕ O(n²)
def print_pairs(arr1, arr2):
    for x in arr1:        # a итераций
        for y in arr2:    # b итераций каждая
            print(x, y)
# Итого: O(a × b)
```

### Правило 4: Вложенные циклы умножаются

```python
# O(n × m)
def nested_loops(n, m):
    for i in range(n):     # n итераций
        for j in range(m): # m итераций каждая
            print(i, j)
# Итого: n × m операций

# Если n = m → O(n²)
```

### Примеры анализа по шагам

**Пример 1: Поиск максимума**
```python
def find_max(arr):      # arr имеет n элементов
    max_val = arr[0]    # 1 операция
    for x in arr:       # n итераций
        if x > max_val: # 1 сравнение
            max_val = x # 1 присваивание (иногда)
    return max_val      # 1 операция

# Подсчёт: 1 + n × 2 + 1 = 2n + 2
# Отбрасываем константы: O(n)
```

**Пример 2: Проверка дубликатов (наивный)**
```python
def has_duplicates(arr):    # n элементов
    for i in range(len(arr)):           # n итераций
        for j in range(i+1, len(arr)):  # от n-1 до 0 итераций
            if arr[i] == arr[j]:
                return True
    return False

# Подсчёт: (n-1) + (n-2) + ... + 1 = n(n-1)/2
# Раскрываем: (n² - n) / 2 = n²/2 - n/2
# Отбрасываем константы и меньшие члены: O(n²)
```

**Пример 3: Binary Search**
```python
def binary_search(arr, target):
    left, right = 0, len(arr) - 1

    while left <= right:
        mid = (left + right) // 2

        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1

    return -1

# Анализ: каждая итерация уменьшает пространство поиска ВДВОЕ
# n → n/2 → n/4 → n/8 → ... → 1
# Сколько раз можно поделить n на 2? log₂(n) раз
# Итого: O(log n)
```

### Space Complexity (Сложность по памяти)

**Time Complexity** — сколько ВРЕМЕНИ занимает алгоритм.
**Space Complexity** — сколько ПАМЯТИ занимает алгоритм.

```python
# O(1) space — константная память
def sum_array(arr):
    total = 0  # 1 переменная
    for x in arr:
        total += x
    return total
# Память не зависит от размера arr

# O(n) space — линейная память
def double_array(arr):
    result = []  # новый массив
    for x in arr:
        result.append(x * 2)
    return result
# Создаём массив размером n

# O(n) space — рекурсия
def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)
# Call stack глубиной n
```

**Trade-off время vs память:**
```
Иногда можно ускорить алгоритм, используя больше памяти.

Пример: Two Sum
- O(n²) time, O(1) space — два вложенных цикла
- O(n) time, O(n) space — с хэш-таблицей

На интервью обычно предпочитают O(n) time с O(n) space.
```

## УРОВЕНЬ 3: Для профессионала (advanced topics)

### Amortized Analysis (Амортизированный анализ)

**Проблема:** ArrayList.add() обычно O(1), но иногда O(n) при resize. Какая реальная сложность?

**Ответ:** Амортизированная O(1) — средняя стоимость по серии операций.

```kotlin
class DynamicArray {
    private var data = IntArray(1)
    private var size = 0

    fun add(element: Int) {
        if (size == data.size) {
            // Resize: O(n) — копируем все элементы
            val newData = IntArray(data.size * 2)
            for (i in data.indices) {
                newData[i] = data[i]
            }
            data = newData
        }
        data[size++] = element  // O(1)
    }
}

/*
АНАЛИЗ за n операций add():

Вставок: n операций по O(1) = n
Resize при размерах 1, 2, 4, 8, ..., n
Копирований: 1 + 2 + 4 + ... + n = 2n - 1

Итого: n + 2n = 3n операций
Амортизированная стоимость: 3n / n = 3 = O(1)
*/
```

### Master Theorem

Для рекуррентных соотношений вида:

> **T(n) = aT(n/b) + f(n)**

где:
- a ≥ 1 — количество подзадач
- b > 1 — во сколько раз уменьшается размер
- f(n) — стоимость разделения/слияния

| Случай | Условие | Результат |
|--------|---------|-----------|
| 1 | f(n) = O(n^(log_b(a) - ε)) | T(n) = Θ(n^(log_b(a))) |
| 2 | f(n) = Θ(n^(log_b(a))) | T(n) = Θ(n^(log_b(a)) × log n) |
| 3 | f(n) = Ω(n^(log_b(a) + ε)) | T(n) = Θ(f(n)) |

**Примеры:**

```
MERGE SORT: T(n) = 2T(n/2) + O(n)
a = 2, b = 2, f(n) = n
log₂(2) = 1 → n^1 = n = f(n) → Case 2
T(n) = O(n log n) ✓

BINARY SEARCH: T(n) = T(n/2) + O(1)
a = 1, b = 2, f(n) = 1
log₂(1) = 0 → n^0 = 1 = f(n) → Case 2
T(n) = O(log n) ✓
```

### Подводные камни (Gotchas)

**1. Скрытая сложность в методах:**
```kotlin
// ВЫГЛЯДИТ как O(n), но на самом деле O(n²)
fun badExample(list: MutableList<Int>) {
    for (i in list.indices) {
        list.removeAt(0)  // removeAt(0) = O(n) для ArrayList!
    }
}
// n итераций × O(n) каждая = O(n²)

// РЕШЕНИЕ:
fun goodExample(list: MutableList<Int>) {
    for (i in list.indices.reversed()) {
        list.removeAt(list.lastIndex)  // O(1)
    }
}
```

```kotlin
// O(n²) — создаёт новую строку каждый раз
fun badConcat(n: Int): String {

        result += "a"  // O(i) — копирует всю строку
    }

}
// 1 + 2 + 3 + ... + n = O(n²)

// O(n) — StringBuilder
fun goodConcat(n: Int): String {
    val sb = StringBuilder()
    for (i in 0 until n) {
        sb.append("a")  // O(1) амортизированно
    }
    return sb.toString()
}
```

**3. Не все вложенные циклы O(n²):**
```kotlin
// Выглядит как O(n²), но на самом деле O(n)
fun twoPointers(arr: IntArray): Int {
    var left = 0
    var right = arr.lastIndex
    var count = 0

    while (left < right) {
        if (arr[left] + arr[right] == 10) {
            count++
            left++
            right--
        } else if (arr[left] + arr[right] < 10) {
            left++
        } else {
            right--
        }
    }
    return count
}
// left и right вместе проходят n элементов → O(n)
```

**4. Рекурсия без мемоизации:**
```kotlin
// O(2^n) — экспоненциальная
fun fibBad(n: Int): Long {
    if (n <= 1) return n.toLong()
    return fibBad(n - 1) + fibBad(n - 2)
}
// fib(50) = 2^50 операций ≈ годы вычислений

// O(n) — с мемоизацией
fun fibGood(n: Int, memo: MutableMap<Int, Long> = mutableMapOf()): Long {
    if (n <= 1) return n.toLong()
    return memo.getOrPut(n) { fibGood(n - 1, memo) + fibGood(n - 2, memo) }
}
// fib(50) = 50 операций → мгновенно
```

## Как объяснять Big O на интервью

### Что ожидают интервьюеры

1. **После написания решения** — сразу назвать Time и Space complexity
2. **Объяснить логику** — почему именно такая сложность
3. **Предложить trade-offs** — можно ли улучшить время за счёт памяти?
4. **Сравнить подходы** — если есть несколько решений

### Шаблон ответа

```
"Давайте проанализирую сложность:

TIME COMPLEXITY: O(n)
- Мы проходим по массиву один раз
- Каждая операция внутри цикла — O(1)
- n итераций × O(1) = O(n)

SPACE COMPLEXITY: O(1)
- Используем только константное количество переменных
- Не создаём новых структур данных, зависящих от n

МОЖНО ЛИ УЛУЧШИТЬ:
- Времянная сложность O(n) оптимальна,
  потому что нам нужно проверить каждый элемент хотя бы раз.
- Если бы массив был отсортирован,
  могли бы использовать binary search → O(log n)."
```

### Частые ошибки на интервью

| Ошибка | Почему плохо | Как правильно |
|--------|--------------|---------------|
| Забыть про space complexity | Интервьюер спросит | Всегда называй и time, и space |
| Сказать O(n) для вложенных циклов | Не всегда верно | Анализируй, сколько итераций в сумме |
| Не учесть встроенные методы | sort() = O(n log n) | Помни сложность стандартных операций |
| Назвать O(1) для HashМap без оговорок | Worst case = O(n) | "O(1) в среднем, O(n) в худшем" |

## Мифы и заблуждения

| Миф | Реальность |
|-----|-----------|
| "O(n) всегда быстрее O(n²)" | **Не всегда.** Для малых n константы важны. O(1000000n) медленнее O(n²) при n < 1000. На практике Big O важен для больших данных. |
| "HashMap всегда O(1)" | **Только в среднем.** При плохой хэш-функции или много коллизий — O(n). Worst case HashMap — все ключи в одном bucket. |
| "Два цикла подряд = O(2n) = O(n²)" | **Нет!** O(n) + O(n) = O(2n) = O(n). Вложенные циклы = O(n²). Последовательные = O(n). |
| "Рекурсия всегда медленнее итерации" | **Не всегда.** Рекурсия с мемоизацией = O(n). Проблема — call stack overflow, не скорость. |
| "Big O = точное время выполнения" | **Нет.** Big O описывает только рост. O(n) может быть 1ms или 1 час — зависит от константы. |
| "Нужно всегда оптимизировать до лучшего Big O" | **Нет.** Premature optimization — зло. O(n²) на 100 элементах часто достаточно. Оптимизируй, когда есть проблема. |
| "Space complexity не важна" | **Важна!** На мобильных устройствах и embedded системах память ограничена. Рекурсия O(n) space может вызвать stack overflow. |
| "Все алгоритмы сортировки O(n log n)" | **Не все.** Bubble Sort = O(n²). Counting Sort = O(n+k). Нижняя граница для сортировки сравнениями — O(n log n), но не все используют сравнения. |

## Вопросы для самопроверки

<details>
<summary>Какая сложность у поиска в HashMap?</summary>

**O(1) в среднем случае.** O(n) в худшем случае (все ключи в одном bucket из-за коллизий). На интервью говори: "O(1) амортизированно, O(n) worst case".

</details>

<details>
<summary>Почему Merge Sort всегда O(n log n)?</summary>

Потому что он **всегда** делит массив пополам — log(n) уровней рекурсии. На каждом уровне сливаем n элементов — O(n) на уровень. Итого: O(n) × O(log n) = O(n log n). В отличие от Quick Sort, который зависит от выбора pivot и может деградировать до O(n²).

</details>

<details>
<summary>Почему O(n) + O(n) = O(n), а не O(2n)?</summary>

По правилу отбрасывания констант. O(2n) = O(n). Константы не влияют на асимптотический рост. При n → ∞ разница между n и 2n несущественна по сравнению с разницей между n и n².

</details>

<details>
<summary>Когда O(n²) алгоритм лучше O(n log n)?</summary>

**При малых n.** Insertion Sort (O(n²)) на практике быстрее Merge Sort (O(n log n)) для массивов < 10-50 элементов из-за меньших констант и лучшей cache locality. Поэтому Tim Sort (используется в Python, Java) переключается на Insertion Sort для малых подмассивов.

</details>

## Связь с другими темами

**[[cs-fundamentals-overview]]** --- Big O нотация является фундаментом всего раздела CS Fundamentals. Каждая структура данных и каждый алгоритм оцениваются через призму Big O: массивы дают O(1) доступ по индексу, linked lists --- O(n), hash tables --- амортизированное O(1). Без понимания Big O невозможно осмысленно выбирать между структурами данных и алгоритмами, а на интервью анализ сложности --- обязательная часть обсуждения любого решения.

**[[sorting-algorithms]]** --- алгоритмы сортировки являются классическим полигоном для применения Big O анализа. Сравнение Bubble Sort O(n^2), Merge Sort O(n log n) и Counting Sort O(n + k) наглядно демонстрирует, как разные подходы масштабируются с ростом данных. Доказательство нижней границы сортировки сравнениями Omega(n log n) --- один из ключевых результатов теории сложности, показывающий, что существуют теоретические пределы эффективности.

**[[searching-algorithms]]** --- поиск элемента в данных иллюстрирует драматическую разницу между O(n) линейного поиска и O(log n) бинарного. На миллионе элементов это разница между миллионом и двадцатью операциями. Binary search требует отсортированных данных, что создаёт trade-off: потратить O(n log n) на сортировку один раз, чтобы потом искать за O(log n) многократно. Hash-based поиск даёт O(1), но ценой дополнительной памяти.

**[[problem-solving-framework]]** --- Big O анализ интегрирован в каждый этап решения алгоритмических задач. На этапе "понять задачу" оценка ограничений (n <= 10^5 обычно требует O(n log n) или лучше) определяет допустимые подходы. На этапе "выбрать стратегию" Big O помогает отсеять заведомо медленные решения. На этапе "оптимизировать" именно Big O показывает, есть ли потенциал для улучшения и где находится bottleneck.

## Проверь себя

> [!question]- Почему O(n) + O(n) = O(n), а не O(n^2)?
> Последовательные операции складываются, а не умножаются. Два отдельных прохода по массиву дают O(2n), а константы отбрасываются, оставляя O(n). Умножение возникает только при вложенных циклах.

> [!question]- У тебя два алгоритма: A с O(n^2) и B с O(n log n). При каких условиях алгоритм A может быть быстрее B?
> При малых n (обычно до 10-50 элементов), когда константные факторы A меньше, чем у B. Insertion Sort O(n^2) на практике быстрее Merge Sort O(n log n) для коротких массивов из-за лучшей cache locality и меньших overhead.

> [!question]- Коллега утверждает, что HashMap всегда работает за O(1). Почему это не совсем верно?
> O(1) — это средний (амортизированный) случай. В худшем случае, при плохой хэш-функции или множестве коллизий, все ключи попадают в один bucket, и операции деградируют до O(n). На интервью правильно говорить: "O(1) амортизированно, O(n) worst case".

> [!question]- Как определить, что рекурсивный алгоритм имеет экспоненциальную сложность O(2^n)?
> Если функция вызывает себя дважды на каждом уровне без мемоизации (как наивный Fibonacci), дерево вызовов удваивается с каждым уровнем. Решение — мемоизация, которая снижает сложность до O(n).

> [!question]- Почему при анализе Big O используют разные переменные для разных входов (O(a + b) вместо O(n))?
> Потому что размеры входов могут быть независимыми. Если функция последовательно обходит два массива разных размеров a и b, сложность зависит от обоих. Запись O(n) была бы неточной, скрывая реальную зависимость.

## Ключевые карточки

Что описывает Big O?
?
Верхнюю границу роста функции — как сильно замедлится алгоритм при увеличении входных данных. Описывает worst case поведение.

Чем отличаются O, Omega и Theta?
?
O (Big O) — верхняя граница (не хуже, чем). Omega — нижняя граница (не лучше, чем). Theta — точная оценка (обе границы совпадают). На интервью используют только Big O.

Почему отбрасывают константы в Big O?
?
При n, стремящемся к бесконечности, константы становятся несущественными. Разница между n и 2n — всего в 2 раза, тогда как разница между n и n^2 — в n раз.

Что такое амортизированная сложность?
?
Средняя стоимость операции по серии операций. Например, ArrayList.add() обычно O(1), но при resize O(n). За n операций суммарно 3n шагов, амортизированно O(1).

Что описывает Master Theorem?
?
Решает рекуррентные соотношения вида T(n) = aT(n/b) + f(n) для Divide and Conquer алгоритмов. Позволяет определить сложность без полного раскрытия рекурсии.

Какова сложность Space Complexity рекурсии?
?
O(глубина рекурсии) из-за call stack. Для factorial(n) — O(n), для binary search — O(log n). Глубокая рекурсия может вызвать stack overflow.

Почему string concatenation в цикле даёт O(n^2)?
?
Каждая конкатенация создаёт новую строку, копируя предыдущую. Суммарно: 1 + 2 + 3 + ... + n = O(n^2). Решение — использовать StringBuilder с амортизированным O(1) на append.

Как отличить O(n) от O(n^2) во вложенных циклах?
?
Не все вложенные циклы дают O(n^2). Если указатели вместе проходят n элементов (как в Two Pointers), сложность O(n). Нужно считать суммарное количество итераций.
