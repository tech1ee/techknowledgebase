# Планирование процессов: как ОС распределяет CPU

Планировщик (scheduler) — компонент ядра, решающий какой процесс получит CPU в следующий момент. На одном CPU может выполняться только один процесс, поэтому планировщик создаёт иллюзию параллельного выполнения, быстро переключаясь между процессами. Выбор алгоритма планирования определяет характеристики системы: throughput, latency, fairness.

---

## Теоретические основы

> **Планирование (scheduling)** — задача распределения ограниченного ресурса (CPU) между конкурирующими потребителями (процессами/потоками) с оптимизацией заданных метрик.

### Формальная модель

Задача планирования:
- **Дано**: N процессов с burst times {b₁, ..., bₙ}, приоритетами, deadlines
- **Найти**: последовательность выполнения, оптимизирующую метрику (throughput, turnaround, waiting time)
- **Ограничение**: в каждый момент на одном CPU выполняется ≤ 1 процесс

### Метрики эффективности

| Метрика | Формула | Что оптимизирует |
|---------|---------|-----------------|
| **Throughput** | Число процессов / время | Системная производительность |
| **Turnaround time** | Completion - Arrival | Время от подачи до завершения |
| **Waiting time** | Turnaround - Burst | Время ожидания в очереди |
| **Response time** | First run - Arrival | Интерактивность |
| **Fairness** | Равномерность распределения CPU | Отсутствие starvation |

### Классификация алгоритмов

| Тип | Примеры | Использование |
|-----|---------|---------------|
| **Non-preemptive** | FCFS, SJF | Batch-системы |
| **Preemptive** | Round Robin, MLFQ, CFS | Интерактивные ОС |
| **Real-time** | EDF, Rate Monotonic | Embedded, RT-системы |

**Completely Fair Scheduler (CFS)** — стандартный планировщик Linux (2007, Ingo Molnar), основан на red-black tree. Каждый процесс получает «виртуальное время» (vruntime); CFS выбирает процесс с наименьшим vruntime.

> **См. также**: [[os-processes-threads]] — процессы и потоки, [[os-synchronization]] — синхронизация, [[concurrency-vs-parallelism]] — конкурентность и параллелизм

## TL;DR

> **Что:** Планировщик — "диспетчер аэропорта", решающий какой самолёт (процесс) взлетит следующим с единственной взлётной полосы (CPU).
>
> **Главные алгоритмы:**
> - **FIFO** — кто первый пришёл, тот и работает (простой, но convoy effect)
> - **Round Robin** — каждому по 10ms, потом следующий (fair, но overhead)
> - **SJF** — короткие задачи первыми (оптимальный avg wait, но starvation длинных)
> - **CFS (Linux)** — "справедливый" scheduler на основе vruntime в red-black tree
>
> **Context switch:** 1-3µs прямых затрат + 10-30µs косвенных (TLB/cache miss).
>
> **Ключевое:** Слишком много потоков на 4 ядрах = overhead на переключение > полезная работа.

## Часть 1: Интуиция без кода

### 🛫 Аналогия 1: Диспетчер аэропорта

**Планировщик — это диспетчер аэропорта** с одной взлётной полосой (CPU):

- **Самолёты (процессы)** ждут в очереди на взлёт
- **Взлётная полоса (CPU)** — только один самолёт может взлетать в момент времени
- **Диспетчер (scheduler)** решает кто взлетит следующим

```
Очередь на взлёт:          Взлётная полоса:
┌─────────────────┐        ┌─────────────────────────┐
│ ✈️ Boeing 747   │        │                         │
│ ✈️ Airbus A320  │   →    │    ✈️ ← Сейчас взлетает │
│ ✈️ Cessna       │        │                         │
│ ✈️ Boeing 777   │        └─────────────────────────┘
└─────────────────┘
```

**Разные стратегии диспетчера:**
- **FIFO:** Кто первый подъехал к полосе — тот и взлетает. Простой, но мелкая Cessna ждёт за огромным Boeing.
- **SJF:** Сначала маленькие самолёты (быстро освободят полосу). Эффективно, но Boeing может ждать вечно.
- **Priority:** VIP-рейсы первыми. Быстро для VIP, но обычные рейсы голодают.
- **Round Robin:** Каждый самолёт получает 5 минут на полосе, потом уступает. Справедливо, но странно для авиации!

### 🍽️ Аналогия 2: Официант в ресторане

Представь **одного официанта** (CPU) на **10 столиков** (процессы):

**FIFO-официант:** Обслуживает столик пока полностью не закончит (принял заказ, принёс еду, десерт, счёт). Другие столики ждут.
```
Столик 1: "Я хочу полный ужин из 5 блюд" → 2 часа
Столик 2-10: Ждут... ждут... голодают...

❌ Convoy Effect: Все застряли за одним долгим заказом
```

**Round Robin официант:** Каждому столику по 2 минуты внимания, потом — следующий.
```
Столик 1: принимает заказ (2 мин) → переключение
Столик 2: принимает заказ (2 мин) → переключение
...
Столик 1: относит напитки (2 мин) → переключение

✅ Все получают внимание, но много "ходьбы" (context switch)
```

**Умный (CFS) официант:** Смотрит кто дольше всех ждал и идёт к нему.
```
vruntime = сколько внимания уже получил столик
Официант идёт к столику с минимальным vruntime
✅ Справедливо + эффективно
```

### 🎤 Аналогия 3: Караоке и quantum

**Time quantum** — это как ограничение на караоке: 3 минуты, потом микрофон у следующего.

```
Quantum слишком МАЛЕНЬКИЙ (10ms):
┌──┐┌──┐┌──┐┌──┐┌──┐┌──┐┌──┐┌──┐
│A ││B ││A ││B ││A ││B ││A ││B │
└──┘└──┘└──┘└──┘└──┘└──┘└──┘└──┘
  ^ context switch каждые 10ms
  Слишком много переключений! Overhead 10-30%

Quantum слишком БОЛЬШОЙ (500ms):
┌─────────────────────────┐┌─────────────────────────┐
│           A             ││           B             │
└─────────────────────────┘└─────────────────────────┘
  Process B ждёт 500ms → плохой response time

Quantum оптимальный (10-100ms):
┌──────────────┐┌──────────────┐┌──────────────┐
│      A       ││      B       ││      A       │
└──────────────┘└──────────────┘└──────────────┘
  Баланс между overhead и responsiveness
```

**Правило большого пальца:**
- Interactive systems (UI): quantum = 10-20ms (быстрый отклик)
- Batch systems (серверы): quantum = 100-200ms (меньше overhead)
- Linux CFS: динамический quantum, зависит от нагрузки

### ⚖️ Аналогия 4: SJF как экспресс-касса в супермаркете

**SJF (Shortest Job First)** — как экспресс-касса "до 10 товаров":

```
Обычная касса (FIFO):
┌─────────────────────────────────────────────────┐
│ 🛒(100)  🛒(50)  🛒(3)  🛒(2)  🛒(5)             │
└─────────────────────────────────────────────────┘
3 товара ждут за 150 товарами = долго!

Экспресс-касса (SJF):
┌─────────────────────────────────────────────────┐
│ 🛒(2)  🛒(3)  🛒(5)  🛒(50)  🛒(100)             │
└─────────────────────────────────────────────────┘
Мелкие покупки первыми = минимум общего ожидания

Average waiting time:
FIFO: (0 + 100 + 150 + 153 + 155) / 5 = 111.6
SJF:  (0 + 2 + 5 + 10 + 60) / 5 = 15.4  ← В 7 раз лучше!
```

**Проблема SJF:** Как узнать "размер корзины" процесса заранее? В реальных ОС используют предсказание на основе истории.

### 🌳 Аналогия 5: CFS как красно-чёрное дерево справедливости

**Linux CFS (Completely Fair Scheduler)** хранит процессы в красно-чёрном дереве, отсортированном по `vruntime` (виртуальное время выполнения):

```
          vruntime = сколько CPU времени процесс УЖЕ получил

                    ┌───────────────┐
                    │   vruntime=50  │
                    │   Process C    │
                    └───────┬───────┘
                   ╱                 ╲
          ┌───────────┐         ┌───────────┐
          │ vruntime=20│         │ vruntime=80│
          │ Process A  │         │ Process D  │
          └───────────┘         └───────────┘
              ↑
         LEFTMOST
         (минимум vruntime)
         = следующий на выполнение!

CFS всегда выбирает самый "обделённый" процесс
(leftmost node = минимальный vruntime)
```

**Почему это справедливо:**
- Новый процесс: vruntime = минимум из всех → сразу получит CPU
- Процесс работал долго: vruntime растёт → уступает другим
- I/O-bound процесс: мало использует CPU → низкий vruntime → приоритет

### 🔢 Численная интуиция

| Метрика | Значение | Что это значит |
|---------|----------|----------------|
| Context switch (прямой) | 1-3 µs | Сохранить/восстановить регистры |
| Context switch (косвенный) | 10-30 µs | TLB miss, cache miss после switch |
| Типичный quantum | 10-100 ms | 1000-10000x дольше context switch |
| Linux CFS minimum granularity | 0.75 ms | Минимальный timeslice |
| Linux CFS latency target | 6 ms | Каждый процесс получит CPU за это время |

**Пример: overhead от context switch**
```
4 ядра, 100 потоков, quantum = 10ms, switch = 30µs

Переключений в секунду = 100 threads × (1000ms / 10ms) = 10,000 switches
Overhead = 10,000 × 30µs = 300ms = 30% времени!

Решение: уменьшить потоки до ~4-8 на ядро
4 ядра × 8 = 32 потока
Переключений = 32 × 100 = 3,200 switches
Overhead = 3,200 × 30µs = 96ms ≈ 10%
```

## Часть 2: Почему это сложно

### ❌ Ошибка 1: "Больше потоков = быстрее"

**СИМПТОМ:** 4-ядерный сервер с 1000 потоками работает медленнее, чем с 16.

**ПОЧЕМУ ВОЗНИКАЕТ:** Интуиция "параллельность = скорость" без учёта overhead.

**РЕАЛЬНОСТЬ:**
```
4 ядра, N потоков:

N = 4:   Каждый поток на своём ядре, 0 context switches
         ✅ Идеально для CPU-bound

N = 8:   2 потока на ядро, минимальные switches
         ✅ Хорошо для смешанной нагрузки

N = 100: 25 потоков на ядро
         Context switch каждые quantum / 25 ≈ каждые 4ms
         Много TLB/cache misses
         ⚠️ Overhead растёт

N = 1000: 250 потоков на ядро
          Память: 1000 × 2MB stack = 2GB только на стеки!
          Context switch overhead: 30-50%
          ❌ Система захлебнулась
```

**РЕШЕНИЕ:**
```
Правило: #threads = #cores × (1 + I/O_ratio)

CPU-bound: threads = cores (4 потока на 4 ядрах)
I/O-bound: threads = cores × 2-4 (8-16 потоков на 4 ядрах)
Mixed:     threads = cores × 1.5-2

Лучше: thread pool фиксированного размера
```

---

### ❌ Ошибка 2: Игнорирование типа нагрузки (CPU-bound vs I/O-bound)

**СИМПТОМ:** "Один алгоритм планирования для всех задач"

**ПОЧЕМУ ВОЗНИКАЕТ:** Непонимание разницы между CPU-bound и I/O-bound.

**ПРИМЕР:**
```java
// CPU-bound: компрессия видео
// Использует CPU на 100%, редко делает I/O
void compressVideo() {
    for (frame in video) {
        // Тяжёлые вычисления
        compress(frame);
    }
}

// I/O-bound: веб-сервер
// 99% времени ждёт сеть, 1% — обработка
void handleRequest() {
    data = waitForNetwork();  // 99% времени здесь
    response = process(data);  // 1% CPU
    sendResponse(response);
}
```

**ТАБЛИЦА ВЫБОРА:**

| Тип задачи | Характеристики | Оптимальный scheduler | Настройки |
|------------|----------------|----------------------|-----------|
| CPU-bound | 100% CPU, редко I/O | FIFO или большой quantum | nice +19 (низкий приоритет) |
| I/O-bound | Много I/O wait | Preemptive, малый quantum | nice -20 (высокий приоритет) |
| Interactive | Ждёт ввод пользователя | CFS с низким latency | По умолчанию |
| Real-time | Жёсткие дедлайны | SCHED_FIFO или SCHED_RR | Real-time priority |

### ❌ Ошибка 3: Priority inversion (инверсия приоритетов)

**СИМПТОМ:** Высокоприоритетная задача ждёт низкоприоритетную.

**ПОЧЕМУ ВОЗНИКАЕТ:** Низкоприоритетный процесс держит lock, который нужен высокоприоритетному.

**КЛАССИЧЕСКИЙ ПРИМЕР (Mars Pathfinder, 1997):**
```
Приоритеты: High (метеоданные) > Medium (коммуникация) > Low (сбор данных)

1. Low захватывает mutex для данных
2. High просыпается, хочет данные → ждёт mutex
3. Medium запускается (выше Low, ниже High)
4. Medium работает бесконечно, блокируя Low
5. High никогда не получит mutex → DEADLOCK!

Timeline:
Low:    [====LOCK====|---blocked by Medium---]
Medium: [------------|============работает============]
High:   [            |ЖДЁТ lock...ЖДЁТ...навсегда    ]
                     ^ priority inversion!
```

**РЕШЕНИЕ — Priority Inheritance:**
```
Когда High ждёт lock, который держит Low:
→ Временно повысить приоритет Low до уровня High
→ Low быстро завершит, отдаст lock
→ Вернуть Low исходный приоритет

Low:    [====LOCK(priority boost)====][done]
High:   [                            ][РАБОТАЕТ]
```

### ❌ Ошибка 4: Непонимание starvation (голодание)

**СИМПТОМ:** Некоторые процессы никогда не получают CPU.

**ПОЧЕМУ ВОЗНИКАЕТ:** Алгоритмы с фиксированными приоритетами.

**ПРИМЕР:**
```
Priority scheduler (без aging):

High priority:   ████████████████████████████████████
Medium priority: |waiting...|waiting...|waiting...|
Low priority:    |never runs|never runs|never runs| ← STARVATION!

Постоянный поток high-priority задач → low никогда не запустится
```

**РЕШЕНИЯ:**

1. **Aging (старение):** Чем дольше ждёт процесс, тем выше его приоритет
```
Initial priority = 10
After 1 sec waiting: priority = 11
After 2 sec waiting: priority = 12
...
Eventually: priority > high-priority → получит CPU
```

2. **CFS (Linux):** Автоматически справедлив — процесс с минимальным vruntime всегда получит шанс

3. **Multilevel Feedback Queue:** Процессы перемещаются между очередями на основе поведения

---

### ❌ Ошибка 5: Неправильный quantum для real-time задач

**СИМПТОМ:** Real-time задача пропускает дедлайны.

**ПОЧЕМУ ВОЗНИКАЕТ:** Использование обычного scheduler для задач с жёсткими временными требованиями.

**ПРИМЕР:**
```
Задача: обработать аудио каждые 10ms (иначе заикание)

Round Robin с quantum = 100ms:
┌───────────────────────────────────────────────────────────────┐
│                      Process A                                 │
└───────────────────────────────────────────────────────────────┘
                                            ^ Audio задача
                                              пропустила 9 дедлайнов!
```

**РЕШЕНИЕ — Linux Real-Time Scheduler:**
```bash
# SCHED_FIFO: run until voluntarily yield
chrt -f 99 ./audio_processor

# SCHED_RR: round-robin среди real-time задач
chrt -r 50 ./video_decoder

# SCHED_DEADLINE: для задач с явными дедлайнами
# runtime=5ms, deadline=10ms, period=10ms
chrt -d --sched-runtime 5000000 --sched-deadline 10000000 --sched-period 10000000 ./task
```

---

### ❌ Ошибка 6: Convoy Effect в I/O-bound системах

**СИМПТОМ:** Все I/O-bound процессы ждут за одним CPU-bound.

**ПОЧЕМУ ВОЗНИКАЕТ:** FIFO scheduling без preemption.

**ПРИМЕР:**
```
FIFO scheduler:

CPU-bound (компиляция):  [████████████████████████████████████████]
I/O-bound (web server):  |waiting...|waiting...|waiting...|waiting|
I/O-bound (database):    |waiting...|waiting...|waiting...|waiting|
I/O-bound (logging):     |waiting...|waiting...|waiting...|waiting|

CPU-bound блокирует всех!
Даже когда I/O-bound нужны всего 1ms CPU.
```

**РЕШЕНИЕ:**
1. Preemptive scheduler (Round Robin, CFS)
2. Повысить приоритет I/O-bound задач (они быстро отдают CPU)
3. Разделить на разные CPU (affinity)

### 🧠 Модель 3: "Gantt chart для визуализации"

```
Процессы: P1(24ms), P2(3ms), P3(3ms)
Arrival time: все в момент 0

FIFO (порядок: P1, P2, P3):
┌────────────────────────────────────────────┬───────┬───────┐
│                    P1 (24ms)               │P2(3ms)│P3(3ms)│
└────────────────────────────────────────────┴───────┴───────┘
0                                            24     27     30

Wait time: P1=0, P2=24, P3=27
Average wait = (0 + 24 + 27) / 3 = 17ms

SJF (порядок: P2, P3, P1):
┌───────┬───────┬────────────────────────────────────────────┐
│P2(3ms)│P3(3ms)│                    P1 (24ms)               │
└───────┴───────┴────────────────────────────────────────────┘
0      3      6                                              30

Wait time: P2=0, P3=3, P1=6
Average wait = (0 + 3 + 6) / 3 = 3ms  ← В 5.7x лучше!
```

### 🧠 Модель 4: "CFS как весы справедливости"

```
CFS балансирует CPU время между процессами

Идеальный мир (бесконечно делимый CPU):
┌─────────────────────────────────────────────────────────────┐
│ P1 ████████████████████████████████████                     │ 33%
│ P2 ████████████████████████████████████                     │ 33%
│ P3 ████████████████████████████████████                     │ 33%
└─────────────────────────────────────────────────────────────┘
Все работают "одновременно", каждый получает 1/3 CPU

Реальный мир (CFS аппроксимация):
┌─────────────────────────────────────────────────────────────┐
│ P1 ████      ████      ████      ████      ████            │
│ P2     ████      ████      ████      ████      ████        │
│ P3         ████      ████      ████      ████      ████    │
└─────────────────────────────────────────────────────────────┘
Быстрое переключение создаёт иллюзию параллельности

vruntime отслеживает "долг" каждого процесса:
- P1 получил больше → vruntime выше → ждёт
- P2 получил меньше → vruntime ниже → следующий на CPU
```

### 🧠 Модель 5: "Multilevel Feedback Queue"

```
MLFQ: Процессы перемещаются между очередями на основе поведения

        Высокий приоритет
        (short quantum)
Queue 0: [Interactive] ←──── Новые процессы начинают здесь
             │
             │ Использовал весь quantum?
             ↓ ДА
Queue 1: [Medium jobs]
             │
             │ Использовал весь quantum?
             ↓ ДА
Queue 2: [CPU-bound]
        Низкий приоритет
        (long quantum)

Правила:
1. Новый процесс → Queue 0 (высший приоритет)
2. Использовал весь quantum → понижение
3. Отдал CPU добровольно (I/O) → остаётся/повышается
4. Периодический boost: все → Queue 0 (против starvation)

Результат:
- Interactive (I/O-bound): остаются в Queue 0, быстрый response
- CPU-bound: опускаются в Queue 2, большой quantum (эффективно)
- Автоматическая адаптация без явных приоритетов!
```

## Терминология для новичков

| Термин | Что это | Аналогия из жизни |
|--------|---------|-------------------|
| **Scheduler** | Компонент ядра, выбирающий кто работает следующим | Диспетчер аэропорта: решает какой самолёт взлетит со взлётной полосы |
| **Context Switch** | Переключение CPU с одного процесса на другой | Переключение внимания между разговорами на вечеринке — надо "вспомнить" контекст |
| **Preemption** | Принудительное прерывание процесса | Учитель отбирает мяч у одного ученика и даёт другому |
| **Quantum / Time Slice** | Максимальное время работы до preemption (обычно 10-100ms) | Ограниченное время на караоке: 3 минуты и микрофон у другого |
| **Throughput** | Количество завершённых задач в единицу времени | Пропускная способность: сколько посетителей обслужил ресторан за вечер |
| **Turnaround Time** | Время от запуска до завершения | Время в ресторане: от входа до выхода (включая ожидание) |
| **Response Time** | Время до первого ответа | Как быстро официант подошёл после того, как вы сели |
| **CPU Burst** | Период непрерывных вычислений | Рывок бегуна между остановками |
| **I/O Burst** | Период ожидания I/O (диск, сеть) | Остановка бегуна — ждёт сигнала светофора |
| **Ready Queue** | Очередь процессов, готовых к работе | Зал ожидания аэропорта — все готовы лететь, ждут своей полосы |
| **Convoy Effect** | Короткие задачи застряли за длинной | Пробка за фурой на узкой дороге |
| **Starvation** | Задача никогда не получает CPU | Посетитель, которого всё время обслуживают последним |
| **vruntime (CFS)** | Виртуальное время выполнения в Linux CFS | "Сколько внимания" процесс уже получил — CFS даёт CPU тем, кто получил меньше |

## Зачем нужен планировщик

### Проблема: один CPU, много процессов

```
Физический CPU: 1 ядро (для простоты)
Процессы: Web server, Database, Logging, Monitoring...

Без планировщика:
  Process A запускается → работает вечно → остальные ждут

С планировщиком:
  10ms: Process A
  10ms: Process B
  10ms: Process C
  10ms: Process A
  ...

Пользователь видит: все процессы работают "одновременно"
```

### Типы процессов

Разные процессы имеют разные паттерны использования CPU:

**CPU-bound:** Много вычислений, мало I/O. Примеры: компиляция, рендеринг, машинное обучение. Хотят максимум CPU time.

**I/O-bound:** Много ожидания I/O, мало вычислений. Примеры: web-сервер (ждёт сеть), database (ждёт диск), GUI (ждёт ввод). Хотят быстрый response time.

```
CPU-bound процесс:
┌────────────────────────────────────────┐
│████████████████████████████████████████│ CPU burst
└────────────────────────────────────────┘

I/O-bound процесс:
┌──┐    ┌──┐    ┌──┐    ┌──┐    ┌──┐
│██│    │██│    │██│    │██│    │██│     CPU bursts
└──┘    └──┘    └──┘    └──┘    └──┘
    ^^^^    ^^^^    ^^^^    ^^^^        I/O wait
```

Хороший планировщик балансирует между CPU-bound и I/O-bound процессами.

## Context Switch: что происходит при переключении

Когда планировщик решает переключить CPU на другой процесс:

```
┌─────────────────────────────────────────────────────────────────┐
│                    CONTEXT SWITCH                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Interrupt/Syscall → CPU переходит в kernel mode             │
│                                                                 │
│  2. Сохранить контекст Process A:                               │
│     - Все регистры CPU (RAX, RBX, RCX, ...)                    │
│     - Program Counter (где остановились)                        │
│     - Stack Pointer                                             │
│     - Флаги процессора                                          │
│     → Сохраняется в PCB (Process Control Block)                 │
│                                                                 │
│  3. Scheduler выбирает следующий процесс (Process B)            │
│                                                                 │
│  4. Загрузить контекст Process B:                               │
│     - Восстановить регистры из PCB                              │
│     - Переключить Page Table (CR3 на x86)                       │
│     - Возможно: TLB flush (дорого!)                             │
│                                                                 │
│  5. Return to user mode → Process B продолжает                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

Время: 1-3 µs (прямые затраты)
      + 10-30 µs (косвенные: TLB miss, cache miss)
```

### Косвенные затраты

Прямые затраты (сохранение/восстановление регистров) — микросекунды. Но косвенные затраты могут быть больше:

**TLB flush:** При смене процесса TLB (кэш трансляции адресов) становится невалидным. Первые обращения к памяти — TLB miss, каждый стоит сотни циклов.

**Cache pollution:** Процесс A заполнил L1/L2/L3 cache своими данными. После switch процесс B видит чужие данные — cache miss.

**Branch predictor pollution:** CPU предсказывает ветвления на основе истории. После switch история от другого процесса — неверные предсказания.

## Когда происходит переключение

### Preemptive vs Non-preemptive

**Non-preemptive (cooperative):** Процесс сам решает отдать CPU. Добровольно вызывает `yield()` или делает syscall. Проблема: зависший процесс блокирует всю систему.

**Preemptive:** ОС может прервать процесс в любой момент. Используется во всех современных ОС.

### Точки переключения

```
1. Процесс завершился (exit)
   → Выбрать следующий из ready queue

2. Процесс заблокировался (I/O, sleep, lock)
   → Переместить в blocked queue, выбрать другой

3. Timer interrupt (preemption)
   → Quantum истёк, процесс принудительно прерывается

4. Высокоприоритетный процесс стал ready
   → Текущий preempted, высокоприоритетный запускается

5. Syscall
   → Процесс в kernel mode, может быть переключение
```

### Timer Interrupt

Аппаратный таймер генерирует прерывания (обычно 100-1000 раз в секунду):

```
Timer interrupt каждые 1ms (1000 Hz):

Process A работает
...
[1ms] Timer interrupt → scheduler → Process A продолжает (quantum не истёк)
...
[1ms] Timer interrupt → scheduler → Process A продолжает
...
[1ms] Timer interrupt → scheduler → Quantum 10ms истёк! → Process B
```

## Алгоритмы планирования

### FIFO (First In, First Out)

Простейший алгоритм: кто первый пришёл, тот первый выполняется до конца.

```
Ready Queue: [A(10ms), B(2ms), C(3ms)]

Выполнение:
|----A(10ms)----|--B(2ms)--|--C(3ms)--|

Turnaround times:
  A: 10ms
  B: 10 + 2 = 12ms
  C: 10 + 2 + 3 = 15ms
  Average: 12.3ms
```

**Проблема — Convoy Effect:** Длинный CPU-bound процесс блокирует короткие I/O-bound процессы. Все ждут, пока "грузовик" (convoy) проедет.

### SJF (Shortest Job First)

Выполнять сначала процессы с наименьшим CPU burst.

```
Ready Queue: [A(10ms), B(2ms), C(3ms)]

Выполнение (SJF):
|--B(2ms)--|--C(3ms)--|----A(10ms)----|

Turnaround times:
  B: 2ms
  C: 2 + 3 = 5ms
  A: 2 + 3 + 10 = 15ms
  Average: 7.3ms (было 12.3ms!)
```

**Проблема:** Как узнать длину следующего CPU burst? В реальности — невозможно. Можно предсказывать на основе истории (exponential averaging).

**Starvation:** Если постоянно приходят короткие процессы, длинные никогда не выполнятся.

### Round Robin (RR)

Каждый процесс получает фиксированный quantum (time slice), затем preemption.

```
Ready Queue: [A, B, C], Quantum = 4ms

|--A(4)--|--B(4)--|--C(4)--|--A(4)--|--B(2)--|--C(3)--|--A(2)--|

A(10ms): 4 + (ждёт 8) + 4 + (ждёт 5) + 2 = выполнен за ~23ms
Но response time отличный — каждый процесс отвечает за 4ms!
```

**Trade-off выбора quantum:**
- Малый quantum (1ms): Отличный response time, но много context switches (overhead)
- Большой quantum (100ms): Мало overhead, но плохой response time
- Типичное значение: 10-100ms

### Priority Scheduling

Каждому процессу присваивается приоритет. Выполняются высокоприоритетные сначала.

```
Priorities: A(low), B(high), C(medium)

Порядок: B → C → A

Проблема — Starvation:
  Высокоприоритетные процессы постоянно приходят
  → Низкоприоритетные никогда не выполняются
```

**Решение — Aging:** Со временем приоритет ожидающего процесса увеличивается. Если процесс ждёт долго, он станет высокоприоритетным.

### Multi-Level Feedback Queue (MLFQ)

Комбинация подходов. Несколько очередей с разными приоритетами:

```
Queue 0 (высший приоритет): Round Robin, quantum = 8ms
Queue 1 (средний):         Round Robin, quantum = 16ms
Queue 2 (низший):          FIFO

Правила:
1. Новый процесс → Queue 0
2. Использовал весь quantum → понижение в следующую очередь
3. Освободил CPU до quantum (I/O) → остаётся в текущей очереди
4. Периодически все процессы поднимаются в Queue 0 (anti-starvation)

Эффект:
- I/O-bound процессы остаются в Queue 0 (высокий приоритет, быстрый response)
- CPU-bound процессы спускаются в Queue 2 (низкий приоритет, длинный quantum)
```

## Linux CFS (Completely Fair Scheduler)

Linux с 2007 года использует CFS — планировщик на основе "справедливости".

### Идея: виртуальное время

CFS отслеживает "виртуальное время выполнения" (vruntime) каждого процесса. Планировщик всегда выбирает процесс с наименьшим vruntime.

```
vruntime = физическое время × (базовый приоритет / nice)

Процесс с nice=-20 (высокий приоритет):
  1 секунда физического времени = меньше vruntime
  → Может работать дольше до переключения

Процесс с nice=19 (низкий приоритет):
  1 секунда физического времени = больше vruntime
  → Быстрее накапливает vruntime, чаще прерывается
```

### Red-Black Tree

CFS хранит процессы в красно-чёрном дереве, отсортированном по vruntime:

```
              (P3: vruntime=50)
              /               \
    (P1: vruntime=30)    (P5: vruntime=70)
    /            \                    \
(P2: 20)      (P4: 40)             (P6: 90)

Выбор следующего процесса: O(1) — самый левый узел (P2)
Вставка/удаление: O(log n)
```

### Targeted Latency

CFS гарантирует, что каждый процесс получит CPU хотя бы раз за "targeted latency" (обычно 20ms).

```
10 процессов, targeted latency = 20ms:
  Каждый процесс получает ~2ms минимум

100 процессов, targeted latency = 20ms:
  Каждый процесс получает ~0.2ms минимум
  (но есть min_granularity ~1ms, чтобы context switch не съел всё время)
```

### Nice Values

```bash
# Запустить с низким приоритетом (больше nice = ниже приоритет)
nice -n 19 ./cpu_intensive_task

# Изменить приоритет работающего процесса
renice -n 10 -p <pid>

# Диапазон nice: -20 (высший приоритет) до 19 (низший)
# Только root может устанавливать отрицательные значения
```

## Real-Time Scheduling

Для задач с жёсткими требованиями к latency (аудио, видео, промышленное управление):

### SCHED_FIFO

Real-time FIFO. Процесс работает пока не заблокируется или не уступит добровольно. Preemption только от процесса с более высоким RT-приоритетом.

### SCHED_RR

Real-time Round Robin. Как SCHED_FIFO, но с quantum для процессов одного приоритета.

```bash
# Запустить с real-time приоритетом
chrt -f 50 ./realtime_task   # SCHED_FIFO, priority 50
chrt -r 50 ./realtime_task   # SCHED_RR, priority 50

# Проверить policy процесса
chrt -p <pid>
```

### Приоритеты

```
Real-time приоритеты (1-99): Всегда выше обычных процессов
Обычные процессы (nice -20 до 19): SCHED_OTHER/SCHED_NORMAL

RT процесс с приоритетом 1 > любой обычный процесс
RT процесс с приоритетом 99 > RT с приоритетом 1
```

**Опасность:** Real-time процесс может заблокировать систему. Один busy-loop в SCHED_FIFO — и система не отвечает.

## Многопроцессорное планирование

### SMP (Symmetric Multiprocessing)

Несколько CPU, общая память. Планировщик должен решать:
- На каком CPU запустить процесс?
- Как балансировать нагрузку?
- Учитывать ли cache affinity?

### Cache Affinity

Если процесс работал на CPU0, его данные в кэше CPU0. Перенос на CPU1 = cache miss.

```
Process A работает на CPU0:
  - L1/L2 cache CPU0 заполнен данными A
  - Следующий запуск на CPU0 → cache hit → быстро
  - Перенос на CPU1 → cache miss → медленно

Решение: "мягкая" привязка (soft affinity)
  - Планировщик предпочитает тот же CPU
  - Но переносит при дисбалансе
```

### NUMA (Non-Uniform Memory Access)

В больших системах память "ближе" к некоторым CPU:

```
┌─────────────────┐    ┌─────────────────┐
│     Node 0      │    │     Node 1      │
│  ┌───┐  ┌───┐  │    │  ┌───┐  ┌───┐  │
│  │CPU│  │CPU│  │    │  │CPU│  │CPU│  │
│  └───┘  └───┘  │    │  └───┘  └───┘  │
│       │        │    │       │        │
│   ┌───────┐    │    │   ┌───────┐    │
│   │ RAM   │    │────│   │ RAM   │    │
│   │ Local │    │slow│   │ Local │    │
│   └───────┘    │    │   └───────┘    │
└─────────────────┘    └─────────────────┘

CPU на Node 0 → RAM Node 0: быстро (~100 cycles)
CPU на Node 0 → RAM Node 1: медленно (~200 cycles)
```

Планировщик учитывает NUMA: держит процесс на CPU рядом с его памятью.

```bash
# Привязать процесс к CPU
taskset -c 0,1 ./process    # Только CPU 0 и 1

# Привязать к NUMA node
numactl --cpunodebind=0 --membind=0 ./process
```

## Границы применимости алгоритмов планирования

### Когда какой алгоритм НЕ подходит

**FIFO/FCFS:**
- ❌ Интерактивные системы — длинная задача блокирует все
- ❌ Real-time — нет гарантий времени отклика
- ✅ Batch processing с похожими задачами

**Round Robin:**
- ❌ Много CPU-bound задач — высокий overhead от context switches
- ❌ Time quantum слишком маленький — больше switching чем работы
- ✅ Интерактивные системы, time-sharing

**Priority Scheduling:**
- ❌ Без aging — starvation низкоприоритетных задач
- ❌ Priority inversion без mitigation
- ✅ Системы с чётким разделением важности задач

**CFS (Linux):**
- ❌ Hard real-time — не гарантирует deadlines
- ❌ Очень короткие задачи — overhead красно-чёрного дерева
- ✅ General-purpose workloads, серверы

## Влияние на приложения

### Thread Pools

Создание потока дорого. Thread pool переиспользует потоки:

```java
// Плохо: создавать поток на каждый запрос
new Thread(() -> handleRequest(request)).start();

// Хорошо: thread pool
ExecutorService executor = Executors.newFixedThreadPool(100);
executor.submit(() -> handleRequest(request));
```

**Размер pool:** Зависит от характера нагрузки.
- CPU-bound: ~количество ядер
- I/O-bound: больше (потоки ждут I/O, CPU простаивает)

### Async I/O и Event Loop

Вместо потока на каждое соединение — один поток обрабатывает много соединений:

```
Thread-per-request (традиционный):
  10,000 соединений = 10,000 потоков = много context switches

Event loop (Node.js, Netty):
  10,000 соединений = 1-few потоков = минимум context switches
```

Подробнее — в [[jvm-executors-futures]] и [[kotlin-coroutines]].

## Кто использует и реальные примеры

### Linux CFS в продакшене

| Компания | Scheduler | Особенности |
|----------|-----------|-------------|
| **Meta (2025)** | SCX-LAVD (поверх CFS) | Деплоят на production серверах. Автоматически определяет latency-sensitive задачи и повышает им приоритет. Снижение variance latency |
| **AWS Lambda** | CFS с cgroups | 2,500+ контейнеров по 128MB на ноде. CFS group scheduling = 5-20% CPU на context switching в dense packed сценариях |
| **Android** | CFS + приоритеты | Foreground app получает больше CPU, background apps — меньше. Плавный UI |
| **Steam Deck** | SCX-LAVD | Игры получают приоритет, фоновые загрузки — нет |

### Эволюция Linux Scheduler

| Год | Scheduler | Изменение |
|-----|-----------|-----------|
| До 2007 | O(1) Scheduler | Сложные эвристики, проблемы с fairness |
| 2007 | **CFS** (Linux 2.6.23) | Red-black tree, vruntime, "completely fair" |
| 2023 | **EEVDF** (Linux 6.6) | Earliest Eligible Virtual Deadline First — лучше для latency-sensitive задач |
| 2025 | SCX (Extensible Scheduler) | Pluggable schedulers из userspace — можно подключить свой! |

### Настройка для разных workloads

| Workload | Проблема | Решение |
|----------|----------|---------|
| **Веб-сервер (I/O-bound)** | Нужен быстрый response time | Default CFS хорошо справляется. Nice-значения для фоновых задач |
| **ML Training (CPU-bound)** | Нужен максимум throughput | `taskset` для привязки к ядрам, избегать context switches |
| **Realtime Audio/Video** | Нужны deadline гарантии | SCHED_FIFO или SCHED_RR, PREEMPT_RT патч |
| **Контейнеры (mixed)** | Fair distribution между контейнерами | cgroups cpu.shares, cpu.cfs_quota |

### Практические числа

| Метрика | Значение | Источник |
|---------|----------|----------|
| CFS targeted latency | ~20ms (default) | Linux kernel docs |
| CFS min_granularity | ~1ms | Минимальный timeslice |
| Context switch (прямой) | 1-3µs | Benchmarks |
| Context switch (полный, с cache miss) | 10-30µs | Production measurements |
| CFS overhead при 100 процессах | <1% CPU | Типичный сервер |
| CFS overhead при 2500+ контейнеров | 5-20% CPU | AWS Lambda research |

### Thread Pool sizing — правила

| Тип нагрузки | Рекомендация | Почему |
|--------------|--------------|--------|
| **CPU-bound** | `threads = num_cores` | Больше потоков = только overhead на switch |
| **I/O-bound** | `threads = num_cores * (1 + wait_time/compute_time)` | Потоки ждут I/O, нужно больше |
| **Mixed** | Отдельные pools для CPU и I/O | Изоляция workloads |

**Пример:**
- 4 ядра, CPU-bound задачи → 4 потока
- 4 ядра, задачи 80% ждут I/O → 4 * (1 + 4) = 20 потоков

## Связь с другими темами

**[[os-overview]]** — Обзор ОС объясняет механизм context switch, который планировщик инициирует при каждом переключении задачи: сохранение регистров текущего потока, загрузка регистров следующего, переключение стека. Без понимания разделения kernel/user mode невозможно осознать, почему timer interrupt — основа preemptive scheduling: hardware таймер генерирует прерывание, CPU переходит в kernel mode, и планировщик получает возможность решить, какой поток запускать следующим. Концепция системных вызовов также связана с планированием: любой блокирующий syscall (read, wait) переводит поток в состояние Blocked, и планировщик выбирает другой Ready-поток.

**[[os-processes-threads]]** — Процессы и потоки — это именно те сущности, которые планировщик распределяет по CPU. Состояния процесса (New, Ready, Running, Blocked, Terminated) формируют основу работы scheduling: планировщик берёт потоки из Ready-очереди и переводит их в Running, а при блокирующей операции или истечении кванта — обратно в Ready или Blocked. В Linux планировщик CFS работает с task_struct, которая представляет как процессы, так и потоки (через clone()), и использует vruntime для отслеживания «справедливости» распределения CPU. Понимание разницы между user-level и kernel-level threads критично: 1:1 модель означает, что ОС видит и планирует каждый поток, тогда как в M:N модели пользовательский runtime выполняет собственное планирование поверх нескольких OS threads.

**[[os-synchronization]]** — Синхронизация и планирование взаимодействуют через проблему priority inversion: когда высокоприоритетный поток ожидает mutex, захваченный низкоприоритетным потоком, а среднеприоритетный поток занимает CPU — высокоприоритетный блокируется на неопределённое время. Решения (priority inheritance, priority ceiling) требуют координации между планировщиком и механизмом блокировок: планировщик временно повышает приоритет владельца mutex до уровня ожидающего потока. Spinlocks также связаны с scheduling: если поток, держащий spinlock, вытесняется планировщиком, все ожидающие потоки тратят свои кванты на бесполезное вращение. Выбор между mutex (блокирующий, вызывает перепланирование) и spinlock (не блокирующий, потребляет CPU) зависит от ожидаемого времени ожидания относительно стоимости context switch.

**[[jvm-executors-futures]]** — JVM Executor framework реализует планирование задач в userspace поверх OS scheduling: ThreadPoolExecutor управляет пулом OS threads и распределяет Runnable/Callable задачи между ними. Это двухуровневое планирование: ОС планирует потоки пула, а Executor планирует задачи на потоки, причём каждый уровень имеет собственные политики — CFS на уровне ОС и work queue на уровне Executor. ForkJoinPool использует work-stealing алгоритм — когда поток завершает свои задачи, он «крадёт» задачи из очередей других потоков, что минимизирует idle time и балансирует нагрузку. Понимание OS scheduling помогает настраивать пулы: core pool size должен соответствовать количеству CPU cores для CPU-bound задач и быть больше для IO-bound (формула N_threads = N_cores × (1 + W/C), где W/C — отношение wait time к compute time).

**Связанные концепции:**
- [[os-memory-management]] — TLB flush при context switch между процессами
- [[jvm-concurrency-overview]] — Thread.yield(), thread priorities в JVM
- [[kotlin-coroutines]] — Dispatchers распределяют корутины по потокам, но сам scheduling корутин — в userspace

## Проверь себя

> [!question]- Веб-сервер обрабатывает 10,000 запросов/сек, каждый запрос на 80% ждёт ответа от базы данных. Сколько потоков оптимально для thread pool на 8-ядерном сервере и почему?
> По формуле: N = N_cores × (1 + W/C) = 8 × (1 + 0.8/0.2) = 8 × 5 = 40 потоков. Объяснение: каждый поток 80% времени заблокирован на I/O (ожидание БД), значит соотношение wait/compute = 4. При 40 потоках в каждый момент ~8 потоков активно используют CPU (остальные 32 ждут I/O), что полностью загружает 8 ядер. Меньше 40 — CPU простаивает; больше 40 — context switch overhead растёт без пользы. Для чисто CPU-bound задач ответ был бы 8 потоков.

> [!question]- Почему Linux заменил CFS на EEVDF (Linux 6.6), хотя CFS считался "completely fair"? Какую проблему EEVDF решает лучше?
> CFS "справедлив" в долгосрочной перспективе, но не гарантирует latency конкретного запроса: процесс с большим nice может ждать долго, пока набежит достаточный vruntime для запуска. EEVDF (Earliest Eligible Virtual Deadline First) добавляет понятие virtual deadline — каждый процесс получает deadline пропорционально весу, и планировщик выбирает процесс с ближайшим deadline среди eligible. Это улучшает worst-case latency для interactive/latency-sensitive задач, сохраняя fairness. CFS мог давать latency spikes до targeted_latency (20ms) × количество процессов.

> [!question]- Priority inversion на Mars Pathfinder (1997) заставлял rover перезагружаться. Опиши проблему и почему priority inheritance её решает.
> На Mars Pathfinder: высокоприоритетная задача (bus management) ждала mutex, захваченный низкоприоритетной задачей (метеорология). Среднеприоритетная задача (communications) вытесняла низкоприоритетную, не давая ей отпустить mutex. Высокоприоритетная задача блокировалась → watchdog timer срабатывал → перезагрузка. **Priority Inheritance** решает: когда высокоприоритетный поток T1 ждёт mutex, держатель T3 временно получает приоритет T1. Теперь T2 не может вытеснить T3, T3 быстро завершает критическую секцию, отпускает mutex, T1 продолжает. Приоритет T3 возвращается к исходному.

> [!question]- AWS Lambda запускает 2500+ контейнеров на одной ноде. Почему CFS теряет 5-20% CPU при такой плотности и какие есть решения?
> При 2500+ задачах CFS тратит значительное время на: (1) O(log N) вставки/удаления в red-black tree — при N=2500 это log₂(2500)≈11 уровней; (2) context switches — при targeted_latency 20ms и 2500 процессах, каждый получает ~8µs (20ms/2500), что сопоставимо со стоимостью switch; (3) TLB flushes при переключении между контейнерами (разные memory spaces). Решения: (1) увеличить sched_min_granularity чтобы снизить частоту переключений; (2) cpu.cfs_quota для ограничения "шумных" контейнеров; (3) CPU pinning (cpuset) для изоляции; (4) SCX-LAVD (extensible scheduler) для custom scheduling policies.

## Ключевые карточки

В чём разница между preemptive и cooperative scheduling?
?
**Preemptive:** ОС прерывает процесс принудительно (по timer interrupt). Процесс не контролирует когда будет прерван. Используется во всех современных ОС. **Cooperative:** Процесс сам отдаёт CPU (yield). Если не отдаёт — другие ждут. Остался в embedded (FreeRTOS) и частично в green threads (goroutines, корутины).

Что такое convoy effect и при каком алгоритме возникает?
?
Convoy effect — ситуация при FIFO scheduling, когда короткие I/O-bound процессы выстраиваются за длинным CPU-bound процессом. CPU-bound процесс занимает CPU, I/O-bound процессы ждут (хотя быстро бы освободили CPU). Результат: низкий CPU utilization, высокое average wait time. Решение: preemptive scheduling (Round Robin, CFS).

Как CFS обеспечивает "справедливость" и какая структура данных используется?
?
CFS отслеживает virtual runtime (vruntime) каждого процесса — сколько "справедливого" CPU-времени он получил. Процессы хранятся в **red-black tree**, отсортированном по vruntime. Планировщик выбирает процесс с минимальным vruntime — O(1). Вставка/удаление — O(log N). Nice value влияет на скорость роста vruntime: nice -20 → vruntime растёт медленнее → больше CPU; nice 19 → быстрее → меньше CPU.

Формула оптимального размера thread pool для I/O-bound задач?
?
N_threads = N_cores × (1 + W/C), где W = время ожидания I/O, C = время вычислений. Пример: 4 ядра, задачи 80% I/O → 4 × (1 + 4) = 20 потоков. Для CPU-bound: N_threads = N_cores (больше = только overhead). Для mixed workloads: отдельные pools для CPU и I/O задач.

Что такое NUMA и почему scheduling должен его учитывать?
?
NUMA (Non-Uniform Memory Access) — архитектура, где память физически "ближе" к определённым CPU. Доступ к локальной RAM: ~100 cycles, к удалённой: ~200 cycles (2x медленнее). Планировщик учитывает: держит процесс на CPU рядом с его памятью (NUMA affinity). `numactl --cpunodebind=0 --membind=0 ./process` — привязка к node 0.

Чем SCHED_FIFO отличается от SCHED_RR в Linux?
?
Оба — real-time scheduling policies (приоритет выше любого обычного процесса). **SCHED_FIFO:** процесс работает пока не заблокируется, не завершится или не будет вытеснен процессом с более высоким RT-приоритетом. Нет quantum. **SCHED_RR:** как FIFO, но с time quantum для процессов одного приоритета — Round Robin между равными. Оба опасны: busy-loop в SCHED_FIFO заблокирует систему.

Что такое cache affinity и зачем нужен soft affinity?
?
Cache affinity — принцип, что процесс лучше запускать на том же CPU, где работал ранее, потому что его данные в L1/L2 кэше этого CPU. Перенос на другой CPU → cache miss → медленный старт. **Soft affinity:** планировщик предпочитает тот же CPU, но переносит при дисбалансе нагрузки. **Hard affinity:** `taskset -c 0,1 ./process` — жёсткая привязка к CPU 0 и 1.

Какие метрики scheduling важны и какие конфликтуют?
?
**Throughput:** количество завершённых задач в единицу времени. **Turnaround time:** от поступления задачи до завершения. **Response time:** от поступления до первого ответа (критично для interactive). **Fairness:** равномерное распределение CPU. **Конфликт:** оптимизация throughput (длинные quantum, batch processing) ухудшает response time; оптимизация response time (короткие quantum, частые switches) снижает throughput из-за context switch overhead.
