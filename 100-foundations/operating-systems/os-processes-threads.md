## Теоретические основы

> **Процесс** — экземпляр выполняющейся программы, обладающий собственным адресным пространством, файловыми дескрипторами и контекстом выполнения. **Поток (thread)** — единица планирования внутри процесса, разделяющая адресное пространство с другими потоками того же процесса.

### Формальная модель

| Свойство | Процесс | Поток |
|----------|---------|-------|
| Адресное пространство | Своё (изолированное) | Общее (shared) |
| Создание | fork() — тяжёлое (copy-on-write) | clone() / pthread_create — лёгкое |
| Коммуникация | IPC: pipe, socket, shared memory | Shared memory (прямой доступ) |
| Изоляция | Полная (один процесс не может читать память другого) | Отсутствует (race conditions) |
| Overhead | ~10-30 MB per process | ~1 MB per thread (stack) |

### Историческая хронология

| Год | Событие | Значение |
|-----|---------|----------|
| 1965 | **Multics** (MIT/GE/Bell Labs) | Первая ОС с мультипрограммностью |
| 1969 | **Unix** (Thompson, Ritchie) | Процесс как центральная абстракция, fork/exec |
| 1971 | **fork()** (Unix) | Создание процесса через копирование |
| 1995 | **POSIX Threads (Pthreads)** | Стандартизация потоков (IEEE 1003.1c) |
| 2003 | **NPTL** (Drepper, Molnar) | Linux threading 1:1 модель, заменила LinuxThreads |
| 2023 | **Virtual Threads** (Java 21) | M:N threading на JVM |

# Процессы и потоки: единицы выполнения

Процесс — это запущенная программа с собственным адресным пространством. Поток (thread) — это последовательность инструкций внутри процесса, которая может выполняться независимо. Один процесс может содержать множество потоков, и все они разделяют память процесса. Это разделение памяти делает потоки эффективными для параллелизма, но создаёт проблемы синхронизации.

Интересный факт: в Linux на низком уровне и процессы, и потоки создаются через один и тот же системный вызов `clone()`. Различие между ними — это просто набор флагов, передаваемых в `clone()`. Процесс создаётся с флагами, которые говорят "скопировать адресное пространство", а поток — с флагами "разделить адресное пространство". Поэтому в ядре Linux потоки иногда называют "lightweight processes" (LWP).

---

## TL;DR

> **Процесс** — запущенная программа с изолированной памятью. Как отдельная квартира: свои стены, своя мебель, соседи не могут зайти без разрешения.
>
> **Поток** — единица выполнения внутри процесса. Как жильцы одной квартиры: делят кухню и ванную, но у каждого своя кровать (stack).
>
> **Когда что:** Процессы — для изоляции и безопасности (Chrome вкладки). Потоки — для быстрого параллелизма с общими данными (MySQL).
>
> **Ключевые числа:** Создание процесса ~100-300µs, потока ~30-50µs. Context switch между потоками ~1.2-1.5µs, между процессами ~2.2µs+.
>
> **Модели:** 1:1 (Linux — 1 user thread = 1 kernel thread), M:N (Go goroutines — много user threads на несколько kernel threads).

## Часть 1: Интуиция без кода

### 🏠 Аналогия 1: Квартиры и жильцы

Представь многоквартирный дом. **Процесс — это отдельная квартира**:
- У квартиры свои стены (границы памяти)
- Своя мебель (данные в heap)
- Своя входная дверь с замком (изоляция)
- Соседи не могут зайти без разрешения (межпроцессная изоляция)

**Поток — это жилец квартиры**:
- Все жильцы одной квартиры делят кухню и гостиную (общий heap)
- Но у каждого своя кровать и тумбочка (свой stack)
- Они могут общаться напрямую, без звонков (быстрая коммуникация)
- Если один жилец устроит пожар — пострадают все (один поток может уронить весь процесс)

**Почему Chrome использует процессы для вкладок?** Если бы вкладки были потоками, упавшая вкладка (пожар в комнате) убила бы весь браузер. С процессами (отдельными квартирами) — упала одна вкладка, остальные работают.

### 🍴 Аналогия 2: fork() как клонирование

`fork()` — это как если бы ты мог мгновенно клонировать свою квартиру со всей мебелью:

```
До fork():                    После fork():
┌─────────────┐               ┌─────────────┐  ┌─────────────┐
│  Квартира   │               │  Родитель   │  │   Ребёнок   │
│   (PID=5)   │      →        │   (PID=5)   │  │   (PID=6)   │
│  x = 42     │               │  x = 42     │  │  x = 42     │
│  arr = [...]│               │  arr = [...]│  │  arr = [...] │ ← копия!
└─────────────┘               └─────────────┘  └─────────────┘
```

После `fork()`:
- Родитель получает PID ребёнка (return > 0)
- Ребёнок получает 0 (return == 0)
- Изменения в одной квартире НЕ влияют на другую

Важно: современные ОС используют **Copy-on-Write (COW)** — реальное копирование происходит только при изменении. Это как если бы клон квартиры был виртуальным, и материализовался только когда кто-то двигает мебель.

### 🎭 Аналогия 3: exec() как трансформация

`exec()` — это как если бы гусеница превратилась в бабочку:
- Тело (PID) остаётся тем же
- Но внутреннее содержимое полностью заменяется
- Старая программа исчезает навсегда

Классический паттерн в Unix:
```
fork()  →  exec()
  │           │
  ↓           ↓
Клон      Превращение клона
себя      в другую программу
```

Это как размножение: родитель создаёт клона, клон становится другой программой, родитель продолжает жить.

### 🏃 Аналогия 4: Context switch как переключение разговоров

Представь, что ты на вечеринке и говоришь с 5 людьми по очереди:

**Переключение между людьми (потоками одного процесса):**
- Ты помнишь общий контекст вечеринки (общая память)
- Нужно только вспомнить, о чём говорили с этим человеком (сохранить/восстановить регистры)
- Быстро: ~1-2 микросекунды

**Переключение между вечеринками (разными процессами):**
- Нужно вспомнить ГДЕ ты (сменить адресное пространство)
- Забыть всё о текущей вечеринке (flush TLB cache)
- Вспомнить контекст новой вечеринки
- Медленно: ~2-3 микросекунды + потеря кэшей

### 🚦 Аналогия 5: Модели потоков как модели такси

**1:1 (один user thread = один kernel thread)** — как обычное такси:
- Один пассажир (поток) = одна машина (kernel thread)
- Дорого, но надёжно
- Так работает Linux pthreads

**M:1 (много user threads на один kernel thread)** — как автобус:
- Много пассажиров, одна машина
- Если автобус застрял (блокирующий syscall) — все ждут
- Так работали старые green threads

**M:N (много user threads на несколько kernel threads)** — как Uber Pool:
- Много пассажиров распределяются по нескольким машинам
- Диспетчер (runtime scheduler) решает кто в какой машине
- Так работают Go goroutines (~2KB) и Kotlin coroutines

```
1:1 модель:          M:N модель:
U U U U U            U U U U U U U U
│ │ │ │ │               \ | | | /
K K K K K                 K K K
                      (scheduler)
```

### 🔢 Численная интуиция

| Операция | Время | Сравнение |
|----------|-------|-----------|
| Создать процесс | ~100-300 µs | Построить квартиру |
| Создать поток | ~30-50 µs | Поставить кровать |
| Создать goroutine | ~0.3 µs | Положить подушку |
| Context switch (потоки) | ~1.2-1.5 µs | Переключить взгляд |
| Context switch (процессы) | ~2.2+ µs | Выйти из комнаты |
| Память на процесс | ~4-10 MB min | Целая квартира |
| Память на поток | ~1-8 MB (stack) | Одна комната |
| Память на goroutine | ~2-8 KB | Один стул |

**Пример:** Веб-сервер с 10,000 соединений
- Потоки: 10,000 × 2MB = 20GB памяти 😱
- Goroutines: 10,000 × 8KB = 80MB памяти ✅

## Часть 2: Почему это сложно

### ❌ Ошибка 1: Путаница process vs thread

**СИМПТОМ:** Студент пишет "создам новый процесс для параллельной обработки данных" когда нужен поток.

**ПОЧЕМУ ВОЗНИКАЕТ:** В быту слова "процесс" и "поток" смешиваются. "Process" в английском = любой процесс выполнения.

**КАК ПРОЯВЛЯЕТСЯ:**
```c
// Студент хочет параллельно обработать массив
for (int i = 0; i < 4; i++) {
    pid_t pid = fork();  // ❌ Создаёт 4 ПРОЦЕССА
    if (pid == 0) {
        process_chunk(data, i);  // Каждый работает с КОПИЕЙ данных!
        exit(0);
    }
}
// Результаты потеряны — каждый процесс изменил свою копию
```

**РЕШЕНИЕ:**
```c
// Для общих данных — потоки
pthread_t threads[4];
for (int i = 0; i < 4; i++) {
    pthread_create(&threads[i], NULL, process_chunk, &args[i]);
}
// Все потоки видят и изменяют ОДНИ данные
```

**ПРАВИЛО:**
- Нужна изоляция и безопасность → процесс
- Нужны общие данные и скорость → поток

### ❌ Ошибка 2: Не ждать завершения потомков (zombie процессы)

**СИМПТОМ:** `ps aux | grep Z` показывает процессы в состоянии "Z" (zombie).

**ПОЧЕМУ ВОЗНИКАЕТ:** Когда процесс завершается, ядро сохраняет его exit status пока родитель не вызовет `wait()`. Если родитель не вызывает — zombie накапливаются.

**КАК ПРОЯВЛЯЕТСЯ:**
```c
for (int i = 0; i < 100; i++) {
    if (fork() == 0) {
        do_work();
        exit(0);  // Ребёнок завершился
    }
    // ❌ Родитель НЕ вызывает wait()
}
// Теперь у нас 100 zombie процессов!
```

**РЕШЕНИЕ:**
```c
// Вариант 1: Явно ждать
for (int i = 0; i < 100; i++) {
    pid_t pid = fork();
    if (pid == 0) {
        do_work();
        exit(0);
    }
}
while (wait(NULL) > 0);  // ✅ Дождаться всех

// Вариант 2: Игнорировать SIGCHLD (zombie auto-reap)
signal(SIGCHLD, SIG_IGN);  // ✅ Ядро само уберёт zombies
```

### ❌ Ошибка 3: Неправильное разделение данных между потоками

**СИМПТОМ:** Race condition, случайные значения, программа работает по-разному при каждом запуске.

**ПОЧЕМУ ВОЗНИКАЕТ:** Потоки делят heap, но каждый имеет свой stack. Передача указателя на локальную переменную — ошибка.

**КАК ПРОЯВЛЯЕТСЯ:**
```c
void* thread_func(void* arg) {
    int id = *(int*)arg;
    printf("Thread %d\n", id);  // Печатает случайные числа!
    return NULL;
}

int main() {
    pthread_t threads[4];
    for (int i = 0; i < 4; i++) {
        pthread_create(&threads[i], NULL, thread_func, &i);  // ❌ &i!
        // i меняется в цикле, а все потоки читают один и тот же адрес
    }
}
// Вывод: "Thread 4 Thread 4 Thread 4 Thread 4" или хуже
```

**РЕШЕНИЕ:**
```c
int ids[4];  // Массив в heap/static
for (int i = 0; i < 4; i++) {
    ids[i] = i;
    pthread_create(&threads[i], NULL, thread_func, &ids[i]);  // ✅
}
// Или передать значение напрямую (cast to void*)
pthread_create(&t, NULL, func, (void*)(intptr_t)i);  // ✅
```

### ❌ Ошибка 4: Блокирующие вызовы в green threads/goroutines

**СИМПТОМ:** Программа на Go "зависает", несмотря на тысячи goroutines.

**ПОЧЕМУ ВОЗНИКАЕТ:** Green threads/goroutines работают поверх ограниченного числа OS threads. Блокирующий syscall блокирует весь OS thread.

**КАК ПРОЯВЛЯЕТСЯ:**
```go
for i := 0; i < 10000; i++ {
    go func() {
        // ❌ CGO вызов или блокирующий syscall
        result := C.blocking_c_function()  // Блокирует OS thread!
    }()
}
// Все GOMAXPROCS OS threads заблокированы → всё висит
```

**РЕШЕНИЕ:**
- Использовать неблокирующие API (net/http вместо CGO)
- Увеличить GOMAXPROCS для CGO-heavy кода
- Использовать worker pool для блокирующих операций:

```go
// ✅ Worker pool для блокирующих операций
workerPool := make(chan struct{}, 100)  // Лимит блокирующих вызовов
for i := 0; i < 10000; i++ {
    go func() {
        workerPool <- struct{}{}  // Acquire
        result := C.blocking_c_function()
        <-workerPool  // Release
    }()
}
```

### ❌ Ошибка 5: Неправильный выбор между fork() и pthread_create()

**СИМПТОМ:** Программа медленная или использует слишком много памяти.

**ПОЧЕМУ ВОЗНИКАЕТ:** Непонимание когда нужна изоляция (процесс) vs общие данные (поток).

**ТАБЛИЦА РЕШЕНИЙ:**

| Сценарий | Выбор | Почему |
|----------|-------|--------|
| Параллельная обработка массива | pthread | Общие данные, минимум overhead |
| Изолированные sandbox'ы | fork() | Падение одного не затронет других |
| Web worker (браузер) | Процесс | Безопасность, изоляция |
| HTTP handlers в сервере | Потоки | Общий кэш, быстрое создание |
| Загрузка плагинов | Процесс | Защита от падения/exploit плагина |
| Параллельные вычисления | Потоки | Нужен доступ к общим данным |

### ❌ Ошибка 6: Думать, что fork() мгновенен с COW

**СИМПТОМ:** "fork() же копирует лениво, значит это бесплатно!"

**ПОЧЕМУ ВОЗНИКАЕТ:** Copy-on-Write создаёт иллюзию бесплатного копирования.

**РЕАЛЬНОСТЬ:**
```
fork() всё равно должен:
1. Создать новый PCB в ядре
2. Скопировать таблицу страниц (не данные, но всё равно работа)
3. Скопировать file descriptors
4. Обновить метаданные

При N MB памяти с M страницами:
- Таблица страниц: M × 8 bytes = значительный размер
- 10GB процесс → ~2.5M страниц → ~20MB на таблицу
```

**КОД ДЕМОНСТРАЦИИ:**
```c
// Процесс с 10GB данных
char* huge = malloc(10ULL * 1024 * 1024 * 1024);
memset(huge, 'x', 10ULL * 1024 * 1024 * 1024);

// fork() будет НЕ мгновенным — копирование page tables
time_t start = clock();
if (fork() == 0) exit(0);
wait(NULL);
printf("fork took: %f ms\n", (clock()-start)*1000.0/CLOCKS_PER_SEC);
// Результат: сотни миллисекунд!
```

```
│    │TLS   │        │TLS   │        │TLS   │            │

```

**Когда использовать:** Когда думаете о том, что разделяется, а что нет.

### 🧠 Модель 2: "Жизненный цикл как конечный автомат"

```
                    ┌─────────────┐
                    │   Created   │
                    └──────┬──────┘
                           │ start()
                           ↓
    ┌──────────────────────────────────────────────┐
    │                    READY                      │
    │  (в очереди планировщика, ждёт CPU)          │
    └───────────────────┬──────────────────────────┘
                        │ scheduler выбрал
        ┌───────────────↓───────────────┐
        │            RUNNING             │
        │   (выполняется на CPU)         │
        └─┬────────────┬────────────────┬┘
          │            │                │
    yield/│      I/O   │           exit │
  preempt │    request │                │
          ↓            ↓                ↓
       READY        WAITING         TERMINATED
                       │                  │
                 I/O done              zombie до
                       │               wait()
                       ↓                  │
                    READY              CLEANUP
```

**Ключевые переходы:**
- READY → RUNNING: только планировщик решает
- RUNNING → WAITING: поток сам запросил I/O
- RUNNING → READY: preemption (квант времени истёк)

### 🧠 Модель 3: "fork() + exec() = spawn() в два шага"

```
Windows CreateProcess():     Unix fork() + exec():
┌──────────────────────┐     ┌──────────────────────────────────┐
│ CreateProcess(       │     │ if (fork() == 0) {               │
│   "program.exe",     │     │     // Ребёнок                   │
│   args,              │     │     setup_environment();         │
│   env,               │     │     redirect_io();               │
│   ...                │     │     exec("program", args);       │
│ )                    │     │ }                                │
│ // Один вызов        │     │ // Два вызова = гибкость!        │
└──────────────────────┘     └──────────────────────────────────┘
```

**Преимущества Unix-подхода:**
1. Между fork() и exec() можно настроить окружение ребёнка
2. Можно перенаправить stdin/stdout (`dup2()`)
3. Можно установить ограничения (`setrlimit()`)
4. Можно сменить пользователя (`setuid()`)

**Это то, как работает shell:**
```bash
$ ls -la | grep foo > output.txt

Shell делает:
1. fork()  →  fork()          # Создать двух детей
2. dup2() для pipe            # Настроить пайп между ними
3. dup2() для output.txt      # Перенаправить вывод
4. exec("ls") / exec("grep")  # Заменить программы
```

|--------|---------|-------------------|

| **Stack** | Память для локальных переменных и вызовов функций (у каждого потока свой) | Стопка тарелок: добавляешь сверху, берёшь сверху (LIFO) |

## Что такое процесс

Процесс — это не просто программа. Программа — это файл на диске (исполняемый код). Процесс — это программа в выполнении: код, загруженный в память, плюс текущее состояние (регистры CPU, открытые файлы, переменные).

### Из чего состоит процесс

```
┌─────────────────────────────────────────────────────────────────┐
│                         ПРОЦЕСС                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Адресное пространство (виртуальная память):                    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Stack (растёт вниз ↓)                                       ││
│  │   - локальные переменные                                    ││
│  │   - адреса возврата из функций                              ││
│  │   - параметры функций                                       ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │                    ↓ свободно ↑                             ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ Heap (растёт вверх ↑)                                       ││
│  │   - malloc(), new                                           ││
│  │   - динамические структуры данных                           ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ BSS (неинициализированные глобальные переменные)            ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ Data (инициализированные глобальные переменные)             ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ Text (код программы) — read-only                            ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  Между Heap и Stack есть ещё одна область:                      │
│  Memory Mapping Segment (mmap) — используется для:              │
│    - Подгрузки динамических библиотек (.so)                     │
│    - Memory-mapped files                                        │
│    - Анонимных маппингов (shared memory, буферы)                │
│    - Стеков дополнительных потоков (pthread)                    │
│                                                                 │
│  + Ресурсы:                                                     │
│    - Открытые файлы (file descriptors)                          │
│    - Сетевые соединения                                         │
│    - Права доступа                                              │
│    - Текущая директория                                         │
│                                                                 │
│  + Состояние:                                                   │
│    - Регистры CPU                                               │
│    - Program Counter (указатель на текущую инструкцию)          │
│    - Stack Pointer                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Process Control Block (PCB)

Ядро хранит информацию о каждом процессе в структуре PCB:

```c
// Упрощённая структура PCB (реальная в Linux — task_struct)
struct PCB {
    int pid;                    // Process ID
    int state;                  // Running, Ready, Blocked, etc.
    int priority;               // Приоритет для планировщика

    // Контекст CPU (сохраняется при переключении)
    registers_t registers;      // Все регистры CPU
    void* program_counter;      // Текущая инструкция
    void* stack_pointer;        // Верх стека

    // Память
    page_table_t* page_table;   // Таблица страниц

    // Ресурсы
    file_descriptor_t* open_files[MAX_FILES];

    // Связи
    struct PCB* parent;         // Родительский процесс
    struct PCB* children;       // Дочерние процессы
};
```

### Состояния процесса

Процесс проходит через несколько состояний:

```
                    ┌──────────────────┐
                    │      New         │
                    │  (создаётся)     │
                    └────────┬─────────┘
                             │ admitted
                             ▼
┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│     Blocked      │◀───│      Ready       │◀───│     Running      │
│  (ждёт событие)  │    │ (готов к работе) │    │  (выполняется)   │
└────────┬─────────┘    └────────┬─────────┘    └────────┬─────────┘
         │                       │                       │
         │ event occurred        │ scheduled             │ preempted
         └──────────────────────▶│◀──────────────────────┘
                                 │
                                 │ exit
                                 ▼
                    ┌──────────────────┐
                    │   Terminated     │
                    │   (завершён)     │
                    └──────────────────┘
```

**Ready:** Процесс готов выполняться, но CPU занят другим процессом. Ждёт своей очереди в ready queue.

**Running:** Процесс выполняется на CPU. В однопроцессорной системе только один процесс может быть в этом состоянии.

**Blocked (Waiting):** Процесс ждёт какого-то события — завершения I/O операции, освобождения ресурса, сигнала. Пока событие не произойдёт, процессу бессмысленно давать CPU.

### Zombie и Orphan процессы

**Zombie (defunct) процесс** — это процесс, который завершил выполнение, но его запись всё ещё остаётся в таблице процессов. Почему так происходит? Когда процесс завершается через `exit()`, ядро сохраняет информацию о нём (exit status, использованное CPU время), чтобы родительский процесс мог её получить через `wait()` или `waitpid()`. Пока родитель не вызовет `wait()`, процесс остаётся zombie.

Zombie процессы не потребляют память или CPU — они занимают только строку в таблице процессов. Но таблица процессов конечна. Если накопится слишком много zombie, новые процессы не смогут создаваться.

```c
// Создание zombie: родитель не вызывает wait()
pid_t pid = fork();
if (pid == 0) {
    // Ребёнок завершается сразу
    exit(0);
}
// Родитель не вызывает wait() — ребёнок становится zombie
sleep(60);  // В течение этих 60 секунд ребёнок — zombie
```

**Orphan процесс** — это работающий процесс, чей родитель завершился раньше него. Когда родитель умирает, ядро "усыновляет" orphan процесс — его новым родителем становится процесс init (PID 1). Когда orphan завершится, init автоматически вызовет `wait()`, поэтому orphan не станут zombie.

```c
pid_t pid = fork();
if (pid == 0) {
    // Ребёнок: спит, пока родитель не умрёт
    sleep(10);
    // После смерти родителя getppid() вернёт 1 (init)
    printf("Мой новый родитель: %d\n", getppid());  // Выведет 1
    exit(0);
}
// Родитель завершается сразу — ребёнок становится orphan
exit(0);
```

**Как предотвратить zombie:**
1. Всегда вызывать `wait()` или `waitpid()` для дочерних процессов
2. Использовать signal handler для SIGCHLD, который вызывает `wait()`
3. Использовать двойной fork (ребёнок сразу завершается, внук усыновляется init)

## Что такое поток (Thread)

Поток — это "легковесный процесс" внутри процесса. Все потоки одного процесса разделяют:
- Адресное пространство (код, heap, глобальные переменные)
- Открытые файлы
- Другие ресурсы процесса

У каждого потока собственные:
- Stack (стек вызовов)
- Регистры CPU
- Program Counter
- Thread-local storage (TLS)

```
┌─────────────────────────────────────────────────────────────────┐
│                         ПРОЦЕСС                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Общая память (все потоки видят):                               │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Heap            │ Data            │ Text (код)              ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  Потоки:                                                        │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
│  │   Thread 1   │ │   Thread 2   │ │   Thread 3   │            │
│  │              │ │              │ │              │            │
│  │ Stack        │ │ Stack        │ │ Stack        │            │
│  │ Registers    │ │ Registers    │ │ Registers    │            │
│  │ PC           │ │ PC           │ │ PC           │            │
│  │ TLS          │ │ TLS          │ │ TLS          │            │
│  └──────────────┘ └──────────────┘ └──────────────┘            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Почему потоки эффективнее процессов для параллелизма

**Создание:** Создание потока значительно быстрее создания процесса. fork() с COW занимает ~100-300µs, pthread_create — ~30-50µs. Разница уменьшилась благодаря Copy-on-Write, но поток всё равно быстрее: не нужно копировать page tables и выделять новые ресурсы.

**Память:** Потоки разделяют heap. Процессы изолированы — для обмена данными нужны специальные механизмы IPC (pipes, shared memory, sockets), которые требуют копирования данных или syscalls.

**Переключение:** Context switch между потоками одного процесса быстрее, чем между процессами. При переключении между процессами нужно:
- Сменить page table (регистр CR3 на x86)
- Сбросить TLB (Translation Lookaside Buffer) — кэш трансляции адресов
- На некоторых архитектурах (ARM) — сбросить L1 кэш полностью!

При переключении между потоками одного процесса адресное пространство не меняется, поэтому TLB остаётся валидным. По измерениям, context switch между потоками занимает 1.2-1.5 микросекунды на одном ядре, между процессами — до 2.2 микросекунды и более из-за TLB miss после переключения.

### Почему потоки опасны

Разделяемая память — это и преимущество, и источник проблем:

```c
// Два потока выполняют этот код одновременно
int counter = 0;  // Общая переменная

void increment() {
    counter++;  // Выглядит атомарно, но это 3 операции:
                // 1. Прочитать counter в регистр
                // 2. Увеличить регистр
                // 3. Записать регистр обратно в counter
}

// Возможный сценарий:
// Thread 1: читает counter (0)
// Thread 2: читает counter (0)
// Thread 1: увеличивает (1)
// Thread 2: увеличивает (1)
// Thread 1: записывает (1)
// Thread 2: записывает (1)
// Результат: 1 вместо 2 — race condition!
```

Подробнее о синхронизации — в [[os-synchronization]].

## Создание процессов: fork() и exec()

В Unix-системах процессы создаются через комбинацию двух syscalls:

### fork() — клонирование процесса

`fork()` создаёт точную копию текущего процесса:

```c
#include <stdio.h>
#include <unistd.h>

int main() {
    printf("Перед fork: PID = %d\n", getpid());

    pid_t pid = fork();  // Здесь процесс раздваивается

    if (pid < 0) {
        // Ошибка
        perror("fork failed");
    } else if (pid == 0) {
        // Это дочерний процесс (fork вернул 0)
        printf("Дочерний процесс: PID = %d, родитель = %d\n",
               getpid(), getppid());
    } else {
        // Это родительский процесс (fork вернул PID ребёнка)
        printf("Родительский процесс: PID = %d, ребёнок = %d\n",
               getpid(), pid);
    }

    return 0;
}

// Вывод:
// Перед fork: PID = 1234
// Родительский процесс: PID = 1234, ребёнок = 1235
// Дочерний процесс: PID = 1235, родитель = 1234
```

**Что копируется при fork():**
- Всё адресное пространство (но с Copy-on-Write оптимизацией)
- Открытые файлы
- Сигналы
- Текущая директория

**Copy-on-Write (COW):** Современные ОС не копируют память сразу. Родитель и ребёнок разделяют одни страницы памяти (read-only). Копия создаётся только когда один из них пытается записать — тогда копируется только изменённая страница. Это делает fork() быстрым даже для процессов с большим heap.

### exec() — замена образа процесса

`exec()` заменяет текущую программу новой:

```c
#include <unistd.h>

int main() {
    printf("Сейчас я программа A\n");

    // Заменить себя на /bin/ls
    execl("/bin/ls", "ls", "-la", NULL);

    // Этот код НЕ выполнится, если exec успешен
    printf("Это никогда не напечатается\n");

    return 0;
}
```

### Типичный паттерн: fork + exec

```c
pid_t pid = fork();

if (pid == 0) {
    // Дочерний процесс — запустить другую программу
    execl("/bin/ls", "ls", "-la", NULL);
    exit(1);  // Сюда попадём только если exec неудачен
}

// Родитель ждёт завершения ребёнка
int status;
waitpid(pid, &status, 0);
printf("Ребёнок завершился с кодом %d\n", WEXITSTATUS(status));
```

Так работает shell: когда вы вводите команду, shell делает fork(), в дочернем процессе exec() запускает команду, а родитель wait() ждёт завершения.

## Создание потоков: POSIX Threads (pthreads)

```c
#include <pthread.h>
#include <stdio.h>

void* thread_function(void* arg) {
    int thread_num = *(int*)arg;
    printf("Поток %d выполняется\n", thread_num);
    return NULL;
}

int main() {
    pthread_t threads[3];
    int thread_args[3] = {1, 2, 3};

    // Создать 3 потока
    for (int i = 0; i < 3; i++) {
        pthread_create(&threads[i], NULL, thread_function, &thread_args[i]);
    }

    // Дождаться завершения всех потоков
    for (int i = 0; i < 3; i++) {
        pthread_join(threads[i], NULL);
    }

    printf("Все потоки завершились\n");
    return 0;
}
```

### Размер стека потока

По умолчанию на Linux каждый поток получает стек размером **8 MB** виртуальной памяти. Это не означает, что сразу выделяется 8 MB физической памяти — страницы выделяются по требованию (demand paging).

Но если у вас 10,000 потоков, это 80 GB виртуального адресного пространства только под стеки. Реальное потребление памяти будет меньше (только использованные страницы), но это накладывает ограничения.

```c
// Уменьшить размер стека потока
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setstacksize(&attr, 64 * 1024);  // 64 KB вместо 8 MB

pthread_t thread;
pthread_create(&thread, &attr, thread_function, NULL);
```

## Процессы vs Потоки: когда что использовать

### Используйте процессы когда:

**Нужна изоляция.** Если один компонент может упасть или зависнуть, он не должен уронить всю систему. Chrome использует отдельный процесс для каждой вкладки — падение одной вкладки не роняет браузер.

**Нужна безопасность.** Процессы изолированы — один процесс не может читать память другого (без явного разрешения). Песочница для недоверенного кода — это отдельный процесс.

**Разные языки/рантаймы.** Микросервисы на разных языках — это разные процессы, общающиеся по сети.

### Используйте потоки когда:

**Нужна общая память.** Потоки эффективно разделяют данные без копирования. Web-сервер с thread-per-request может хранить кэш в памяти, доступный всем потокам.

**Нужно много параллельных задач.** 10,000 потоков создаются за секунды; 10,000 процессов — за минуты.

**Latency критична.** Context switch между потоками быстрее.

### Сравнение

| Критерий | Процессы | Потоки |
|----------|----------|--------|
| Создание (fork/pthread) | ~100-300µs | ~30-50µs |
| Память | Изолирована | Разделяемая |
| Коммуникация | IPC (медленно) | Общая память (быстро) |
| Отказоустойчивость | Высокая (изоляция) | Низкая (один упал — все упали) |
| Context switch | Медленный (TLB flush) | Быстрый |
| Синхронизация | Не нужна (изоляция) | Обязательна (race conditions) |

## Когда НЕ использовать многопоточность

Многопоточность — не серебряная пуля. Иногда она усложняет код без реального выигрыша.

### Ситуации где многопоточность НЕ поможет

**1. CPU-bound задача на одном ядре:**
```
❌ 4 потока на 1 ядре = context switch overhead + те же вычисления
✅ 1 поток = никакого overhead
```
Многопоточность ускоряет CPU-bound задачи только если есть свободные ядра.

**2. I/O-bound задача без параллельных запросов:**
```
❌ 1 поток читает файл → 2 потока не ускорят чтение с одного диска
✅ Async I/O на 1 потоке = меньше памяти, тот же результат
```

**3. Простая последовательная логика:**
```
❌ parse → validate → save: 3 потока с синхронизацией
✅ Последовательный код: проще, быстрее, надёжнее
```

### Признаки неправильного применения

| Симптом | Вероятная проблема |
|---------|-------------------|
| Код сложнее, но не быстрее | Overhead > benefit |
| Постоянные deadlock/race conditions | Архитектура требует пересмотра |
| 1000 потоков, но 4 ядра | Потоки = не решение |
| Lock contention >30% времени | Данные надо разделить иначе |

### Альтернативы многопоточности

| Вместо | Попробуй |
|--------|----------|
| Thread per request | Event loop (Node.js, Netty) |
| Много OS threads | Coroutines / Green threads |
| Shared mutable state | Message passing (Actors, channels) |
| Lock everything | Lock-free structures / immutable data |

## Модели многопоточности

### User-level threads vs Kernel threads

**Kernel threads:** ОС знает о каждом потоке и планирует их. Это то, что используют большинство современных приложений.

**User-level threads:** Потоки управляются библиотекой в user space. ОС видит только один процесс. Преимущество — быстрое создание и переключение (нет syscalls). Недостаток — если один user thread заблокируется на syscall, заблокируются все.

### Модели отображения

```
1:1 (One-to-One)          M:1 (Many-to-One)       M:N (Many-to-Many)

User      Kernel          User      Kernel        User      Kernel
┌─┐  ───  ┌─┐             ┌─┐                     ┌─┐
└─┘       └─┘             ├─┤  ───  ┌─┐           ├─┤  ───  ┌─┐
┌─┐  ───  ┌─┐             ├─┤       └─┘           ├─┤       ├─┤
└─┘       └─┘             ├─┤                     └─┘       └─┘
┌─┐  ───  ┌─┐             └─┘                     ┌─┐  ───  ┌─┐
└─┘       └─┘                                     └─┘       └─┘

Linux, Windows            Green Threads           Go goroutines
Простая модель            Быстро, но блокирует    Баланс
```

**1:1 (Linux, Windows):** Каждый user thread = один kernel thread. Простая модель, но создание потока требует syscall.

**M:1 (Green threads):** Много user threads на одном kernel thread. Очень быстрое создание, но если один поток делает blocking syscall — все ждут.

**M:N (Go goroutines, Erlang):** Много user threads распределяются по нескольким kernel threads. Runtime балансирует нагрузку и обходит blocking syscalls.

## Как это работает в JVM и Kotlin

### JVM Threads = OS Threads (1:1)

В HotSpot JVM каждый `java.lang.Thread` — это отдельный OS thread:

```java
// Создание Java thread = создание OS thread
Thread thread = new Thread(() -> {
    System.out.println("Выполняется в OS thread");
});
thread.start();  // pthread_create() под капотом
```

Поэтому создание 10,000 Java threads = создание 10,000 OS threads = ~80 GB виртуальной памяти под стеки.

### Kotlin Coroutines = User-level concurrency

Coroutines — это M:N модель:

```kotlin
// 100,000 coroutines на нескольких OS threads
runBlocking {
    repeat(100_000) {
        launch {
            delay(1000)  // Не блокирует OS thread!
            println("Coroutine $it")
        }
    }
}
```

Coroutine — это просто объект в heap (~несколько сотен байт). Dispatchers распределяют coroutines по пулу OS threads. `delay()` не блокирует OS thread — coroutine приостанавливается, и thread выполняет другие coroutines.

Подробнее — в [[kotlin-coroutines]].

## Кто использует и реальные примеры

### Браузеры: Process-per-tab

| Браузер | Архитектура | Почему так |
|---------|-------------|------------|
| **Google Chrome** | Каждая вкладка = отдельный процесс | Безопасность + изоляция: если вкладка упадёт, другие не пострадают. 10 вкладок = 10 процессов |
| **Mozilla Firefox** | 4 процесса контента + дополнительные по необходимости | Баланс между памятью и изоляцией. Первые 4 вкладки = 4 процесса, остальные используют потоки внутри |

**Почему Chrome использует процессы, а не потоки:**
- Вкладка с вредоносным JS не может прочитать память другой вкладки (процессы изолированы)
- Зависшая вкладка не вешает весь браузер
- Можно убить один процесс без потери других вкладок

**Цена:** Chrome потребляет больше RAM, чем Firefox (каждый процесс имеет overhead ~30-50MB).

### Базы данных: Processes vs Threads

| СУБД | Модель | Характеристики |
|------|--------|----------------|
| **PostgreSQL** | Process-per-connection | Каждое подключение = fork() нового процесса. Надёжно, но ~100KB overhead на соединение. Лучше для <500 соединений |
| **MySQL** | Thread-per-connection | Каждое подключение = pthread. Быстрее, меньше памяти. Может обрабатывать 500+ threads |
| **Oracle** | Shared Server (M:N) | Пул процессов обслуживает много соединений. Масштабируется лучше |

**Практический совет:**
- PostgreSQL: используйте connection pooling (PgBouncer) при >100 соединениях
- MySQL: настройте thread_pool для high concurrency

### Модели многопоточности в языках

| Язык/Платформа | Модель | Thread overhead | Max practical threads |
|----------------|--------|-----------------|----------------------|
| **Java (HotSpot)** | 1:1 | ~8MB stack | ~10K |
| **Java (Loom)** | M:N virtual threads | ~несколько KB | ~1M+ |
| **Go** | M:N goroutines | ~2KB | ~1M+ |
| **Kotlin** | M:N coroutines | ~несколько сотен байт | ~1M+ |
| **Python** | 1:1 + GIL | ~8MB stack | ~100-1000 (GIL ограничивает) |
| **Node.js** | Single-threaded event loop | 0 (1 thread) | N/A (async) |

**Benchmark примеры:**
- Создание OS thread (pthread): ~20-50µs
- Создание goroutine: ~1-20µs (в 10-50 раз быстрее)
- Переключение OS threads: ~5-10µs
- Переключение goroutines: ~несколько наносекунд - 5µs

### Когда что выбирать (практические правила)

| Сценарий | Рекомендация | Почему |
|----------|--------------|--------|
| Веб-сервер с 10K+ соединений | Event loop (Node.js) или coroutines (Kotlin/Go) | Меньше overhead на context switch |
| CPU-bound вычисления | Processes или threads = числу ядер | Больше потоков = только overhead |
| Изоляция ненадёжного кода | Отдельный процесс (sandbox) | Процессы изолированы, потоки — нет |
| Микросервисы разных команд | Отдельные процессы (контейнеры) | Независимый деплой, разные языки |
| Параллельная обработка данных | Thread pool фиксированного размера | Контроль над parallelism |

## Связь с другими темами

**[[os-overview]]** — Обзор операционных систем закладывает фундамент для понимания процессов и потоков: концепции kernel mode vs user mode объясняют, почему системные вызовы fork() и pthread_create() имеют ощутимый overhead — каждый из них требует перехода в пространство ядра. Понимание архитектуры ОС (монолитное ядро vs микроядро) определяет, как именно процессы и потоки обслуживаются: в монолитном ядре Linux все операции происходят в едином адресном пространстве ядра, а в микроядре Minix — через message passing между серверными процессами. Без этого контекста невозможно понять, почему context switch стоит 2-5 µs и куда уходит это время.

**[[os-scheduling]]** — Планирование CPU неразрывно связано с процессами и потоками, поскольку именно планировщик определяет, какой поток получит процессорное время и на какой срок. CFS в Linux работает с task_struct — той же структурой, которая представляет поток в ядре, — распределяя vruntime пропорционально весам nice-значений. Понимание scheduling объясняет, почему создание 1000 потоков на 4 ядрах приводит к деградации: overhead от context switch и конкуренции за кэш-линии перевешивает выигрыш от параллелизма. Также планирование определяет поведение приоритетов: real-time потоки (SCHED_FIFO) могут вытеснять обычные потоки CFS.

**[[os-memory-management]]** — Управление памятью непосредственно определяет структуру адресного пространства процесса: сегменты text, data, heap, stack, а также механику page tables для трансляции виртуальных адресов в физические. Именно memory management объясняет, почему fork() использует Copy-on-Write: вместо дублирования всех страниц ядро помечает их read-only и копирует только при записи, что делает fork() почти мгновенным для read-heavy процессов. Каждый поток получает собственный стек (обычно 8 MB виртуальной памяти), и это напрямую влияет на лимит потоков: 10 000 потоков потребуют 80 GB виртуального адресного пространства. Понимание TLB и page table switching также объясняет, почему переключение между процессами дороже, чем между потоками одного процесса.

**[[os-synchronization]]** — Синхронизация является прямым следствием многопоточности: когда несколько потоков разделяют адресное пространство процесса, возникают race conditions, требующие mutex, semaphore и других примитивов. Без корректной синхронизации даже простая операция counter++ превращается в data race, поскольку состоит из трёх машинных инструкций (load, increment, store), между которыми может произойти переключение потока. Проблема deadlock — когда два потока блокируют друг друга, захватив ресурсы в разном порядке — является одной из классических опасностей многопоточного программирования. Понимание модели памяти (memory ordering, happens-before) критично для написания корректного lock-free кода.

**[[jvm-concurrency-overview]]** — JVM реализует потоки поверх нативных OS threads через 1:1 модель: каждый java.lang.Thread соответствует одному pthread в Linux или NSThread в macOS. Это означает, что все ограничения ОС — стоимость context switch, размер стека, лимиты на количество потоков — напрямую влияют на Java-приложения. Создание 10 000 Java threads приводит к потреблению ~80 GB виртуальной памяти (8 MB стек × 10K) и тысячам context switches в секунду. JVM добавляет собственный overhead: safepoint polling, GC stop-the-world паузы блокируют все потоки, а synchronized/volatile реализуются через CAS-инструкции и memory barriers процессора. Проект Loom (virtual threads) в Java 21+ переходит к модели M:N, аналогичной корутинам.

**[[kotlin-coroutines]]** — Корутины Kotlin реализуют M:N модель потоков: множество легковесных корутин мультиплексируются на небольшой пул OS threads через Dispatchers. В отличие от OS threads, корутина не привязана к конкретному потоку и может приостановиться (suspend) без блокировки нижележащего потока — вместо блокирующего системного вызова она сохраняет continuation и освобождает поток для другой работы. Это решает проблему масштабируемости: если 10 000 OS threads потребляют 80 GB памяти и генерируют тысячи context switches, то 10 000 корутин работают на пуле из 4-8 потоков с минимальным overhead (~200 bytes на корутину). Понимание OS threads необходимо для отладки корутин: structured concurrency, dispatchers и их взаимодействие с ОС определяют реальную производительность.

## Проверь себя

> [!question]- Почему PostgreSQL использует модель process-per-connection, а MySQL — thread-per-connection, и в каких сценариях каждый подход выигрывает?
> PostgreSQL выбрал процессы ради изоляции: падение одного соединения не может повредить общую память другого. MySQL выбрал потоки ради производительности: меньше overhead на создание и переключение, общий кэш запросов. PostgreSQL выигрывает в надёжности (длинные транзакции, расширения на C), MySQL — в throughput при тысячах коротких соединений. Поэтому PostgreSQL нуждается в connection pooling (PgBouncer) раньше, чем MySQL.

> [!question]- При каких условиях M:N модель потоков (goroutines, coroutines) деградирует до уровня обычных OS threads, и как этого избежать?
> M:N деградирует, когда goroutines выполняют блокирующие syscalls или CGO-вызовы: каждый такой вызов блокирует целый OS thread из пула, и при достаточном количестве блокирующих вызовов все OS threads заняты — планировщик не может запускать другие goroutines. Избежать можно через неблокирующие API, worker pool для блокирующих операций и увеличение GOMAXPROCS для CGO-heavy кода.

> [!question]- Сравните стоимость context switch между потоками одного процесса и между процессами. Почему разница не в 10 раз, а всего в ~2 раза?
> Основная разница — TLB flush при переключении процессов (смена CR3 регистра). Но благодаря PCID (Process Context ID) на современных CPU полный TLB flush не нужен — записи тегируются. Поэтому реальная разница ~1.2-1.5µs vs ~2.2µs. Основной скрытый cost не в самом switch, а в последующих TLB miss и cache miss при "прогреве" кэшей нового процесса.

> [!question]- Почему zombie процессы опаснее на практике, чем orphan процессы, хотя zombie не потребляют ни CPU, ни память?
> Zombie занимают запись в таблице процессов ядра, которая конечна (обычно 32768 по умолчанию). Если программа порождает тысячи дочерних процессов без wait() — PID пространство исчерпается, и система не сможет создавать новые процессы. Orphan процессы безопасны — init (PID 1) автоматически "усыновляет" их и вызывает wait() при завершении.

## Ключевые карточки

Чем поток отличается от процесса?
?
Поток — единица выполнения внутри процесса, разделяющая heap и ресурсы с другими потоками. Процесс — изолированная программа с собственным адресным пространством. Потоки дешевле (~50µs создание, ~2MB stack), но один упавший поток роняет весь процесс.

Что делает fork() в Unix?
?
fork() создаёт точную копию процесса через системный вызов clone(). Использует Copy-on-Write — реальное копирование страниц памяти происходит только при записи. Родитель получает PID ребёнка, ребёнок получает 0.

Что такое Copy-on-Write (COW)?
?
Оптимизация fork(): вместо копирования памяти обе копии разделяют одни страницы (помечены read-only). При попытке записи происходит page fault, и ядро копирует только изменённую страницу. Экономит время и память.

Чем модель 1:1 отличается от M:N для потоков?
?
1:1 (Linux pthreads): один user thread = один kernel thread. Надёжно, но дорого (~8MB stack). M:N (Go goroutines, Kotlin coroutines): много user threads на несколько kernel threads. Дёшево (~2-8KB), масштабируется до миллионов, но блокирующий syscall блокирует весь OS thread.

Что такое zombie процесс и как его предотвратить?
?
Zombie — завершённый процесс, чей родитель не вызвал wait(). Занимает запись в таблице процессов. Предотвращение: вызывать wait()/waitpid(), установить signal(SIGCHLD, SIG_IGN), или использовать двойной fork.

Почему context switch между потоками быстрее, чем между процессами?
?
При переключении между процессами нужно сменить page table (CR3), сбросить TLB и прогреть кэши. Между потоками одного процесса адресное пространство общее — TLB остаётся валидным. Разница: ~1.2-1.5µs (потоки) vs ~2.2+µs (процессы).

Когда лучше использовать процессы вместо потоков?
?
Процессы нужны для изоляции и безопасности: sandbox для недоверенного кода, отдельные вкладки в Chrome, микросервисы на разных языках. Потоки — когда нужна общая память, высокая скорость создания и низкая latency context switch.

Какой overhead у goroutine по сравнению с OS thread?
?
Goroutine: ~2-8KB памяти, ~0.3µs создание. OS thread: ~1-8MB stack, ~30-50µs создание. Разница в 1000x по памяти и 100x по скорости создания. Это позволяет запускать миллион горутин vs ~10K OS threads.
