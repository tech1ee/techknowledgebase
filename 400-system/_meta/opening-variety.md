# Вариативность структуры: альтернативы шаблонам

> Проблема: TL;DR, "крючок", фиксированная структура — сами становятся клише.
> Решение: 4 типа открытий по цели читателя (не 8 — paradox of choice).

---

## Ключевой принцип

**Не "какой шаблон использовать", а "что нужно читателю в этой конкретной заметке".**

> **Научная база:** Paradox of Choice (Schwartz, 2004) — 8 вариантов создают cognitive overload. 4 варианта — оптимум для выбора.

---

## 4 типа открытий по цели читателя

> **Правило выбора:** Определи цель читателя → выбери соответствующий тип.

### Тип 1: "Что это?" → Прямой ответ

**Когда использовать:** Читатель слышал термин и хочет понять суть.

```
Вопрос: Как работает GC в JVM?

Открытие:
"GC находит объекты без ссылок и удаляет их.
Всё остальное — оптимизации этого процесса."
```

**Варианты:**
- Краткое определение + ключевой принцип
- Факт/цифра, которая объясняет суть

### Тип 2: "Зачем мне это?" → Проблема → Решение

**Когда использовать:** Читатель не уверен, нужна ли ему эта тема.

```
"Деплой в 2010: у тебя Python 3, у клиента Python 2.
Через неделю — 'у меня работает'.
Docker появился чтобы это убить."
```

**Варианты:**
- Боль → решение
- История проблемы → как решили
- Личный опыт → урок

### Тип 3: "Как сделать X?" → Результат сразу

**Когда использовать:** Читатель пришёл за практическим руководством.

```
"К концу статьи у тебя будет работающий CI/CD пайплайн.
Время: 30 минут. Требования: GitHub аккаунт."
```

**Варианты:**
- Что получишь + сколько времени
- Пошаговый план в 3 строки
- "Сделай X, получи Y"

### Тип 4: "A или B?" → Контраст

**Когда использовать:** Читатель выбирает между вариантами.

```
"Все говорят: микросервисы — будущее.
Но 29% компаний вернулись к монолиту.
Разберём когда что выбирать."
```

**Варианты:**
- Противопоставление + почему так
- Таблица сравнения в 3 строки
- "X если..., Y если..."

## Варианты структуры (вместо одинаковых разделов)

### По цели читателя

| Цель | Структура |
|------|-----------|
| "Что это?" | Проблема → Решение → Как работает |
| "Как сделать?" | Результат → Шаги → Проверка |
| "Почему так?" | История → Эволюция → Текущее состояние |
| "A или B?" | Критерии → Сравнение → Решение |
| "Что внутри?" | Обзор → Детали → Связи |

### Гибкие блоки

Вместо фиксированных разделов — набор блоков, которые комбинируются:

```
БЛОКИ (выбирать нужные):

□ Суть (1-2 предложения) — почти всегда
□ Контекст/история — когда важно "почему так"
□ Визуальная схема — для сложных концепций
□ Код с комментариями — для технических тем
□ Пример из практики — когда абстрактно
□ Подводные камни — когда есть неочевидные проблемы
□ Ограничения — когда важен контекст применимости
□ Сравнение с X — когда есть альтернативы
□ Дерево решений — когда выбор зависит от контекста
□ Частые вопросы — когда много недопониманий
□ Связи — когда тема часть большей картины
```

## Анти-шаблоны (что делает контент мёртвым)

### 1. Одинаковая структура везде
```
❌ Каждая заметка: TL;DR → Что это → Как работает → Примеры → Связи

✅ Структура адаптируется под контент:
   - Tutorial: сразу к делу, шаги, результат
   - Deep dive: история, контекст, потом детали
   - Reference: таблица, примеры, ctrl+f
```

### 2. Механические открытия
```
❌ "TL;DR: X делает Y."
   (каждый раз одинаково)

✅ Варьировать:
   - Иногда: прямой ответ
   - Иногда: вопрос
   - Иногда: история
   - Иногда: факт
```

### 3. Обязательные разделы ради структуры
```
❌ "Связи: [[X]]" (потому что "надо")

✅ Связи только когда они добавляют ценность.
   Если заметка самодостаточна — связи не нужны.
```

### 4. Дешёвые трюки
```
❌ "Вы не поверите что случилось дальше..."
❌ "Секрет, о котором молчат..."
❌ "Это изменит вашу жизнь..."

✅ Обещай только то, что даёшь.
   Если факт удивительный — он удивит сам.
   Не нужно говорить "удивительный факт".
```

## Как выбирать структуру

```
1. Кто читатель? Что он уже знает?

2. Зачем он здесь? Что хочет получить?
   - Ответ на вопрос → прямой ответ в начале
   - Научиться делать → шаги с результатом
   - Понять глубже → история и контекст
   - Выбрать → сравнение и критерии

3. Что уникального в этой теме?
   - Есть интересная история → начни с неё
   - Есть неочевидный факт → начни с него
   - Есть частая ошибка → начни с неё
   - Ничего особенного → просто дай ответ

4. Какие блоки реально нужны?
   - Не добавляй раздел потому что "так принято"
   - Добавляй только если он даёт ценность
```

## Примеры: одна тема — 4 типа открытий

**Тема: Docker**

### Тип 1: "Что это?" → Прямой ответ
```
Docker упаковывает приложение со всеми зависимостями в контейнер.
Контейнер работает одинаково везде — на твоём маке, на сервере, в облаке.
Docker-образ: 15MB. Виртуалка: 1.5GB. Разница в 100 раз.
```

### Тип 2: "Зачем мне это?" → Проблема → Решение
```
2010 год. Ты деплоишь на сервер клиента.
У тебя Python 3.8, PostgreSQL 12, Ubuntu 20.
У него Python 2.7, PostgreSQL 9, CentOS 6.
Docker это убил — "у меня работает" больше не проблема.
```

### Тип 3: "Как сделать X?" → Результат сразу
```
К концу статьи у тебя будет работающий Docker-контейнер.
Время: 15 минут. Требования: Docker Desktop установлен.
3 команды: build → run → push.
```

### Тип 4: "A или B?" → Контраст
```
Docker vs VM: что выбрать?
Docker: старт за секунды, 15MB, shared kernel.
VM: старт за минуты, 1.5GB, полная изоляция.
Docker для dev/prod consistency, VM для security isolation.
```

## Чеклист вариативности

> 6 пунктов — в пределах эффективности (Gawande, 2009)

```
□ Определил цель читателя (что/зачем/как/выбор)
□ Выбрал соответствующий тип открытия из 4
□ Структура соответствует цели, не шаблону
□ Открытие отличается от последних 3 заметок
□ Нет механических элементов "для галочки"
□ Каждый раздел добавляет ценность (не "так принято")
```

### Дерево решений

```
Цель читателя:
    │
    ├─► "Что это такое?"
    │       └─► Тип 1: Прямой ответ
    │
    ├─► "Зачем мне это нужно?"
    │       └─► Тип 2: Проблема → Решение
    │
    ├─► "Как это сделать?"
    │       └─► Тип 3: Результат сразу
    │
    └─► "Что выбрать: A или B?"
            └─► Тип 4: Контраст
```
