# Communication Styles: понимай других, адаптируй себя

> **TL;DR:** DISC делит людей на 4 стиля по двум осям: task/people и fast/slow pace. D (Dominance) хочет результаты, I (Influence) — признание, S (Steadiness) — стабильность, C (Conscientiousness) — точность. Ключевая идея: адаптируй свой стиль к собеседнику — говори на его языке. Применяется в: 1-on-1, stakeholder management, team collaboration, code review.

---

## Зачем это нужно?

### Представьте ситуацию

Ты подходишь к коллеге с идеей: "Привет! Хочу обсудить новый подход к архитектуре. Это классная возможность, команде понравится!" Коллега холодно: "Какие данные? Какие риски? Давай конкретику." Ты расстроен — "он всегда такой сухой". Но дело не в нём — ты I-style (influence), он C-style (conscientiousness). Ты продаёшь энтузиазмом, он покупает данными.

**Без понимания стилей:**
- "Он меня не понимает" / "Она слишком эмоциональная"
- Постоянные трения с определёнными людьми
- Неудачные pitch'и идей
- Frustration от "неправильных" реакций

**С пониманием стилей:**
- Адаптируешь message под audience
- Знаешь что важно каждому
- Эффективнее влияешь
- Меньше личных конфликтов

## Терминология

| Термин | Что это | IT-аналогия |
|--------|---------|-------------|
| **DISC** | Модель 4 поведенческих стилей (Dominance, Influence, Steadiness, Conscientiousness) | Как 4 разных API — один результат, разные интерфейсы |
| **D-style** | Доминирующий: прямой, результат-ориентированный | Как CLI — быстро, без лишних слов |
| **I-style** | Влияющий: энергичный, people-oriented | Как GUI — визуальный, интерактивный, engaging |
| **S-style** | Стабильный: терпеливый, supportive | Как reliable service — consistent, predictable |
| **C-style** | Добросовестный: точный, detail-oriented | Как static analysis — проверяет всё, требует точности |
| **Versatility** | Способность адаптировать стиль | Как adapter pattern — работает с разными interfaces |

## Пошаговый процесс

### Как определить стиль человека

**Шаг 1: Наблюдай за поведением**

| Наблюдение | Вероятный стиль |
|------------|-----------------|
| Говорит быстро, к делу | D или I |
| Говорит медленно, обдуманно | S или C |
| Много о людях и feelings | I или S |
| Много о данных и фактах | D или C |

**Шаг 2: Обрати внимание на email стиль**

```
D-style email:
"Нужен статус по проекту. Когда будет готово?"
[3 строки, no greeting]

I-style email:
"Привет! Надеюсь хорошо провёл выходные!
Хотел спросить как дела с проектом — уверен, идёт
отлично! Будет круто увидеть результат! 😊"

S-style email:
"Привет, надеюсь не отвлекаю. Когда будет удобно,
можешь рассказать как идёт проект? Без спешки,
просто хочу быть в курсе если могу чем-то помочь."

C-style email:
"Добрый день. Прошу предоставить статус по проекту:
1. Процент выполнения
2. Риски и blockers
3. Planned vs actual timeline
Приложи документацию если есть."
```

**Шаг 3: Задай диагностические вопросы**

| Вопрос | D отвечает | I отвечает | S отвечает | C отвечает |
|--------|------------|------------|------------|------------|
| "Как тебе идея X?" | "Какой ROI?" | "Классно! Команде понравится!" | "Как это повлияет на людей?" | "Какие данные? Риски?" |

### Как адаптировать свой стиль

**К D-style:**
```
ДО (если ты I/S):
"Привет! У меня классная идея, думаю команде понравится!
Мы могли бы попробовать новый подход к деплойментам..."

ПОСЛЕ:
"Есть предложение сократить deploy time на 40%.
Нужно: 2 недели работы одного dev. ROI: экономия
10 часов/неделю. Одобряешь?"
```

**К I-style:**
```
ДО (если ты D/C):
"Проект задерживается на 3 дня. Причины:
1. Интеграция сложнее expected
2. Dependency от внешнего API"

ПОСЛЕ:
"Хочу поделиться статусом — есть challenge, но
мы справимся! Интеграция интереснее чем думали,
учимся новому. Сдвинем на 3 дня — зато качество
будет отличное! Команда мотивирована."
```

**К S-style:**
```
ДО (если ты D):
"Переходим на новую систему с понедельника.
Разберётесь по ходу."

ПОСЛЕ:
"Планируем перейти на новую систему. Хочу услышать
твоё мнение — как это повлияет на твою работу?
Какая поддержка нужна? Можем обсудить timeline."
```

**К C-style:**
```
ДО (если ты I):
"Это будет круто! Давай попробуем, разберёмся
по ходу!"

ПОСЛЕ:
"Вот proposal с данными:
- Текущие метрики: [данные]
- Expected improvement: [цифры]
- Риски и mitigation: [список]
- Timeline с milestones: [план]
Что думаешь? Какие gaps в анализе?"
```

## Скрипты и Templates

### Use Case 1: Pitch идеи разным стилям

**Одна идея — 4 подачи:**

**Идея:** Внедрить feature flags

**Pitch для D:**
```
"Feature flags сократят время релиза на 50%.
Внедрение: 1 sprint. Решение: LaunchDarkly.
Одобряешь бюджет?"
```

**Pitch для I:**
```
"Представь: команда сможет релизить фичи без страха!
Feature flags дадут нам flexibility и confidence.
Другие команды уже используют — отзывы отличные.
Давай попробуем?"
```

**Pitch для S:**
```
"Хочу обсудить идею — feature flags. Это поможет
команде чувствовать себя увереннее при релизах.
Изменения будут постепенными, с поддержкой.
Как думаешь, это подойдёт нашей команде?"
```

**Pitch для C:**
```
"Предлагаю feature flags. Вот анализ:
- Сравнение 3 решений: [таблица]
- Риски и mitigation: [документ]
- POC план: [timeline]
- Метрики успеха: [KPIs]
Готов обсудить детали."
```

### Use Case 2: Code Review комментарии

**Один issue — 4 формулировки:**

**Issue:** Неэффективный SQL запрос

**Для D:**
```
"N+1 query. Замени на batch load. Performance issue."
```

**Для I:**
```
"Классный прогресс! Заметил одну штуку — этот query
может создать проблемы на scale. Давай вместе
посмотрим как оптимизировать?"
```

**Для S:**
```
"Хороший код! Есть момент который стоит обсудить —
этот запрос может замедлиться при росте данных.
Когда удобно — могу показать альтернативный подход?"
```

**Для C:**
```
"Query complexity: O(n) вместо O(1).
При 10K records: ~500ms vs ~5ms.
Рекомендация: batch loading (см. паттерн в ServiceX).
Готов обсудить implementation details."
```

### Use Case 3: 1-on-1 feedback

**Одна проблема — 4 подхода:**

**Проблема:** Опоздания на митинги

**Для D:**
```
"Три последних митинга — опоздания. Это задерживает
команду. Что нужно чтобы приходить вовремя?"
```

**Для I:**
```
"Заметил что последние митинги было сложно начать
вовремя. Понимаю, много всего происходит! Давай
подумаем как организовать время чтобы всё успевать
и быть на связи с командой?"
```

**Для S:**
```
"Хотел обсудить один момент — не как критику,
просто хочу понять. На последних митингах было
сложно начать вовремя. Всё в порядке? Может,
чем-то могу помочь?"
```

**Для C:**
```
"Вот данные по последним 10 митингам: 3 опоздания
от 5 до 10 минут. Это влияет на эффективность
команды — теряем ~15 минут суммарно. Какие
факторы влияют? Давай найдём системное решение."
```

### Use Case 4: Сообщение о проблеме

**Одна проблема — 4 эскалации:**

**Проблема:** Проект задерживается

**Для D-style менеджера:**
```
"Проект задерживается на неделю.
Причина: dependency от Team X.
Варианты: 1) сократить scope, 2) добавить ресурс.
Рекомендую вариант 1. Нужно твоё решение."
```

**Для I-style менеджера:**
```
"Хочу поделиться ситуацией — есть challenge с
проектом, но команда мотивирована решить!
Dependency от Team X заняла больше времени.
Видим два пути — давай обсудим что лучше?"
```

**Для S-style менеджера:**
```
"Есть момент который хочу обсудить — проект
задерживается. Не хочу скрывать — лучше решить
вместе. Команда работает хорошо, просто внешняя
зависимость оказалась сложнее. Какие thoughts?"
```

**Для C-style менеджера:**
```
"Статус проекта:
- Original deadline: [дата]
- New projected: [дата + 7 дней]
- Root cause: [детальный анализ]
- Risk assessment: [матрица]
- Mitigation options: [3 варианта с pros/cons]
Готов обсудить в удобное время."
```

## Распространённые ошибки

### Ошибка 1: Stereotyping

**Неправильно:**
"Он D-style, значит ему плевать на людей"

**Почему:**
DISC — про поведение, не про ценности. D может deeply care о команде, просто выражает иначе.

**Правильно:**
DISC — инструмент для адаптации коммуникации, не для судить людей.

### Ошибка 2: Оправдание своего стиля

**Неправильно:**
"Я C-style, поэтому мне нужна неделя на анализ" (когда нужно решение сегодня)

**Правильно:**
Понимать свой стиль = знать свои tendencies и компенсировать когда нужно.

### Ошибка 3: Не адаптироваться

**Неправильно:**
Всегда общаться в своём стиле независимо от audience.

**Правильно:**
Versatility — способность shift стиль под ситуацию и человека.

### Ошибка 4: Overcorrection

**Неправильно:**
Полностью менять себя до неузнаваемости.

**Правильно:**
Адаптация — это adjustment, не перерождение. Core message остаётся, меняется delivery.

## Когда использовать / НЕ использовать

### Используй style adaptation, когда:

| Ситуация | Почему важно |
|----------|--------------|
| **Pitch stakeholder'у** | Говоришь на его языке |
| **1-on-1 с direct report** | Персонализированный подход |
| **Cross-team collaboration** | Меньше friction |
| **Conflict resolution** | Понимаешь underlying needs |

### Ограничения:

| Ситуация | Что учесть |
|----------|------------|
| **Crisis** | Нет времени на адаптацию — будь direct |
| **Не знаешь человека** | Начни neutral, observe, adjust |
| **Группа с разными стилями** | Balance approach |

## Практика

### Упражнение 1: Style Identification

**Определи стили:**

1. "Пришли мне TL;DR. Что решаем?"
2. "Это классная идея! Команде понравится! Когда начинаем?"
3. "Можешь подробнее объяснить? Какие данные это подтверждают?"
4. "Как это повлияет на team workload? Не хочу чтобы люди выгорели."

<details><summary>Ответы</summary>

1. D — direct, results-focused
2. I — enthusiastic, people-focused
3. C — analytical, data-focused
4. S — caring, harmony-focused
</details>

### Упражнение 2: Message Adaptation

**Задание:**
Адаптируй это сообщение для каждого стиля:

Original: "Нужно внедрить новую систему мониторинга"

<details><summary>Ответы</summary>

**Для D:**
"Новый мониторинг сократит MTTR на 60%. Нужно $X и 2 недели. Одобряешь?"

**Для I:**
"Есть идея которая сделает жизнь команды легче — новый мониторинг! Будем быстрее находить проблемы. DevRel рекомендуют. Давай обсудим?"

**Для S:**
"Хочу предложить новую систему мониторинга. Она поможет команде меньше стрессовать при инцидентах. Внедрим постепенно, с поддержкой. Что думаешь?"

**Для C:**
"Proposal: новая система мониторинга.
- Сравнение 3 решений: [таблица]
- ROI analysis: [цифры]
- Implementation plan: [timeline]
- Risks: [assessment]
Готов обсудить детали."
</details>

### Упражнение 3: Self-Assessment

**Вопросы для определения своего стиля:**

1. Когда принимаю решения, я обычно:
   - a) Быстро, на основе gut feeling
   - b) Консультируюсь с другими
   - c) Анализирую данные детально
   - d) Даю время на обдумывание

2. В meetings я обычно:
   - a) Веду, направляю, решаю
   - b) Генерирую идеи, энергизирую
   - c) Слушаю, поддерживаю
   - d) Задаю уточняющие вопросы

3. Мой идеальный email:
   - a) Короткий, по делу
   - b) Friendly, с emoji
   - c) Тёплый, заботливый
   - d) Структурированный, детальный

### Ежедневная практика

| День | Фокус | Действие |
|------|-------|----------|
| Пн | Observe | Определи стиль 2-3 коллег |
| Вт | Self-aware | В каком стиле ты сегодня общался? |
| Ср | Adapt | Намеренно адаптируй один message |
| Чт | Ask | Спроси коллегу как ему комфортнее общаться |
| Пт | Reflect | Какие adaptations сработали? |
