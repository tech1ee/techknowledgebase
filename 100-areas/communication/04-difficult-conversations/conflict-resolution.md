# Conflict Resolution для IT-специалистов

> **TL;DR:** Конфликт — не зло, а источник лучших решений. 5 стилей (Thomas-Kilmann): Competing, Collaborating, Compromising, Avoiding, Accommodating — каждый уместен в своей ситуации. Ключ: фокус на интересах, не позициях + "disagree and commit" после принятия решения.

---

## Зачем это нужно?

### Представьте ситуацию

**Architecture review.** Два Senior Developer'а спорят о подходе: microservices vs monolith.

**Без навыка:**
```
Dev A: "Microservices — единственный правильный подход для масштабирования."
Dev B: "Это over-engineering. Monolith проще и быстрее."
Dev A: "Ты просто не понимаешь distributed systems."
Dev B: "А ты гонишься за хайпом."
...
Результат: Личный конфликт, решение не принято, проект застрял.
Или: Tech Lead принимает решение авторитарно → один из devs демотивирован.
```

**С навыком:**
```
Facilitator: "Давайте разделим проблему и людей. Оба подхода имеют
              право на жизнь. Какие ИНТЕРЕСЫ за каждой позицией?"

Dev A: "Хочу чтобы система масштабировалась когда вырастем в 10x."
Dev B: "Хочу чтобы мы выпустили MVP за 3 месяца, а не за год."

Facilitator: "То есть оба хотят успех продукта — один фокусируется
              на долгосрочном scaling, другой на time-to-market.
              Какие варианты учитывают оба интереса?"

Dev A: "Можно начать с modular monolith с чёткими boundaries..."
Dev B: "...который потом можно будет разделить если понадобится."

Решение: Modular monolith с defined interfaces для возможного split.
Оба: "Disagree and commit" — даже если не идеально, двигаемся вперёд.
```

## Для кого этот материал

| Уровень | Подходит? | Рекомендация |
|---------|-----------|--------------|
| **Junior** | Полезен | Понять свой стиль, не избегать конфликтов |
| **Middle** | Важен | Tech disagreements требуют навыка |
| **Senior** | Критичен | Architecture debates, cross-team conflicts |
| **Tech Lead** | Обязателен | Разрешение конфликтов в команде — твоя работа |

## Терминология

| Термин | Что это (простыми словами) | IT-аналогия |
|--------|---------------------------|-------------|
| **Conflict Mode** | Стиль поведения в конфликте | Default behavior pattern |
| **Competing** | Отстаиваю свою позицию любой ценой | Force push |
| **Collaborating** | Ищем решение выгодное всем | Pair programming |
| **Compromising** | Каждый уступает частично | Merge conflict resolution |
| **Avoiding** | Игнорирую конфликт | git stash |
| **Accommodating** | Уступаю в пользу другого | Accept all incoming changes |
| **Disagree and commit** | Не согласен, но поддерживаю решение | Approved PR despite concerns |
| **Escalation** | Передача конфликта на уровень выше | Raise to manager |

## Как это работает?

### Thomas-Kilmann Conflict Mode Instrument (TKI)

Модель 1974 года, до сих пор стандарт в corporate training. Описывает 5 стилей поведения по двум осям: **assertiveness** (насколько защищаешь свои интересы) и **cooperativeness** (насколько учитываешь интересы другого).

```
┌──────────────────────────────────────────────────────────────────┐
│                THOMAS-KILMANN MODEL                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   HIGH    │  COMPETING        │  COLLABORATING                  │
│           │  ▓▓▓▓▓▓▓▓▓        │  ▓▓▓▓▓▓▓▓▓▓▓▓                   │
│ ASSERTIVE │  "Я выиграю"      │  "Мы оба выиграем"              │
│ (свои     │                   │                                  │
│ интересы) │      ─────────────┼───────────────                   │
│           │        COMPROMISING                                  │
│           │        ▓▓▓▓▓▓▓▓▓▓▓                                  │
│           │        "Поделим"                                     │
│           │      ─────────────┼───────────────                   │
│   LOW     │  AVOIDING         │  ACCOMMODATING                   │
│           │  ▓▓▓▓▓▓▓          │  ▓▓▓▓▓▓▓▓▓▓                     │
│           │  "Не сейчас"      │  "Как скажешь"                   │
│           │                   │                                  │
│           └───────────────────┴──────────────────────────────── │
│                 LOW                    HIGH                      │
│                   COOPERATIVE (интересы другого)                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5 стилей подробно

---

#### 1. COMPETING (Конкуренция)

**Характеристика:** High assertive, Low cooperative. "Моя позиция победит."

```
IT-ПРИМЕРЫ:
├── Security engineer настаивает на стандартах несмотря на deadline pressure
├── Tech Lead принимает окончательное решение после долгой дискуссии
└── Архитектор отклоняет PR с критичной уязвимостью

КОГДА ИСПОЛЬЗОВАТЬ:
✓ Критичная ситуация (security breach, production down)
✓ Непопулярное но правильное решение
✓ Защита команды от нереалистичных требований
✓ Когда уверен на 100% и stakes высокие

КОГДА НЕ ИСПОЛЬЗОВАТЬ:
✗ Обычные technical disagreements
✗ Когда отношения важнее результата
✗ Когда не уверен в своей позиции
```

**Скрипт:**
```
"Понимаю альтернативную точку зрения, но в данном случае [причина]
требует именно такого решения. Моё решение: [X]. Готов обсудить
implementation details, но направление — финальное."
```

---

#### 2. COLLABORATING (Сотрудничество)

**Характеристика:** High assertive, High cooperative. "Найдём решение выгодное всем."

```
IT-ПРИМЕРЫ:
├── Architecture brainstorm где интегрируются разные идеи
├── Совместная работа над requirements с PM
└── Retrospective где ищут root cause проблем

КОГДА ИСПОЛЬЗОВАТЬ:
✓ Обе стороны имеют важные insights
✓ Есть время на глубокую дискуссию
✓ Отношения и результат одинаково важны
✓ Сложная проблема требует разных perspectives

КОГДА НЕ ИСПОЛЬЗОВАТЬ:
✗ Срочные решения (production issue)
✗ Тривиальные вопросы (overkill)
✗ Одна сторона не заинтересована в collaboration
```

**Скрипт:**
```
"У нас разные взгляды на проблему. Давай разберём интересы каждого:
- Что для тебя важно в этом решении?
- Какие concerns хочешь чтобы были учтены?
Я хочу [X], потому что [причина].
Какой вариант мог бы учесть и то, и другое?"
```

---

#### 3. COMPROMISING (Компромисс)

**Характеристика:** Medium assertive, Medium cooperative. "Каждый уступает часть."

```
IT-ПРИМЕРЫ:
├── "Ты делаешь этот модуль по-своему, я — свой по-своему"
├── "Берём твой подход, но с моими тестами"
└── Разделение scope между командами

КОГДА ИСПОЛЬЗОВАТЬ:
✓ Collaboration не сработал — нужен quick resolution
✓ Оба варианта примерно одинаково хороши
✓ Временное решение пока нет лучшего
✓ Равные по власти стороны

КОГДА НЕ ИСПОЛЬЗОВАТЬ:
✗ Есть явно лучшее решение
✗ Компромисс создаёт "худшее из двух миров"
✗ Важные принципы под вопросом
```

**Скрипт:**
```
"Похоже мы оба частично правы. Предлагаю компромисс:
берём [часть A] из твоего подхода и [часть B] из моего.
Это не идеально для обоих, но позволяет двигаться вперёд.
Согласен?"
```

---

#### 4. AVOIDING (Избегание)

**Характеристика:** Low assertive, Low cooperative. "Не сейчас."

```
IT-ПРИМЕРЫ:
├── Откладывание bikeshedding discussion
├── "Давай обсудим это после релиза"
└── Игнорирование trolling в code review

КОГДА ИСПОЛЬЗОВАТЬ:
✓ Вопрос тривиален / не стоит времени
✓ Нужно остыть (эмоции высоки)
✓ Больше информации нужно для решения
✓ Другие решат лучше тебя

КОГДА НЕ ИСПОЛЬЗОВАТЬ:
✗ Конфликт будет escalate если не решить
✗ Вопрос важен для команды
✗ Твой input критичен для решения
```

**Скрипт:**
```
"Это важный вопрос, но сейчас не лучшее время для обсуждения.
Предлагаю [вернуться к этому после X / поставить в agenda
на Y / собрать больше данных]."
```

---

#### 5. ACCOMMODATING (Уступка)

**Характеристика:** Low assertive, High cooperative. "Пусть будет по-твоему."

```
IT-ПРИМЕРЫ:
├── Junior уступает Senior'у в техническом споре
├── Принять чужой стиль кода ради consistency
└── "Мне не принципиально, делай как считаешь нужным"

КОГДА ИСПОЛЬЗОВАТЬ:
✓ Вопрос важнее для другого чем для тебя
✓ Ты не прав или недостаточно компетентен
✓ Отношения важнее результата в данном вопросе
✓ Building goodwill для будущего

КОГДА НЕ ИСПОЛЬЗОВАТЬ:
✗ Ты уверен что прав и stakes высокие
✗ Это создаёт плохой прецедент
✗ Другая сторона использует это как манипуляцию
```

**Скрипт:**
```
"Я вижу что это для тебя важно. Мои concerns: [кратко].
Но в данном случае готов довериться твоему judgement.
Давай сделаем по-твоему."
```

### Когда какой стиль использовать

```

│              ВЫБОР СТИЛЯ ПО СИТУАЦИИ                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  COMPETING        когда: Emergency, security, clear authority   │
│  ▓▓▓▓▓▓▓▓▓       пример: "Production down — делаем по моему     │
│                          плану, обсуждения потом."              │
│                                                                  │
│  COLLABORATING    когда: Complex problem, time available        │
│  ▓▓▓▓▓▓▓▓▓▓▓▓    пример: Architecture design session            │
│                                                                  │
│  COMPROMISING     когда: Equal parties, quick resolution needed │

│  AVOIDING         когда: Trivial, need cooldown, bad timing     │

│  ACCOMMODATING    когда: Lower stakes for you, build goodwill   │
│  ▓▓▓▓▓▓▓▓        пример: "Мне не принципиально, делай как       │
│                          считаешь правильным."                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## "Disagree and Commit"

### Суть принципа

Amazon leadership principle: можно спорить до принятия решения, но после — все коммитятся на 100%.

```
┌──────────────────────────────────────────────────────────────────┐
│                 DISAGREE AND COMMIT                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ФАЗА 1: DISAGREE              ФАЗА 2: COMMIT                  │
│   ┌─────────────────────┐       ┌─────────────────────┐         │
│   │ Спорь                │       │ Поддерживай         │         │
│   │ Приводи аргументы    │ ───►  │ Не саботируй        │         │
│   │ Отстаивай позицию    │       │ Помогай успеху      │         │
│   │ Используй данные     │       │ Никаких "I told you │         │
│   └─────────────────────┘       │  so" потом          │         │
│                                  └─────────────────────┘         │
│                                                                  │
│   "Я не согласен с решением X, но раз команда выбрала этот      │
│    путь — я делаю всё для его успеха."                          │
│                                                                  │
│   ─────────────────────────────────────────────────────────────  │
│                                                                  │
│   ВАЖНО: Disagree ≠ быть aggressive                             │
│          Commit ≠ молча страдать                                │
│                                                                  │
│   Если после commit видишь что решение катастрофично —          │
│   обязанность поднять вопрос снова (с новыми данными)           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Когда применять

| Ситуация | Действие |
|----------|----------|
| Architecture decision, ты в меньшинстве | Disagree → Commit |
| Tech stack choice, не твой preferred | Disagree → Commit |
| Process change, тебе не нравится | Disagree → Commit |
| Явно неэтичное или нелегальное решение | Disagree → Escalate (НЕ commit) |
| Решение создаст security vulnerability | Disagree → Escalate |

### Скрипт для Disagree and Commit

```
ДО РЕШЕНИЯ (Disagree):
"Хочу зафиксировать мои concerns:
1. [Concern 1 с обоснованием]
2. [Concern 2 с обоснованием]

Риски которые вижу: [риски]
Альтернатива которую предлагаю: [альтернатива]

Но понимаю что решение за командой/лидом."

ПОСЛЕ РЕШЕНИЯ (Commit):
"Решение принято. Мои concerns остаются, но я полностью
поддерживаю выбранный путь и буду работать на его успех.
Что я могу сделать чтобы помочь?"
```

## Interest-Based Conflict Resolution

### Positions vs Interests

```
┌──────────────────────────────────────────────────────────────────┐
│              POSITIONS vs INTERESTS                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   POSITION (видимое):           INTEREST (скрытое):             │
│   ┌─────────────────────┐       ┌─────────────────────┐         │
│   │ "Хочу микросервисы" │       │ "Хочу масштабируемость"│       │
│   │ "Нужен релиз в      │       │ "Хочу показать progress│       │
│   │  пятницу"           │       │  CEO до его отпуска"  │        │
│   │ "Не буду делать     │       │ "Чувствую что меня    │        │
│   │  этот таск"         │       │  не уважают"          │        │
│   └─────────────────────┘       └─────────────────────┘         │
│                                                                  │
│   Позиция — это ТРЕБОВАНИЕ                                      │
│   Интерес — это ПОЧЕМУ за требованием                           │
│                                                                  │
│   Конфликт позиций: Win-Lose                                    │
│   Конфликт интересов: часто Win-Win возможен                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Как выявить interests

**Задавай вопросы:**
```
"Почему для тебя важно [position]?"
"Что произойдёт если мы сделаем/не сделаем X?"
"Какой результат ты хочешь получить?"
"Какие concerns стоят за этим требованием?"
"Что тебя беспокоит в альтернативном подходе?"
```

**Пример:**

```
Position A: "Нужен TypeScript"
Position B: "Останемся на JavaScript"

Исследуем interests:
"Почему TypeScript важен?" → "Меньше runtime errors, лучше refactoring"
"Почему JavaScript?" → "Быстрее начать, команда не знает TS"

Interests:
A: Надёжность кода, maintainability
B: Скорость разработки, learning curve

Возможные решения учитывающие оба interest:
- Gradual migration: начать с JS, критичные модули на TS
- Training: инвестировать в обучение команды
- Strict JS: JSDoc + strict ESLint как промежуточный шаг
```

## Пошаговый процесс разрешения конфликта

### Шаг 1: Acknowledge and Separate

```
ДЕЙСТВИЯ:
1. Признай что конфликт существует
2. Отдели людей от проблемы
3. Создай безопасную среду для обсуждения

СКРИПТ:
"Вижу что у нас разные взгляды на [тема]. Это нормально и
даже полезно. Давай разберёмся в проблеме — не друг с другом,
а вместе против проблемы."
```

### Шаг 2: Explore Interests

```
ДЕЙСТВИЯ:
1. Спроси о interests каждой стороны
2. Слушай активно (не готовь ответ пока слушаешь)
3. Paraphrase для подтверждения понимания

СКРИПТ:
"Помоги мне понять твою позицию.
Почему [X] важно для тебя?
Какие concerns стоят за этим?"

После ответа:
"Если я правильно понял, тебе важно [paraphrase].
Это верно? Что-то упустил?"
```

### Шаг 3: Generate Options

```
ДЕЙСТВИЯ:
1. Brainstorm без оценки
2. Комбинируй идеи
3. Ищи варианты win-win

СКРИПТ:
"Теперь когда мы понимаем интересы обеих сторон,
давай сгенерируем варианты. Пока без оценки —
любые идеи, даже странные."
```

### Шаг 4: Evaluate and Decide

```
ДЕЙСТВИЯ:
1. Оцени варианты по объективным критериям
2. Выбери оптимальный
3. Если не можете договориться — эскалируй или используй authority

СКРИПТ:
"Давай оценим варианты по критериям:
- Насколько это решает concerns A?
- Насколько это решает concerns B?
- Риски?
- Effort?

По этим критериям [вариант X] выглядит оптимальным.
Согласны двигаться с ним?"
```

### Шаг 5: Commit and Document

```
ДЕЙСТВИЯ:
1. Получи explicit commitment от всех сторон
2. Зафиксируй решение письменно
3. Определи follow-up actions

СКРИПТ:
"Договорились на [решение]. [Имя], ты согласен двигаться
с этим подходом? [Имя], ты?

Отлично. Я зафиксирую в [Jira/email/ADR].
Next steps: [кто что делает]."
```

## Скрипты для типичных IT-конфликтов

### Use Case 1: Technical Disagreement

**Ситуация:** Два Senior'а спорят об архитектуре

**Скрипт (Facilitator):**
```
1. "Давайте сначала определим критерии успеха. Что нам важно
    в этом решении? Performance? Maintainability? Time to market?"

2. "Теперь оценим оба подхода по этим критериям. [Approach A],
    какие плюсы и минусы по каждому критерию? [Approach B]?"

3. "Есть ли вариант который комбинирует сильные стороны обоих?"

4. Если не удаётся договориться:
   "Предлагаю spike: 2 дня на POC обоих подходов, потом
    сравниваем результаты. Или я приму решение как [Lead/Architect]
    на основе представленных аргументов."
```

### Use Case 2: Scope Creep Conflict

**Ситуация:** PM добавляет scope, dev team pushback

**Скрипт (Dev Lead):**
```
1. "Понимаю что эти features важны для клиента. Давай разберёмся
    как это влияет на текущие commitments."

2. "Наш capacity — X points. Сейчас committed — Y. Это добавляет Z.
    Какие варианты у нас есть?"

   Варианты (Iron Triangle):
   A) Добавить scope → сдвинуть deadline
   B) Добавить scope → убрать что-то из текущего sprint
   C) Сохранить deadline → не брать новый scope сейчас

3. "Какой вариант лучше соответствует бизнес-приоритетам?
    Это решение PM/PO, я даю данные для informed decision."
```

### Use Case 3: Code Review Conflict

**Ситуация:** Reviewer блокирует PR, автор не согласен

**Скрипт (Автор):**
```
1. "Спасибо за детальный review. Хочу понять твои concerns глубже.
    Вижу комментарий про [X]. Какие риски ты видишь?"

2. "Мой reasoning был [Y]. Но я слышу что ты concerned о [Z].
    Есть ли вариант который учитывает и то, и другое?"

3. Если не удаётся договориться:
   "Предлагаю позвать [третьего человека] для fresh perspective."

   Или (если time-sensitive):
   "Предлагаю merge с TODO и follow-up task для рефакторинга.
    Так мы разблокируемся и не теряем твой input."
```

### Use Case 4: Interpersonal Conflict

**Ситуация:** Два члена команды не могут работать вместе

**Скрипт (Lead/Manager):**
```
1. Сначала 1-on-1 с каждым:
   "[Имя], расскажи что происходит с твоей стороны.
    Что тебя беспокоит? Что бы ты хотел изменить?"

2. Joint meeting (после индивидуальных):
   "Я поговорил с каждым из вас. Вижу что есть [проблема].
    Цель этой встречи — найти способ эффективно работать вместе.
    Не выяснять кто прав."

3. "Давай договоримся о working agreements:
    - Как будем коммуницировать?
    - Как будем разрешать разногласия?
    - Какие behaviors unacceptable?"

4. "Я проверю как идут дела через 2 недели. Если ситуация
    не улучшится, будем думать о других решениях."
```

## Распространённые ошибки

### Ошибка 1: Avoiding всегда

**Неправильно:**
Игнорировать конфликты в надежде что сами разрешатся.

**Почему это неправильно:**
Конфликты escalate со временем. Маленькое недопонимание → большой конфликт.

**Правильно:**
Address конфликты рано, когда они ещё маленькие.

### Ошибка 2: Competing всегда

**Неправильно:**
Всегда продавливать свою позицию.

**Почему это неправильно:**
Теряешь goodwill, люди перестают делиться идеями, team morale падает.

**Правильно:**
Competing только когда stakes высокие И ты уверен на 100%.

### Ошибка 3: Фокус на позициях

**Неправильно:**
"Я хочу X" vs "Я хочу Y" → торговля.

**Почему это неправильно:**
Не находится win-win, результат suboptimal для всех.

**Правильно:**
Копай в interests: "Почему X важен?" → найди underlying needs.

### Ошибка 4: Не документировать решения

**Неправильно:**
Договорились устно → через месяц "мы так не договаривались".

**Почему это неправильно:**
Memory unreliable. Без документации — разные интерпретации.

**Правильно:**
Всегда фиксируй: email summary, ADR, Jira ticket.

### Ошибка 5: "I told you so"

**Неправильно:**
После commit — саботировать или злорадствовать если решение не сработало.

**Почему это неправильно:**
Разрушает trust, делает будущие disagreements токсичными.

**Правильно:**
Commit полностью. Если решение провалилось — constructive retrospective, не blame.

## Когда использовать

### Используй conflict resolution, когда:

| Ситуация | Стиль | Почему |
|----------|-------|--------|
| Architecture debate | Collaborating | Оба эксперта, нужен best solution |
| Production incident | Competing | Speed критичен, authority needed |
| Tabs vs spaces | Avoiding | Не стоит времени |
| Junior vs Senior на tech | Accommodating (Junior) | Learning opportunity |
| Scope creep | Compromising | Quick resolution needed |

### Escalate, когда:

| Ситуация | Действие |
|----------|----------|
| Этический конфликт | Escalate to management/HR |
| Safety/security risk | Escalate immediately |
| Harassment/discrimination | Escalate to HR |
| Conflict между managers | Escalate to their manager |
| Deadlock после attempts | Escalate for decision |

## Практика

### Упражнение 1: Определи стиль

**Сценарий:** В каждой ситуации определи оптимальный TKI стиль.

1. Production down, ты on-call engineer
2. Обсуждение naming convention для нового проекта
3. Коллега хочет использовать его подход, тебе не принципиально
4. Architecture decision с долгосрочными последствиями
5. Конфликт с тем же человеком третий раз подряд

**Эталонный ответ:**
<details><summary>Показать</summary>

1. **Competing** — emergency, speed критичен
2. **Avoiding или Compromising** — low stakes, bikeshedding
3. **Accommodating** — тебе не принципиально, build goodwill
4. **Collaborating** — high stakes, нужен best solution
5. **Collaborating с элементом Competing** — pattern требует addressing, не avoiding

</details>

### Упражнение 2: Найди interests

**Сценарий:** Developer говорит "Я не буду делать этот таск."

**Задача:** Какие могут быть interests за этой позицией?

**Эталонный ответ:**
<details><summary>Показать</summary>

**Возможные interests:**
1. Таск technically невозможен / плохо defined
2. Не соответствует skill set → страх failure
3. Перегрузка → burnout risk
4. Считает таск бессмысленным → lack of purpose
5. Конфликт с другим членом команды на этом таске
6. Не чувствует ownership / не был вовлечён в planning

**Как выяснить:**
"Расскажи больше. Что конкретно беспокоит в этом таске?"

</details>

### Ежедневная практика

| День | Упражнение | Контекст |
|------|------------|----------|
| 1 | Определи свой default TKI стиль | Рефлексия |
| 2 | Используй другой стиль | Сознательно выбери uncommon для себя |
| 3 | Копай в interests | При любом disagreement |
| 4 | Практикуй disagree and commit | Когда решение не твоё |
| 5 | Document resolution | Зафиксируй любое решение |
