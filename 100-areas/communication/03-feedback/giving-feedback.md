# Giving Feedback: обратная связь, которая меняет поведение

> **TL;DR:** Эффективная обратная связь = конкретное описание поведения + его влияние + ожидаемое изменение. Ключевая идея: feedback о поведении, не о личности. SBI (Situation-Behavior-Impact) — базовый фреймворк. Применяется в: code review, 1-on-1, performance review, daily collaboration.

---

## Зачем это нужно?

### Представьте ситуацию

Code review. Ты пишешь: "Этот код плохой, переделай". Автор PR обижается, защищается, начинается конфликт. Или хуже — молча соглашается, но отношения испорчены. Через месяц та же проблема в коде повторяется, потому что человек не понял *что именно* исправить и *почему*.

**Без структурированного feedback:**
- Получатель не понимает что менять
- Защитная реакция вместо изменения
- Повторение тех же ошибок
- Ухудшение отношений в команде

**С правильным feedback:**
- Ясно что и почему менять
- Фокус на улучшении, не на вине
- Реальное изменение поведения
- Укрепление доверия

## Терминология

| Термин | Что это | IT-аналогия |
|--------|---------|-------------|
| **Feedback** | Информация о результатах действий для корректировки | Как логи — показывают что произошло для debugging |
| **SBI** | Situation-Behavior-Impact — структура feedback | Как stack trace — context, action, result |
| **Radical Candor** | Care Personally + Challenge Directly | Как pair programming — поддержка + честный review |
| **Constructive feedback** | Feedback направленный на улучшение | Как PR review — не "плохо", а "вот как лучше" |
| **Defensiveness** | Защитная реакция на критику | Как race condition — мешает нормальной работе |

## Как это работает?

### SBI Model (Center for Creative Leadership)

Базовый фреймворк для структурированного feedback:

```
┌─────────────────────────────────────────────────────────────────┐
│                        SBI MODEL                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  S — SITUATION (Ситуация)                                       │
│      ─────────────────────                                      │
│      Конкретное время и место                                   │
│      "На вчерашнем code review..."                              │
│      "Во время sprint planning в понедельник..."                │
│                                                                 │
│  B — BEHAVIOR (Поведение)                                       │
│      ────────────────────                                       │
│      Наблюдаемые действия (не интерпретации)                    │
│      "...ты прервал Сашу три раза..."                           │
│      "...ты написал комментарий 'это плохой код'..."            │
│                                                                 │
│  I — IMPACT (Влияние)                                           │
│      ──────────────────                                         │
│      Конкретный эффект на тебя, команду, проект                 │
│      "...из-за этого она потеряла мысль и митинг затянулся"     │
│      "...автор PR не понял что исправить и потратил час"        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Источник:** [Center for Creative Leadership: SBI Model](https://www.ccl.org/articles/leading-effectively-articles/closing-the-gap-between-intent-vs-impact-sbii/)

### Почему SBI работает

1. **Конкретика** — не "ты грубый", а конкретное поведение
2. **Факты** — наблюдаемые действия, а не интерпретации
3. **Связь причина-следствие** — человек понимает *почему* это проблема
4. **Без атаки на личность** — поведение можно изменить, характер — нет

### Radical Candor (Kim Scott)

Матрица стилей feedback от бывшего руководителя в Apple и Google:

```
                     CHALLENGE DIRECTLY
                            ↑
                            │
           ┌────────────────┼────────────────┐
           │                │                │
           │   OBNOXIOUS    │   RADICAL      │
           │   AGGRESSION   │   CANDOR       │
           │                │                │
           │  "Brutal       │  "Care + Truth"│
           │   honesty"     │   IDEAL        │
           │                │                │
CARE    ───┼────────────────┼────────────────┼─── NOT CARE
PERSONALLY │                │                │    PERSONALLY
           │                │                │
           │   RUINOUS      │  MANIPULATIVE  │
           │   EMPATHY      │  INSINCERITY   │
           │                │                │
           │  "Nice but     │  "Fake nice"   │
           │   unhelpful"   │   TOXIC        │
           │                │                │
           └────────────────┼────────────────┘
                            │
                            ↓
                     DON'T CHALLENGE
```

**Источник:** [Radical Candor by Kim Scott](https://www.radicalcandor.com/our-approach/)

### Четыре квадранта

| Квадрант | Описание | Пример в code review |
|----------|----------|---------------------|
| **Radical Candor** | Care + Challenge | "Я хочу помочь тебе вырасти. Этот код работает, но есть способ сделать его понятнее. Давай разберём?" |
| **Obnoxious Aggression** | Challenge без Care | "Этот код ужасен. Как можно было так написать?" |
| **Ruinous Empathy** | Care без Challenge | "Всё хорошо, одобряю!" (хотя есть серьёзные проблемы) |
| **Manipulative Insincerity** | Ни Care, ни Challenge | "Норм" в комментарии, но жалуешься за спиной |

### Соотношение 5:1

Исследование Losada & Heaphy на 60 бизнес-юнитах:

```
┌─────────────────────────────────────────────────────────────────┐
│              СООТНОШЕНИЕ ПОЗИТИВ : НЕГАТИВ                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Высокоэффективные команды:     5 : 1                           │
│  Средние команды:               2 : 1                           │
│  Низкоэффективные команды:      1 : 3                           │
│                                                                 │
│  ⚠️ Это НЕ значит "хвали 5 раз перед критикой"                  │
│     Это общий баланс взаимодействий в команде                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Интерпретация:** В здоровой команде позитивных взаимодействий (признание, благодарность, поддержка) должно быть в 5 раз больше, чем негативных (критика, указание на ошибки).

## Пошаговый процесс

### Подготовка (до разговора)

1. **Определи цель** — чего хочешь добиться?
2. **Собери факты** — конкретные примеры поведения
3. **Проверь timing** — сейчас подходящий момент?
4. **Выбери место** — приватно для критики, публично для похвалы

### Шаг 1: Situation (Ситуация)

**Что делать:**
- Укажи конкретное время и место
- Создай контекст для получателя
- Дай понять о чём разговор

**Пример:**
```
"На вчерашнем code review PR #1234..."
"Во время нашего 1-on-1 в среду..."
"На sprint planning в понедельник утром..."
```

**Скрипт:**
```
"Хочу обсудить с тобой [ситуация]. Найдётся 10 минут?"
```

### Шаг 2: Behavior (Поведение)

**Что делать:**
- Описывай наблюдаемые действия
- Избегай интерпретаций и оценок
- Будь конкретен

**Правило:** Описывай то, что видеокамера могла бы записать.

```
❌ "Ты был агрессивным"        → интерпретация
✅ "Ты повысил голос и сказал..." → наблюдаемое поведение

❌ "Ты не уважаешь коллег"     → интерпретация
✅ "Ты прервал Машу три раза"  → наблюдаемое поведение

❌ "Код плохой"                → оценка
✅ "В этом методе 150 строк    → наблюдаемое
    без комментариев"
```

### Шаг 3: Impact (Влияние)

**Что делать:**
- Опиши конкретный эффект
- Используй "я" или "команда" (не "ты сделал плохо")
- Покажи связь причина-следствие

**Пример:**
```
"...из-за этого автор PR потратил дополнительный час,
пытаясь понять что именно исправить."

"...команда потеряла 20 минут на выяснение того,
что можно было решить за 5 минут."

"...у меня сложилось впечатление, что моё мнение не важно."
```

### Шаг 4: Пауза и диалог

**Что делать:**
- Дай человеку осмыслить
- Спроси о его видении ситуации
- Используй active listening

**Скрипт:**
```
"Как ты это видишь?"
"Что ты думаешь об этом?"
"Расскажи, что происходило с твоей стороны?"
```

### Шаг 5: Forward (Вперёд)

**Что делать:**
- Обсуди желаемое поведение
- Договорись о конкретных действиях
- Предложи поддержку

**Скрипт:**
```
"В следующий раз было бы полезно..."
"Как думаешь, что можно сделать по-другому?"
"Чем могу помочь?"
```

## Скрипты и Templates

### Use Case 1: Code Review — конструктивный комментарий

**Ситуация:** PR с проблемным кодом

**❌ Неправильно:**
```
"Плохой код, переделай."
"WTF is this?"
"Это не работает."
```

**✅ Правильно:**
```
"Этот метод делает три вещи: валидацию, трансформацию
и сохранение. Если разбить на три метода, будет проще
тестировать и поддерживать. Что думаешь?"
```

**Template для code review:**
```
[Наблюдение]: "Вижу, что [конкретное наблюдение]"
[Почему]: "Это может привести к [конкретная проблема]"
[Предложение]: "Альтернатива: [конкретное решение]"
[Вопрос]: "Как тебе такой подход?"
```

**Чеклист code review feedback:**
□ Фокус на коде, не на авторе
□ Объяснил "почему", не только "что"
□ Предложил альтернативу
□ Задал вопрос (не приказ)
□ Нет сарказма и passive-aggressive тона

### Use Case 2: 1-on-1 — критический feedback

**Ситуация:** Разработчик опаздывает на митинги

**Скрипт (SBI):**
```
"Хотел обсудить последние две недели.
[S] На трёх последних стендапах — в понедельник, среду и пятницу —
[B] ты присоединился через 5-10 минут после начала.
[I] Из-за этого команда ждала или пересказывала то, что уже обсудили.
    Это занимает время у всех.

Как ты это видишь? Что происходит?"
```

**Важно:** После SBI — пауза и listening. Возможно, есть причины.

### Use Case 3: Позитивный feedback (SBI для похвалы)

**Ситуация:** Разработчик помог коллеге с багом

**Скрипт:**
```
"Хотел сказать кое-что.
[S] Вчера, когда Саша застрял на том баге с concurrency,
[B] ты отложил свою задачу и провёл с ним два часа debugging.
[I] Благодаря этому мы не задержали релиз, и Саша научился
    новому подходу к отладке.

Это реально ценно. Спасибо!"
```

**Почему позитивный feedback важен:**
- Закрепляет желательное поведение
- Баланс 5:1
- Показывает что ты замечаешь хорошее

### Use Case 4: Feedback "наверх" (менеджеру)

**Ситуация:** Менеджер отменяет 1-on-1 в последний момент

**Скрипт:**
```
"Есть момент, который хотел обсудить.
[S] За последний месяц
[B] три наших 1-on-1 были отменены менее чем за час до начала.
[I] Мне сложно планировать — я готовлюсь к разговору, а потом
    нужно срочно переключаться на другое.

Понимаю что у тебя много urgent дел. Может, есть способ
сделать встречи более стабильными?"
```

### Use Case 5: Feedback в конфликтной ситуации

**Ситуация:** Коллега публично раскритиковал твоё решение

**Скрипт:**
```
"Хотел поговорить о митинге.
[S] Когда я представлял архитектуру,
[B] ты сказал 'это не будет работать' и перечислил
    пять причин перед всей командой.
[I] Я почувствовал, что меня атакуют, а не что мы ищем
    лучшее решение вместе.

Я открыт к критике, но было бы комфортнее обсудить
concerns один на один сначала. Как ты видишь ситуацию?"
```

## Распространённые ошибки

### Ошибка 1: Sandwich Method

**Неправильно:**
```
"Ты молодец! Но этот код ужасен. Зато у тебя хороший README."
```

**Почему это неправильно:**
- Люди ждут "но" после похвалы
- Обесценивает и похвалу, и критику
- Неискренне — это чувствуется

**Правильно:**
```
Разделяй похвалу и критику. Дай каждому свой момент.
```

### Ошибка 2: Vague Feedback

**Неправильно:**
```
"Нужно работать над коммуникацией."
"Будь более проактивным."
"Код нужно улучшить."
```

**Почему это неправильно:**
- Непонятно что конкретно менять
- Нет actionable steps
- Создаёт тревожность без направления

**Правильно:**
```
"На последних трёх митингах ты не задавал вопросов.
Попробуй на следующем задать хотя бы один clarifying question."
```

### Ошибка 3: Личность вместо поведения

**Неправильно:**
```
"Ты невнимательный."
"Ты не командный игрок."
"Ты грубый."
```

**Почему это неправильно:**
- Атака на идентичность — вызывает defensiveness
- Личность не меняется, поведение — да
- Закрывает диалог

**Правильно:**
```
"В этом PR три места где ты пропустил edge cases.
Давай посмотрим, как можно это ловить на review?"
```

### Ошибка 4: Публичная критика

**Неправильно:**
```
[На митинге]: "Кстати, Вася, твой код из прошлого спринта
               создал нам кучу проблем."
```

**Почему это неправильно:**
- Унижение перед коллегами
- Defensiveness вместо learning
- Токсичная культура

**Правильно:**
```
Praise in public, correct in private.
```

### Ошибка 5: Feedback как monologue

**Неправильно:**
```
Ты: [5 минут говоришь о проблемах]
Коллега: [молчит, чувствует себя атакованным]
```

**Почему это неправильно:**
- Нет диалога — нет понимания context
- Нет buy-in на изменения
- Человек не чувствует себя услышанным

**Правильно:**
```
SBI → Пауза → "Как ты видишь ситуацию?" → Слушаешь → Диалог
```

## Когда использовать / НЕ использовать

### Используй структурированный feedback, когда:

| Ситуация | Почему важно |
|----------|--------------|
| **Code review** | Автор понимает что и почему менять |
| **1-on-1** | Регулярный feedback = development |
| **Performance review** | Конкретика вместо общих фраз |
| **После инцидента** | Blameless learning |
| **Mentoring** | Помогаешь человеку расти |

### НЕ используй (или адаптируй), когда:

| Ситуация | Что делать вместо |
|----------|-------------------|
| **Человек эмоционален** | Подожди пока остынет |
| **Ты эмоционален** | Подожди пока остынешь |
| **Публично** | Перенеси в private (для критики) |
| **Нет конкретики** | Собери факты сначала |
| **Слишком много feedback сразу** | 1-2 пункта за раз |

### Timing для feedback

```
┌─────────────────────────────────────────────────────────────────┐
│                    КОГДА ДАВАТЬ FEEDBACK                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ✅ ХОРОШЕЕ ВРЕМЯ:                                              │
│  • Как можно раньше после события (свежая память)               │
│  • Когда человек готов услышать (не в стрессе)                  │
│  • Приватно для критики                                         │
│  • Регулярно (не копить)                                        │
│                                                                 │
│  ❌ ПЛОХОЕ ВРЕМЯ:                                               │
│  • Сразу после конфликта (эмоции высоки)                        │
│  • Перед важным дедлайном                                       │
│  • Публично (для критики)                                       │
│  • Когда сам зол                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Практика

### Упражнение 1: SBI Practice

**Сценарий:**
Вспомни последний раз, когда хотел дать feedback но не дал (или дал плохо).

**Задача:**
Переформулируй используя SBI:
- S: Когда и где это произошло?
- B: Что конкретно человек сделал? (факты, не интерпретации)
- I: Какой был эффект на тебя/команду/проект?

**Эталонный ответ:**
<details><summary>Показать пример</summary>
Было: "Ты всегда опаздываешь на митинги"

SBI: "На трёх последних стендапах (S) ты присоединился через 5-10 минут после начала (B). Из-за этого мы ждали или повторяли информацию, что занимало время у всей команды (I)."
</details>

### Упражнение 2: Code Review Rewrite

**Сценарий:**
Перепиши эти code review комментарии в конструктивные:

1. "This is wrong"
2. "Why would you do this?"
3. "Bad naming"

**Эталонные ответы:**
<details><summary>Показать</summary>

1. "This is wrong" →
   "Этот подход может привести к race condition когда два пользователя
   обновляют запись одновременно. Рассмотри добавление оптимистичной
   блокировки или транзакции. Что думаешь?"

2. "Why would you do this?" →
   "Интересно понять логику этого решения. Какой use case ты
   рассматривал? Может, я что-то упускаю."

3. "Bad naming" →
   "Имя `processData` не говорит что именно делает метод.
   Может, `validateAndSaveUserProfile` точнее отразит назначение?"
</details>

### Упражнение 3: Radical Candor Check

**Сценарий:**
Для каждой ситуации определи квадрант:

1. "Всё хорошо, не волнуйся" (когда есть серьёзная проблема)
2. "Твой код — мусор" (публично на митинге)
3. "Одобряю" (а потом жалуешься коллегам)
4. "Я верю в тебя, и именно поэтому скажу прямо: эта архитектура не масштабируется"

**Ответы:**
<details><summary>Показать</summary>

1. **Ruinous Empathy** — заботишься о чувствах, но не challenge'ишь
2. **Obnoxious Aggression** — challenge без care
3. **Manipulative Insincerity** — ни care, ни challenge
4. **Radical Candor** — care + direct challenge
</details>
