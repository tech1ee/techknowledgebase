# Receiving Feedback: искусство слышать правду и расти

> **TL;DR:** Получение feedback — навык, а не пассивный процесс. Ключевая идея: defensiveness блокирует рост; growth mindset превращает критику в топливо развития. Формула: пауза → уточнение → благодарность → действие. Применяется в: code review, 1-on-1, performance review, peer feedback.

---

## Зачем это нужно?

### Представьте ситуацию

Code review. Коллега пишет: "Эта архитектура не масштабируется, нужно переделать". Мгновенная реакция: "Он не понимает контекст. У нас было мало времени. Он всегда критикует." Закрываешься, защищаешься, отвечаешь "у нас дедлайн, потом посмотрим". Через полгода система падает под нагрузкой — ровно там, где предупреждали.

**Без навыка принимать feedback:**
- Защитная реакция вместо обучения
- Пропущенные возможности для роста
- Повторение одних ошибок
- Репутация "человека, которого нельзя критиковать"

**С развитым навыком:**
- Каждый feedback = data point для улучшения
- Быстрый рост как специалиста
- Коллеги не боятся говорить правду
- Репутация "открытого к росту"

## Терминология

| Термин | Что это | IT-аналогия |
|--------|---------|-------------|
| **Defensiveness** | Защитная реакция на критику: отрицание, оправдание, контратака | Как firewall с deny all — блокирует даже полезный traffic |
| **Growth Mindset** | Вера что способности развиваются через усилия и обучение | Как agile — итеративное улучшение, не waterfall "сразу идеально" |
| **Fixed Mindset** | Вера что способности врождённые и неизменные | Как legacy system — "работает, не трогай" |
| **Feedback Loop** | Цикл: действие → feedback → корректировка → действие | Как CI/CD pipeline — автоматическое улучшение после каждого commit |
| **Seek Feedback** | Активный запрос обратной связи | Как pull request review — сам просишь, а не ждёшь |

## Как это работает?

### Growth Mindset vs Fixed Mindset (Carol Dweck)

Исследование Stanford показало фундаментальную разницу в том, как люди реагируют на критику:

```
┌─────────────────────────────────────────────────────────────────┐
│                    FIXED vs GROWTH MINDSET                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  FIXED MINDSET                  GROWTH MINDSET                  │
│  ─────────────────              ──────────────────              │
│                                                                 │
│  "Способности врождённые"       "Способности развиваются"       │
│                                                                 │
│  Feedback = оценка личности     Feedback = данные для роста     │
│                                                                 │
│  Критика = угроза               Критика = возможность           │
│                                                                 │
│  Ошибка = провал                Ошибка = обучение               │
│                                                                 │
│  "Я не могу это"                "Я пока не могу это"            │
│                                                                 │
│  Избегает challenge             Ищет challenge                  │
│                                                                 │
│  Мозг "выключается"             Мозг активно обрабатывает       │
│  при ошибках                    ошибки                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Ключевое исследование:** Dweck показала через brain imaging, что люди с fixed mindset демонстрируют минимальную мозговую активность при анализе своих ошибок. Они фокусируются только на том, правильно или неправильно. Люди с growth mindset показывают высокую активность — мозг обрабатывает информацию об ошибке для обучения.

**Источник:** [Carol Dweck: Mindset](https://fs.blog/carol-dweck-mindset/)

Почему мы защищаемся при критике:

```
┌─────────────────────────────────────────────────────────────────┐
│                   АНАТОМИЯ DEFENSIVENESS                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. TRIGGER                                                     │
│     Критика воспринимается как угроза идентичности              │

│                                                                 │
│  2. AMYGDALA HIJACK                                             │
│     Миндалина активирует fight-or-flight                        │
│     Рациональное мышление "выключается"                         │
│                                                                 │
│  3. ЗАЩИТНАЯ РЕАКЦИЯ                                            │
│     • Отрицание: "Это не проблема"                              │
│     • Оправдание: "У меня были причины"                         │
│     • Контратака: "А твой код тоже не идеальный"                │
│     • Минимизация: "Это мелочь"                                 │
│                                                                 │
│  4. РЕЗУЛЬТАТ                                                   │
│     Feedback отвергнут, обучение не произошло                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
│                                                                 │
```

## Пошаговый процесс

### Подготовка (mindset перед feedback)

1. **Напомни себе:** Feedback о поведении/работе, не о личности
2. **Установи цель:** "Я хочу узнать что-то полезное"
3. **Признай эмоции:** "Возможно, будет некомфортно — и это нормально"

### Шаг 1: Пауза (не реагируй сразу)

**Что делать:**
- Сделай глубокий вдох
- Не отвечай в первые 3-5 секунд
- Напомни: "Это data, не attack"

**Почему важно:**
Пауза даёт время prefrontal cortex "включиться" и модерировать эмоциональную реакцию amygdala.

**Скрипт (внутренний диалог):**
```
"Окей, это feedback. Мне не нужно соглашаться или не соглашаться
прямо сейчас. Моя задача — понять."
```

### Шаг 2: Слушай и уточняй (не защищайся)

**Что делать:**
- Дай говорящему закончить
- Задавай clarifying questions
- Перефразируй для проверки понимания

**Скрипт:**
```
"Помоги мне понять точнее — ты имеешь в виду [X] или [Y]?"
"Можешь привести конкретный пример?"
"Если я правильно понял, проблема в [перефразировка]. Верно?"
```

**Чего НЕ делать:**
```
❌ "Да, но у меня были причины..."
❌ "Ты не понимаешь контекст..."
❌ "А вот твой код тоже..."
```

### Шаг 3: Поблагодари (да, даже за критику)

**Что делать:**
- Поблагодари за честность
- Признай усилие дать feedback

**Почему это работает:**
- Снижает напряжение
- Поощряет будущий honest feedback
- Сигнализирует openness

**Скрипт:**
```
"Спасибо, что сказал это. Понимаю, что это не всегда легко."
"Ценю, что ты нашёл время это обсудить."
"Спасибо за прямоту."
```

### Шаг 4: Возьми время на обработку (если нужно)

**Что делать:**
- Попроси время, если эмоции сильны
- Договорись о follow-up

**Скрипт:**
```
"Это важный feedback. Дай мне время обдумать — вернусь к тебе завтра?"
"Хочу properly осмыслить это. Можем продолжить разговор в [время]?"
```

### Шаг 5: Оцени и действуй

**Что делать:**
- Отдели эмоции от содержания
- Определи actionable items
- Создай план изменений
- Сообщи о прогрессе

**Framework для оценки:**

```
┌─────────────────────────────────────────────────────────────────┐
│               ОЦЕНКА ПОЛУЧЕННОГО FEEDBACK                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Есть ли в feedback правда? (даже частичная)                 │
│                                                                 │
│  2. Это pattern (повторяется от разных людей)?                  │
│                                                                 │
│  3. Источник заслуживает доверия в этой области?                │
│                                                                 │
│  4. Какие конкретные действия я могу предпринять?               │
│                                                                 │
│  5. Что могу изменить в следующий раз?                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Скрипты и Templates

### Use Case 1: Code Review — критика архитектуры

**Ситуация:** Reviewer пишет: "Это решение не масштабируется. Надо переделывать."

**❌ Defensive реакция:**
```
"У нас был дедлайн. Ты не понимаешь контекст."
"А у тебя в прошлом PR тоже были проблемы."
"Это работает, что ещё надо?"
```

**✅ Growth реакция:**
```
"Спасибо за комментарий. Можешь уточнить, какой сценарий
масштабирования ты имеешь в виду? 10x users? 100x requests?
Хочу понять boundaries проблемы."
```

**Follow-up после анализа:**
```
"Посмотрел детальнее. Ты прав — при N+1 запросах на каждого
пользователя это развалится. Вижу два варианта: batch loading
или caching. Какой предпочитаешь обсудить первым?"
```

### Use Case 2: 1-on-1 — критика communication

**Ситуация:** Менеджер говорит: "На митингах ты иногда перебиваешь коллег."

**❌ Defensive реакция:**
```
"Я? Никогда не замечал. Может, ты неправильно понял?"
"Это потому что они говорят слишком долго."
```

**✅ Growth реакция:**
```
"Спасибо, что сказал. Я не осознавал этого. Можешь вспомнить
конкретный пример — когда и кого я перебил? Хочу понять
pattern чтобы работать над этим."
```

**Action plan:**
```
"Окей, вижу. План такой: на следующих митингах буду сознательно
ждать 2 секунды после того как человек замолчит, прежде чем
говорить. Если увидишь что я снова это делаю — дай знать сразу."
```

### Use Case 3: Performance Review — негативный feedback

**Ситуация:** "За этот квартал твоя продуктивность ниже ожиданий."

**❌ Defensive реакция:**
```
"Это несправедливо! У меня было много отвлечений!"
"А что насчёт того проекта который я спас?"
```

**✅ Growth реакция:**
```
"Это важный feedback. Помоги понять: по каким метрикам
ты это оцениваешь? Хочу убедиться что мы смотрим на одно и то же.
И какой уровень ты ожидаешь?"
```

**Constructive follow-up:**
```
"Понял критерии. Вижу gap. Хочу составить план — можем
обсудить конкретные milestones на следующий квартал?
И договориться о check-ins каждые две недели?"
```

### Use Case 4: Peer Feedback — неожиданная критика

**Ситуация:** Коллега: "Когда ты делаешь код review, иногда звучишь грубо."

**Скрипт:**
```
"Спасибо, что решился сказать это — знаю, это непросто.
Я точно не хочу быть грубым. Можешь показать пример —
какой комментарий так прозвучал? Хочу понять как это
воспринимается с другой стороны."
```

### Use Case 5: Активный запрос feedback

**Ситуация:** Хочешь получить feedback о своей работе

**Скрипты для запроса:**

```
После code review:
"Как тебе общий подход в этом PR? Что бы ты сделал иначе?"

После презентации:
"Что сработало? Что можно улучшить в следующий раз?"

После проекта:
"Если бы делали снова — что бы ты предложил делать иначе?"

Общий запрос:
"Над чем мне стоит поработать чтобы быть эффективнее?"
```

**Чеклист для запроса feedback:**
□ Конкретный вопрос (не "как я?")
□ Безопасная обстановка для честности
□ Готовность услышать критику
□ Follow-up действие после feedback

## Распространённые ошибки

### Ошибка 1: Instant Defense

**Неправильно:**
```
Feedback: "Твой код сложно читать"
Реакция: "У меня был дедлайн, и вообще это работает!"
```

**Почему это неправильно:**
- Отключает возможность услышать полезное
- Сигнализирует "не давай мне feedback"
- Упускаешь шанс на рост

**Правильно:**
```
"Понял. Что конкретно было сложно — naming, структура, или
что-то ещё? Хочу улучшить."
```

### Ошибка 2: Yes, but...

**Неправильно:**
```
"Да, понимаю, НО у меня были причины..."
"Согласен, НО это из-за требований..."
```

**Почему это неправильно:**
- "Но" стирает всё сказанное до него
- Это скрытая форма отрицания
- Человек перестаёт давать feedback

**Правильно:**
```
"Понимаю твою точку. И вот контекст, который может быть
полезен: [факты]. Как думаешь, с этим в виду — что лучший путь?"
```

### Ошибка 3: Минимизация

**Неправильно:**
```
"Это мелочь, не о чем беспокоиться."
"Один раз — не pattern."
```

**Почему это неправильно:**
- Обесценивает усилие давшего feedback
- Пропускаешь early warning signals
- Мелочи складываются в большое

**Правильно:**
```
"Спасибо что заметил. Буду следить за этим."
```

### Ошибка 4: Контратака

**Неправильно:**
```
"А вот ты сам на прошлой неделе..."
"Может, сначала свой код исправишь?"
```

**Почему это неправильно:**
- Переводит фокус с твоего роста
- Разрушает доверие
- Создаёт toxic dynamics

**Правильно:**
```
[Обработай feedback о себе отдельно]
[Если есть feedback для другого — дай отдельно, в другое время]
```

### Ошибка 5: Fishing for Compliments

**Неправильно:**
```
"Ну а что хорошего было?" (сразу после критики)
"А в целом-то я справился, да?"
```

**Почему это неправильно:**
- Выглядит неуверенно
- Обесценивает критику
- Не помогает росту

**Правильно:**
```
[Прими критику]
[Позитивный feedback придёт отдельно или сам попросишь later]
```

## Когда использовать / НЕ использовать

### Используй техники принятия feedback, когда:

| Ситуация | Почему важно |
|----------|--------------|
| **Code review** | Каждый comment = возможность научиться |
| **1-on-1** | Регулярный feedback = быстрый рост |
| **Performance review** | Определяет карьеру |
| **Peer feedback** | Коллеги видят то, что ты не видишь |
| **Post-mortem** | Системный learning из ошибок |

### НЕ применяй (или адаптируй), когда:

| Ситуация | Что делать вместо |
|----------|-------------------|
| **Feedback явно деструктивный** | "Спасибо за точку зрения" и закончи |
| **Ты слишком эмоционален** | "Дай вернусь к этому завтра" |
| **Публичное унижение** | Перенеси в private для обсуждения |
| **Feedback от non-expert** | Оцени источник, возьми зерно если есть |

## Практика

### Упражнение 1: Reframe Practice

**Сценарий:**
Переформулируй эти defensive реакции в growth реакции:

1. "Это не моя вина, requirements были неясные"
2. "Работает же, что ещё надо?"
3. "Ты сам так делаешь"

**Эталонные ответы:**
<details><summary>Показать</summary>

1. "Понимаю проблему. В следующий раз буду задавать уточняющие вопросы до начала работы. Какие clarifications помогли бы в этот раз?"

2. "Спасибо за feedback. Помоги понять — какой аспект вызывает concern? Performance, readability, maintainability?"

3. "Понял твой point. Давай сфокусируемся на этом сейчас. Если есть feedback для меня по другим темам — обсудим отдельно."
</details>

### Упражнение 2: SARA Awareness

**Сценарий:**
Вспомни последний раз, когда получил критику. Определи через какие стадии SARA прошёл.

**Задача:**
- Как долго был в каждой стадии?
- Что помогло перейти в Acceptance?
- Что задержало в Anger/Resistance?

### Упражнение 3: Seek Feedback Practice

**Задача на неделю:**
Активно запроси feedback у 3 разных людей:
1. Коллега (peer)
2. Более опытный (mentor/senior)
3. Менее опытный (junior — они видят иначе)

**Скрипты для запроса:**
```
"Работаю над [навык]. Что из того что ты видишь — работает?
Что стоит изменить?"
```
