# Переключение контекста: скрытый убийца продуктивности

Каждое переключение между задачами стоит 23 минуты на возврат к предыдущему состоянию. Человеческий мозг — не многозадачная система, а быстрый переключатель. Чем чаще переключаешься — тем меньше успеваешь.

---

## Почему это важно для разработчиков

Типичный день программиста:
- 9:00 — Код задачи A
- 9:15 — Slack сообщение → ответ
- 9:20 — Возврат к задаче A → "где я остановился?"
- 9:35 — Stand-up meeting
- 10:00 — Код задачи A → наконец вошёл в flow
- 10:15 — "Срочная" задача B от PM
- 10:20 — Начинаешь B, но думаешь об A

**Результат:** 8 часов в офисе → 2-3 часа реальной работы

---

## Нейробиология переключения

### Attention Residue (Остаточное внимание)

Когда ты переключаешься с задачи A на задачу B, часть твоего внимания остаётся на A.

```
Задача A (код)     Задача B (Slack)
    100% ──────────────→ 0%        ← ожидание
     70% ←─────────────  30%       ← реальность
```

**Исследование Sophie Leroy (2009):**
- Участники переключались между задачами
- Производительность на задаче B падала на 20-40%
- Причина: attention residue от задачи A

### Стоимость переключения

| Метрика | Значение | Источник |
|---------|----------|----------|
| Время возврата к сложной задаче | 23 минуты | UC Irvine |
| Потеря продуктивности при multitasking | 40% | APA |
| Снижение IQ при постоянных отвлечениях | 10 пунктов | University of London |
| Количество переключений в час (среднее) | 10-15 | Gloria Mark |

### Почему мозг не может быть multitasking

Prefrontal cortex (PFC) — "управляющий" мозга — может фокусироваться только на одной когнитивно сложной задаче.

**Что выглядит как multitasking:**
- Идти и разговаривать → одна задача автоматическая
- Печатать и думать → одна моторная, одна когнитивная

**Что НЕ работает как multitasking:**
- Писать код и слушать митинг → обе когнитивные → PFC переключается

## Типы переключений

### 1. Вынужденные переключения (External)
- Slack/email уведомления
- Вопросы коллег
- Митинги
- "Срочные" задачи

### 2. Добровольные переключения (Internal)
- "Проверю соцсети на минутку"
- "Отвечу на это сообщение сейчас"
- Прокрастинация через переключение

**Исследование:** 44% переключений — добровольные (Gloria Mark)

Это значит: ты сам — главный источник своих переключений.

## Стратегии минимизации

### 1. Context Batching

Группируй похожие задачи в блоки:

```
❌ Плохо:
9:00  Код
9:15  Email
9:30  Код
9:45  Slack
10:00 Код
10:15 Email

✅ Хорошо:
9:00-11:00  Код (Deep Work блок)
11:00-11:30 Коммуникации (Email + Slack)
11:30-13:00 Код
```

### 2. Time Blocking

Резервируй время для разных типов работы:

```
Утро (9-12):    Deep Work — код, архитектура
Полдень (12-14): Shallow Work — митинги, review
День (14-17):   Mixed — код + коммуникации
```

### 3. Context Stack (для неизбежных переключений)

Когда переключение неизбежно — записывай контекст:

```
Переключаюсь с задачи A:
- Где остановился: функция validateUser, строка 45
- Следующий шаг: добавить проверку email
- Открытые вопросы: нужен ли regex для email?

Время записи: 30 секунд
Время восстановления без записи: 15-23 минуты
ROI: x30-x46
```

### 4. Notification Management

```
Уровень 1: Полное отключение (Deep Work)
- Телефон в другой комнате
- Slack закрыт
- Только таймер

Уровень 2: Фильтрация (Focus Work)
- Уведомления только от конкретных людей
- Scheduled "Do Not Disturb"

Уровень 3: Batch Checking
- Проверка сообщений каждые 2 часа
- Не "реагировать", а "обрабатывать"
```

## Когда переключение полезно

### Interleaving (чередование при обучении)

Парадокс: при **обучении** переключение между темами улучшает долгосрочное запоминание.

```
Изучение React:
❌ 4 часа только hooks → потом 4 часа только routing
✅ 45 мин hooks → 45 мин routing → 45 мин hooks → ...
```

**Почему:**
- Мозг учится **распознавать** тип задачи
- Создаёт связи между концепциями
- Имитирует реальные условия применения

**Но:** Это работает для **обучения**, не для **production работы**.

### Творческое застревание

Когда застрял на проблеме > 30 минут:
1. Переключись на другую задачу
2. Дай подсознанию работать (incubation)
3. Вернись через 20-30 минут

Это называется **productive switching** — намеренное переключение для инкубации идей.

## Практический протокол для разработчика

### Утренний Deep Work блок

```
8:00-8:30   Планирование дня (какие задачи, в каком порядке)
8:30-11:30  Deep Work блок
            - Уведомления OFF
            - Одна задача
            - Если застрял > 30 мин → productive switch
11:30-12:00 Batch коммуникаций
```

### Обработка прерываний

Когда кто-то подходит с вопросом:

```
Вариант A: Вопрос срочный (production down)
→ Записать context stack (30 сек)
→ Переключиться

Вариант B: Вопрос не срочный
→ "Могу ответить через [время конца блока]"
→ Записать вопрос
→ Продолжить работу
```

### Метрики для отслеживания

| Метрика | Цель | Как измерять |
|---------|------|--------------|
| Длина непрерывного блока | > 90 мин | Таймер |
| Количество переключений в день | < 10 | Счётчик |
| Время до first deep work | < 30 мин от начала дня | Таймер |
