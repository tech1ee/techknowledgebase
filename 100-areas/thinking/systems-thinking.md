# Системное мышление: видеть лес за деревьями

Система — не сумма частей, а результат их взаимодействия. Изменить одного разработчика в команде — почти ничего не изменится. Изменить процесс ревью — влияние будет в разы больше.

---

## История: почему это важно

В 1999 году NASA потеряла Mars Climate Orbiter — аппарат стоимостью $125 миллионов. Причина? Одна команда использовала метрическую систему, другая — имперскую. Никто не проверил совместимость.

Это не ошибка одного человека. Это системный провал: отсутствие проверок, разрозненные команды, нет единых стандартов.

1️⃣ Классический подход: "Найти виновного и наказать"

2️⃣ Системный подход: "Какие условия позволили ошибке произойти? Как изменить систему, чтобы такое не повторилось?"

*Второй вопрос намного сложнее, но только он ведёт к реальным улучшениям.*

---

## Что такое система

Система — это набор элементов, связанных между собой для достижения цели.

Ключевое свойство: **целое больше суммы частей**.

```
Автомобиль ≠ двигатель + колёса + кузов
Автомобиль = взаимодействие всех компонентов

Команда ≠ разработчик + дизайнер + менеджер
Команда = их взаимодействие, культура, процессы
```

### Три главных элемента любой системы

1️⃣ **Элементы** — видимые части (люди, код, сервера)

2️⃣ **Связи** — как элементы взаимодействуют (API, процессы, коммуникация)

3️⃣ **Цель** — для чего система существует (часто неочевидна!)

*Важно: изменение элементов обычно меньше влияет на систему, чем изменение связей или цели.*

Пример: заменить одного разработчика другим — система почти не изменится. Изменить процесс код-ревью — влияние будет гораздо больше.

## Cynefin Framework: какие проблемы бывают

Не все проблемы одинаковы. И это критически важно понять. Дэйв Сноуден создал фреймворк Cynefin (читается "кы-НЕ-вин", валлийское слово), который помогает понять, с каким типом ситуации ты имеешь дело — и какой подход использовать.

**Главная идея:** Применять сложный подход к простой проблеме — overengineering. Применять простой подход к сложной проблеме — провал. Сначала определи тип проблемы.

### 1️⃣ Clear (Простые) — "Есть инструкция, следуй ей"

**Характеристика:** Причина и следствие очевидны. Все согласны, что делать.

**Подход:** Sense → Categorize → Respond. Увидел → понял категорию → применил стандартное решение.

**Примеры из IT:**
```
Проблема: Пользователь забыл пароль
→ Стандартное решение: отправить email для сброса
→ Не нужен эксперт, не нужен анализ

Проблема: 404 ошибка на странице
→ Проверить роутинг, исправить путь
→ Есть чёткий алгоритм

Проблема: Нужно добавить новое поле в форму
→ Добавить в модель, миграция, UI
→ Делали 100 раз, всё понятно
```

**Ловушка:** Если все проблемы кажутся Clear — ты что-то упускаешь. Или недооцениваешь ситуацию.

### 2️⃣ Complicated (Сложные) — "Нужен эксперт"

**Характеристика:** Причина и следствие есть, но требуют анализа. Несколько правильных решений.

**Подход:** Sense → Analyze → Respond. Увидел → проанализировал (с экспертами) → выбрал лучшее решение.

**Примеры из IT:**
```
Проблема: База данных тормозит
→ DBA анализирует: индексы, план запросов, конфигурация, железо
→ Нужна экспертиза, но есть чёткий ответ

Проблема: Как спроектировать API для нового сервиса
→ Архитектор рассматривает варианты: REST vs GraphQL, структура endpoints
→ Несколько хороших решений, выбираем оптимальное

Проблема: Миграция на новую версию фреймворка
→ Нужно понять breaking changes, спланировать шаги
→ Сложно, но предсказуемо
```

**Ключевое:** Здесь работают эксперты и лучшие практики. Можно спросить "как правильно" и получить ответ.

### 3️⃣ Complex (Комплексные) — "Нужны эксперименты"

**Характеристика:** Причина и следствие понятны только в ретроспективе. Нельзя предсказать результат.

**Подход:** Probe → Sense → Respond. Попробуй → посмотри что вышло → адаптируйся.

**Примеры из IT:**
```
Проблема: Почему пользователи не конвертируются в платящих?
→ Нет однозначного ответа
→ Нужны гипотезы, A/B тесты, данные
→ То, что работает у конкурента, может не работать у тебя

Проблема: Какую архитектуру выбрать для нового продукта?
→ Зависит от того, как продукт будет расти (а этого никто не знает)
→ Начни с простого, эволюционируй

Проблема: Как улучшить производительность команды?
→ Люди — сложная система
→ Эксперимент: ввели daily standup. Посмотрели. Адаптировали.

Проблема: Почему модель галлюцинирует на определённых запросах?
→ LLM — чёрный ящик
→ Пробуем разные промпты, температуру, контекст
→ Смотрим что работает
```

**Ключевое:** Здесь НЕ работает "спросить эксперта". Эксперт может дать направление, но гарантий нет. Нужны маленькие эксперименты с быстрой обратной связью.

### 4️⃣ Chaotic (Хаотичные) — "Сначала действуй!"

**Характеристика:** Нет видимой связи причины и следствия. Кризис.

**Подход:** Act → Sense → Respond. Сначала стабилизируй, потом разбирайся.

**Примеры из IT:**
```
Ситуация: Production упал в 3 часа ночи, причина неизвестна
→ Сначала: rollback на работающую версию
→ Потом: разбор полётов

Ситуация: Взлом, данные утекают
→ Сначала: отключить доступ, остановить утечку
→ Потом: расследование

Ситуация: DDoS-атака
→ Сначала: включить защиту, перенаправить трафик
→ Потом: анализ источника
```

**Ключевое:** В хаосе нет времени на анализ. Любое решение лучше паралича. После стабилизации — перейди в Complex или Complicated домен.

### Почему это важно для разработчиков

```
Ошибка: Лечить Complex проблему как Complicated
"Почему продукт не взлетает?" → "Давайте наймём консультанта, он скажет"
→ Консультант даст красивую презентацию, но это не работает
→ Нужны эксперименты, не экспертиза

Ошибка: Лечить Complicated как Clear
"База тормозит" → "Перезагрузим сервер"
→ Проблема вернётся
→ Нужен анализ

Ошибка: Лечить Chaotic как Complex
Сервер горит → "Давайте обсудим архитектурные улучшения"
→ Пока обсуждаете, бизнес теряет деньги
→ Сначала потуши, потом думай
```

*Большинство интересных проблем в IT — Complex. Но мы часто пытаемся решать их как Complicated (найти эксперта) или Clear (найти инструкцию). Это не работает.*

## Петли обратной связи

Системы живут через петли обратной связи. Понимание этих петель — ключ к пониманию системы.

### Усиливающие петли (Reinforcing)

```
Больше пользователей → Больше контента → Привлекает больше пользователей
                                    ↑__________________________|
```

Пример: социальные сети, marketplace'ы. Рост усиливает рост.

*Работает и в негатив: меньше пользователей → меньше контента → ещё меньше пользователей.*

### Балансирующие петли (Balancing)

```
Много багов → Больше времени на фиксы → Меньше новых фичей → Меньше новых багов
                                                        ↑_______________|
```

Система стремится к равновесию.

### Задержки (Delays)

Вот где всё усложняется. Между действием и результатом — задержка.

**Пример:** Нанял 5 разработчиков сегодня. Производительность вырастет через 3-6 месяцев (онбординг). Но если не знаешь про задержку — через месяц решишь "нанять ещё 5", потому что роста не видно.

*Mythical Man-Month Брукса — именно про это: добавление людей в опаздывающий проект делает его ещё более опаздывающим (из-за задержек на коммуникацию и онбординг).*

## Системные архетипы

Есть типичные паттерны поведения систем. Зная их, можно предсказывать проблемы.

### "Fixes that Fail"

Быстрое решение создаёт новые проблемы.

```
Симптом → Быстрый фикс → Симптом исчезает
              ↓
        Побочный эффект → Симптом возвращается (хуже)
```

**Пример:** Сервис тормозит → добавили кэш → работает быстро. Через месяц: кэш инвалидируется неправильно, данные рассинхронизированы, пользователи видят устаревшую информацию.

### "Shifting the Burden"

Вместо решения корневой причины — перекладывание проблемы.

**Пример:** Тесты падают → отключили тесты. Проблема "решена", но реальная причина (плохой код) никуда не делась.

### "Limits to Growth"

Рост упирается в ограничение.

**Пример:** Стартап растёт → нанимает людей → процессы не масштабируются → хаос → рост замедляется.

Решение: заранее видеть ограничения и инвестировать в их снятие до того, как они станут критичными.

## Практика: как применять

### Шаг 1: Нарисуй систему

Буквально. На бумаге или в Miro.

```
[Пользователь] → [Frontend] → [API Gateway] → [Backend] → [Database]
                                    ↓
                              [Auth Service]
```

Добавь:
- Кто/что влияет на систему извне?
- Какие петли обратной связи существуют?
- Где задержки?

### Шаг 2: Найди рычаги

Не все части системы равноценны. Есть точки, где маленькое изменение даёт большой эффект.

**Пример:** В системе код-ревью рычаг — не "нанять больше ревьюеров", а "автоматизировать проверки, которые не требуют человека".

### Шаг 3: Думай о последствиях второго порядка

Любое решение имеет последствия. У этих последствий тоже есть последствия.

```
Решение: Добавить кэш
Последствие 1: Быстрее ответы
Последствие 2: Сложнее инвалидация
Последствие 3: Нужен мониторинг кэша
Последствие 4: Больше инфраструктуры = больше точек отказа
```

*Не значит, что не нужно добавлять кэш. Значит, что нужно заранее думать о полной картине.*

### Шаг 4: Экспериментируй в Complex-доменах

Если проблема комплексная — не пытайся найти "правильное" решение. Делай маленькие эксперименты, собирай данные, итерируй.

## Типичные ошибки

### "Оптимизация части вместо целого"

**Ошибка:** Каждая команда оптимизирует свою часть.

**Результат:** Локальные оптимумы, глобальный субоптимум.

**Пример:** Backend оптимизирован, frontend оптимизирован, но API между ними — боттлнек. Общая производительность не улучшилась.

### "Игнорирование задержек"

**Ошибка:** Ожидать мгновенных результатов от изменений.

**Результат:** Преждевременные выводы, дополнительные изменения до того, как проявился эффект первых.

### "Борьба с симптомами"

**Ошибка:** Исправлять то, что видно, а не то, что является причиной.

**Пример:** "У нас много багов" → "Давайте наймём QA". Но причина — в процессе разработки, а не в тестировании.
