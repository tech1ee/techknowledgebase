## Зачем это нужно

В 2024 году глобальные затраты от киберпреступлений достигли $9.2 триллиона. 75% организаций столкнулись с атаками ransomware. APT-активность против США выросла на 136% только в Q1 2025. Это не абстрактные угрозы — это реальность, в которой работает каждый разработчик.

**Проблема:** Большинство разработчиков воспринимают безопасность как "что-то, что делает отдел security". Они пишут код, не понимая базовых принципов защиты информации. Результат — уязвимости, которые обнаруживаются через месяцы или годы, когда ущерб уже нанесён.

**Решение:** Этот материал даёт фундамент — те концепции, на которых строится вся информационная безопасность. Без понимания CIA Triad, Defense in Depth и Principle of Least Privilege невозможно принимать правильные архитектурные решения.


## Терминология

| Термин | Что это | Аналогия |
|--------|---------|----------|
| **Threat** | Потенциальная опасность, которая может эксплуатировать уязвимость | Грабитель, который может попытаться взломать дом |
| **Vulnerability** | Слабость в системе, которую можно эксплуатировать | Незапертая дверь или слабый замок |
| **Risk** | Вероятность × Ущерб от реализации угрозы | Шанс, что грабитель войдёт × стоимость украденного |
| **Asset** | То, что мы защищаем (данные, системы, репутация) | Ценности в доме |
| **Control** | Механизм защиты от угроз | Замок, сигнализация, собака |
| **Attack Vector** | Путь, которым атакующий достигает цели | Дверь, окно, крыша — способы проникновения |
| **Attack Surface** | Совокупность всех точек, через которые можно атаковать | Все двери, окна и другие входы в дом |
| **Exploit** | Код или техника, использующая уязвимость | Отмычка для конкретного типа замка |

## CS-фундамент: История информационной безопасности

### 1883: Принципы Керкгоффса

История информационной безопасности начинается не с компьютеров. В 1883 году Огюст Керкгоффс, лингвист и профессор, опубликовал статью в Journal of Military Science, где сформулировал принцип, на котором строится вся современная криптография:

> "Система должна оставаться безопасной, даже если всё, кроме ключа, известно противнику"

**Почему это важно:** Этот принцип противоположен "security through obscurity" (безопасность через сокрытие). Нельзя полагаться на то, что атакующий не знает, как работает система. Единственный секрет — ключ.

### 1940-е: Enigma и рождение криптоанализа

Во время Второй мировой войны немецкая машина Enigma шифровала военные коммуникации. Успешный взлом в Блетчли-Парк (Алан Тьюринг и команда) показал: даже сложные системы шифрования уязвимы, если есть достаточно ресурсов и понимание паттернов.

```
┌─────────────────────────────────────────────────────────────────┐
│                    УРОКИ ENIGMA                                  │
├─────────────────────────────────────────────────────────────────┤
│  1. Complexity ≠ Security                                        │
│     Сложная система может быть уязвимой                          │
│                                                                  │
│  2. Patterns Are Dangerous                                       │
│     Повторяющиеся паттерны (приветствия, подписи) помогли взлому │
│                                                                  │
│  3. Human Factor                                                 │
│     Операторы допускали ошибки, упрощавшие взлом                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1952: Создание NSA

Создание Агентства национальной безопасности США (NSA) ознаменовало первые формальные усилия по защите цифровой информации на государственном уровне.

### 1960-е: Физическая безопасность

В эту эпоху безопасность фокусировалась на физических мерах: ограничение доступа к компьютерам, пароли, противопожарная защита. Компьютеры были дорогими и редкими — достаточно было не пускать посторонних в комнату.

### 1969: ARPANET и начало сетевой эры

Создание ARPANET — предшественника интернета — изменило всё. Теперь компьютеры могли общаться, а значит — атакующий мог быть где угодно.

### 1971: Creeper и Reaper — первый вирус и антивирус

Боб Томас создал программу Creeper, которая перемещалась по сети ARPANET с сообщением "I'M THE CREEPER: CATCH ME IF YOU CAN". Рэй Томлинсон написал Reaper для её удаления — первый пример антивирусного ПО.

### 1983: Термин "Cybersecurity"

Уильям Гибсон ввёл термин "cyberspace" в романе "Neuromancer". Три года спустя США приняли Computer Fraud and Abuse Act — первый закон против компьютерных преступлений.

### 1986: The 414s — первые знаменитые хакеры

Группа подростков из Милуоки взломала более 60 военных и корпоративных компьютеров. Это показало: угроза исходит не только от государств, но и от любопытных энтузиастов.

### 1990-е: Интернет и SSL

Расширение интернета принесло новые угрозы. В 1995 году появился SSL (Secure Sockets Layer) — первый протокол для защищённой передачи данных в вебе.

### 2000-е: Промышленность кибербезопасности

Создание Department of Homeland Security и National Cyber Security Division в США. Появление индустрии антивирусов, файрволов, IDS/IPS как отдельного рынка.

### 2010: Stuxnet — кибероружие

Stuxnet стал первым примером вредоносного ПО, используемого для физического разрушения (иранские центрифуги). Кибератаки вышли за пределы кражи данных в реальный мир.

### 2013: Сноуден и массовая слежка

Утечки Эдварда Сноудена показали масштаб государственной слежки. Это изменило отношение к приватности и подтолкнуло развитие end-to-end шифрования.

### 2020-е: AI, Zero Trust, Supply Chain

Современная эра: AI-driven атаки, Zero Trust Architecture, атаки на цепочку поставок (SolarWinds, Log4j). Безопасность стала непрерывным процессом, а не набором инструментов.

## CIA Triad: Три столпа информационной безопасности

CIA Triad — Confidentiality, Integrity, Availability — фундаментальная модель информационной безопасности. Она встроена практически в каждый современный фреймворк: от ISO 27001 до GDPR.

```
                         ┌─────────────────┐
                         │                 │
                         │ CONFIDENTIALITY │
                         │                 │
                         └────────┬────────┘
                                  │
                    "Только авторизованные
                     имеют доступ к данным"
                                  │
              ┌───────────────────┴───────────────────┐
              │                                       │
              │            CIA TRIAD                  │
              │                                       │
              └───────────────────┬───────────────────┘
                     /                       \
                    /                         \
    ┌──────────────┐                           ┌──────────────┐
    │              │                           │              │
    │  INTEGRITY   │                           │ AVAILABILITY │
    │              │                           │              │
    └──────────────┘                           └──────────────┘
           │                                          │
    "Данные точны                              "Данные доступны
     и не изменены                              когда нужны"
     без разрешения"
```

### Confidentiality (Конфиденциальность)

**Что это:** Гарантия, что информация доступна только авторизованным лицам.

**Почему важно:** Утечка конфиденциальных данных — одна из самых дорогих проблем. Средняя стоимость утечки в 2024 году — $4.45 миллиона.

**Угрозы конфиденциальности:**
- Перехват данных при передаче (man-in-the-middle)
- Несанкционированный доступ к базам данных
- Социальная инженерия и фишинг
- Кража учётных данных
- Инсайдерские угрозы

**Механизмы защиты:**

| Механизм | Описание | Пример |
|----------|----------|--------|
| **Encryption** | Шифрование данных at rest и in transit | AES-256, TLS 1.3 |
| **Access Control** | Ограничение доступа по принципу need-to-know | RBAC, ABAC |
| **Authentication** | Проверка личности пользователя | MFA, биометрия |
| **Data Classification** | Маркировка данных по уровню секретности | Public, Internal, Confidential, Secret |

```python
# Пример: Шифрование данных для конфиденциальности
from cryptography.fernet import Fernet

class SecureDataStore:
    def __init__(self):
        # В реальности ключ хранится в HSM или Key Management Service
        self.key = Fernet.generate_key()
        self.cipher = Fernet(self.key)

    def store_sensitive_data(self, data: str) -> bytes:
        """
        Шифруем данные перед сохранением.
        Даже если БД скомпрометирована, данные защищены.
        """
        return self.cipher.encrypt(data.encode())

    def retrieve_sensitive_data(self, encrypted: bytes) -> str:
        """
        Расшифровываем только для авторизованных операций.
        """
        return self.cipher.decrypt(encrypted).decode()
```

### Integrity (Целостность)

**Что это:** Гарантия, что данные точны, полны и не были изменены без авторизации.

**Почему важно:** Если атакующий может изменить данные незаметно, он может:
- Изменить банковские переводы
- Подменить медицинские записи
- Модифицировать код в production

**Угрозы целостности:**
- Несанкционированная модификация данных
- Man-in-the-middle атаки с изменением данных
- Ошибки в ПО, приводящие к повреждению данных
- Malware, изменяющее файлы

**Механизмы защиты:**

| Механизм | Описание | Пример |
|----------|----------|--------|
| **Checksums** | Контрольные суммы для обнаружения изменений | MD5, SHA-256 |
| **Digital Signatures** | Криптографическое подтверждение авторства | RSA, ECDSA |
| **Version Control** | Отслеживание изменений | Git, audit logs |
| **Input Validation** | Проверка входных данных | Sanitization, type checking |

```kotlin
// Пример: Проверка целостности данных
import java.security.MessageDigest

class DataIntegrityChecker {

    /**
     * Вычисляем хеш данных при сохранении.
     * При чтении сравниваем — если хеши не совпадают, данные изменены.
     */
    fun computeHash(data: ByteArray): String {
        val digest = MessageDigest.getInstance("SHA-256")
        val hashBytes = digest.digest(data)
        return hashBytes.joinToString("") { "%02x".format(it) }
    }

    fun verifyIntegrity(data: ByteArray, expectedHash: String): Boolean {
        val actualHash = computeHash(data)

        // Используем constant-time comparison для защиты от timing attacks
        return MessageDigest.isEqual(
            actualHash.toByteArray(),
            expectedHash.toByteArray()
        )
    }
}
```

### Availability (Доступность)

**Что это:** Гарантия, что системы и данные доступны авторизованным пользователям когда нужно.

**Почему важно:** Недоступная система может стоить миллионы. Например, простой Amazon на 1 час в Prime Day обошёлся бы в ~$100 миллионов потерянных продаж.

**Угрозы доступности:**
- DDoS атаки
- Ransomware (шифрует данные, делая их недоступными)
- Аппаратные сбои
- Природные катастрофы
- Ошибки конфигурации

**Механизмы защиты:**

| Механизм | Описание | Пример |
|----------|----------|--------|
| **Redundancy** | Дублирование компонентов | Multi-AZ, RAID |
| **Load Balancing** | Распределение нагрузки | AWS ALB, nginx |
| **DDoS Protection** | Защита от атак на отказ | CloudFlare, AWS Shield |
| **Backup & Recovery** | Резервное копирование | 3-2-1 rule |
| **Disaster Recovery** | План восстановления | Hot/warm/cold standby |

```yaml
# Пример: Kubernetes deployment для высокой доступности
apiVersion: apps/v1
kind: Deployment
metadata:
  name: critical-service
spec:
  # Минимум 3 реплики для отказоустойчивости
  replicas: 3

  strategy:
    type: RollingUpdate
    rollingUpdate:
      # Во время деплоя минимум 2 пода работают
      maxUnavailable: 1
      maxSurge: 1

  template:
    spec:
      # Распределяем поды по разным нодам
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchLabels:
                  app: critical-service
              topologyKey: kubernetes.io/hostname

      containers:
        - name: app
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"

          # Health checks для автоматического восстановления
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10

          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
```

### Баланс между C, I и A

Важно понимать: усиление одного аспекта может ослабить другие.

```
┌─────────────────────────────────────────────────────────────────┐
│                    TRADE-OFFS В CIA TRIAD                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MFA усиливает Confidentiality                                   │
│  НО: Если пользователь потеряет 2FA-устройство → Availability ↓  │
│                                                                  │
│  Шифрование защищает Confidentiality                             │
│  НО: Если ключи потеряны → Availability = 0                      │
│                                                                  │
│  Строгий Access Control повышает Confidentiality и Integrity     │
│  НО: Слишком строгий → пользователи не могут работать            │
│                                                                  │
│  Репликация данных улучшает Availability                         │
│  НО: Больше копий → больше точек утечки (Confidentiality ↓)      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Расширение CIA: Authenticity и Non-repudiation

Современные фреймворки добавляют ещё два принципа:

**Authenticity (Подлинность):** Уверенность, что данные, транзакция или коммуникация исходят от заявленного источника.

**Non-repudiation (Неотказуемость):** Невозможность отрицать совершённое действие. Например, цифровая подпись доказывает, что именно этот человек подписал документ.

## AAA Framework: Authentication, Authorization, Accounting

AAA — фреймворк для контроля и отслеживания доступа в компьютерных сетях.

```
┌─────────────────────────────────────────────────────────────────┐
│                       AAA FRAMEWORK                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌──────────────────┐                                           │
│   │  AUTHENTICATION  │ → "КТО ты?"                               │
│   │   (Аутентификация)│   Проверка личности                      │
│   └────────┬─────────┘                                           │
│            │                                                     │
│            ▼                                                     │
│   ┌──────────────────┐                                           │
│   │  AUTHORIZATION   │ → "ЧТО тебе разрешено?"                   │
│   │   (Авторизация)  │   Проверка прав доступа                   │
│   └────────┬─────────┘                                           │
│            │                                                     │
│            ▼                                                     │
│   ┌──────────────────┐                                           │
│   │   ACCOUNTING     │ → "ЧТО ты делал?"                         │
│   │   (Учёт)         │   Логирование действий                    │
│   └──────────────────┘                                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Authentication (Аутентификация)

**Что это:** Процесс проверки личности пользователя или устройства.

**Факторы аутентификации:**

| Фактор | Что это | Примеры |
|--------|---------|---------|
| **Something you know** | То, что знаешь | Пароль, PIN, секретный вопрос |
| **Something you have** | То, что имеешь | Телефон, токен, смарт-карта |
| **Something you are** | То, чем являешься | Отпечаток пальца, лицо, голос |
| **Somewhere you are** | Где находишься | Геолокация, IP-адрес |
| **Something you do** | Как действуешь | Паттерн набора, подпись |

**MFA (Multi-Factor Authentication):** Комбинация двух и более факторов. Если один скомпрометирован, остальные защищают.

### Authorization (Авторизация)

**Что это:** Определение, какие ресурсы и действия разрешены аутентифицированному пользователю.

**Модели авторизации:**

| Модель | Описание | Когда использовать |
|--------|----------|-------------------|
| **RBAC** | Role-Based Access Control — права по ролям | Организации с чёткой иерархией |
| **ABAC** | Attribute-Based Access Control — права по атрибутам | Сложные динамические правила |
| **ACL** | Access Control Lists — списки доступа | Файловые системы, сети |
| **PBAC** | Policy-Based Access Control — по политикам | Микросервисы, API |

```python
# Пример: RBAC реализация
from enum import Enum
from typing import Set

class Permission(Enum):
    READ = "read"
    WRITE = "write"
    DELETE = "delete"
    ADMIN = "admin"

class Role:
    def __init__(self, name: str, permissions: Set[Permission]):
        self.name = name
        self.permissions = permissions

# Определяем роли с их разрешениями
ROLES = {
    "viewer": Role("viewer", {Permission.READ}),
    "editor": Role("editor", {Permission.READ, Permission.WRITE}),
    "admin": Role("admin", {Permission.READ, Permission.WRITE, Permission.DELETE, Permission.ADMIN}),
}

class User:
    def __init__(self, username: str, roles: Set[str]):
        self.username = username
        self.roles = roles

    def has_permission(self, permission: Permission) -> bool:
        """
        Проверяем, есть ли у пользователя нужное разрешение
        через любую из его ролей.
        """
        for role_name in self.roles:
            role = ROLES.get(role_name)
            if role and permission in role.permissions:
                return True
        return False

# Использование
user = User("john", {"editor"})
print(user.has_permission(Permission.READ))   # True
print(user.has_permission(Permission.DELETE)) # False
```

### Accounting (Учёт)

**Что это:** Отслеживание и запись действий пользователей для аудита, анализа и расследований.

**Что логировать:**
- Время входа/выхода
- IP-адреса и геолокация
- Доступ к ресурсам
- Изменения данных
- Неудачные попытки входа
- Административные действия

## Defense in Depth: Многослойная защита

Defense in Depth (DiD) — стратегия, предполагающая множество независимых уровней защиты. Если один уровень преодолён, остальные продолжают защищать.

### Почему это работает

Cyber Security Breaches Survey 2024 показал: 59% средних и 69% крупных компаний в UK столкнулись с кибератаками за год. 31% бизнесов атакуют еженедельно.

Ни один инструмент не даёт 100% защиты. Но комбинация инструментов на разных уровнях делает атаку экспоненциально сложнее.

### Уровни защиты

#### 1. Physical Security (Физическая безопасность)

Защита аппаратного обеспечения и инфраструктуры от физического доступа.

- Контроль доступа в здания и серверные
- Видеонаблюдение
- Биометрические замки
- Защита от стихийных бедствий

#### 2. Network Security (Сетевая безопасность)

- **Firewalls:** Контроль входящего и исходящего трафика
- **IDS/IPS:** Обнаружение и предотвращение вторжений
- **Network Segmentation:** Изоляция критичных систем
- **VPN:** Защищённые соединения

#### 3. Endpoint Security (Защита конечных точек)

- **Antivirus/Anti-malware:** Базовая защита от вредоносного ПО
- **EDR (Endpoint Detection and Response):** Продвинутый мониторинг и реагирование
- **XDR (Extended Detection and Response):** EDR + сетевая телеметрия + облако
- **Hardening:** Отключение ненужных сервисов, минимальные привилегии

#### 4. Application Security (Безопасность приложений)

- **Secure coding practices:** OWASP guidelines
- **Input validation:** Защита от injection
- **MFA:** Многофакторная аутентификация
- **Vulnerability scanning:** Регулярное сканирование

#### 5. Data Security (Защита данных)

- **Encryption:** At rest и in transit
- **DLP (Data Loss Prevention):** Предотвращение утечек
- **Backup:** Регулярное резервное копирование
- **Access controls:** Минимальные привилегии

#### 6. Human Layer (Человеческий фактор)

По некоторым оценкам, 95% утечек данных связаны с человеческим фактором.

- **Security awareness training:** Обучение сотрудников
- **Simulated phishing:** Тестирование на фишинг
- **Clear policies:** Понятные политики безопасности

## Principle of Least Privilege (PoLP)

**Определение NIST:** "Принцип безопасности, согласно которому система должна ограничивать привилегии доступа пользователей (или процессов, действующих от имени пользователей) до минимума, необходимого для выполнения назначенных задач."

### Почему это критично

**Реальный пример (2024):** Утечка данных Snowflake затронула ~165 корпоративных клиентов, включая AT&T, Ticketmaster и Santander Bank. Хакеры украли учётные данные, которые имели более высокий уровень доступа, чем необходимо. Результат: доступ к банковским данным 30+ миллионов человек и 28 миллионам номеров кредитных карт.

```
┌─────────────────────────────────────────────────────────────────┐
│                    SNOWFLAKE BREACH (2024)                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ❌ КАК БЫЛО:                                                    │
│     Service account с admin-правами на все данные                │
│     Credentials украдены через phishing                          │
│     → Полный доступ ко всем данным клиентов                      │
│                                                                  │
│  ✅ КАК ДОЛЖНО БЫТЬ:                                             │
│     Отдельные accounts с минимальными правами                    │
│     Каждый видит только свои данные                              │
│     → Компрометация одного account не даёт доступ к другим       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Реализация Least Privilege

#### 1. Role-Based Access Control (RBAC)

```sql
-- PostgreSQL: Создание ролей с минимальными привилегиями

-- Роль только для чтения данных
CREATE ROLE readonly_user;
GRANT CONNECT ON DATABASE production TO readonly_user;
GRANT USAGE ON SCHEMA public TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;

-- Роль для приложения — только CRUD на своих таблицах
CREATE ROLE app_user;
GRANT CONNECT ON DATABASE production TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON users, orders, products TO app_user;
-- НЕ даём: DROP, ALTER, TRUNCATE, доступ к другим таблицам

-- Роль для analytics — только чтение агрегированных данных
CREATE ROLE analytics_user;
GRANT CONNECT ON DATABASE production TO analytics_user;
GRANT USAGE ON SCHEMA analytics TO analytics_user;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics TO analytics_user;
-- НЕ даём доступ к schema public с PII
```

#### 2. Just-in-Time Access (JIT)

Вместо постоянных привилегий — временный доступ по запросу.

```python
# Концептуальный пример JIT Access
from datetime import datetime, timedelta
from typing import Optional

class JITAccessManager:
    def __init__(self):
        self.elevated_access: dict[str, datetime] = {}

    def request_elevated_access(
        self,
        user_id: str,
        reason: str,
        duration_minutes: int = 60
    ) -> bool:
        """
        Пользователь запрашивает временные повышенные права.
        В реальной системе: требуется approve от manager.
        """
        # Логируем запрос для аудита
        print(f"[AUDIT] User {user_id} requested elevated access: {reason}")

        # Устанавливаем время истечения
        expiry = datetime.utcnow() + timedelta(minutes=duration_minutes)
        self.elevated_access[user_id] = expiry

        return True

    def has_elevated_access(self, user_id: str) -> bool:
        """
        Проверяем, есть ли у пользователя активные повышенные права.
        """
        expiry = self.elevated_access.get(user_id)
        if not expiry:
            return False

        if datetime.utcnow() > expiry:
            # Время истекло — отзываем доступ
            del self.elevated_access[user_id]
            print(f"[AUDIT] Elevated access for {user_id} expired")
            return False

        return True
```

#### 4. Privileged Access Management (PAM)

Для административных учётных записей:
- Хранение credentials в защищённом vault
- Checkout/checkin процедуры
- Session recording
- Automatic credential rotation

#### Preventive Controls (Превентивные)

**Цель:** Предотвратить инцидент до его возникновения.

| Контроль | Описание | Пример |
|----------|----------|--------|
| Access Control | Ограничение доступа | RBAC, ABAC |
| Encryption | Шифрование данных | AES-256, TLS |
| Firewall | Блокировка нежелательного трафика | AWS Security Groups |
| Authentication | Проверка личности | MFA, SSO |
| Security Training | Обучение персонала | Phishing awareness |

#### Detective Controls (Детективные)

**Цель:** Обнаружить инцидент во время или после его возникновения.

| Контроль | Описание | Пример |
|----------|----------|--------|
| IDS/IPS | Обнаружение вторжений | Snort, Suricata |
| SIEM | Агрегация и анализ логов | Splunk, ELK |
| Audit Logs | Запись действий | CloudTrail, Audit logs |
| Monitoring | Непрерывный мониторинг | Prometheus, Datadog |
| Vulnerability Scanning | Поиск уязвимостей | Nessus, Qualys |

#### Corrective Controls (Корректирующие)

**Цель:** Исправить последствия инцидента.

| Контроль | Описание | Пример |
|----------|----------|--------|
| Antivirus | Удаление malware | CrowdStrike, Defender |
| Backup & Recovery | Восстановление данных | 3-2-1 backup strategy |
| Patching | Исправление уязвимостей | Automated patching |
| Incident Response | Реагирование на инциденты | IR playbooks |

#### Compensating Controls (Компенсирующие)

**Цель:** Обеспечить защиту, когда основной контроль невозможен.

Пример: Legacy система не поддерживает современное шифрование → добавляем firewall, изолируем в отдельный сегмент сети, усиливаем мониторинг.

### Complete Controls

Best practice: использовать комбинацию preventive + detective + corrective на каждом уровне защиты.

```
Пример для защиты базы данных:

PREVENTIVE:
  • Network segmentation (DB в отдельном VLAN)
  • Access control (только app servers)
  • Encryption at rest (TDE)

DETECTIVE:
  • Query logging (все SQL-запросы)
  • Anomaly detection (необычные паттерны)
  • Failed login alerts

CORRECTIVE:
  • Automated backup (point-in-time recovery)
  • Incident response runbook
  • Forensic data collection
```

### Формула риска

```
RISK = LIKELIHOOD × IMPACT

Где:
- Likelihood = вероятность реализации угрозы
- Impact = ущерб при реализации угрозы
```

### Risk Treatment Strategies

После оценки риска есть четыре стратегии:

| Стратегия | Описание | Когда использовать |
|-----------|----------|-------------------|
| **Mitigate** | Снизить вероятность или ущерб | Риск высокий, но управляемый |
| **Transfer** | Передать риск третьей стороне | Страхование, outsourcing |
| **Accept** | Принять риск без действий | Риск низкий, стоимость митигации выше |
| **Avoid** | Избежать риска полностью | Отказаться от рискованной деятельности |


### Ограничения Risk Matrix

Risk matrix — полезный инструмент, но имеет ограничения:

1. **Субъективность:** Разные эксперты оценивают одинаковые риски по-разному
2. **Статичность:** Показывает snapshot, а риски меняются
3. **Упрощение:** Сложные риски не всегда укладываются в простую матрицу

### MITRE ATT&CK

MITRE ATT&CK — матрица из 500+ задокументированных техник реальных атакующих. Более детальна, чем Kill Chain.

**Структура:**
- **Tactics** (колонки): "ЧТО" хочет достичь атакующий (Initial Access, Execution, Persistence...)
- **Techniques** (строки): "КАК" это достигается (Phishing, PowerShell, Registry Run Keys...)
- **Procedures**: Конкретные примеры использования техник APT-группами

**Использование:**
- Threat Hunting: поиск индикаторов конкретных техник
- Red Team: планирование упражнений
- Detection: построение правил для SIEM
- Gap Analysis: проверка покрытия защиты

### ISO 27001:2022

Международный стандарт для Information Security Management System (ISMS).

**Особенности:**
- Сертификация (в отличие от NIST CSF)
- Акцент на risk-based management
- 93 контроля в Annex A
- Фокус на cloud security, data leakage, threat detection

**Сравнение с NIST:**
- Организация с ISO 27001 покрывает ~83% требований NIST CSF
- Организация с NIST CSF покрывает ~61% ISO 27001

### SOC 2

Стандарт для сервисных организаций, разработанный AICPA.

**5 Trust Services Criteria:**

| Критерий | Обязательный | Описание |
|----------|--------------|----------|
| **Security** | Да | Защита от несанкционированного доступа |
| **Availability** | Нет | Доступность систем |
| **Processing Integrity** | Нет | Корректность обработки данных |
| **Confidentiality** | Нет | Защита конфиденциальных данных |
| **Privacy** | Нет | Защита персональных данных |

**Типы отчётов:**
- **Type 1:** Оценка на момент времени (design)
- **Type 2:** Оценка за период 3-12 месяцев (operating effectiveness)

## Мифы и заблуждения

### Миф 1: "Мы слишком маленькие, нас не атакуют"

**Реальность:** Киберпреступники часто предпочитают маленькие цели — у них слабее защита. 43% кибератак направлены на малый бизнес. Автоматизированные атаки не разбирают размер компании.

### Миф 2: "Кибербезопасность — это задача IT"

**Реальность:** 74% CISO считают человеческий фактор главным риском (Proofpoint 2024). Безопасность — это ответственность каждого сотрудника. От CEO до intern.

### Миф 3: "Сильный пароль — это всё, что нужно"

**Реальность:** Пароли могут быть скомпрометированы через:
- Phishing
- Data breaches (password reuse)
- Brute force
- Social engineering

MFA снижает риск компрометации на 99.9% (Microsoft).

### Миф 4: "Больше инструментов = лучше защита"

**Реальность:** Избыток инструментов приводит к:
- Alert fatigue (слишком много алертов)
- Gaps в покрытии (инструменты не интегрированы)
- Увеличению attack surface

Важна интеграция и orchestration, а не количество.

### Миф 5: "Нас ещё не атаковали — значит, мы защищены"

**Реальность:** Среднее время обнаружения breach — 200+ дней. Вас могли атаковать, но вы ещё не знаете об этом.

### Миф 6: "Compliance = Security"

**Реальность:** Compliance — это минимальный базовый уровень. Организации, соответствующие всем стандартам, всё равно бывают взломаны. Compliance ≠ Security, но Security → Compliance.

### Миф 7: "Антивирус защитит от всего"

**Реальность:** Антивирус не защитит от:
- Zero-day эксплойтов
- Social engineering
- Insider threats
- Advanced Persistent Threats
- Misconfiguration

### Миф 8: "Кибербезопасность слишком дорогая"

**Реальность:** Средняя стоимость data breach — $4.45M. Стоимость базовой безопасности — значительно меньше. Это инвестиция, а не расход.

## Практические рекомендации

### Для разработчиков

```
┌─────────────────────────────────────────────────────────────────┐
│              SECURITY CHECKLIST ДЛЯ РАЗРАБОТЧИКОВ                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  INPUT VALIDATION                                                │
│  □ Валидируй все входные данные (never trust user input)         │
│  □ Используй parameterized queries (защита от SQL injection)     │
│  □ Encode output (защита от XSS)                                 │
│                                                                  │
│  AUTHENTICATION & AUTHORIZATION                                  │
│  □ Реализуй MFA для критичных операций                           │
│  □ Используй проверенные библиотеки (не пиши crypto сам)         │
│  □ Проверяй авторизацию на каждом endpoint                       │
│                                                                  │
│  DATA PROTECTION                                                 │
│  □ Шифруй sensitive data at rest и in transit                    │
│  □ Не храни секреты в коде (используй vault/env vars)            │
│  □ Минимизируй хранение PII                                      │
│                                                                  │
│  SECURE CODING                                                   │
│  □ Обновляй зависимости (dependency scanning)                    │
│  □ Используй SAST/DAST в CI/CD                                   │
│  □ Следуй OWASP guidelines                                       │
│                                                                  │
│  LOGGING & MONITORING                                            │
│  □ Логируй security-relevant события                             │
│  □ Не логируй sensitive data (пароли, tokens)                    │
│  □ Настрой алерты на аномалии                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Для архитекторов

1. **Design with security in mind** — Security не добавляется в конце
2. **Assume breach** — Проектируй так, будто периметр уже пробит
3. **Defense in depth** — Множество независимых слоёв защиты
4. **Least privilege** — Минимальные права везде
5. **Fail secure** — При ошибке система должна быть в безопасном состоянии
6. **Zero Trust** — Не доверяй ничему по умолчанию

### Для организаций

1. **Security awareness training** — Регулярное обучение всех сотрудников
2. **Incident response plan** — Готовый план на случай инцидента
3. **Regular assessments** — Pentest, vulnerability scanning
4. **Backup & recovery** — Тестируемые backup'ы
5. **Vendor management** — Оценка безопасности третьих сторон
