# Сетевые технологии: от битов до микросервисов

> **Энциклопедия сетевых технологий для разработчиков** — 23 материала, от базовых концепций до production-уровня

---

## Теоретические основы

> **Компьютерная сеть** — система взаимосвязанных вычислительных устройств, обменивающихся данными по общим протоколам через каналы связи (Tanenbaum & Wetherall 2011).

**Фундаментальные модели сетевого стека:**

| Модель | Год | Уровней | Происхождение | Применение |
|--------|-----|---------|---------------|------------|
| **OSI** | 1984 | 7 | ISO 7498, академическая | Эталонная модель, обучение |
| **TCP/IP** | 1974 | 4 | Cerf & Kahn, DARPA | Реальный Internet |
| **Гибридная** | ~ | 5 | Kurose & Ross учебник | Практическое обучение |

**Ключевые принципы проектирования Internet:**

- **End-to-End Principle** (Saltzer, Reed & Clark 1984) — интеллект на краях сети, ядро простое
- **Layering** (Dijkstra 1968) — каждый уровень предоставляет сервис вышестоящему, скрывая детали реализации
- **Packet Switching** (Baran 1964, Davies 1965) — данные разбиваются на пакеты, маршрутизируемые независимо
- **Best-Effort Delivery** — сеть не гарантирует доставку, надёжность обеспечивается протоколами верхних уровней

**Хронология ключевых событий:**

```
1969  ARPANET (4 узла)          1991  World Wide Web (Berners-Lee)
1974  TCP/IP (Cerf & Kahn)      1995  HTTP/1.0, коммерческий Internet
1983  DNS (Mockapetris)         2015  HTTP/2 (SPDY → RFC 7540)
1984  OSI Model (ISO 7498)      2022  HTTP/3 (QUIC → RFC 9114)
```

> **Закон Меткалфа** (1980) — ценность сети пропорциональна квадрату числа подключённых узлов: V ∝ n².

Связанные материалы: [[network-physical-layer]], [[network-transport-layer]], [[network-ip-routing]], [[network-dns-tls]], [[network-http-evolution]]

## Часть 1: Интуиция без кода

> Сетевые технологии кажутся магией, пока не понимаешь, что это просто очень организованная почтовая служба для компьютеров.

### Аналогия 1: Почтовая служба (Сетевой стек)

Представьте, что вы отправляете письмо другу в другой город:

```
ВЫ ПИШЕТЕ ПИСЬМО (Application Layer):
┌─────────────────────────────────────────────────────────────────┐
│  "Привет, Вася! Как дела? Приезжай в гости!"                    │
│                                                                  │
│  Это СОДЕРЖИМОЕ — то, что вы хотите передать                    │
│  В сети: HTTP запрос, JSON данные, HTML страница                │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
КЛАДЁТЕ В КОНВЕРТ (Transport Layer):
┌─────────────────────────────────────────────────────────────────┐
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ Кому: Вася (порт 443)                                    │    │
│  │ От кого: Петя (порт 52431)                               │    │
│  │ Тип: Заказное с уведомлением (TCP)                       │    │
│  │      или Обычное без подтверждения (UDP)                 │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  TCP = заказное письмо с подтверждением получения               │
│  UDP = открытка — быстро, но без гарантий                       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
ПИШЕТЕ АДРЕС НА КОНВЕРТЕ (Network Layer):
┌─────────────────────────────────────────────────────────────────┐
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ Откуда: Москва, ул. Ленина 10 (192.168.1.100)           │    │
│  │ Куда: Питер, ул. Невского 5 (93.184.216.34)             │    │
│  │ Индекс: 190000 (маршрутизация)                          │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  IP-адрес = почтовый адрес в мире компьютеров                   │
│  Маршрутизаторы = сортировочные центры почты                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
ПОЧТАЛЬОН ЗАБИРАЕТ (Data Link / Physical Layer):
┌─────────────────────────────────────────────────────────────────┐
│  Почтальон знает: "Это письмо для соседнего дома" (MAC адрес)   │
│  Или: "Нужно отнести на почту для дальней отправки"             │
│                                                                  │
│  MAC = адрес конкретного почтового ящика на вашей улице         │
│  Физический уровень = грузовики, самолёты, дороги               │
└─────────────────────────────────────────────────────────────────┘
```

**Ключевой инсайт:** Каждый уровень добавляет свой "конверт" поверх предыдущего. Это называется **инкапсуляция**.

### Аналогия 2: Аэропорт (Маршрутизация)

Как ваш чемодан попадает из Москвы в Токио?

```
Ваш чемодан (пакет данных):
┌─────────────────────────────────────────────────────────────────┐
│  Бирка: FROM: Москва (SVO)  TO: Токио (NRT)                     │
└─────────────────────────────────────────────────────────────────┘

Путь чемодана (маршрутизация):

Москва (SVO) ─────► Дубай (DXB) ─────► Токио (NRT)
     │                   │                  │
     │                   │                  │
     ▼                   ▼                  ▼
"Следующий рейс     "Следующий рейс    "Конечная
 на Дубай"           на Токио"          точка!"

КАЖДЫЙ АЭРОПОРТ (роутер) знает:
• НЕ весь маршрут целиком
• А только: "Куда отправить ДАЛЬШЕ?"

Таблица маршрутизации аэропорта Дубай:
┌────────────────┬───────────────────┐
│ Если в Азию    │ → Терминал 3      │
│ Если в Европу  │ → Терминал 1      │
│ Если в Африку  │ → Терминал 2      │
└────────────────┴───────────────────┘

В сети: роутер смотрит IP-адрес назначения и решает,
        через какой интерфейс отправить пакет дальше.
```

### Аналогия 3: Телефонный звонок vs Рация (TCP vs UDP)

```
TCP = ТЕЛЕФОННЫЙ ЗВОНОК:
┌─────────────────────────────────────────────────────────────────┐
│  1. Набираете номер (SYN)                                       │
│  2. Гудки... (ожидание)                                         │
│  3. "Алло?" — соединение установлено (SYN-ACK, ACK)            │
│  4. Разговор с подтверждением: "Понял" — "Да" (ACK на данные)  │
│  5. "Пока!" — "Пока!" — завершение (FIN)                       │
│                                                                  │
│  ✅ Гарантия доставки (если связь есть)                         │
│  ✅ Порядок сообщений сохраняется                               │
│  ❌ Накладные расходы на подтверждения                          │
└─────────────────────────────────────────────────────────────────┘

UDP = РАЦИЯ / ГРОМКОГОВОРИТЕЛЬ:
┌─────────────────────────────────────────────────────────────────┐
│  1. Нажал кнопку — говоришь (отправил пакет)                    │
│  2. Услышали? Не услышали? — не знаю                            │
│  3. Нет "соединения" — просто кричишь в эфир                    │
│                                                                  │
│  ✅ Минимальная задержка                                        │
│  ✅ Нет накладных расходов                                      │
│  ❌ Пакеты могут потеряться                                     │
│  ❌ Порядок не гарантирован                                     │
│                                                                  │
│  Применение: видеозвонки, онлайн-игры, DNS запросы              │
│  "Лучше пропустить кадр, чем ждать его 2 секунды"               │
└─────────────────────────────────────────────────────────────────┘
```

### Аналогия 4: Ресторан с доставкой (Client-Server)

```
ВЫ (Client/Браузер)              РЕСТОРАН (Server)
       │                               │
       │  "Хочу пиццу Маргарита"       │
       │  (HTTP GET /pizza/margherita) │
       ├──────────────────────────────►│
       │                               │
       │                               │ Повар готовит...
       │                               │
       │   "Вот ваша пицца!"           │
       │   (HTTP 200 OK + данные)      │
       │◄──────────────────────────────┤
       │                               │

Что может пойти не так?

HTTP 200 — "Ваш заказ готов!" (успех)
HTTP 301 — "Мы переехали, вот новый адрес" (redirect)
HTTP 404 — "Такой пиццы нет в меню" (not found)
HTTP 500 — "Кухня сломалась" (server error)
HTTP 502 — "Курьер не смог забрать с кухни" (bad gateway)
HTTP 503 — "Слишком много заказов, попробуйте позже" (overloaded)
```

## Часть 2: Почему это сложно

### Ошибка 1: "Не знаю, где искать проблему"

**СИМПТОМ:** "Сайт не открывается" — и дальше тупик.

```
НЕПРАВИЛЬНЫЙ ПОДХОД:
┌─────────────────────────────────────────────────────────────────┐
│  "Не работает" → Гуглю ошибку → Копирую решение со StackOverflow│
│  → Не помогло → Гуглю другое → 3 часа потеряно                  │
└─────────────────────────────────────────────────────────────────┘

ПРАВИЛЬНЫЙ ПОДХОД (диагностика по уровням):
┌─────────────────────────────────────────────────────────────────┐
│  1. ФИЗИЧЕСКИЙ: Есть ли вообще сеть?                            │
│     → ping 8.8.8.8                                              │
│     Если нет — проблема с кабелем/WiFi/провайдером              │
│                                                                  │
│  2. DNS: Резолвится ли имя?                                     │
│     → dig example.com                                           │
│     Если нет — проблема с DNS (попробуй 8.8.8.8 или 1.1.1.1)   │
│                                                                  │
│  3. ТРАНСПОРТ: Открыт ли порт?                                  │
│     → curl -v https://example.com                               │
│     → telnet example.com 443                                    │
│     Если нет — firewall, security group, порт закрыт           │
│                                                                  │
│  4. ПРИЛОЖЕНИЕ: Что отвечает сервер?                            │
│     → curl -I https://example.com                               │
│     HTTP 502? 503? 500? — проблема на стороне сервера          │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Диагностируй от нижних уровней к верхним: Physical → Network → Transport → Application.

### Ошибка 2: "It's always DNS" — но почему?

**СИМПТОМ:** Все шутят про DNS, но ты не понимаешь почему это так часто проблема.

```
ПОЧЕМУ DNS ТАК ЧАСТО "ВИНОВАТ":
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  1. DNS — ПЕРВЫЙ шаг каждого запроса                            │
│     Браузер: "api.example.com это какой IP?"                    │
│     Если DNS не работает — ВСЁ не работает                      │
│                                                                  │
│  2. DNS кэшируется ВЕЗДЕ (и по-разному!)                        │
│     • Браузер: ~1 минута                                        │
│     • ОС: ~1-30 минут                                           │
│     • Роутер: может быть часами                                 │
│     • ISP DNS: может быть ДНЯМИ                                 │
│                                                                  │
│  3. TTL (Time To Live) игнорируется                             │
│     Ты поменял DNS запись, TTL=60 сек                           │
│     Но ISP кэширует на 24 часа → "не работает!"                │
│                                                                  │
│  4. DNS резолверы разные                                        │
│     У тебя: 8.8.8.8 (Google) — работает                        │
│     У клиента: ISP DNS — показывает старое                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:**
```bash
# Проверь DNS с разных резолверов
dig @8.8.8.8 example.com      # Google DNS
dig @1.1.1.1 example.com      # Cloudflare DNS
dig @dns.your-isp.com example.com  # ISP DNS

# Очисти локальный кэш
# macOS:
sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
# Linux:
sudo systemd-resolve --flush-caches
# Windows:
ipconfig /flushdns
```

### Ошибка 3: "HTTPS = полная безопасность"

**СИМПТОМ:** "У нас HTTPS, значит всё зашифровано и безопасно!"

```
ЧТО HTTPS ШИФРУЕТ:
┌─────────────────────────────────────────────────────────────────┐
│  ✅ Тело запроса (POST данные, JSON)                            │
│  ✅ URL path (/api/users/123)                                   │
│  ✅ HTTP заголовки (Authorization, Cookie)                      │
│  ✅ Ответ сервера полностью                                     │
└─────────────────────────────────────────────────────────────────┘

ЧТО HTTPS НЕ СКРЫВАЕТ:
┌─────────────────────────────────────────────────────────────────┐
│  ❌ IP-адрес клиента и сервера (видны всем на пути)            │
│  ❌ Домен сервера (SNI в TLS handshake — до TLS 1.3 ECH)       │
│  ❌ Размер передаваемых данных                                  │
│  ❌ Время запросов (timing attacks)                             │
│  ❌ Факт соединения (metadata)                                  │
│                                                                  │
│  Пример: ISP видит что ты ходишь на bank.com,                   │
│          хоть и не видит что именно делаешь                     │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** HTTPS защищает данные в транзите, но для полной приватности нужен VPN/Tor (скрывает IP) + ECH (скрывает домен).

### Ошибка 4: "TCP гарантирует доставку"

**СИМПТОМ:** "Я отправил данные по TCP, значит они точно дошли!"

```
ЧТО TCP РЕАЛЬНО ГАРАНТИРУЕТ:
┌─────────────────────────────────────────────────────────────────┐
│  ✅ Если данные доставлены — они доставлены ПРАВИЛЬНО          │
│     (без ошибок, в правильном порядке)                          │
│  ✅ Отправитель УЗНАЕТ, если доставка не удалась               │
│     (connection reset, timeout)                                  │
└─────────────────────────────────────────────────────────────────┘

ЧТО TCP НЕ ГАРАНТИРУЕТ:
┌─────────────────────────────────────────────────────────────────┐
│  ❌ Что данные БУДУТ доставлены                                 │
│     Сеть упала? TCP вернёт ошибку, не будет бесконечно ждать   │
│                                                                  │
│  ❌ Что получатель ОБРАБОТАЛ данные                             │
│     send() вернул OK = данные в буфере ОС                      │
│     НЕ значит что приложение их получило                        │
│                                                                  │
│  ❌ Что данные дошли за разумное время                          │
│     Retransmissions могут добавить секунды задержки             │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Для критичных данных используй application-level acknowledgement (приложение подтверждает получение), а не полагайся только на TCP.

### Ошибка 5: "Локально работает, в проде — нет"

**СИМПТОМ:** "На моём компьютере всё работает!" 🤷

```
ТИПИЧНЫЕ ПРИЧИНЫ:
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  1. FIREWALL / SECURITY GROUPS                                  │
│     Локально: все порты открыты                                 │
│     Прод: открыт только 443, остальное DROP                     │
│                                                                  │
│  2. DNS РЕЗОЛВИНГ                                               │
│     Локально: localhost, 127.0.0.1                              │
│     Прод: нужен internal DNS, service discovery                 │
│                                                                  │
│  3. NETWORK POLICIES (Kubernetes)                               │
│     Локально: docker-compose, всё в одной сети                  │
│     Прод: pods в разных namespaces не видят друг друга          │
│                                                                  │
│  4. NAT / PRIVATE NETWORKS                                      │
│     Локально: прямой доступ к базе                              │
│     Прод: база в private subnet, нужен bastion/VPN              │
│                                                                  │
│  5. TLS CERTIFICATES                                            │
│     Локально: self-signed или HTTP                              │
│     Прод: нужен валидный сертификат                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Чеклист перед деплоем:
```
□ Security Groups разрешают нужные порты?
□ DNS записи настроены?
□ Network Policies в K8s разрешают трафик?
□ TLS сертификаты валидны и не просрочены?
□ Env variables для endpoints правильные?
```

### Ошибка 6: "HTTP/2 всегда быстрее HTTP/1.1"

**СИМПТОМ:** "Включили HTTP/2, а быстрее не стало (или стало хуже)!"

```
КОГДА HTTP/2 БЫСТРЕЕ:
┌─────────────────────────────────────────────────────────────────┐
│  ✅ Много параллельных запросов (страница с 50 ресурсами)      │
│  ✅ Мультиплексирование: все запросы через одно соединение      │
│  ✅ Header compression (HPACK) экономит байты                   │
│  ✅ Server Push (сервер отправляет ресурсы заранее)            │
└─────────────────────────────────────────────────────────────────┘

КОГДА HTTP/2 НЕ ПОМОГАЕТ ИЛИ ХУЖЕ:
┌─────────────────────────────────────────────────────────────────┐
│  ❌ Один-два запроса — разница минимальна                       │
│                                                                  │
│  ❌ Packet loss в сети:                                         │
│     HTTP/1.1: потерян 1 запрос из 6 = остальные 5 OK           │
│     HTTP/2: потерян 1 пакет TCP = ВСЕ 6 запросов ждут          │
│     (Head-of-Line blocking на уровне TCP)                       │
│                                                                  │
│  ❌ Плохой WiFi / мобильная сеть                                │
│     HTTP/2 хуже из-за TCP HOL blocking                          │
│     HTTP/3 (QUIC) решает эту проблему!                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Для плохих сетей (mobile) используй HTTP/3 (QUIC). Он решает HOL blocking, работая поверх UDP с собственным контролем надёжности.

```
Сетевой пакет — это матрёшка:

┌─────────────────────────────────────────────────────────────────┐
│  Ethernet Frame (Data Link)                                      │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │  IP Packet (Network)                                         ││
│  │  ┌─────────────────────────────────────────────────────────┐││
│  │  │  TCP Segment (Transport)                                 │││
│  │  │  ┌─────────────────────────────────────────────────────┐│││
│  │  │  │  HTTP Request (Application)                          ││││
│  │  │  │  ┌─────────────────────────────────────────────────┐││││
│  │  │  │  │                                                  │││││
│  │  │  │  │    {"name": "Вася", "age": 25}                  │││││
│  │  │  │  │    (Ваши данные — самая маленькая матрёшка)     │││││
│  │  │  │  │                                                  │││││
│  │  │  │  └─────────────────────────────────────────────────┘││││
│  │  │  └─────────────────────────────────────────────────────┘│││
│  │  └─────────────────────────────────────────────────────────┘││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘

Каждый уровень:
• Добавляет свой header (обёртку)
• НЕ знает и НЕ заботится о внутренностях
• Общается только со своим уровнем на другой стороне

IP не знает, что внутри TCP. TCP не знает, что внутри HTTP.
Это называется РАЗДЕЛЕНИЕ ОТВЕТСТВЕННОСТИ (separation of concerns).
```

### Модель 3: Труба и вода (Bandwidth vs Latency)

```
BANDWIDTH (пропускная способность) = ДИАМЕТР трубы
LATENCY (задержка) = ДЛИНА трубы

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Тонкая короткая труба:          Толстая длинная труба:        │
│  ┌───────────┐                   ┌═══════════════════════════┐ │
│  │░░░░░░░░░░░│                   ║░░░░░░░░░░░░░░░░░░░░░░░░░░░║ │
│  └───────────┘                   └═══════════════════════════┘ │
│  Bandwidth: 10 Mbps              Bandwidth: 1 Gbps             │
│  Latency: 5ms                    Latency: 100ms                │
│                                                                  │
│  Скачать 1 MB:                   Скачать 1 MB:                 │
│  Время ≈ 5ms + (1MB/10Mbps)      Время ≈ 100ms + (1MB/1Gbps)   │
│       ≈ 5ms + 800ms = 805ms           ≈ 100ms + 8ms = 108ms    │
│                                                                  │
│  Скачать 1 KB (маленький запрос):                              │
│  Тонкая: 5ms + ~1ms = 6ms        Толстая: 100ms + ~0ms = 100ms │
│                                                                  │
│  ВЫВОД: Для маленьких запросов LATENCY важнее BANDWIDTH!       │
│         Для больших файлов — наоборот.                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

Для API запросов (обычно маленькие):
• Снижай latency (ближе сервер, меньше hops)
• Bandwidth почти не важен

Для видеостриминга (большие данные):
• Bandwidth критичен
• Небольшая latency терпима
```

## Почему разработчику нужно понимать сети?

**"It's always DNS"** — шутка, которая перестаёт быть смешной, когда ты не можешь понять, почему приложение не работает.

Большинство современных приложений — это **сетевые приложения**. Даже "локальный" код часто обращается к API, базам данных, кэшам. Когда что-то ломается, понимание сетевого стека — разница между "гуглить ошибку 3 часа" и "понять проблему за 5 минут".

### Реальные сценарии, где это критично

```
Сценарий 1: "Приложение тормозит"
├─ Без знания сетей: "Наверное, сервер перегружен"
└─ С пониманием: Открываешь DevTools → Network tab
   ├─ Видишь: TTFB (Time to First Byte) 2 секунды
   ├─ Копаешь глубже: DNS resolution 500ms
   └─ Решение: Включить DNS prefetch или сменить резолвер

Сценарий 2: "API возвращает 502 Bad Gateway"
├─ Без знания сетей: "Что-то сломалось на сервере"
└─ С пониманием: 502 = прокси не получил ответ от upstream
   ├─ Проверяешь: Жив ли backend? → curl напрямую
   ├─ Смотришь: Health checks load balancer'а
   └─ Находишь: Backend упал из-за OOM

Сценарий 3: "Работает локально, не работает в проде"
├─ Без знания сетей: "Магия"
└─ С пониманием:
   ├─ Проверяешь: Security Groups / Firewall rules
   ├─ Сравниваешь: Network policies в K8s
   └─ Находишь: Порт закрыт в production VPC
```

## Модели сетевого взаимодействия

### OSI vs TCP/IP: почему две модели?

**OSI Model (1984)** — академическая, создана ISO для стандартизации. 7 уровней, но на практике некоторые "склеиваются".

**TCP/IP Model (1970-е)** — практическая, выросла из ARPANET. 4 уровня, отражает реальный интернет.

```
┌─────────────────────────────────────────────────────────────────┐
│                    OSI Model (7 уровней)                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  7. Application    │                                            │
│  6. Presentation   │ ──► Application Layer (HTTP, DNS, TLS)     │
│  5. Session        │                                            │
│                    │                                            │
│  4. Transport      │ ──► Transport Layer (TCP, UDP, QUIC)       │
│                    │                                            │
│  3. Network        │ ──► Internet Layer (IP, ICMP, ARP)         │
│                    │                                            │
│  2. Data Link      │                                            │
│  1. Physical       │ ──► Network Access (Ethernet, WiFi)        │
│                    │                                            │
├─────────────────────────────────────────────────────────────────┤
│                    TCP/IP Model (4 уровня)                      │
└─────────────────────────────────────────────────────────────────┘
```

**Почему OSI до сих пор изучают?** Она даёт более детальную классификацию. Когда говоришь "L7 load balancer" или "L4 firewall" — это терминология OSI.

### Как данные проходят через стек

```
Отправитель (твоё приложение)
         │
         ▼
┌─────────────────┐
│   Application   │  HTTP Request: GET /api/users
│     Layer       │  Headers: Host, Content-Type, Authorization
└────────┬────────┘
         │ Добавляется: HTTP headers
         ▼
┌─────────────────┐
│   Transport     │  TCP Segment
│     Layer       │  Src Port: 52431, Dst Port: 443
└────────┬────────┘  Seq: 1000, Ack: 500, Flags: ACK,PSH
         │ Добавляется: TCP header (20+ bytes)
         ▼
┌─────────────────┐
│   Internet      │  IP Packet
│     Layer       │  Src IP: 192.168.1.100
└────────┬────────┘  Dst IP: 93.184.216.34, TTL: 64
         │ Добавляется: IP header (20+ bytes)
         ▼
┌─────────────────┐
│   Network       │  Ethernet Frame
│   Access        │  Src MAC: AA:BB:CC:DD:EE:FF
└────────┬────────┘  Dst MAC: 11:22:33:44:55:66
         │ Добавляется: Ethernet header (14 bytes) + FCS (4 bytes)
         ▼
    ═══════════════
    Физическая среда (провод, WiFi, оптика)
    ═══════════════
```

**Инкапсуляция**: каждый уровень "оборачивает" данные предыдущего, добавляя свой заголовок.

**Декапсуляция**: получатель "разворачивает" в обратном порядке.

## Эволюция сетевых технологий: хронология

```
1969    ARPANET — первая пакетная сеть (4 узла)
        │
1974    TCP/IP — Vint Cerf & Bob Kahn публикуют спецификацию
        │
1983    DNS — Domain Name System заменяет hosts.txt
        │
1989    HTTP — Tim Berners-Lee изобретает WWW
        │
1995    HTTP/1.0 — RFC 1945
        │
1997    HTTP/1.1 — RFC 2068 (persistent connections, chunked transfer)
        │
1999    TLS 1.0 — замена SSL 3.0
        │
2009    WebSocket — RFC 6455 draft, bidirectional communication
        │
2012    SPDY — Google экспериментирует с HTTP/2 прототипом
        │
2015    HTTP/2 — RFC 7540 (multiplexing, header compression)
        │
2016    QUIC — Google начинает эксперименты
        │
2018    TLS 1.3 — RFC 8446 (1-RTT handshake, forward secrecy)
        │
2022    HTTP/3 — RFC 9114 (QUIC-based, UDP transport)
        │
2024    ~30% top websites поддерживают HTTP/3
        QUIC становится стандартом для mobile-first приложений
```

**Ключевой паттерн эволюции**: каждое поколение решает проблемы предыдущего:
- HTTP/1.1 → HTTP/2: решает проблему head-of-line blocking на уровне HTTP
- HTTP/2 → HTTP/3: решает HOL blocking на уровне TCP через переход на QUIC

## Частые заблуждения

### "TCP гарантирует доставку"

**Неточно.** TCP гарантирует, что если данные доставлены, они доставлены в правильном порядке и без ошибок. Но TCP **не гарантирует** доставку в случае обрыва соединения. Приложение должно обрабатывать connection errors.

### "HTTPS шифрует всё"

**Частично.** HTTPS шифрует **содержимое** запроса (body, headers, path). Но **не скрывает**:
- IP-адреса клиента и сервера
- Размер передаваемых данных
- Timing информацию
- SNI (Server Name Indication) — hostname до TLS 1.3 ESNI/ECH

### "UDP быстрее TCP"

**Зависит от контекста.** UDP имеет меньший overhead, но:
- Нет congestion control → может перегрузить сеть
- Нет retransmission → потеря данных
- QUIC (поверх UDP) добавляет reliability и часто быстрее TCP из-за 0-RTT

### "HTTP/2 всегда быстрее HTTP/1.1"

**Не всегда.** HTTP/2 выигрывает при множестве параллельных запросов. Для одиночного запроса разница минимальна. При packet loss HTTP/2 может быть **медленнее** из-за HOL blocking на уровне TCP.

## Проверь себя

> [!question]- Почему при потере одного TCP-пакета HTTP/2 блокирует ВСЕ параллельные потоки, а HTTP/1.1 — только один из нескольких соединений? Как HTTP/3 решает эту проблему?
> HTTP/2 мультиплексирует все потоки через одно TCP-соединение. TCP гарантирует порядок доставки, поэтому потеря одного пакета блокирует весь поток данных до его ретрансмиссии — это Head-of-Line (HOL) blocking на уровне TCP. HTTP/1.1 открывает 6 отдельных TCP-соединений, и потеря пакета в одном не влияет на остальные 5. HTTP/3 использует QUIC поверх UDP, где каждый поток имеет собственный контроль надёжности — потеря пакета одного потока не блокирует другие.

> [!question]- Разработчик настроил DNS-запись с TTL=60 секунд, но клиенты видят старый IP ещё сутки. Объясни все уровни кэширования DNS, из-за которых это происходит.
> DNS кэшируется на нескольких уровнях: браузер (~1 минута), ОС (~1-30 минут), домашний роутер (может часами), DNS-резолвер ISP (может игнорировать TTL и кэшировать до 24 часов). Даже если вы установили TTL=60 секунд, ISP-резолвер может не уважать этот TTL и хранить запись значительно дольше. Поэтому при миграции используют стратегию: заранее снижают TTL до минимума, ждут распространения, затем меняют запись.

> [!question]- Для API-запросов (маленькие данные) важнее latency, а для стриминга видео — bandwidth. Как эта разница влияет на выбор архитектуры: расположение серверов, CDN, протоколы?
> Для API (маленькие запросы) latency доминирует: пакет данных передаётся за доли миллисекунды, но RTT до сервера может занимать 100+ мс. Поэтому критично размещать edge-серверы ближе к пользователям, использовать CDN для кэширования, применять 0-RTT (TLS 1.3, QUIC), DNS prefetch. Для видеостриминга нужен высокий bandwidth — здесь важнее пропускная способность канала, буферизация, адаптивный битрейт. Это связано с архитектурой распределённых систем ([[architecture-overview]]) и паттернами API ([[api-design]]).

> [!question]- Почему `send()` вернул OK в TCP-соединении не означает, что получатель обработал данные? Какой механизм нужен на уровне приложения?
> `send()` возвращает OK, когда данные записаны в буфер ОС отправителя. Далее ОС асинхронно передаёт их по TCP. Даже получение TCP ACK означает лишь, что данные попали в буфер ОС получателя — приложение могло ещё не прочитать их или упасть до обработки. Для критичных данных нужен application-level acknowledgement: приложение-получатель явно подтверждает обработку (например, HTTP 200 с телом ответа, или ACK-сообщение в очереди сообщений).

> [!question]- HTTPS шифрует URL path и заголовки, но не скрывает домен сервера. Почему это так и какие технологии решают эту проблему?
> При TLS handshake клиент отправляет Server Name Indication (SNI) в открытом виде — это нужно, чтобы сервер с несколькими доменами на одном IP знал, какой сертификат предъявить. IP-адреса клиента и сервера также видны, так как они находятся на уровне IP (L3), ниже шифрования TLS. Для скрытия SNI разработана технология ECH (Encrypted Client Hello) в TLS 1.3. Для скрытия IP используются VPN или Tor.

## Ключевые карточки

Что такое инкапсуляция в сетевом стеке?
?
Каждый уровень стека "оборачивает" данные предыдущего уровня, добавляя свой заголовок. Ethernet Frame содержит IP Packet, который содержит TCP Segment, который содержит HTTP-данные. Получатель разворачивает в обратном порядке (декапсуляция).

Чем TCP отличается от UDP? Когда использовать каждый?
?
TCP — надёжная доставка с подтверждением, порядок сохраняется, есть handshake (SYN, SYN-ACK, ACK). UDP — без подтверждения, без гарантии порядка, минимальная задержка. TCP — для веб, файлов, API. UDP — для видеозвонков, онлайн-игр, DNS, где потеря кадра лучше задержки.

Что такое Head-of-Line (HOL) blocking и почему это проблема HTTP/2?
?
HOL blocking — ситуация, когда потеря одного пакета блокирует обработку всех последующих. HTTP/2 мультиплексирует все запросы через одно TCP-соединение, поэтому потеря одного пакета TCP останавливает все потоки. HTTP/3 решает это через QUIC (UDP), где каждый поток независим.

Как диагностировать сетевую проблему по уровням стека?
?
Снизу вверх: 1) Physical — `ping 8.8.8.8` (есть ли сеть?), 2) DNS — `dig example.com` (резолвится ли имя?), 3) Transport — `curl -v` / `telnet` (открыт ли порт?), 4) Application — `curl -I` (что отвечает сервер? 4xx/5xx?).

В чём разница между Bandwidth и Latency?
?
Bandwidth (пропускная способность) — сколько данных можно передать за единицу времени (диаметр трубы). Latency (задержка) — сколько времени данные в пути (длина трубы). Для маленьких API-запросов важнее latency, для больших файлов и стриминга — bandwidth.

Какие данные HTTPS НЕ скрывает?
?
IP-адреса клиента и сервера, домен сервера (SNI до ECH в TLS 1.3), размер передаваемых данных, время запросов (timing), факт установления соединения (метаданные). HTTPS шифрует только содержимое: body, headers, URL path.

Что такое Three-Way Handshake в TCP?
?
Установление TCP-соединения: 1) Клиент отправляет SYN, 2) Сервер отвечает SYN-ACK, 3) Клиент подтверждает ACK. После этого соединение установлено. Завершение — через FIN.

Чем отличаются модели OSI (7 уровней) и TCP/IP (4 уровня)?
?
OSI — академическая, 7 уровней (Physical, Data Link, Network, Transport, Session, Presentation, Application). TCP/IP — практическая, 4 уровня: Network Access, Internet, Transport, Application. Верхние 3 уровня OSI объединены в Application TCP/IP. Терминология OSI (L4, L7) используется повсеместно.

Почему DNS так часто является причиной сетевых проблем?
?
DNS — первый шаг каждого запроса; если DNS не работает — ничего не работает. DNS кэшируется на разных уровнях (браузер, ОС, роутер, ISP) с разными TTL, что приводит к рассинхронизации. ISP может игнорировать установленный TTL.

Что такое CIDR-нотация /24, /16, /8?
?
CIDR указывает размер подсети через количество фиксированных бит маски: /24 = 256 адресов (одна "улица"), /16 = 65 536 адресов (один "район"), /8 = ~16 миллионов адресов (один "город"). Чем больше число после /, тем меньше подсеть.
