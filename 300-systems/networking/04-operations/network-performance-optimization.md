# Network Performance Optimization

> TCP tuning, benchmarking и системная оптимизация для production

---

## Теоретические основы

> **Bandwidth-Delay Product (BDP)** — фундаментальная величина в теории сетевой производительности: BDP = Bandwidth * RTT. Показывает объём данных "в полёте" между отправителем и получателем. TCP-буферы должны быть >= BDP для полного использования канала. Пример: 1 Gbps * 50 ms = 6.25 MB.

### TCP Congestion Control: историческая эволюция

| Алгоритм | Автор, год | Принцип | Реакция на потерю |
|----------|-----------|---------|-------------------|
| TCP Tahoe | Jacobson, 1988 | Slow Start + Congestion Avoidance | cwnd = 1 MSS (полный сброс) |
| TCP Reno | Jacobson, 1990 | Fast Retransmit + Fast Recovery | cwnd = cwnd/2 (быстрое восстановление) |
| TCP NewReno | Floyd, Henderson, 1999 | Улучшенный Fast Recovery | Корректная обработка множественных потерь |
| CUBIC | Ha, Rhee, Xu, 2008 | Кубическая функция роста cwnd | Дефолт в Linux с 2.6.19 |
| BBR | Cardwell et al. (Google), 2016 | Модель bottleneck bandwidth + RTT | Не реагирует на потерю напрямую |
| BBRv2 | Google, 2019+ | Улучшенная fairness | Учитывает ECN и потери |

### Работа Jacobson (1988)

> Van Jacobson. "Congestion Avoidance and Control" (1988, ACM SIGCOMM) — одна из наиболее влиятельных работ в сетевых технологиях. Описывает механизмы **Slow Start** и **Congestion Avoidance**, которые предотвратили "congestion collapse" интернета в октябре 1986 года. Все современные TCP-реализации основаны на принципах, изложенных в этой работе.

### Bufferbloat

- **Bufferbloat** (Gettys, Nichols, 2011) — чрезмерная буферизация в сетевых устройствах, вызывающая высокую latency без видимой потери пакетов. Маршрутизаторы с большими буферами поглощают пакеты вместо того, чтобы сигнализировать о перегрузке (через drop или ECN)
- **Решения:** Active Queue Management (AQM): CoDel (Nichols, Jacobson, 2012), FQ-CoDel (RFC 8290), PIE (RFC 8033)
- **Диагностика:** `ping` показывает высокую latency под нагрузкой, но 0% packet loss

### Window Scaling (RFC 7323)

TCP Window Scaling (RFC 7323, Borman et al., 2014) расширяет максимальное значение TCP Receive Window с 65535 байт (16 бит) до 1 ГБ (30-бит значение через shift). Необходим для эффективного использования высокоскоростных каналов с большим RTT (без scaling: max throughput на 100ms RTT = 65535/0.1 = 5.2 Mbps).

> **Формула пропускной способности TCP:** Throughput <= Window Size / RTT. Без Window Scaling и оптимизации буферов пропускная способность TCP ограничена размером окна, а не физической полосой канала.

**См. также:** [[network-transport-layer]] (TCP congestion control подробно), [[network-latency-optimization]] (latency-специфичная оптимизация), [[network-observability]] (метрики для оценки эффекта оптимизации)

```
|   +----------+                             +----------+            |
|   | Фабрика  |                             | Магазин  |            |

|   |  buffer) |                             |  buffer) |            |
|   +----------+                             +----------+            |
|                                                                    |
|   Если склад маленький:                                            |
|   - Фабрика ждёт пока грузовики вернутся                          |

|                                                                    |
|   Если склад огромный:                                            |

```

**Формула BDP:**
```
Bandwidth-Delay Product = Bandwidth × RTT

Пример:
1 Gbps × 50ms = 1,000,000,000 × 0.050 = 50,000,000 бит = 6.25 MB

Нужен буфер минимум 6.25 MB чтобы заполнить канал!
Default в Linux часто 4 MB — недостаточно для WAN.
```

### Аналогия 3: Congestion Control как стили вождения

```
+------------------------------------------------------------------+
|                    CONGESTION CONTROL                             |
+------------------------------------------------------------------+
|                                                                    |
|   CUBIC (дефолт Linux):                                           |
|   "Осторожный водитель"                                           |
|   +----------------------------------------------------------+   |
|   | • Разгоняется постепенно                                  |   |
|   | • При ДТП (packet loss) резко тормозит                   |   |
|   | • Потом снова осторожно ускоряется                       |   |
|   | • Хорошо работает, когда все так ездят                   |   |
|   +----------------------------------------------------------+   |
|                                                                    |
|   BBR (Google):                                                   |
|   "Умный autopilot с GPS"                                         |
|   +----------------------------------------------------------+   |
|   | • Измеряет реальную пропускную способность               |   |
|   | • Измеряет минимальный RTT                               |   |
|   | • Едет на оптимальной скорости, не по packet loss        |   |
|   | • Не тормозит резко при одиночной потере                 |   |
|   | • Лучше на lossy сетях (WiFi, WAN)                       |   |
|   +----------------------------------------------------------+   |
|                                                                    |
|   СРАВНЕНИЕ:                                                      |
|                                                                    |
|   Сценарий              CUBIC           BBR                       |
|   ─────────────────────────────────────────────                   |
|   Low latency LAN       ✅ Отлично       ✅ Отлично               |
|   High latency WAN      ⚠️ Медленный     ✅ Отлично               |
|   Lossy network         ❌ Тормозит      ✅ Устойчив              |
|   Shared fairness       ✅ Честный       ⚠️ Может быть жадным     |
+------------------------------------------------------------------+
```

**Когда какой использовать:**

| Сценарий | Рекомендация | Почему |
|----------|--------------|--------|
| Локальная сеть, < 5ms RTT | CUBIC | Default работает отлично |
| WAN, 50ms+ RTT | BBR | Лучше использует канал |
| Мобильная/WiFi сеть | BBR | Устойчив к потерям |
| Shared hosting | CUBIC | Честнее к соседям |
| CDN, раздача контента | BBR | Максимизирует throughput |

## Часть 2: Почему это сложно

> 🎯 **Цель секции**: Изучить 6 типичных ошибок, которые совершают при оптимизации сети.

### Ошибка 1: Слепое копирование sysctl с интернета

**СИМПТОМ:**
```bash
# "Нашёл на StackOverflow, добавил в sysctl.conf"
net.core.rmem_max=268435456
net.core.wmem_max=268435456
net.ipv4.tcp_rmem=4096 87380 268435456
net.ipv4.tcp_wmem=4096 87380 268435456
net.ipv4.tcp_congestion_control=bbr
net.ipv4.tcp_notsent_lowat=16384
net.ipv4.tcp_low_latency=1

# Результат: память закончилась, сервер упал
```

**Почему происходит:**
- Настройки для 10Gbps сети применены на VPS с 1 Gbps
- Буферы по 256MB × 1000 соединений = 256GB RAM
- Не понимание что делает каждый параметр

**РЕШЕНИЕ:**
```bash
# Шаг 1: Узнай текущие значения
sysctl net.core.rmem_max
sysctl net.ipv4.tcp_rmem

# Шаг 2: Рассчитай нужный BDP
# Bandwidth: 1 Gbps, RTT: 50ms
# BDP = 1,000,000,000 * 0.050 / 8 = 6.25 MB

# Шаг 3: Установи разумные значения
net.core.rmem_max=16777216      # 16 MB (с запасом от BDP)
net.ipv4.tcp_rmem="4096 87380 16777216"

# Шаг 4: Измерь throughput до и после
iperf3 -c server -t 30
```

**Правило**: Никогда не копируй настройки без понимания. Каждый параметр — это trade-off.

### Ошибка 2: BBR везде без тестирования

**СИМПТОМ:**
```bash
# "BBR лучше, включу везде"
sysctl -w net.ipv4.tcp_congestion_control=bbr

# Результат:
# - На одних серверах стало лучше
# - На других — хуже
# - На третьих — проблемы с fairness
```

**Почему происходит:**
- BBR v1 может быть агрессивным к CUBIC потокам
- Некоторые middleboxes ломают BBR
- BBR на LAN может работать хуже CUBIC

**РЕШЕНИЕ:**
```bash
# Шаг 1: Тестируй в staging environment
# До BBR:
iperf3 -c server -t 60 -P 4
# Запиши: throughput, retransmits, RTT

# Шаг 2: Включи BBR
sysctl -w net.ipv4.tcp_congestion_control=bbr

# Шаг 3: Тестируй снова
iperf3 -c server -t 60 -P 4
# Сравни с baseline

# Шаг 4: Тестируй при конкуренции (несколько потоков)
# Если BBR забирает весь канал — это проблема

# BBR v3 (kernel 6.x) решает многие проблемы fairness
# Проверь версию:
cat /proc/sys/net/ipv4/tcp_congestion_control
modinfo tcp_bbr
```

### Ошибка 3: Отключение tcp_timestamps

**СИМПТОМ:**
```bash
# "Timestamps добавляют overhead, отключу"
sysctl -w net.ipv4.tcp_timestamps=0

# Результат:
# - PAWS (Protection Against Wrapped Sequences) сломан
# - RTT estimation неточная
# - На high-speed networks могут быть проблемы
```

**Почему происходит:**
- Старые статьи про "оптимизацию" рекомендовали отключать
- 12 байт overhead кажутся значимыми

**РЕШЕНИЕ:**
```bash
# НЕ отключай timestamps!
net.ipv4.tcp_timestamps=1  # default и правильно

# Почему timestamps важны:
# 1. PAWS защищает от old duplicate segments
# 2. Точный RTT = лучший congestion control
# 3. Без них на 10Gbps+ sequence numbers wrap слишком быстро

# Единственный случай отключения:
# Если middlebox ломает timestamps (очень редко)
```

### Ошибка 4: Огромные буферы "на всякий случай"

**СИМПТОМ:**
```bash
# "Больше буфер = лучше, поставлю 1GB"
net.core.rmem_max=1073741824
net.core.wmem_max=1073741824

# Результат:
# - Bufferbloat: latency выросла с 5ms до 500ms
# - Memory exhaustion при высокой нагрузке
# - Приложение "подвисает" из-за полных буферов
```

**Почему происходит:**
- Непонимание что буферы увеличивают latency
- "Больше = лучше" ментальность

**РЕШЕНИЕ:**
```
Правильный размер буфера = 2-4 × BDP

BDP для разных сценариев:
┌──────────────────┬─────────┬────────────────┐
│ Сценарий         │ BDP     │ Буфер (max)    │
├──────────────────┼─────────┼────────────────┤
│ LAN (1ms RTT)    │ 125 KB  │ 256 KB         │
│ Regional (20ms)  │ 2.5 MB  │ 8 MB           │
│ WAN (100ms)      │ 12.5 MB │ 32 MB          │
│ Satellite (600ms)│ 75 MB   │ 256 MB         │
└──────────────────┴─────────┴────────────────┘

# Для большинства случаев достаточно:
net.core.rmem_max=16777216   # 16 MB
net.core.wmem_max=16777216   # 16 MB
```

### Ошибка 5: Оптимизация без baseline

**СИМПТОМ:**
```bash
# "Сделал оптимизацию, стало быстрее... кажется?"

# Не было измерений ДО
# Не было измерений ПОСЛЕ
# Не понятно что изменилось и насколько
```

**Почему происходит:**
- Желание "сделать быстрее" без понимания текущего состояния
- Нет culture of measurement

**РЕШЕНИЕ:**
```bash
# BEFORE any changes — collect baseline

# 1. Throughput
iperf3 -c target -t 60 -J > baseline_throughput.json

# 2. Latency
ping -c 100 target | tee baseline_ping.txt
# Обрати внимание на min/avg/max/mdev

# 3. TCP stats
ss -ti > baseline_tcp_stats.txt

# 4. Retransmissions
netstat -s | grep -i retrans > baseline_retrans.txt

# AFTER changes — compare

# 5. Measure again with SAME parameters
iperf3 -c target -t 60 -J > after_throughput.json

# 6. Compare
jq -r '.end.sum_received.bits_per_second' baseline_throughput.json
jq -r '.end.sum_received.bits_per_second' after_throughput.json

# Если не лучше на 10%+ — изменение не имеет смысла
```

### Ошибка 6: Использование tcp_tw_recycle

**СИМПТОМ:**
```bash
# "Нашёл в старой статье"
sysctl -w net.ipv4.tcp_tw_recycle=1

# Результат в kernel < 4.12:
# - Соединения через NAT случайно дропаются
# - "Connection reset" у части пользователей

# В kernel >= 4.12:
sysctl: cannot stat /proc/sys/net/ipv4/tcp_tw_recycle:
No such file or directory
```

**Почему происходит:**
- Старые статьи (до 2017) рекомендовали
- tcp_tw_recycle удалён из ядра 4.12 из-за багов с NAT

**РЕШЕНИЕ:**
```bash
# НЕ используй tcp_tw_recycle — он удалён и был сломан

# Если нужно быстрее переиспользовать TIME_WAIT:
# Используй tcp_tw_reuse (безопаснее)
net.ipv4.tcp_tw_reuse=1

# Но лучше:
# 1. Используй connection pooling в приложении
# 2. Используй HTTP keepalive
# 3. Увеличь TIME_WAIT если нужно:
net.ipv4.tcp_fin_timeout=30  # default 60
```

## Зачем оптимизировать сеть

### Философия сетевой оптимизации

**Premature optimization — корень всех зол.** Это утверждение Дональда Кнута применимо и к сетям. Большинство приложений отлично работают с default настройками TCP/IP. Оптимизация нужна, когда:
1. Вы измерили и доказали, что сеть — bottleneck
2. У вас high-bandwidth (10Gbps+) или high-latency (WAN) links
3. Вы обслуживаете тысячи concurrent connections
4. Ваше приложение чувствительно к latency (trading, gaming, real-time)

**Опасность слепой оптимизации.** В интернете много советов типа "добавьте эти sysctl и будет быстрее". Но каждый параметр — это trade-off. Увеличение буферов увеличивает memory usage. Агрессивный congestion control может вызвать unfairness. BBR может не работать с некоторыми middleboxes. Без понимания что делает каждый параметр — легко сделать хуже.

**Правильный порядок действий:**
1. **Measure first.** Соберите baseline метрики: latency, throughput, packet loss
2. **Identify bottleneck.** Где именно проблема: сеть? диск? CPU?
3. **Change one thing.** Меняйте один параметр за раз
4. **Measure again.** Сравните с baseline
5. **Document.** Зафиксируйте что изменили и какой эффект

**Типичные проблемы, которые решает оптимизация:**
- High latency при нормальном bandwidth — часто проблема в congestion control
- Throughput не достигает линейной скорости — маленькие буферы или старый congestion control
- Intermittent slowdowns под нагрузкой — exhaustion listen backlog или file descriptors
- Connection timeouts при high concurrency — маленький somaxconn или SYN backlog

**Когда НЕ нужна оптимизация:**
- Default работает нормально
- Проблема в application, не в сети
- Throughput ограничен физическим каналом (1Gbps link не станет 10Gbps от sysctl)
- У вас меньше 1000 concurrent connections

**Цели оптимизации (в порядке приоритета):**
- Максимизировать throughput — использовать весь доступный канал
- Минимизировать latency — особенно tail latency (p99, p99.9)
- Обеспечить стабильность под нагрузкой — graceful degradation, не падения
- Эффективно использовать ресурсы — не тратить память на гигантские буферы без нужды

## TCP Tuning (Linux)

### Зачем понимать TCP Tuning

**TCP был создан для надёжности, не для скорости.** Когда Vint Cerf и Bob Kahn проектировали TCP в 1970-х, интернет состоял из медленных, ненадёжных линий. Default настройки TCP оптимизированы для совместимости и безопасности — они работают везде, но не максимизируют производительность.

**Что такое TCP tuning?** Это настройка параметров TCP стека для конкретных условий сети. Буферы определяют, сколько данных может быть "в полёте" между отправителем и получателем. Congestion control решает, как быстро увеличивать скорость передачи. Backlog управляет очередью входящих соединений.

**Когда default недостаточно:**
- **High-bandwidth links (10Gbps+):** Default буферы слишком маленькие — не можете заполнить канал
- **High-latency links (WAN, 50ms+ RTT):** Маленькие буферы = низкий throughput (bandwidth-delay product)
- **High-concurrency servers (тысячи connections):** Default backlog = dropped connections под нагрузкой
- **Packet loss networks:** Loss-based congestion control (CUBIC) слишком агрессивно снижает скорость

**Почему нельзя просто "выкрутить всё на максимум":**
- Большие буферы = больше memory per connection
- Агрессивный congestion control = unfairness к другим потокам
- Слишком большой backlog = память на полу-открытые соединения
- Некоторые оптимизации ломают совместимость с middleboxes

### Buffer Sizes

```bash
# Увеличить максимальные буферы
# Для high-bandwidth networks (10Gbps+)
sudo sysctl -w net.core.rmem_max=16777216
sudo sysctl -w net.core.wmem_max=16777216

# TCP буферы: min default max (в байтах)
sudo sysctl -w net.ipv4.tcp_rmem="4096 87380 16777216"
sudo sysctl -w net.ipv4.tcp_wmem="4096 87380 16777216"

# Для high-throughput links
sudo sysctl -w net.core.netdev_max_backlog=5000
```

**Формула для буфера:**
```
Optimal buffer = Bandwidth (bytes/sec) × RTT (sec)

Пример: 1 Gbps link, 50ms RTT
Buffer = (1,000,000,000 / 8) × 0.05 = 6.25 MB
```

> **Предупреждение:** Увеличивать default value (второе число) осторожно. Слишком большой default может вызвать buffer collapsing и latency spikes.

### Connection Backlog

**Что такое backlog?** Когда клиент отправляет SYN для установления TCP соединения, сервер помещает это "полу-открытое" соединение в очередь (SYN backlog). После завершения handshake соединение переходит в accept queue (somaxconn). Если очереди переполнены — новые соединения отбрасываются.

**Симптомы маленького backlog:**
- Connection refused или timeouts при burst нагрузки
- `netstat -s | grep -i listen` показывает dropped connections
- Проблема появляется внезапно при пиках трафика

```bash
# Увеличить для high-concurrency серверов
sudo sysctl -w net.core.somaxconn=4096
sudo sysctl -w net.ipv4.tcp_max_syn_backlog=4096

# Persistent — добавить в /etc/sysctl.conf
net.core.somaxconn = 4096
net.ipv4.tcp_max_syn_backlog = 4096
```

### Congestion Control: BBR

**Проблема congestion control.** Интернет — shared resource. Если все будут отправлять данные максимально быстро — сеть перегрузится и никто не получит нормальный throughput. Congestion control — механизм, который определяет "безопасную" скорость передачи.

**Как работает классический подход (CUBIC, Reno).** Loss-based: отправляем всё быстрее, пока не начнутся потери пакетов. Потеря = сигнал перегрузки → резко снижаем скорость. Проблема: на современных сетях потери часто не связаны с перегрузкой (wireless, noisy links), а на буферизованных сетях потери приходят слишком поздно (bufferbloat).

**Как работает BBR (Bottleneck Bandwidth and Round-trip propagation time).** Model-based: измеряем реальную bandwidth и RTT, строим модель сети, отправляем данные со скоростью, которую сеть реально может обработать. Не ждём потерь — реагируем на изменения latency.

**Когда BBR даёт преимущество:**
- High-latency links (WAN, international) — не ждёт потерь для достижения max throughput
- Lossy networks — не снижает скорость из-за случайных потерь
- Буферизованные сети — избегает bufferbloat

**Когда BBR может быть хуже:**
- LAN с очень низкой latency — overhead модели не окупается
- Сети с агрессивными middleboxes — некоторые устройства не понимают BBR паттерны

```bash
# Проверить текущий алгоритм
sysctl net.ipv4.tcp_congestion_control
# Обычно: cubic

# Включить BBR (требует kernel 4.9+)
sudo sysctl -w net.core.default_qdisc=fq
sudo sysctl -w net.ipv4.tcp_congestion_control=bbr

# Проверить
sysctl net.ipv4.tcp_congestion_control
# bbr
```

**BBR vs CUBIC:**

| Характеристика | CUBIC | BBR |
|----------------|-------|-----|
| Подход | Loss-based | Model-based |
| High latency links | Хуже | Лучше |
| Packet loss | Агрессивно снижает window | Более стабильно |
| Fairness | Хорошо | Может быть агрессивен |

## Connection Pooling

### Зачем нужен Pooling

**Почему создание соединения дорого.** Каждое новое TCP соединение требует:
1. **DNS lookup** (10-100ms) — если не закэширован
2. **TCP handshake** (1 RTT, 20-100ms) — SYN → SYN-ACK → ACK
3. **TLS handshake** (1-2 RTT, 40-200ms) — key exchange, certificate verification
4. **Server-side resources** — память, file descriptor, возможно thread

Для локального сервиса это миллисекунды. Для WAN с TLS — сотни миллисекунд на каждый запрос.

**Что такое connection pooling.** Вместо создания нового соединения на каждый запрос, держим пул готовых ("прогретых") соединений. Запрос берёт соединение из пула, использует, возвращает. Следующий запрос переиспользует то же соединение.

**Экономия на практике:**
```
Без пула:
Request 1: [DNS][TCP Connect][TLS][Request][Response][Close]
Request 2: [DNS][TCP Connect][TLS][Request][Response][Close]
Request 3: [DNS][TCP Connect][TLS][Request][Response][Close]

С пулом:
Request 1: [DNS][TCP Connect][TLS][Request][Response]
Request 2:                        [Request][Response]
Request 3:                        [Request][Response]

Экономия: ~300ms+ на каждый последующий запрос
```

**Benchmarks:** Reusing connections reduces total run time by factor of ~3x.

### Pool Configuration Best Practices

```yaml
# Пример для HikariCP (Java DB pool)
hikari:
  maximum-pool-size: 100      # Увеличить для production
  minimum-idle: 10            # Минимум готовых соединений
  connection-timeout: 30000   # 30s — max ожидание соединения
  idle-timeout: 600000        # 10m — закрыть idle connections
  max-lifetime: 1800000       # 30m — max время жизни connection
  leak-detection-threshold: 60000  # 60s — логировать если не вернули
```

### HTTP Client Pool

```python
# Python requests с connection pooling
import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

session = requests.Session()
adapter = HTTPAdapter(
    pool_connections=100,    # Пул на хост
    pool_maxsize=100,        # Max соединений в пуле
    max_retries=Retry(total=3, backoff_factor=0.1)
)
session.mount('https://', adapter)

# Использовать session для всех запросов
response = session.get('https://api.example.com')
```

```java
// Java HttpClient (11+)
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(10))
    .executor(Executors.newFixedThreadPool(10))
    .build();
// Reuse client instance — он управляет пулом внутри
```

## TLS Optimization

### Почему TLS влияет на производительность

**TLS добавляет latency.** Перед передачей данных нужно согласовать ключи шифрования. В TLS 1.2 это занимает 2 round-trips (200-400ms на WAN). TLS 1.3 сократил до 1 round-trip. 0-RTT resumption позволяет отправлять данные сразу — но с ограничениями безопасности.

**Почему нельзя просто отключить TLS.** Современный интернет требует шифрования: HTTP/2 и HTTP/3 работают только через TLS, браузеры помечают HTTP как небезопасный, Google понижает HTTP сайты в поиске, многие API вообще недоступны без TLS.

**Цель TLS optimization — минимизировать overhead без потери безопасности:**
- Использовать TLS 1.3 вместо 1.2 (меньше round-trips)
- Включить session resumption (переиспользовать согласованные ключи)
- Настроить OCSP stapling (избежать дополнительных запросов к CA)
- Выбрать быстрые cipher suites (ChaCha20 быстрее AES на устройствах без AES-NI)

### TLS 1.3 Advantages

| Feature | TLS 1.2 | TLS 1.3 |
|---------|---------|---------|
| Handshake RTT | 2 | 1 |
| 0-RTT resumption | Нет | Да |
| Forward Secrecy | Опционально | Обязательно |
| Cipher Suites | ~30+ | 5 |

### Session Resumption

```
Первое соединение:
Client ←→ Server: Full TLS Handshake (1 RTT)
Server → Client: Session Ticket

Последующие соединения:
Client → Server: ClientHello + Session Ticket
Client → Server: [Early Data] (0-RTT)
Server → Client: Finish
```

**~40% HTTPS соединений — resumptions.** С 0-RTT экономим 1 RTT на каждом.

### 0-RTT Security

```
ВАЖНО: 0-RTT уязвим к replay attacks!

Безопасно для 0-RTT:
✅ GET запросы
✅ HEAD запросы
✅ Idempotent операции

НЕ использовать для 0-RTT:
❌ POST с побочными эффектами
❌ Финансовые транзакции
❌ Любые non-idempotent операции
```

### Cipher Suite Optimization

```nginx
# Nginx TLS optimization
ssl_protocols TLSv1.2 TLSv1.3;

# TLS 1.3 ciphers (fast + secure)
ssl_ciphersuites TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256;

# TLS 1.2 fallback
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;

# Session cache
ssl_session_cache shared:SSL:50m;
ssl_session_timeout 1d;

# Session tickets (для resumption)
ssl_session_tickets on;

# OCSP Stapling
ssl_stapling on;
ssl_stapling_verify on;
```

## DNS Optimization

### Почему DNS критичен для производительности

**DNS — первый шаг каждого соединения.** Прежде чем браузер отправит HTTP запрос, он должен узнать IP адрес сервера. Медленный DNS = медленный старт каждой страницы. На странице с 50 разными доменами (main site, CDN, analytics, ads) — это 50 DNS запросов.

**Типичные проблемы DNS:**
- **Slow resolver** — ISP resolver перегружен или далеко географически
- **Cache miss** — TTL истёк, нужен новый запрос к authoritative серверу
- **Cold start** — первый запрос к домену всегда медленнее (ничего не закэшировано)
- **DNS over HTTPS overhead** — дополнительная задержка на TLS, если используется DoH

**Стратегии оптимизации:**
- **Prefetching** — браузер заранее резолвит домены, которые понадобятся
- **Correct TTL** — баланс между быстрым failover и cache эффективностью
- **Fast DNS provider** — anycast DNS с низкой глобальной latency
- **Connection reuse** — минимизировать количество разных доменов

### TTL Strategy

| Тип записи | Рекомендуемый TTL | Причина |
|------------|-------------------|---------|
| Static (корпоративный сайт) | 86400 (24h) | Редко меняется |
| Dynamic (load balancer) | 300-900 (5-15m) | Нужна гибкость |
| Failover | 60-120 (1-2m) | Быстрое переключение |

> **Важно:** TTL <300s часто игнорируется ISP кэшами.

### DNS Prefetching

```html
<!-- В HTML head -->
<link rel="dns-prefetch" href="//api.example.com">
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="preconnect" href="https://api.example.com" crossorigin>
```

### DNS Provider Selection

```
Факторы выбора:
- Anycast deployment (глобальная latency)
- DDoS protection
- Response time (<50ms)
- Uptime SLA

Популярные: Cloudflare, AWS Route 53, Google Cloud DNS
Managed DNS на ~40% быстрее self-hosted.
```

## HTTP Compression

### Зачем сжимать HTTP трафик

**Trade-off: CPU vs bandwidth.** Сжатие тратит CPU на сервере и клиенте, но уменьшает объём передаваемых данных. На медленных соединениях (мобильный интернет) экономия bandwidth перевешивает CPU overhead. На быстрых LAN — compression может даже замедлить передачу.

**Где compression критично:**
- **Text-based content:** HTML, CSS, JavaScript — сжимается в 5-10 раз
- **JSON API responses:** Типичное сжатие 70-90%
- **Mobile users:** Bandwidth ограничен, latency высокая
- **CDN billing:** Платите за переданные байты

**Где compression не нужно или вредно:**
- **Already compressed:** JPEG, PNG, WebP, WOFF2 — уже сжаты, повторное сжатие бесполезно
- **Tiny files (<1KB):** Overhead сжатия больше экономии
- **High-throughput LAN:** CPU bottleneck при гигабитных скоростях
- **Real-time streams:** Задержка на compression неприемлема

### Algorithm Comparison

| Algorithm | Compression Ratio | Speed | Browser Support |
|-----------|-------------------|-------|-----------------|
| **gzip** | Good | Fast | 100% |
| **Brotli** | Better (+19%) | Slower | 98% |
| **Zstandard** | Similar to Brotli | Faster | Chrome, Firefox |

### Configuration

```nginx
# Nginx compression
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 6;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml;

# Brotli (требует модуль)
brotli on;
brotli_comp_level 6;
brotli_types text/plain text/css application/json application/javascript text/xml application/xml;
```

### Best Practices

```
✅ Сжимать:
- HTML, CSS, JavaScript
- JSON, XML
- SVG

❌ НЕ сжимать:
- JPEG, PNG, WebP (уже сжаты)
- WOFF2 fonts (уже сжаты)
- Файлы <1KB (overhead > benefit)
```

### Caching Headers

```nginx
# Cache-Control + Compression
location ~* \.(js|css)$ {
    gzip_static on;
    brotli_static on;

    add_header Cache-Control "public, max-age=31536000, immutable";
    add_header Vary "Accept-Encoding";
}
```

## QUIC / HTTP/3

### Зачем нужен QUIC

**Проблема TCP+TLS.** TCP был создан в 1970-х, TLS добавлен позже как отдельный слой. Это означает:
- Два отдельных handshake (TCP + TLS) = 2-3 round-trips до первых данных
- Head-of-line blocking: потеря одного пакета блокирует ВСЕ streams в HTTP/2
- Connection привязан к IP — при смене сети (Wi-Fi → мобильный) соединение умирает

**Что такое QUIC.** Google создал новый транспортный протокол поверх UDP, который объединяет функции TCP и TLS:
- **Один handshake** для transport и encryption — 0-RTT или 1-RTT
- **Independent streams** — потеря пакета в одном stream не блокирует другие
- **Connection ID** — соединение переживает смену IP (mobile handoff)
- **Улучшенный congestion control** — встроен в протокол, не ограничен legacy TCP

**HTTP/3 = HTTP поверх QUIC.** Семантика HTTP (GET, POST, headers, status codes) та же, но transport совершенно другой.

### Performance Characteristics

**Преимущества QUIC:**
- No Head-of-Line Blocking (multiplexed streams)
- 0-RTT connection establishment
- Connection migration (mobile handoff)
- Better performance at high latency/packet loss

**Где QUIC выигрывает:**
```
✅ Mobile users (network switching)
✅ High-latency connections (>100ms RTT)
✅ Networks with packet loss (>1%)
✅ Many parallel requests (no HOL blocking)
```

**Где TCP может быть лучше:**
```
⚠️ Very high bandwidth, low latency LANs
⚠️ Single large file transfers
⚠️ CPU-constrained servers (QUIC более expensive)
```

### Benchmarks Summary

```
HTTP/3 vs HTTP/2:
- Connection establishment: 33% faster
- High packet loss scenarios: Significant advantage
- Streaming video: 69% vs 56% достигают 5Mbps+

Предупреждение (ACM 2024):
На очень быстрых сетях QUIC может быть до 45% медленнее
из-за user-space overhead.
```

## Benchmarking Methodology

### Зачем нужна методология бенчмаркинга

**"Measure twice, cut once."** Оптимизация без измерений — это гадание. Интуиция часто ошибается: "очевидный" bottleneck может быть не там, где кажется. Правильный бенчмаркинг показывает реальную картину и доказывает эффект изменений.

**Типичные ошибки бенчмаркинга:**
- **Too short tests:** 5 секунд недостаточно — не учитываются warmup и variance
- **Single run:** Один запуск нерепрезентативен — нужно минимум 3, берём median
- **Wrong metric:** Средняя latency скрывает tail latency — смотрите p95/p99
- **Different conditions:** Тестировали утром, применили изменения вечером — несопоставимо
- **Local testing:** Результаты на localhost не переносятся на production

**Правильный подход:**
1. **Establish baseline** — измерьте текущую производительность ДО изменений
2. **Control conditions** — тестируйте в одинаковых условиях (время, load, версии)
3. **Long duration** — 60+ секунд для стабильных результатов
4. **Multiple runs** — минимум 3 прогона, используйте median
5. **Document everything** — записывайте условия, параметры, результаты

### iperf3 для Throughput

```bash
# Сервер
iperf3 -s

# Клиент — базовый тест
iperf3 -c server-ip

# Длительный тест (60 sec)
iperf3 -c server-ip -t 60

# Parallel streams (для 10G+ links)
iperf3 -c server-ip -P 4 -t 60

# Reverse direction
iperf3 -c server-ip -R

# UDP тест с target bandwidth
iperf3 -c server-ip -u -b 1G

# JSON output для автоматизации
iperf3 -c server-ip -J > results.json
```

### HTTP Benchmarking

```bash
# wrk — HTTP load testing
wrk -t12 -c400 -d30s https://api.example.com/endpoint

# hey (Go-based)
hey -n 10000 -c 100 https://api.example.com/endpoint

# ab (Apache Bench)
ab -n 10000 -c 100 https://api.example.com/endpoint
```

### Latency Testing

```bash
# curl timing
curl -o /dev/null -s -w "
DNS:     %{time_namelookup}s
Connect: %{time_connect}s
TLS:     %{time_appconnect}s
TTFB:    %{time_starttransfer}s
Total:   %{time_total}s
" https://example.com

# Continuous monitoring
while true; do
  curl -o /dev/null -s -w "%{time_total}\n" https://api.example.com
  sleep 1
done | tee latency.log
```

### Best Practices

```
1. Baseline first — измерь "чистую" производительность железа
2. Long tests — 60+ секунд для стабильных результатов
3. Bidirectional — тестируй оба направления
4. Multiple runs — minimum 3 прогона, берём median
5. Document conditions — время, load, network state
```

## Связь с другими темами

**[[network-latency-optimization]]** — Latency optimization фокусируется на минимизации задержек отдельных запросов и соединений, тогда как performance optimization охватывает более широкий спектр: TCP tuning, системные буферы, congestion control и benchmarking. Многие техники пересекаются, но performance optimization делает акцент на серверной стороне (sysctl tuning, socket options), а latency optimization — на клиентской (prefetching, connection pooling, CDN). Рекомендуется сначала изучить latency optimization для понимания метрик и целей, затем переходить к системной оптимизации.

**[[network-transport-layer]]** — TCP tuning, который составляет основу performance optimization, требует глубокого понимания TCP-протокола: window scaling, congestion control (CUBIC, BBR), Nagle's algorithm, delayed ACK. Без знания механизмов TCP невозможно осознанно настраивать sysctl-параметры или выбирать между TCP и QUIC для конкретного сценария. Транспортный уровень — обязательный prerequisite для этой темы.

**[[network-tcpdump-wireshark]]** — Packet analysis является основным инструментом для диагностики performance-проблем: retransmissions, window scaling issues, slow start behavior и congestion events видны только на уровне пакетов. Без навыков работы с tcpdump и Wireshark невозможно измерить реальный эффект от sysctl-оптимизаций или найти bottleneck в TCP-соединении. Изучите основы packet analysis перед глубоким погружением в performance optimization.

## Проверь себя

> [!question]- Ты нашел в интернете набор sysctl-параметров для "ускорения сети" и хочешь применить их на production-сервер. Почему это опасно и какой правильный подход?
> Слепое копирование sysctl опасно по нескольким причинам: 1) Параметры оптимальны для конкретных условий (bandwidth, RTT, load pattern) — то, что ускоряет 10Gbps сервер, может навредить 100Mbps. 2) Некоторые параметры взаимозависимы (буферы, backlog, congestion control). 3) Параметр tcp_tw_recycle удален из ядра 4.12, а его применение на старых ядрах ломает NAT. Правильный подход: 1) Измерить baseline (iperf3, curl timing). 2) Менять по одному параметру за раз. 3) Тестировать под нагрузкой (wrk, hey). 4) Сравнивать результаты с baseline. 5) Документировать изменения.

> [!question]- Сервер использует CUBIC congestion control. Инженер предлагает переключиться на BBR. В каких сценариях BBR будет лучше, а в каких -- хуже?
> BBR лучше на соединениях с высоким RTT и packet loss (межконтинентальные, мобильные сети), потому что он моделирует bottleneck bandwidth и не снижает скорость так агрессивно при потере пакетов. BBR хуже (или не лучше) на: 1) LAN с низким RTT — CUBIC уже достигает max throughput. 2) При конкуренции с CUBIC-потоками — BBR может забирать непропорционально много bandwidth. 3) На CPU-ограниченных серверах — BBR требует больше вычислений. Правило: тестировать с iperf3 -t 60 с BBR и CUBIC, сравнивать throughput и retransmissions.

> [!question]- iperf3 показывает throughput 950 Mbps на гигабитном линке, но реальные HTTP-запросы работают медленно. Почему throughput и application performance -- это разные вещи?
> iperf3 измеряет raw TCP throughput в идеальных условиях: один длинный поток данных, нет application logic, нет TLS overhead, нет DNS lookup. Реальные HTTP-запросы имеют: 1) Маленький размер (не успевают выйти из TCP slow start). 2) TLS handshake overhead. 3) Application processing time (БД, бизнес-логика). 4) Множество коротких соединений вместо одного длинного. 5) Head-of-line blocking (HTTP/1.1). Поэтому нужно бенчмаркить на уровне приложения (wrk, hey), а не только raw TCP.

## Ключевые карточки

BBR vs CUBIC
?
CUBIC — дефолтный congestion control Linux, реагирует на потерю пакетов снижением скорости. BBR — модель от Google, оценивает bottleneck bandwidth и RTT, менее агрессивен при потерях. BBR лучше на высокоlatency соединениях с потерями.

Bandwidth-Delay Product (BDP)
?
BDP = Bandwidth x RTT. Показывает сколько данных "в полете" между отправителем и получателем. TCP буферы должны быть >= BDP для полного использования канала. Пример: 1 Gbps x 50ms = 6.25 MB.

sysctl net.core.rmem_max / wmem_max
?
Максимальный размер receive/send буферов сокетов. По умолчанию слишком маленькие для высокоскоростных соединений. Рекомендация: 16 MB (16777216) для серверов. Автонастройка TCP масштабирует буферы до этого лимита.

TLS 1.3 vs TLS 1.2
?
TLS 1.3: 1-RTT handshake (вместо 2-RTT в TLS 1.2), поддержка 0-RTT для повторных соединений, меньше cipher suites (только безопасные). Ускоряет каждое новое соединение на 1 RTT.

gzip vs Brotli
?
Brotli дает на 15-25% лучшее сжатие чем gzip при схожей скорости декомпрессии. Поддерживается 98% браузеров. Используй Brotli для статики (предварительно сжатые файлы), gzip как fallback. Не сжимай уже сжатое (JPEG, PNG, WOFF2).

QUIC vs TCP для production
?
QUIC выигрывает на мобильных сетях (connection migration), высоком packet loss (independent streams), и первом подключении (0-RTT). TCP лучше на LAN (меньше CPU overhead), для bulk transfers, и когда middleboxes блокируют UDP. Тестировать обязательно.

Методология бенчмаркинга
?
1) Измерить baseline до изменений. 2) Тесты 60+ секунд (short tests нерепрезентативны). 3) Минимум 3 прогона, брать median. 4) Контролировать условия (время, load). 5) Измерять на уровне приложения (wrk), а не только raw TCP (iperf3).
