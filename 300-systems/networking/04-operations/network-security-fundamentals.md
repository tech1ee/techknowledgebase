### Терминология для новичков

> 💡 **Network Security** = защита данных при передаче по сети. Кто-то может подслушать, подменить, или заблокировать — от этого защищаемся.

| Термин | Значение | Аналогия для новичка |
|--------|----------|---------------------|
| **Firewall** | Фильтр трафика по правилам | **Охранник на входе** — кого пускать |
| **Encryption** | Шифрование данных | **Секретный код** — без ключа не прочитать |
| **TLS/SSL** | Шифрование соединения | **Конверт с печатью** — никто не вскроет |
| **DDoS** | Distributed Denial of Service | **Толпа блокирует вход** — никто не пройдёт |
| **Zero Trust** | Не доверяй никому | **Проверяй каждого** — даже "своих" |
| **VPN** | Защищённый туннель | **Труба внутри трубы** — никто не видит что внутри |
| **IDS/IPS** | Обнаружение/предотвращение вторжений | **Сигнализация** — кричит при взломе |
| **CIA Triad** | Confidentiality, Integrity, Availability | **Три кита безопасности** |
| **Man-in-the-Middle** | Атака посредника | **Подслушивание разговора** |
| **Spoofing** | Подмена источника | **Поддельное удостоверение** |

## Теоретические основы: формальный базис информационной безопасности

### Принцип Керкгоффса (1883)

> **Принцип (Auguste Kerckhoffs, 1883):** Криптосистема должна быть безопасной, даже если всё о системе, кроме **ключа**, известно противнику.

Переформулировка Шеннона (1949): "Враг знает систему" (The enemy knows the system). Это фундаментальный принцип — **security through obscurity не работает**. Вся безопасность должна опираться на секретность ключа, а не алгоритма.

Следствия для практики:
- Открытые алгоритмы (AES, RSA, TLS) **безопаснее** проприетарных
- Код шифрования должен быть публично проверяемым (open-source)
- Обфускация ≠ безопасность

### Совершенная секретность Шеннона (1949)

> **Определение (Shannon, 1949, "Communication Theory of Secrecy Systems"):** Криптосистема обладает **совершенной секретностью** (perfect secrecy), если для любого открытого текста M и шифротекста C: P(M|C) = P(M) — наблюдение шифротекста не даёт **никакой** информации об открытом тексте.

**Теорема Шеннона:** Совершенная секретность достижима ⟺ пространство ключей ≥ пространство сообщений. Единственная практическая реализация — **One-Time Pad** (одноразовый блокнот). Все реальные шифры (AES, ChaCha20) — **вычислительно безопасны** (computationally secure), но не совершенно секретны.

### CIA Triad: формальная модель

Saltzer & Schroeder (1975), "The Protection of Information in Computer Systems" — формализация трёх свойств:

| Свойство | Формальное определение | Типичные атаки | Типичные защиты |
|----------|----------------------|----------------|-----------------|
| **Confidentiality** | Информация доступна только авторизованным субъектам: ∀s ∉ Auth(r): read(s, r) = ⊥ | Перехват, утечка | Шифрование, ACL |
| **Integrity** | Информация не изменена неавторизованно: hash(data_received) = hash(data_sent) | MITM, подмена | HMAC, цифровые подписи |
| **Availability** | Ресурс доступен авторизованным субъектам в разумное время: response_time(r) ≤ T_max | DDoS, resource exhaustion | Rate limiting, redundancy |

Расширения: **CIA + Non-Repudiation** (невозможность отрицания авторства) + **Authenticity** (подтверждение подлинности источника).

### Модель угроз Долева-Яо (1983)

> **Модель (Dolev & Yao, 1983):** Атакующий может **перехватывать, модифицировать, удалять и создавать** любые сообщения в канале связи. Единственное ограничение — он не может сломать криптографические примитивы (шифрование, подпись) без знания ключа.

Это стандартная модель для анализа протоколов безопасности (TLS, SSH, OAuth). Все протоколы безопасности проектируются исходя из того, что **канал полностью контролируется противником**.

### Принципы защиты (Saltzer & Schroeder, 1975)

| Принцип | Суть | Пример |
|---------|------|--------|
| **Least Privilege** | Минимальные необходимые привилегии | Microservice с read-only доступом к БД |
| **Defense in Depth** | Несколько слоёв защиты | Firewall + TLS + auth + encryption at rest |
| **Fail-Safe Defaults** | По умолчанию — запрет | deny-all firewall rule |
| **Complete Mediation** | Каждый доступ проверяется | Zero Trust: verify every request |
| **Economy of Mechanism** | Простота = безопасность | Минимальная attack surface |

### Аналогия 5: Firewall как охранник с чек-листом

**Firewall — это охранник на входе:**
```
┌─────────────────────────────────────────────────────┐
│                                                     │
│     ИНТЕРНЕТ           FIREWALL            ВАШ     │
│    (снаружи)          (охранник)          СЕРВЕР   │
│                                                     │
│      🌐  ──────►   ┌─────────────┐  ──────►  🖥️    │
│                    │ ЧЕК-ЛИСТ:   │                  │
│                    │             │                  │
│                    │ ✅ Порт 443 │ HTTPS — пускаем  │
│                    │ ✅ Порт 80  │ HTTP — пускаем   │
│                    │ ❌ Порт 22  │ SSH — НЕ пускаем │
│                    │ ❌ Всё остальное│               │
│                    └─────────────┘                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**Что firewall умеет:**
```
✅ Блокировать по порту (закрыть SSH извне)
✅ Блокировать по IP (заблокировать подозрительные адреса)
✅ Ограничивать частоту (rate limiting)
✅ Логировать попытки (видеть кто стучится)
```

**Что firewall НЕ умеет:**
```
❌ Проверять содержимое зашифрованного трафика
❌ Понимать, что запрос к /api/users — это SQL injection
❌ Защищать от атак на открытые порты (для этого WAF)
❌ Защищать от атакующего, который УЖЕ внутри
```

## Часть 2: Почему это сложно

> 6 типичных ошибок в сетевой безопасности

### Ошибка 1: Security by obscurity

**СИМПТОМ:**
```
"Мы спрятали admin-панель по адресу /admin-xyz123-secret
 Никто не найдёт!"

"Мы используем нестандартный порт 2222 для SSH.
 Атакующие не догадаются!"

"Наш API не документирован — никто не узнает endpoints."
```

**ПОЧЕМУ ЭТО НЕ РАБОТАЕТ:**
```
Атакующие используют автоматические сканеры:

$ nmap -p 1-65535 target.com    # Найдёт ВСЕ открытые порты
$ dirb http://target.com        # Перебор директорий
$ nuclei -u target.com          # Автоматический поиск уязвимостей

Результат за минуты:
- Порт 2222 → SSH обнаружен
- /admin-xyz123-secret → Admin panel found
- /api/v1/internal/users → Hidden API endpoint
```

**РЕШЕНИЕ:**
```
Принцип Керкгоффса: безопасность системы должна зависеть
ТОЛЬКО от секретности ключа, а не от незнания алгоритма.

✅ Предполагай, что атакующий ЗНАЕТ структуру системы
✅ Защищай endpoints аутентификацией, а не скрытием
✅ Используй firewall для ограничения доступа к admin
✅ Реальная защита: MFA + rate limiting + WAF
```

---

### Ошибка 2: Доверие внутренней сети

**СИМПТОМ:**
```
"Это ВНУТРЕННИЙ сервис, зачем ему аутентификация?"
"Redis/MongoDB без пароля — он же не смотрит наружу!"
"Микросервисы общаются внутри VPC — всё безопасно."
```

**ПОЧЕМУ ЭТО ОПАСНО:**
```
Сценарий атаки (lateral movement):

1. Атакующий находит уязвимость в публичном API
   └── SSRF или RCE в веб-приложении

2. Получает доступ к внутренней сети
   └── Теперь он "внутри периметра"

3. Сканирует внутреннюю сеть
   └── Находит Redis без пароля на 6379

4. Использует Redis для дальнейшей атаки
   └── Данные, credentials, RCE через Lua scripts

Вся защита периметра — бесполезна.
```

**РЕШЕНИЕ:**
```
┌─────────────────────────────────────────────────────┐
│  ZERO TRUST ВНУТРИ СЕТИ                             │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ✅ Аутентификация между микросервисами (mTLS)      │
│  ✅ Network policies (allow только нужное)          │
│  ✅ Пароли на Redis, MongoDB, Elasticsearch         │
│  ✅ Сегментация: DB не видит frontend напрямую      │
│  ✅ Audit logs внутреннего трафика                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### Ошибка 3: Хранение секретов в коде

**СИМПТОМ:**
```python
# config.py
DATABASE_URL = "postgresql://admin:SuperSecret123@db.prod.com/main"
AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
API_KEY = "sk-prod-abc123def456..."

# .env (но .env случайно попал в git)
```

**ПОЧЕМУ ЭТО ОПАСНО:**
```
1. Git history хранит ВСЁ навсегда
   └── Даже если удалили — можно найти

2. GitHub сканирует публичные репо
   └── Боты собирают credentials за секунды

3. Сотрудник ушёл → у него копия репо с секретами

4. CI/CD логи могут показать переменные окружения
```

**РЕШЕНИЕ:**
```bash
# 1. Используй secrets manager
AWS_SECRET_KEY=$(aws secretsmanager get-secret-value --secret-id prod/db)

# 2. .env НИКОГДА в git
echo ".env" >> .gitignore

# 3. Сканируй репо на секреты
git secrets --scan
trufflehog git file://./

# 4. Ротация секретов
# Если секрет утёк — немедленно меняй ВСЕ связанные credentials

# 5. Разные секреты для dev/staging/prod
# Утечка dev-credentials не компрометирует production
```

### Ошибка 4: Логирование sensitive data

**СИМПТОМ:**
```python
# "Для отладки"
logger.info(f"Login attempt: user={username}, password={password}")
logger.debug(f"API request: {request.headers}")  # Включая Authorization
logger.info(f"Processing payment: card={card_number}, cvv={cvv}")
```

**ПОЧЕМУ ЭТО ОПАСНО:**
```
1. Логи часто хранятся без шифрования
2. Логи доступны команде поддержки, DevOps, аналитикам
3. Логи отправляются в ELK/Splunk — расширяется круг доступа
4. Старые логи могут храниться годами
5. При взломе log-сервера — массовая утечка credentials
6. GDPR/PCI DSS запрещают логировать такие данные
```

**РЕШЕНИЕ:**
```python
# НИКОГДА не логируй:
# - Пароли (даже хешированные)
# - Токены и API ключи
# - Номера карт, CVV
# - Персональные данные (без маскирования)

# Маскирование
def mask_card(card: str) -> str:
    return f"****-****-****-{card[-4:]}"

logger.info(f"Payment: card={mask_card(card_number)}")
# Output: Payment: card=****-****-****-1234

# Структурированное логирование без sensitive fields
logger.info("Login attempt", extra={
    "user_id": user.id,
    "ip": request.ip,
    "success": False,
    # НЕТ password!
})
```

### Ошибка 5: Отключение проверки сертификатов

**СИМПТОМ:**
```python
# "Сертификат самоподписанный, так быстрее"
requests.get(url, verify=False)

# "В тестовом окружении работает"
ssl_context.check_hostname = False
ssl_context.verify_mode = ssl.CERT_NONE

# curl без проверки
curl -k https://api.example.com
```

**ПОЧЕМУ ЭТО ОПАСНО:**
```
Без проверки сертификата:

    Клиент                 Атакующий                Сервер
       │                      │                        │
       │──── HTTPS запрос ───►│                        │
       │                      │──── HTTPS запрос ──────│
       │                      │◄─── Ответ ─────────────│
       │◄── "Ответ" ──────────│                        │
       │    (изменённый)                               │

Man-in-the-Middle:
- Атакующий видит ВСЮ информацию
- Может подменить ответы
- verify=False отключает ЕДИНСТВЕННУЮ защиту
```

**РЕШЕНИЕ:**
```python
# НИКОГДА verify=False в production

# Если самоподписанный сертификат — добавь в trust store
requests.get(url, verify='/path/to/ca-bundle.crt')

# Для тестов — используй mkcert для локальных сертификатов
# mkcert localhost 127.0.0.1 ::1

# Если "не работает с verify=True" — ИСПРАВЬ СЕРТИФИКАТ:
# - Обнови корневые CA
# - Проверь цепочку сертификатов
# - Убедись, что hostname совпадает
```

## Основные векторы атак (2024)

### 1. Фишинг и Social Engineering

**#1 вектор атак в 2024 году.** Microsoft и Google — самые имитируемые бренды.

**AI-усиление:**
- Генерация персонализированных писем
- Deepfake голосовых сообщений
- Автоматический анализ жертв

**Защита:**
- MFA (не SMS!)
- Security Awareness Training
- Email-фильтрация (SPF, DKIM, DMARC)

### 2. Компрометация учётных данных

Атаки с использованием **valid credentials** переместились на 1 место.

```
Атакующий получает credentials → Входит легитимно →
→ Повышает привилегии → Lateral movement → Exfiltration
```

**Защита:**
- Principle of Least Privilege
- Just-in-Time Access
- Anomaly Detection (UEBA)

### 3. Supply Chain атаки

**Пример 2024:** Вредоносные зависимости в GitHub Actions → компрометация CI/CD.

```
Уязвимая зависимость
       ↓
Ваш package.json / requirements.txt
       ↓
Код выполняется в вашем окружении
       ↓
Атакующий получает доступ
```

**Защита:**
- SBOM (Software Bill of Materials)
- Dependency scanning (Dependabot, Snyk)
- Lockfiles с хешами

### 4. SSRF (Server-Side Request Forgery)

OWASP Top 10, критичный вектор для cloud-native приложений.

```
Атакующий → Ваш сервер → Внутренний сервис (metadata API, Redis, DB)
```

**Пример (AWS):**
```bash
# Получение credentials через SSRF
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

**Защита:**
- Whitelist разрешённых URL
- Network segmentation
- IMDSv2 в AWS (требует токен)

## DDoS: Типы и защита

### Почему DDoS так эффективен

**Асимметрия атаки.** Атакующему нужно мало ресурсов для создания огромной нагрузки. DNS Amplification позволяет превратить 1 запрос в 50 ответов к жертве. Ботнет из тысяч заражённых устройств генерирует терабиты трафика. Защищающемуся нужно обрабатывать весь этот трафик — или иметь механизм фильтрации.

**Эволюция DDoS:**
- **2000-е:** Простые flood атаки, достаточно было большого канала
- **2010-е:** Application-layer атаки (Slowloris), требовали умной фильтрации
- **2020-е:** Смешанные multi-vector атаки, IoT ботнеты (Mirai), атаки на API

**Почему это важно для разработчиков:** Даже если у вас Cloudflare впереди, application-layer DDoS может пройти. Ваш код должен уметь gracefully degradе под нагрузкой, а не падать полностью.

### Три категории DDoS

| Тип | Уровень | Примеры | Цель |
|-----|---------|---------|------|
| **Volumetric** | L3-4 | UDP flood, ICMP flood, DNS amplification | Забить канал |
| **Protocol** | L3-4 | SYN flood, Ping of Death, Smurf | Исчерпать ресурсы стека |
| **Application** | L7 | HTTP flood, Slowloris, DNS query flood | Исчерпать ресурсы приложения |

### DNS Amplification

```
1. Атакующий → DNS-сервер: "Дай ВСЮ зону для domain.com"
   (IP отправителя подменён на IP жертвы)

2. DNS-сервер → Жертва: Ответ 50x больше запроса

Amplification factor: до 50x
```

### Slowloris

Держит тысячи соединений открытыми, отправляя данные очень медленно.

```python
# Концепция Slowloris
while True:
    sock.send(b"X-a: b\r\n")  # Частичный HTTP header
    time.sleep(10)            # Никогда не заканчивается
```

### Стратегии защиты

**1. Rate Limiting:**
```nginx
# Nginx: 10 запросов/сек с burst 20
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

location /api/ {
    limit_req zone=api burst=20 nodelay;
}
```

**2. SYN Cookies:**
```bash
# Включить SYN cookies (защита от SYN flood)
echo 1 > /proc/sys/net/ipv4/tcp_syncookies
```

**3. Cloud-based DDoS protection:**
- Cloudflare, AWS Shield, Akamai
- Anycast: трафик распределяется по глобальной сети
- Scrubbing centers: фильтрация вредоносного трафика

**4. Failover и capacity planning:**
- CDN с геораспределением
- Auto-scaling при повышенной нагрузке

## Zero Trust Architecture

### История: почему периметр больше не работает

**Классическая модель "castle and moat"** предполагала чёткую границу: всё внутри защищено, всё снаружи — враждебно. Это работало, когда:
- Все сотрудники сидели в офисе
- Все серверы стояли в собственном датацентре
- VPN был единственным способом удалённого доступа

**Что изменилось:**
- **Cloud:** Данные и сервисы распределены по AWS, Azure, GCP
- **Remote work:** Сотрудники работают отовсюду, на личных устройствах
- **SaaS:** Корпоративные данные в Slack, Google Workspace, Salesforce
- **Взломы:** Атакующий внутри периметра может перемещаться свободно

**Результат:** Периметра больше нет. Каждый сервис, каждый пользователь, каждый запрос — потенциально враждебный.

### Принцип: Never Trust, Always Verify

Традиционная модель: **доверяем всему внутри периметра.**
Zero Trust: **не доверяем ничему по умолчанию.**

```
Традиционно:           Zero Trust:
┌──────────────┐       ┌──────────────┐
│   Trusted    │       │   Verify     │
│   Network    │       │   Every      │
│   ┌──────┐   │       │   ┌──────┐   │
│   │Users │   │       │   │Users │←── Auth каждый запрос
│   └──────┘   │       │   └──────┘   │
│              │       │              │
│   ┌──────┐   │       │   ┌──────┐   │
│   │Apps  │   │       │   │Apps  │←── Auth + Encrypt
│   └──────┘   │       │   └──────┘   │
└──────────────┘       └──────────────┘
```

### 5 Pillars of Zero Trust (Microsoft/NIST)

| Pillar | Что защищаем | Как |
|--------|-------------|-----|
| **Identity** | Пользователи, сервисы | MFA, SSO, conditional access |
| **Devices** | Endpoints | MDM, compliance checks |
| **Network** | Трафик | Microsegmentation, encryption |
| **Applications** | Workloads | API security, least privilege |
| **Data** | Информация | Classification, DLP, encryption |

### Ключевые компоненты

**1. Identity-based access:**
```yaml
# Пример политики (OPA/Rego стиль)
allow {
    input.user.mfa_verified == true
    input.device.compliant == true
    input.request.resource in user_allowed_resources
}
```

**2. Microsegmentation:**
```
Вместо одной большой сети:
[All services] ←→ [All services]

Изолированные сегменты:
[Web] ←→ [API] ←→ [DB]
  ↑         ↑        ↑
 Policies  Policies  Policies
```

**3. Continuous validation:**
- Проверка при каждом запросе, не только при логине
- Session timeout + re-authentication
- Behavioral analytics

### С чего начать

1. **Identity** — внедрить SSO + MFA
2. **Inventory** — знать все устройства и сервисы
3. **Microsegmentation** — начать с критичных систем
4. **Monitoring** — логировать все access attempts

## DNS Security

### Угрозы DNS

| Атака | Описание | Последствия |
|-------|----------|-------------|
| **DNS Spoofing/Cache Poisoning** | Подмена DNS-ответов | Редирект на вредоносные сайты |
| **DNS Amplification** | Использование DNS для DDoS | Усиление атаки в 50x |
| **DNS Tunneling** | Передача данных через DNS | Bypass firewall, exfiltration |
| **DNS Hijacking** | Перехват DNS-запросов | MITM, credential theft |

### DNSSEC

**DNSSEC** добавляет криптографические подписи к DNS-записям:

```
Без DNSSEC:
Resolver → DNS → Ответ (можно подменить!)

С DNSSEC:
Resolver → DNS → Ответ + Подпись
                  ↓
         Проверка подписи через chain of trust
```

**Ограничение:** DNSSEC НЕ шифрует трафик, только подтверждает подлинность.

### DNS over HTTPS (DoH) vs DNS over TLS (DoT)

| Протокол | Порт | Особенности |
|----------|------|-------------|
| **Обычный DNS** | 53/UDP | Открытый текст, виден ISP |
| **DoT** | 853/TCP | Шифрование TLS, можно заблокировать |
| **DoH** | 443/TCP | Неотличим от HTTPS-трафика |

**DoH конфигурация в Linux:**
```bash
# Установка systemd-resolved с DoH
# /etc/systemd/resolved.conf
[Resolve]
DNS=1.1.1.1#cloudflare-dns.com
DNSOverTLS=yes
```

### Enterprise-рекомендации (NSA)

> Используйте **внутренний DoH resolver** вместо внешних.
> Блокируйте known external DoH resolvers.

**Причина:** DoH к внешнему резолверу делает невозможным DNS-filtering и аудит.

## TLS Security

### TLS 1.3 vs TLS 1.2

| Аспект | TLS 1.2 | TLS 1.3 |
|--------|---------|---------|
| Handshake | 2 RTT | 1 RTT (0-RTT возможен) |
| Cipher suites | Много (включая слабые) | Только сильные |
| Forward Secrecy | Опционально | Обязательно |
| RSA key exchange | Поддерживается | Удалён |

### Certificate Pinning

**Проблема:** Скомпрометированный CA может выпустить поддельный сертификат.

**Решение:** Приложение "знает" какой сертификат ожидать.

```kotlin
// Android: Certificate Pinning с OkHttp
val certificatePinner = CertificatePinner.Builder()
    .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
    .build()

val client = OkHttpClient.Builder()
    .certificatePinner(certificatePinner)
    .build()
```

**Важно:** Pinning требует ротации сертификатов — планируйте заранее!

> **HPKP deprecated** в пользу Certificate Transparency (CT).

### Mutual TLS (mTLS)

Стандартный TLS: клиент проверяет сервер.
**mTLS:** обе стороны проверяют друг друга.

```
Client                           Server
   │                                │
   │←── Server Certificate ────────│
   │─── Client Certificate ────────→│
   │                                │
   ════════ Обе стороны аутентифицированы ════════
```

**Когда использовать:**
- Microservices (service mesh)
- API между организациями
- Zero Trust сети

### Best Practices

```nginx
# Nginx: Secure TLS configuration
ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers off;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;

# HSTS
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;

# OCSP Stapling
ssl_stapling on;
ssl_stapling_verify on;
resolver 1.1.1.1 8.8.8.8 valid=300s;
```

## Firewall: iptables vs nftables

### Зачем разработчику понимать firewall

**Firewall — не только для sysadmin.** Если ваше приложение не работает в production, первый вопрос: "порт открыт?". Если вы деплоите в cloud, вы настраиваете Security Groups. Если у вас Docker или Kubernetes — там свои network policies.

**Типичные проблемы, которые решает firewall:**
- **Уменьшение attack surface:** Нет открытого порта — нет атаки
- **Network segmentation:** База данных не должна быть доступна из интернета
- **Rate limiting:** Защита от brute-force на уровне сети
- **Logging:** Видеть попытки подключения к закрытым портам

**Чего firewall НЕ делает:**
- Не защищает от атак на открытые порты (для этого WAF, application security)
- Не защищает от атак изнутри (lateral movement)
- Не проверяет содержимое зашифрованного трафика

### nftables — современный стандарт

> **nftables заменил iptables** как дефолтный firewall в Debian/Ubuntu.

| Аспект | iptables | nftables |
|--------|----------|----------|
| Синтаксис | Отдельные команды (iptables, ip6tables) | Единый nft |
| Производительность | Медленнее при большом количестве правил | Оптимизирован |
| Атомарные обновления | Нет | Да |
| Sets и Maps | Ограничено | Нативная поддержка |

### Базовая конфигурация nftables

```bash
#!/usr/sbin/nft -f
# /etc/nftables.conf

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;

        # Разрешить established/related
        ct state established,related accept

        # Loopback
        iif lo accept

        # ICMP (ping)
        ip protocol icmp accept
        ip6 nexthdr icmpv6 accept

        # SSH (только из trusted network)
        tcp dport 22 ip saddr 10.0.0.0/8 accept

        # HTTP/HTTPS
        tcp dport { 80, 443 } accept

        # Логировать dropped
        log prefix "nftables drop: " counter drop
    }

    chain forward {
        type filter hook forward priority 0; policy drop;
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

### Best Practices

1. **Default deny** — начинай с `policy drop`
2. **Whitelist approach** — разрешай только нужное
3. **Логирование** — отслеживай заблокированное
4. **Rate limiting** — защита от brute-force

```bash
# Rate limiting для SSH (max 3 подключения в минуту)
tcp dport 22 ct state new limit rate 3/minute accept
```

### SQL Injection

**Уязвимый код:**
```python
# НИКОГДА так не делай!
query = f"SELECT * FROM users WHERE id = {user_input}"
cursor.execute(query)
```

**Безопасный код (Parameterized Queries):**
```python
# Правильно: параметры отделены от SQL
cursor.execute("SELECT * FROM users WHERE id = ?", (user_input,))
```

### Cross-Site Scripting (XSS)

**Три типа:**
- **Reflected:** Payload в URL, отражается в ответе
- **Stored:** Payload сохраняется в БД
- **DOM-based:** Payload обрабатывается в клиентском JS

**Защита:**
```html
<!-- Content Security Policy -->
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self'">
```

```python
# Экранирование вывода
from markupsafe import escape
safe_output = escape(user_input)
```

### Cross-Site Request Forgery (CSRF)

**Атака:** Жертва заходит на вредоносный сайт → сайт делает запрос к банку от её имени.

**Защита — CSRF Token:**
```html
<form method="POST">
    <input type="hidden" name="csrf_token" value="random_unique_token">
    ...
</form>
```

```python
# Проверка на сервере
if request.form['csrf_token'] != session['csrf_token']:
    abort(403)
```

**SameSite Cookies:**

### Ограничения reactive blocking

> Атакующие легко обходят блокировку по IP (proxy, Tor).

**Лучший подход:**

## Подводные камни

### 1. Security by obscurity

**Ошибка:** Надежда, что атакующий не найдёт скрытый endpoint.

```
/admin-panel-secret-xyz123  ← Легко находится сканерами
```

**Правильно:** Защита должна работать, даже если атакующий знает всё.

### 2. Доверие внутренней сети

**Ошибка:** "Это внутренний сервис, зачем ему аутентификация?"

**Реальность:** Lateral movement после первого взлома.

**Правильно:** Zero Trust — verify everywhere.

### 3. Устаревшие TLS-версии

**Ошибка:** Поддержка TLS 1.0/1.1 "для совместимости".

```nginx
# Плохо!
ssl_protocols TLSv1 TLSv1.1 TLSv1.2 TLSv1.3;
```

**Правильно:** Только TLS 1.2+ с современными cipher suites.

### 4. Хранение секретов в коде

**Ошибка:**
```python
API_KEY = "sk-prod-123abc..."  # Утекает в git
```

**Правильно:**
- Environment variables
- Secrets manager (Vault, AWS Secrets Manager)
- `.env` в `.gitignore`

### 5. Логирование sensitive data

**Ошибка:**
```python
logger.info(f"User login: {username}, password: {password}")
```

**Правильно:** Никогда не логируй пароли, токены, PII.

### 6. Игнорирование security headers

**Ошибка:** Приложение без security headers.

**Правильно:**
```nginx
add_header X-Frame-Options "DENY";
add_header X-Content-Type-Options "nosniff";
add_header X-XSS-Protection "1; mode=block";
add_header Referrer-Policy "strict-origin-when-cross-origin";
```

### 7. Отключение проверки сертификатов

**Ошибка:**
```python
requests.get(url, verify=False)  # НИКОГДА в production!
```

**Правильно:** Исправь сертификат, а не отключай проверку.

## Проверь себя

> [!question]- Разработчик добавил в код `requests.get(url, verify=False)` чтобы "быстро починить" ошибку с TLS-сертификатом. Почему это опасно и каков правильный подход?
> `verify=False` отключает проверку TLS-сертификата, что делает соединение уязвимым для Man-in-the-Middle атаки: атакующий может перехватить трафик, подставить свой сертификат, и клиент не заметит разницу. Все данные (включая пароли, токены, персональные данные) передаются "в открытую" для посредника. Правильный подход: 1) Диагностировать ошибку: `openssl s_client -connect host:443 -showcerts`. 2) Если incomplete chain — попросить admin добавить intermediate cert. 3) Если self-signed (dev) — добавить CA cert в trust store: `requests.get(url, verify='/path/to/ca-cert.pem')`. 4) Никогда не отключать verify в production.

> [!question]- Компания использует подход "все в одной сети доверия" — все сервисы общаются без аутентификации во внутренней сети. Какие риски это несет и какой подход рекомендуется?
> Риски: 1) Lateral movement — если атакующий скомпрометирует один сервис, он получает доступ ко всем остальным без препятствий. 2) Insider threat — недобросовестный сотрудник имеет доступ ко всему. 3) Blast radius — компрометация одного компонента = компрометация всей системы. Рекомендуемый подход — Zero Trust: 1) Каждый сервис аутентифицирует каждый запрос (mTLS, JWT). 2) Network segmentation (VPC, Network Policies). 3) Principle of Least Privilege — доступ только к необходимым ресурсам. 4) Мониторинг внутреннего трафика.

> [!question]- На сервере настроен firewall с policy DROP, открыты порты 22, 80, 443. Администратор случайно заблокировал порт 22 при обновлении правил. Как предотвратить такую ситуацию?
> 1) Перед применением правил всегда создавать cron-задачу для отката: `at now + 5 minutes` с командой восстановления предыдущих правил. 2) Использовать nftables с атомарными обновлениями — все правила применяются одной транзакцией или не применяются вовсе. 3) Тестировать правила через `iptables-apply` (применяет на 10 секунд, откатывает если нет подтверждения). 4) Иметь out-of-band доступ (IPMI, serial console, облачная консоль). 5) Использовать IaC (Infrastructure as Code) для firewall rules с version control.

> [!question]- Чем отличается DDoS на уровне L3/L4 от DDoS на уровне L7? Какие инструменты защиты применяются для каждого?
> L3/L4 DDoS (volumetric): SYN flood, UDP flood, ICMP flood — цель перегрузить пропускную способность или исчерпать ресурсы (SYN table). Защита: upstream filtering (ISP/CDN), SYN cookies, rate limiting, blackhole routing, anycast. L7 DDoS (application): HTTP flood, slowloris, API abuse — цель перегрузить application logic при относительно малом трафике. Защита: WAF (Web Application Firewall), rate limiting по IP/session, CAPTCHA, geographic filtering, challenge-response. L7 атаки сложнее обнаружить, потому что каждый запрос выглядит легитимным.

## Ключевые карточки

CIA Triad
?
Три столпа информационной безопасности: Confidentiality (секретность — данные видят только авторизованные), Integrity (целостность — данные не изменены), Availability (доступность — система работает когда нужно). Каждое решение по безопасности балансирует между ними.

Zero Trust
?
Принцип "никому не доверяй, всегда проверяй". Каждый запрос аутентифицируется и авторизуется, независимо от источника (внутренняя сеть или внешняя). Реализация: mTLS, microsegmentation, least privilege, continuous verification.

Defense in Depth
?
Многослойная защита: firewall, network segmentation, TLS, authentication, authorization, input validation, monitoring. Если один слой пробит, следующий останавливает атаку. Аналогия: средневековый замок — ров, стена, башни, внутренняя крепость.

mTLS (Mutual TLS)
?
Двусторонняя TLS-аутентификация: не только клиент проверяет сервер, но и сервер проверяет клиент. Используется в service mesh (Istio), API между организациями, Zero Trust архитектурах. Каждый сервис имеет свой сертификат.

nftables vs iptables
?
nftables — современная замена iptables в Linux. Преимущества: единый синтаксис (вместо iptables/ip6tables), атомарные обновления правил, нативная поддержка sets и maps, лучшая производительность при большом количестве правил.

DDoS защита
?
L3/L4: upstream filtering, SYN cookies, rate limiting, anycast. L7: WAF, rate limiting по session, CAPTCHA, challenge-response. Комбинация CDN (Cloudflare, AWS Shield) + application-level protection. Невозможно защититься только одним методом.
