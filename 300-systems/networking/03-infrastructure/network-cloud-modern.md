# Облачные сети: VPC, Load Balancing, CDN

---

## Теоретические основы

> **SDN (Software-Defined Networking)** — архитектурная парадигма, основанная на формальном разделении плоскости управления (control plane) и плоскости данных (data plane). SDN-контроллер централизованно управляет правилами forwarding через программируемые интерфейсы (OpenFlow, RFC 7426). Эта модель лежит в основе всех облачных сетевых абстракций.

### Формальная модель SDN

| Уровень | Компонент | Функция | Примеры |
|---------|-----------|---------|---------|
| Application Plane | Сетевые приложения | Бизнес-логика, политики | Firewall, Load Balancer, IDS |
| Control Plane | SDN-контроллер | Централизованное управление | OpenDaylight, ONOS, AWS VPC Controller |
| Data Plane | Коммутаторы/маршрутизаторы | Forwarding пакетов по правилам | Open vSwitch, DPDK, SmartNIC |

### NFV и overlay-сети

- **NFV (Network Functions Virtualization)** — замена аппаратных сетевых устройств программными реализациями (виртуальные маршрутизаторы, файерволы, балансировщики). ETSI NFV ISG стандартизировал архитектуру в 2012 году
- **VXLAN (RFC 7348, Mahalingam et al., 2014)** — протокол overlay-сети, инкапсулирующий L2-кадры в UDP-пакеты (порт 4789). Поддерживает до 16 млн виртуальных сегментов (24-бит VNI). Основа сетевой виртуализации в AWS VPC, Azure VNet
- **GENEVE (RFC 8926, 2020)** — эволюция VXLAN с расширяемыми метаданными, поддерживается AWS Gateway Load Balancer

### Service Mesh архитектура

> **Service Mesh** — инфраструктурный уровень для управления межсервисной коммуникацией, реализованный через sidecar-прокси (Envoy, linkerd2-proxy) или eBPF (Cilium). Формально: L7-aware прокси, обеспечивающий mTLS, observability, traffic management и policy enforcement без модификации кода приложений.

### Хронология облачных сетевых технологий

- **2008** — McKeown et al. "OpenFlow: Enabling Innovation in Campus Networks" — основа SDN
- **2009** — AWS VPC — первая коммерческая реализация виртуальных частных сетей в облаке
- **2012** — ETSI NFV ISG — стандартизация Network Functions Virtualization
- **2014** — RFC 7348 (VXLAN) — стандарт overlay-сетей для дата-центров
- **2017** — Istio 0.1 — первый production-ready service mesh
- **2023** — Cilium graduated CNCF — eBPF-based networking и service mesh

### Терминология для новичков

> 💡 **VPC (Virtual Private Cloud)** = ваша "частная сеть" внутри облака. Как офисная сеть, только виртуальная и в AWS/GCP/Azure.

| Термин | Значение | Аналогия для новичка |
|--------|----------|---------------------|
| **VPC** | Изолированная виртуальная сеть | **Частный офис** — своя сеть, свои правила |
| **Subnet** | Часть VPC с диапазоном IP | **Этаж в здании** — отдельная зона |
| **Public Subnet** | Подсеть с доступом в интернет | **Рецепция** — открыта для посетителей |
| **Private Subnet** | Подсеть без прямого интернета | **Серверная** — только для сотрудников |
| **Security Group** | Файрвол для инстансов | **Охранник у двери** — кого пускать |
| **NAT Gateway** | Выход в интернет для private subnet | **Прокси-сервер** — выходим от имени одного IP |
| **ALB** | L7 балансировщик (HTTP) | **Умный администратор** — читает запрос и направляет |
| **NLB** | L4 балансировщик (TCP/UDP) | **Простой швейцар** — быстро направляет по порту |
| **CDN** | Сеть доставки контента | **Кэш рядом с пользователем** — картинки ближе |
| **PoP** | Точка присутствия CDN | **Филиал почты** — ближайший к тебе |
| **PrivateLink** | Приватный доступ к сервисам | **Прямой провод** — без выхода в интернет |

## Часть 1: Интуиция без кода

> 🎯 **Цель секции**: Понять облачные сети через повседневные аналогии — без единой строки кода или конфигурации.

### Аналогия 1: VPC как офисное здание

```
+------------------------------------------------------------------+
|                        РЕГИОН AWS (Москва)                        |
|                     = Бизнес-центр "Облако"                       |
|                                                                    |
|  +----------------------------+  +----------------------------+   |
|  |     AVAILABILITY ZONE A    |  |     AVAILABILITY ZONE B    |   |
|  |     = Корпус "Альфа"       |  |     = Корпус "Бета"        |   |
|  |                            |  |                            |   |
|  |  +----------------------+  |  |  +----------------------+  |   |
|  |  |      VPC             |  |  |  |      VPC             |  |   |
|  |  |   = Офис компании    |  |  |  |   = Офис компании    |  |   |
|  |  |                      |  |  |  |                      |  |   |
|  |  | +--------+ +-------+ |  |  |  | +--------+ +-------+ |  |   |
|  |  | |Public  | |Private| |  |  |  | |Public  | |Private| |  |   |
|  |  | |Subnet  | |Subnet | |  |  |  | |Subnet  | |Subnet | |  |   |
|  |  | |=Рецеп- | |=Сервер| |  |  |  | |=Рецеп- | |=Сервер| |  |   |
|  |  | | ция    | | ная   | |  |  |  | | ция    | | ная   | |  |   |
|  |  | +--------+ +-------+ |  |  |  | +--------+ +-------+ |  |   |
|  |  +----------------------+  |  |  +----------------------+  |   |
|  +----------------------------+  +----------------------------+   |
+------------------------------------------------------------------+
```

**Что это означает:**
- **Регион** = бизнес-центр в конкретном городе
- **Availability Zone** = отдельный корпус (независимое электропитание, охлаждение)
- **VPC** = офис вашей компании внутри корпуса
- **Public Subnet** = рецепция — открыта для посетителей
- **Private Subnet** = серверная — только для сотрудников
- **Security Group** = охранник у двери кабинета

**Почему это важно**: Если в корпусе "Альфа" отключат электричество, корпус "Бета" продолжит работать. Multi-AZ deployment = офисы в двух корпусах.

### Аналогия 2: Load Balancer как администратор бизнес-центра

```
                      Посетители (запросы)
                             |
                             v
                    +------------------+
                    |  АДМИНИСТРАТОР   |
                    |  (Load Balancer) |
                    +------------------+
                      /      |       \
                     v       v        v
              +-------+  +-------+  +-------+
              |Офис 1 |  |Офис 2 |  |Офис 3 |
              |Server1|  |Server2|  |Server3|
              +-------+  +-------+  +-------+

    ALB (L7) = "Умный администратор"
    - Читает запрос: "Мне нужен бухгалтер"
    - Направляет в нужный офис по содержимому
    - Может отказать: "Бухгалтерия закрыта"

    NLB (L4) = "Простой охранник"
    - Смотрит только номер кабинета (порт)
    - Быстро направляет, не вникая в суть
    - Не читает содержимое запроса
```

**Когда какой использовать:**
| Сценарий | Тип LB | Почему |
|----------|--------|--------|
| HTTP API с роутингом по path | ALB | Нужно читать URL |
| WebSocket соединения | ALB | Нужен HTTP upgrade |
| TCP база данных | NLB | Чистый TCP, низкая latency |
| UDP игровой сервер | NLB | UDP не поддерживает ALB |
| Миллион соединений | NLB | Меньше overhead |

### Аналогия 3: CDN как сеть почтовых отделений

```
                         ИНТЕРНЕТ
                            |
    +-------------------------------------------------------+
    |                                                       |
    |   Москва        Лондон        Нью-Йорк       Токио   |
    |     📮            📮            📮            📮      |
    |    PoP           PoP           PoP           PoP     |
    |     |             |             |             |       |
    +-----|-------------|-------------|-------------|-------+
          |             |             |             |
          v             v             v             v
       Юзер в        Юзер в        Юзер в       Юзер в
       Москве        Париже        Бостоне      Сеуле

    Без CDN: Все письма идут из главного офиса (origin)
             Москва -> Origin (Франкфурт) -> Москва = 50ms

    С CDN:   Копии писем хранятся в ближайшем отделении
             Москва -> PoP (Москва) = 5ms
```

**Что кэширует CDN:**
- ✅ Статика: картинки, CSS, JS, шрифты, видео
- ✅ API ответы с Cache-Control заголовками
- ❌ Динамика: корзина, личный кабинет
- ❌ POST/PUT/DELETE запросы

**Ключевые метрики:**
- **Cache Hit Ratio** = % запросов, отданных из кэша (цель: >90%)
- **Time to First Byte** = время до первого байта (цель: <50ms)
- **Origin Shield** = дополнительный кэш перед origin (снижает нагрузку)

### Аналогия 4: Security Group vs NACL как охранник vs турникет

```
+------------------------------------------------------------------+
|                         ЗДАНИЕ (VPC)                              |
|                                                                    |
|   +----------------------------------------------------------+   |
|   |              ТУРНИКЕТ НА ВХОДЕ (NACL)                     |   |
|   |                                                           |   |
|   |   - Проверяет всех входящих И выходящих                  |   |
|   |   - Не запоминает: "Ты уже заходил"                      |   |
|   |   - Правила по порядку (первое сработавшее)              |   |
|   |   - Уровень: весь этаж (subnet)                          |   |
|   +----------------------------------------------------------+   |
|                              |                                    |
|                              v                                    |
|   +----------------------------------------------------------+   |
|   |              ОХРАННИК У КАБИНЕТА (Security Group)         |   |
|   |                                                           |   |
|   |   - Проверяет только входящих                            |   |
|   |   - Запоминает: "Если впустил — выпущу обратно"          |   |
|   |   - Все правила проверяются вместе (OR)                  |   |
|   |   - Уровень: конкретный кабинет (instance)               |   |
|   +----------------------------------------------------------+   |
+------------------------------------------------------------------+
```

**Практическое различие:**

| Характеристика | Security Group | NACL |
|---------------|----------------|------|
| **Stateful/Stateless** | Stateful | Stateless |
| **Return traffic** | Автоматически | Нужно явное правило |
| **Уровень** | Instance/ENI | Subnet |
| **Правила** | Allow only | Allow + Deny |
| **Порядок правил** | Все оцениваются | По номерам (первое wins) |
| **Когда использовать** | Основная защита | Блокировка IP, compliance |

```
+------------------------------------------------------------------+
|                         VPC (Офис)                                |
|                                                                    |
|  PRIVATE SUBNET                          PUBLIC SUBNET            |
|  (Серверная)                             (Рецепция)               |
|                                                                    |
|  +-------------+                        +---------------+         |
|  | DB Server   |                        | NAT Gateway   |         |
|  | 10.0.2.50   |----> Хочу в интернет   | 52.10.20.30   |         |
|  +-------------+      обновить пакеты   +---------------+         |
|        |                                       |                  |
|        |                                       |                  |
|        +----------> Маршрутизация ------------>+                  |
|                                                |                  |
+------------------------------------------------|------------------+
                                                 |
                                                 v
                                         +---------------+
                                         |   ИНТЕРНЕТ    |
                                         | Видит только  |
                                         | 52.10.20.30   |
                                         +---------------+

```

**Когда что использовать:**

| Компонент | Направление | Использование |
|-----------|-------------|---------------|
| **Internet Gateway** | Двустороннее | Public subnet, web-серверы |
| **NAT Gateway** | Только outbound | Private subnet, обновления, API calls |
| **NAT Instance** | Только outbound | Бюджетная альтернатива (устаревает) |
| **VPC Endpoint** | Без интернета | Доступ к AWS сервисам приватно |

## Часть 2: Почему это сложно

> 🎯 **Цель секции**: Изучить 6 типичных ошибок, с которыми сталкиваются разработчики при работе с облачными сетями.

### Ошибка 1: Слишком узкий CIDR блок для VPC

**СИМПТОМ:**
```
Terraform apply:
Error: Error creating subnet: InvalidSubnet.Range:
The CIDR '10.0.0.0/24' has only 256 addresses.
Cannot create subnet for 128 instances + reserves.
```

**Почему происходит:**
Разработчики выбирают /24 (256 IP) "потому что достаточно", забывая:
- AWS резервирует 5 IP на subnet (network, gateway, DNS, future, broadcast)
- Blue-green deployment удваивает количество инстансов
- Auto-scaling может добавить 3x инстансов в пике

**РЕШЕНИЕ:**
```
# ПЛОХО: тесный CIDR
VPC CIDR:    10.0.0.0/24  (256 IP на ВСЮ VPC)
Subnet:      10.0.0.0/26  (64 IP на subnet)
Доступно:    59 IP (64 - 5 reserved)

# ХОРОШО: с запасом
VPC CIDR:    10.0.0.0/16  (65536 IP на VPC)
Subnet:      10.0.1.0/24  (256 IP на subnet)
Доступно:    251 IP на subnet, можно создать 256 subnets
```

**Правило**: Начинайте с /16 для VPC, /24 для subnet. Сужать проще, чем расширять.

### Ошибка 2: Security Group разрешает 0.0.0.0/0

**СИМПТОМ:**
```bash
# AWS Security Hub / GuardDuty alert:
[CRITICAL] Security group sg-12345 allows unrestricted
inbound traffic on port 22 from 0.0.0.0/0
```

**Почему происходит:**
"Для тестирования быстро открыл SSH со всего мира, потом забыл закрыть."

**РЕШЕНИЕ:**
```
# ПЛОХО: открыто миру
Type: SSH
Port: 22
Source: 0.0.0.0/0  # ВЕСЬ ИНТЕРНЕТ!

# ХОРОШО: только с bastion/VPN
Type: SSH
Port: 22
Source: sg-bastion-host  # Только через jump-сервер

# ЕЩЁ ЛУЧШЕ: AWS Session Manager (без открытых портов)
# Вообще не нужен порт 22!
```

**Правило**: Никогда не открывайте 0.0.0.0/0 для SSH (22), RDP (3389), или баз данных. Используйте bastion host или Session Manager.

### Ошибка 3: ALB в private subnet

**СИМПТОМ:**
```
curl https://my-alb.region.elb.amazonaws.com
curl: (7) Failed to connect: Connection timed out
```

**Почему происходит:**
ALB создан в private subnet без доступа к интернету, поэтому клиенты из интернета не могут до него достучаться.

**РЕШЕНИЕ:**
```
+------------------------------------------------------------------+
|                            VPC                                    |
|                                                                    |
|   PUBLIC SUBNETS (2+ AZ)          PRIVATE SUBNETS (2+ AZ)        |
|   +-------------------+           +---------------------+         |
|   |     ALB ✅        |           |    EC2 instances    |         |
|   | (интернет-facing) |---------->|    (target group)   |         |
|   +-------------------+           +---------------------+         |
|           |                                                       |
|           v                                                       |
|   Internet Gateway                                                |
+------------------------------------------------------------------+

Правила размещения:
- Internet-facing ALB → Public Subnets (минимум 2 AZ!)
- Internal ALB → Private Subnets
- Target instances → Private Subnets (безопасность)
```

### Ошибка 4: CDN кэширует персональные данные

**СИМПТОМ:**
```
Пользователь А видит корзину пользователя Б!
Cache-Control: public, max-age=3600
```

**Почему происходит:**
Разработчик не настроил правильные заголовки кэширования, и CDN закэшировал страницу с персональными данными.

**РЕШЕНИЕ:**
```
# ПЛОХО: кэшируется для всех
GET /api/cart
Cache-Control: public, max-age=3600  # CDN кэширует!

# ХОРОШО: персональные данные не кэшируются
GET /api/cart
Cache-Control: private, no-store     # Только браузер клиента
Vary: Authorization                   # Разный кэш для разных юзеров

# CDN bypass для динамики:
Cloudflare Page Rule: /api/* → Cache Level: Bypass
CloudFront Behavior: /api/* → TTL: 0, Forward all headers
```

**Правило**: `Cache-Control: public` только для статики. Всё с авторизацией — `private` или `no-store`.

### Ошибка 5: DNS loop при настройке CDN

**СИМПТОМ:**
```
curl https://example.com
curl: (7) Failed to connect: Too many redirects
# или
502 Bad Gateway
```

**Почему происходит:**
CDN настроен на домен example.com, а origin тоже указывает на example.com:
```
                  +----------+
                  |   CDN    |
         +------->| example  |<-------+
         |        | .com     |        |
         |        +----------+        |
         |              |             |
         |              v             |
         |        +----------+        |
         +--------|  Origin  |--------+
                  | example  |
                  | .com     |  <- LOOP!
                  +----------+
```

**РЕШЕНИЕ:**
```
# ПЛОХО: origin = тот же домен
CDN Domain:    example.com
Origin:        example.com  # LOOP!

# ХОРОШО: origin = отдельный домен или IP
CDN Domain:    example.com
Origin:        origin.example.com  # или IP напрямую
               (DNS запись указывает на Load Balancer)

# Структура DNS:
example.com         → CNAME → cdn.cloudflare.com
origin.example.com  → A     → 52.1.2.3 (ALB IP)
```

### Ошибка 6: Забыли egress rules для NAT Gateway

**СИМПТОМ:**
```
# Из EC2 в private subnet:
curl https://api.github.com
curl: (7) Failed to connect: Connection timed out

# Но NAT Gateway создан и route table настроена!
```

**Почему происходит:**
NACL по умолчанию разрешает весь трафик, но если вы кастомизировали — забыли разрешить ephemeral ports для return traffic.

**РЕШЕНИЕ:**
```
# NACL для Private Subnet:

# Outbound (от инстансов):
Rule 100: ALLOW | HTTPS (443) | 0.0.0.0/0      # К интернету
Rule 110: ALLOW | HTTP  (80)  | 0.0.0.0/0      # К интернету

# Inbound (return traffic!):
Rule 100: ALLOW | TCP | 1024-65535 | 0.0.0.0/0 # Ephemeral ports!
          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
          БЕЗ ЭТОГО ПРАВИЛА NAT НЕ РАБОТАЕТ!

# Почему 1024-65535?
# Это ephemeral (временные) порты, которые ОС выбирает
# для исходящих соединений. Return traffic приходит на них.
```

**Правило**: Если используете NACL — всегда разрешайте ephemeral ports (1024-65535) для inbound в private subnet.

### Модель 5: Чеклист для production-ready VPC

```
□ VPC DESIGN
  ├── □ CIDR достаточно большой (/16 для VPC)
  ├── □ Не пересекается с on-premises или другими VPC
  ├── □ Subnets в минимум 2 AZ
  └── □ Отдельные subnets: public, private, isolated (DB)

□ CONNECTIVITY
  ├── □ Internet Gateway для public subnets
  ├── □ NAT Gateway для private subnets (в каждой AZ!)
  ├── □ VPC Endpoints для AWS сервисов (S3, DynamoDB)
  └── □ Transit Gateway если >3 VPC

□ SECURITY
  ├── □ Security Groups: минимальные права, нет 0.0.0.0/0
  ├── □ NACL: только для compliance или IP blocking
  ├── □ Flow Logs включены (CloudWatch или S3)
  └── □ WAF перед публичными ALB

□ DNS
  ├── □ Route 53 private hosted zone для внутренних имён
  ├── □ DNS hostnames и resolution включены в VPC
  └── □ Записи для internal ALB

□ MONITORING
  ├── □ VPC Flow Logs → CloudWatch Insights
  ├── □ ALB access logs → S3
  ├── □ CloudWatch alarms на NAT Gateway
  └── □ AWS Config rules для Security Groups

□ COST OPTIMIZATION
  ├── □ NAT Gateway в каждой AZ vs один shared (trade-off)
  ├── □ VPC Endpoints вместо NAT для AWS сервисов
  ├── □ Reserved capacity для ALB/NLB если предсказуемо
  └── □ Удаление неиспользуемых EIP, NAT, endpoints
```

```

### Ключевые проблемы, которые решают облачные сети

**1. Изоляция и безопасность**
- Логическая изоляция ресурсов разных проектов/команд
- Сегментация по уровням доступа (public/private/isolated)
- Zero-trust networking с mTLS между сервисами
- Compliance requirements (PCI DSS, HIPAA, SOC2)

**2. Масштабируемость**
- Горизонтальное масштабирование через auto-scaling groups
- Балансировка нагрузки между инстансами и регионами
- CDN для глобальной доставки контента

**3. Доступность и отказоустойчивость**
- Multi-AZ развертывание (99.99% SLA)
- Автоматический failover при сбоях
- Geographic redundancy для disaster recovery

**4. Производительность**
- Низкая latency через edge computing
- Оптимизированные пути между сервисами
- Высокопроизводительные inter-connect для гибридных сред

**5. Стоимость**
- Pay-as-you-go вместо капитальных затрат
- Reserved instances для предсказуемых нагрузок
- FinOps практики для оптимизации расходов

## Современные тренды 2024-2025

### AWS VPC Lattice: замена App Mesh

#### Почему AWS отказался от App Mesh

**История создания и проблемы App Mesh.** В 2019 году AWS запустил App Mesh как ответ на растущую популярность Istio и service mesh в целом. Идея была правильной: в мире микросервисов нужен стандартизированный способ управления трафиком между сервисами — балансировка, ретраи, таймауты, mTLS шифрование.

**Но реализация оказалась сложной.** App Mesh работал через Envoy sidecar прокси — отдельный контейнер, который запускался рядом с каждым подом вашего приложения. Это создавало несколько серьёзных проблем:

1. **Операционная сложность.** Нужно было управлять версиями Envoy, мониторить их здоровье, отлаживать проблемы маршрутизации. У каждого пода теперь было два контейнера вместо одного.

2. **Overhead ресурсов.** Каждый Envoy sidecar потреблял память и CPU. При 100 подах это уже ощутимые расходы.

3. **Ограниченная совместимость.** App Mesh работал только с ECS и EKS. Lambda, EC2 Auto Scaling Groups — всё это требовало отдельных решений.

4. **Сложность cross-VPC коммуникации.** Чтобы сервисы в разных VPC общались через App Mesh, нужно было настраивать Transit Gateway, VPC Peering, NAT — огромная инфраструктурная работа.

**Adoption был низким.** AWS признал это в 2024 году, объявив о прекращении App Mesh с 30 сентября 2026 года. Новые клиенты не могут подключиться с 24 сентября 2024 года.

**VPC Lattice — это принципиально другой подход.** Вместо sidecar прокси в каждом поде, Lattice работает на уровне самой сети AWS. Это "managed data plane" — вы не управляете прокси, AWS делает это за вас. Трафик проходит через инфраструктуру AWS напрямую, без дополнительных контейнеров в ваших подах.

**Ключевое преимущество:** VPC Lattice изначально спроектирован для мира, где сервисы могут быть Lambda-функциями, ECS-задачами, EKS-подами и EC2-инстансами одновременно. И все они могут жить в разных VPC и даже разных AWS-аккаунтах. Lattice решает cross-VPC routing нативно, через так называемые Service Networks.

### Cilium и eBPF Networking

#### Почему iptables и kube-proxy больше не справляются

**Исторический контекст.** Когда создавался Kubernetes в 2014 году, для реализации сетевой маршрутизации выбрали iptables — проверенную технологию, которая работала в Linux десятилетиями. Kube-proxy использует iptables для направления трафика к нужным подам при обращении к сервисам. Это было прагматичное решение: iptables понятен, хорошо документирован, "просто работает".

**Но архитектура iptables имеет фундаментальные ограничения:**

1. **O(n) сложность.** Iptables реализует правила как связанный список. Каждый пакет проходит через все правила последовательно. В кластере с 5,000 сервисов это означает 5,000+ проверок на каждый пакет.

2. **Атомарные обновления.** При изменении правил iptables пересоздаёт всю цепочку правил целиком, в одной транзакции. Добавление одного сервиса требует перезаписи всех правил.

3. **Экспоненциальный рост.** По мере роста кластера количество правил растёт экспоненциально. Измерения показывают непредсказуемую latency и деградацию производительности с ростом числа сервисов.

4. **Отсутствие L7-awareness.** Iptables работает на уровне IP и портов. Он не понимает HTTP, gRPC, или других прикладных протоколов. Для L7-политик нужны дополнительные прокси.

**IPVS был попыткой решить проблему.** Kubernetes добавил режим IPVS для kube-proxy, который использует хеш-таблицы вместо списков и масштабируется лучше. Но IPVS всё ещё работает в рамках традиционной модели kernel-модулей, которая достигает потолка производительности при высокодинамичных нагрузках.

**eBPF (extended Berkeley Packet Filter) — это революция в сетевом стеке Linux.** Это технология, позволяющая запускать программы прямо в ядре Linux, в безопасной песочнице. Вместо статических правил, которые ядро интерпретирует, eBPF загружает скомпилированные программы, которые ядро выполняет напрямую.

**Cilium использует eBPF для полной замены kube-proxy:**

- **O(1) lookup.** Вместо списка правил — хеш-таблица. Время поиска не зависит от количества сервисов.
- **Socket-level load balancing.** eBPF перехватывает трафик на уровне сокета, минуя часть сетевого стека ядра.
- **XDP (eXpress Data Path).** Обработка пакетов прямо в драйвере сетевой карты, до того как пакет попал в полный сетевой стек.
- **L7-политики без sidecar.** eBPF может инспектировать HTTP-заголовки прямо в ядре.

**Результаты впечатляют:** 10-100x меньше latency, возможность обрабатывать миллионы пакетов в секунду на одном ноде. Benchmark'и показывают, что eBPF-решения превосходят даже baseline "нода-к-ноде" на современных ядрах — потому что eBPF обходит слой iptables полностью.

### Zero Trust Networking в Multi-Cloud

#### Почему периметр безопасности больше не работает

**Традиционная модель "замок и ров".** Десятилетиями корпоративная безопасность строилась по принципу крепости: снаружи — враждебный мир, внутри — безопасная зона. Firewall на периметре сети, VPN для удалённого доступа, и полное доверие ко всему трафику внутри корпоративной сети. Если ты прошёл firewall — значит, ты свой.

**Эта модель катастрофически провалилась.**

**Случай SolarWinds (2020)** стал переломным моментом. 13 декабря 2020 года FireEye обнаружила вредоносный код в обновлении ПО SolarWinds Orion — системы мониторинга, которую использовали более 18,000 организаций, включая Министерства финансов, торговли и национальной безопасности США, а также Microsoft и Deloitte. Хакеры имели доступ к системам до 14 месяцев.

**Почему традиционная защита не сработала:**

1. **Доверие к внутреннему трафику.** SolarWinds Orion считался доверенным ПО внутри периметра. Его исходящие соединения не блокировались — блокировались только "известные плохие" адреса.

2. **Lateral movement.** Получив начальный доступ через скомпрометированное ПО, атакующие свободно перемещались по сети, потому что внутренний трафик не проверялся.

3. **Implicit trust.** Все системы внутри периметра доверяли друг другу по умолчанию.

**Как это могло бы работать с Zero Trust:** архитектура была бы инвертирована. Коммуникация разрешена только с известными и аутентифицированными сущностями, независимо от того, откуда исходит трафик — "изнутри" или "снаружи". Каждому компоненту предоставлены минимальные права. Даже если хакер получил доступ, его возможности были бы резко ограничены.

**Концепция Zero Trust появилась в 2010 году,** когда аналитик Forrester Джон Киндерваг опубликовал первый отчёт, предупреждая об опасности слепого доверия внутри сетей. Но adoption был медленным — индустрия была скептична, а изменение парадигмы требовало значительных усилий.

**SolarWinds всё изменил.** В мае 2021 года администрация Байдена выпустила указ "Об улучшении кибербезопасности страны", требующий от федеральных агентств перехода на Zero Trust. К концу 2024 года все федеральные агентства США должны были внедрить принципы Zero Trust.

**Статистика 2024 года подтверждает необходимость:** 71% рост атак с использованием украденных или скомпрометированных credentials по сравнению с прошлым годом. Атакующие используют легитимные учётные данные, чтобы "войти через парадную дверь". 68% взломов в 2024 году включали человеческий фактор — ошибки, фишинг, злоупотребления инсайдеров.

**Принципы Zero Trust для облачной инфраструктуры:**

- **Never trust, always verify.** Каждый запрос проверяется, независимо от источника.
- **Least privilege access.** Минимальные права, достаточные для выполнения конкретной задачи.
- **Assume breach.** Проектируй так, будто взлом уже произошёл — ограничь blast radius.
- **Verify explicitly.** Проверяй identity, device, location, behavior — всё вместе.
- **Micro-segmentation everywhere.** Сегментация на уровне отдельных workloads, не сетей.

### Serverless VPC Networking

#### Проблема холодного старта Lambda в VPC

**Изначальный design Lambda (2014-2019).** Когда AWS Lambda нужен доступ к ресурсам внутри VPC — RDS базе данных, ElastiCache, или внутреннему API — функция должна получить сетевой интерфейс (ENI) в этой VPC. В первоначальной архитектуре каждый новый execution environment Lambda создавал собственный ENI. Это было логично с точки зрения изоляции, но катастрофически для производительности.

**Почему это было так медленно:**

1. **Создание ENI — тяжёлая операция.** Elastic Network Interface создаётся через EC2 API, получает IP-адрес, привязывается к security groups, получает route к подсети. Всё это занимает секунды.

2. **10-15 секунд cold start.** Эксперименты показывали, что Lambda с 3GB памяти запускалась в 30 раз дольше при подключении к VPC по сравнению с Lambda без VPC.

3. **Исчерпание IP-адресов.** Каждый ENI занимает IP-адрес в подсети. При масштабировании сотен Lambda-функций подсети /24 (251 доступный IP) заканчивались быстро.

4. **Лимиты ENI.** AWS накладывает ограничения на количество ENI в регионе. При высоких нагрузках упирались в эти лимиты.

5. **Rate limiting EC2 API.** Массовое создание ENI могло упереться в throttling EC2 API.

**Результат: разработчики избегали VPC.** Если можно было обойтись без подключения к VPC — обходились. Это ограничивало архитектурные возможности и иногда приводило к менее безопасным решениям (открытие баз данных в интернет).

**Решение 2019 года: AWS Hyperplane ENI.** На re:Invent 2018 AWS анонсировал план решения проблемы, а в сентябре 2019 выкатил изменения. Ключевая идея: вместо создания ENI для каждого execution environment, использовать общие ENI, pre-provisioned заранее.

**Как работает Hyperplane:** это внутренняя технология AWS (также используется в Network Load Balancer и NAT Gateway), которая действует как распределённый пакетный forwarder. ENI в вашей VPC подключается к Hyperplane, и множество Lambda execution environments подключаются к нему через туннели.

**Технические детали:**

- ENI создаётся при создании Lambda функции или при изменении VPC-настроек, а не при invocation.
- Функции с одинаковой комбинацией security group + subnet делят один ENI.
- При invocation execution environment просто устанавливает сетевой туннель к уже существующему ENI.
- Первоначальная настройка занимает до 90 секунд (один раз при deploy), но последующие cold starts — менее секунды.

**Реальные результаты:** cold start снизился с 12 секунд до 5 секунд сразу после внедрения, и продолжает улучшаться. Для latency-critical приложений AWS добавил Provisioned Concurrency (pre-warmed execution environments) и SnapStart для Java (JVM snapshot restoration).

**Современные рекомендации:**

- DynamoDB, S3, SQS — не требуют VPC, используйте VPC Endpoints если нужна приватность
- RDS, ElastiCache, внутренние API — используйте VPC, Hyperplane ENI делает это приемлемым
- Для latency-critical — Provisioned Concurrency или SnapStart
- ARM64 (Graviton2) даёт лучший cold start благодаря более быстрой инициализации

### Network Cost Optimization (FinOps)

#### Скрытые расходы на сетевую инфраструктуру в облаке

**"Ваш AWS-счёт продолжает расти, и вы не понимаете почему."** Это типичная ситуация для многих организаций. Compute-ресурсы (EC2, Lambda) обычно хорошо понятны и оптимизированы. Но сетевые расходы часто остаются невидимыми — до тех пор, пока не становятся проблемой.

**NAT Gateway — главный виновник.** Вместо того чтобы давать каждому инстансу публичный IP, организации используют NAT Gateway для исходящего интернет-трафика из приватных подсетей. Это правильно с точки зрения безопасности. Но цена: $0.045 в час за сам gateway + $0.045 за каждый обработанный гигабайт.

**Пример расчёта:** 50TB трафика через NAT Gateway в месяц:
- NAT Gateway hourly: $0.045 × 24 × 30 = $32.40
- Data processing: 50,000 GB × $0.045 = $2,250
- **Итого: $2,282.40 в месяц только за NAT**

**Типичные ошибки конфигурации:**

1. **Трафик к AWS-сервисам через NAT вместо VPC Endpoints.** Ваши Lambda-функции обращаются к S3 через интернет (NAT), хотя могли бы использовать бесплатный Gateway Endpoint.

2. **Cross-AZ трафик.** Передача данных между зонами доступности стоит $0.01 за GB в каждом направлении. Если ваш сервис в AZ-a обращается к базе в AZ-b — это платно.

3. **Неиспользуемые Elastic IP.** AWS не берёт деньги за EIP, привязанный к работающему инстансу. Но за неиспользуемый EIP — $0.005 в час (≈$3.60 в месяц за каждый).

4. **Interface VPC Endpoints "на всякий случай".** Interface Endpoints стоят $0.01 в час + $0.01 за GB. Gateway Endpoints (S3, DynamoDB) — бесплатны.

**FinOps-подход к networking:** это не просто "сокращение расходов", а операционная модель, которая приносит финансовую ответственность в переменные cloud-расходы. Ключевые принципы:

1. **Visibility.** Вы не можете оптимизировать то, что не видите. CloudWatch metrics, Cost Explorer с группировкой по сервисам и тегам.

2. **Allocation.** Распределение расходов по командам, приложениям, линиям бизнеса. Кто генерирует трафик — тот и отвечает за оптимизацию.

3. **Optimization.** Архитектурные решения с учётом стоимости: co-location сервисов в одной AZ, VPC Endpoints, кэширование.

**Реальные результаты:** компании сообщают о снижении NAT Gateway costs на 40% после анализа паттернов трафика и правильной конфигурации VPC Endpoints. Типичный enterprise может тратить $2,250/месяц на NAT processing для 50TB данных — и сократить это в разы.

**Инструменты мониторинга:** CloudWatch VPC Flow Logs показывают паттерны трафика. AWS Cost Explorer позволяет группировать по сервисам. Специализированные инструменты (Kentik, CloudHealth, Spot.io) предоставляют глубокую аналитику и рекомендации.

```
Cloud Networking Cost Breakdown:

## Best Practices Summary

### VPC Design
1. **Plan CIDR early** - избегайте пересечений с on-premises и другими VPC
2. **Multi-AZ deployment** - минимум 2 AZ для production
3. **Subnet segmentation** - public/private/isolated по назначению
4. **VPC Flow Logs** - включите для troubleshooting и compliance
5. **IPv6 ready** - планируйте dual-stack с начала

### Security
1. **Least privilege** - открывайте только необходимые порты
2. **Security groups as primary** - NACLs как дополнительный слой
3. **Reference SGs, not CIDRs** - используйте security group ID как source
4. **No 0.0.0.0/0 for SSH/RDP** - только VPN или bastion
5. **Encrypt in transit** - TLS везде, mTLS между сервисами

### Load Balancing
1. **ALB for HTTP/HTTPS** - path routing, header routing
2. **NLB for TCP/UDP** - низкая latency, static IP
3. **Health checks tuning** - подберите interval и thresholds
4. **Graceful shutdown** - deregistration_delay = connection drain time

### CDN
1. **Version static assets** - hash в имени файла
2. **Separate behaviors** - разные cache policies для static/dynamic
3. **Security headers** - CSP, HSTS, X-Frame-Options
4. **Monitor hit ratio** - должен быть >90% для static

### Cost Optimization
1. **VPC Endpoints** - избегайте NAT для AWS services
2. **Single NAT for dev** - не нужен HA в non-production
3. **Reserved capacity** - для стабильных workloads
4. **Data transfer awareness** - cross-AZ/region transfers cost money

## Проверь себя

> [!question]- Почему Security Group со входящим правилом 0.0.0.0/0 на порту 22 --- критическая уязвимость, даже если SSH требует ключ?
> Открытый SSH на весь интернет подвергает сервер brute-force атакам, сканированию уязвимостей SSH-демона, эксплуатации 0-day. Ключевая аутентификация защищает от подбора пароля, но не от уязвимостей в самом OpenSSH. Решение: ограничить SSH конкретными IP/VPN, использовать bastion host, AWS Systems Manager Session Manager.

> [!question]- Чем ALB (Application Load Balancer) отличается от NLB (Network Load Balancer) и когда выбрать каждый?
> ALB работает на L7 (HTTP/HTTPS): path-based routing, host-based routing, WAF интеграция, WebSocket. NLB на L4 (TCP/UDP): ultra-low latency, static IP, millions of RPS, TLS passthrough. ALB для веб-приложений и API. NLB для: gRPC, IoT, gaming, когда нужен реальный IP клиента или статический IP.

> [!question]- CDN настроен, но персонализированные страницы показывают данные других пользователей. Что не так?
> CDN кэширует ответы по URL. Если персонализированный контент (профиль, корзина) имеет тот же URL для всех, CDN закэширует ответ первого пользователя и покажет всем. Решение: добавить Cache-Control: private для персонализированного контента, использовать Vary: Cookie, или исключить dynamic paths из CDN.

## Ключевые карточки

Что такое VPC и из чего он состоит?
?
VPC (Virtual Private Cloud) --- изолированная виртуальная сеть в облаке. Компоненты: subnets (public/private в разных AZ), Internet Gateway (выход в интернет), NAT Gateway (для private subnets), Route Tables, Security Groups (stateful firewall), NACLs (stateless firewall).

Чем Security Group отличается от NACL?
?
Security Group --- stateful (ответный трафик разрешён автоматически), на уровне ENI/instance, только ALLOW правила. NACL --- stateless (нужны отдельные правила для входящего и исходящего), на уровне subnet, ALLOW и DENY правила с приоритетами. SG --- основная защита, NACL --- дополнительный уровень.

Как работает CDN?
?
CDN (Content Delivery Network) --- сеть edge-серверов по всему миру. Статический контент кэшируется на PoP (Point of Presence) ближайшем к пользователю. Первый запрос идёт к origin, последующие --- из кэша. Снижает latency, разгружает origin, защищает от DDoS.

Что такое NAT Gateway?
?
NAT Gateway позволяет инстансам в private subnet выходить в интернет (скачивать обновления, обращаться к API) без собственного публичного IP. Входящие соединения из интернета невозможны. Альтернатива: NAT Instance (дешевле, но ручное управление, single point of failure).

Что такое Service Mesh?
?
Service Mesh (Istio, Linkerd, Cilium) --- инфраструктурный уровень для управления межсервисной коммуникацией. Sidecar proxy (Envoy) перехватывает весь трафик. Обеспечивает: mTLS, retries, circuit breaking, observability, traffic splitting без изменения кода приложений.
