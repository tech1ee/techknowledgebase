# SQL Fundamentals: от SELECT до оконных функций

> SQL — язык для работы с реляционными базами данных. Знание SQL — обязательный навык для любого разработчика, независимо от специализации.

---

## Теоретические основы

> **SQL (Structured Query Language)** — декларативный язык для управления данными в реляционных СУБД. Основан на **реляционной алгебре** (Codd, 1970) и **реляционном исчислении** (Codd, 1972).

### Реляционная модель (Codd, 1970)

E.F. Codd в работе «A Relational Model of Data for Large Shared Data Banks» определил:
- **Relation** = таблица (набор кортежей одной структуры)
- **Attribute** = столбец (именованный домен значений)
- **Tuple** = строка (набор значений атрибутов)
- **Key** = минимальный набор атрибутов, уникально идентифицирующий кортеж

### Реляционная алгебра: 6 базовых операций

| Операция | SQL-аналог | Описание |
|----------|-----------|----------|
| **σ (Selection)** | WHERE | Фильтрация строк по условию |
| **π (Projection)** | SELECT columns | Выбор столбцов |
| **× (Cartesian product)** | CROSS JOIN | Все комбинации строк |
| **⋈ (Join)** | JOIN ON | Соединение по условию |
| **∪ (Union)** | UNION | Объединение множеств строк |
| **− (Difference)** | EXCEPT | Разность множеств строк |

> **См. также**: [[databases-overview]] — карта раздела, [[database-design-optimization]] — оптимизация запросов

## TL;DR

- **SELECT** — выборка данных, основа всего
- **JOIN** — объединение таблиц (INNER, LEFT, RIGHT, FULL)
- **Подзапросы** — запрос внутри запроса (WHERE, FROM, SELECT)
- **CTE (WITH)** — именованные подзапросы для читаемости
- **Оконные функции** — агрегация без группировки (ROW_NUMBER, RANK, LAG/LEAD)
- **Агрегация** — GROUP BY + HAVING для сводных данных

## Часть 2: Почему это сложно — типичные ошибки

### Ошибка 1: NULL — "чёрная дыра" в сравнениях

**СИМПТОМ:** "Запрос возвращает 0 строк, хотя данные точно есть!"

```sql
-- ❌ НЕПРАВИЛЬНО: NULL не равен ничему, даже себе!
SELECT * FROM users WHERE status = NULL;    -- 0 строк!
SELECT * FROM users WHERE status != 'active'; -- пропустит NULL!

-- Почему? NULL = "неизвестно"
-- "неизвестно" = "неизвестно" → "неизвестно" (не TRUE!)
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    NULL — ЭТО НЕ ЗНАЧЕНИЕ                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   NULL ≠ 0                                                       │
│   NULL ≠ ""  (пустая строка)                                    │
│   NULL ≠ NULL  (!)                                               │
│                                                                  │
│   NULL означает: "Я НЕ ЗНАЮ"                                    │
│                                                                  │
│   Вопрос: "Неизвестное = Неизвестному?"                         │
│   Ответ:  "Не знаю" (NULL, не TRUE и не FALSE)                  │
│                                                                  │
│   ⚠️ ОСОБАЯ ОПАСНОСТЬ: NOT IN с NULL                            │
│                                                                  │
│   SELECT * FROM users                                            │
│   WHERE id NOT IN (1, 2, NULL);  -- Вернёт 0 строк!             │
│                                                                  │
│   Почему? NOT IN разворачивается в:                             │
│   id != 1 AND id != 2 AND id != NULL                            │
│                           ↑                                      │
│                      Всегда NULL → всё выражение NULL           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:**
```sql
-- ✅ ПРАВИЛЬНО
SELECT * FROM users WHERE status IS NULL;
SELECT * FROM users WHERE status IS NOT NULL;
SELECT * FROM users WHERE status != 'active' OR status IS NULL;

-- ✅ NOT IN без NULL
SELECT * FROM users WHERE id NOT IN (
    SELECT user_id FROM banned WHERE user_id IS NOT NULL
);

-- ✅ Лучше: NOT EXISTS
SELECT * FROM users u
WHERE NOT EXISTS (SELECT 1 FROM banned b WHERE b.user_id = u.id);
```

### Ошибка 2: Cartesian Product — случайный взрыв данных

**СИМПТОМ:** "Запрос вернул миллионы строк, хотя в таблицах по тысяче"

```sql
-- ❌ КАТАСТРОФА: забыли условие JOIN
SELECT * FROM orders, products;  -- 1000 × 1000 = 1,000,000 строк!

-- или эквивалент
SELECT * FROM orders
CROSS JOIN products;  -- То же самое!
```

```
│   ┌────────┐                ┌────────────┐                      │
│   │ Заказ 1│                │ Товар A    │                      │
│   │ Заказ 2│                │ Товар B    │                      │
│   │ Заказ 3│                │ Товар C    │                      │
│   └────────┘                └────────────┘                      │
│                                                                  │
│   БЕЗ условия JOIN (Cartesian):     С условием JOIN:            │
│   ┌────────────────────────┐        ┌────────────────────────┐  │
│   │ Заказ 1 │ Товар A      │        │ Заказ 1 │ Товар A      │  │
│   │ Заказ 1 │ Товар B      │        │ Заказ 2 │ Товар B      │  │

│   │ Заказ 2 │ Товар A      │        └────────────────────────┘  │
│   │ Заказ 2 │ Товар B      │        3 строки (правильно!)       │
│   │ Заказ 2 │ Товар C      │                                    │
│   │ Заказ 3 │ Товар A      │                                    │

│   │ Заказ 3 │ Товар C      │                                    │
```

```sql
-- ✅ ВСЕГДА указывай условие JOIN
SELECT * FROM orders o
INNER JOIN products p ON o.product_id = p.id;

-- ⚠️ Признак проблемы: результат >> ожидаемого
-- Если ждал 1000, а получил 1,000,000 — проверь JOIN!
```

### Ошибка 4: Функции на колонках в WHERE — убийца индексов

**СИМПТОМ:** "Индекс есть, но EXPLAIN показывает Seq Scan"

```sql
-- ❌ ПЛОХО: функция на колонке = индекс бесполезен
SELECT * FROM orders WHERE YEAR(created_at) = 2024;
SELECT * FROM users WHERE LOWER(email) = 'ivan@mail.ru';
SELECT * FROM products WHERE price + 100 > 500;
```

```
┌─────────────────────────────────────────────────────────────────┐
│            ФУНКЦИИ УБИВАЮТ ИНДЕКСЫ (не SARGable)                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Индекс работает как телефонный справочник:                    │
│   • Найти "Иванов" — быстро (индекс по алфавиту)               │
│   • Найти "все фамилии из 6 букв" — медленно (читать всё)      │
│                                                                  │
│   YEAR(created_at) = 2024                                        │
│           ↓                                                      │
│   БД должна вычислить YEAR() для КАЖДОЙ строки                  │
│   → Индекс бесполезен → Full Table Scan                         │
│                                                                  │
│   ❌ WHERE YEAR(date) = 2024                                     │
│   ✅ WHERE date >= '2024-01-01' AND date < '2025-01-01'         │
│                                                                  │
│   ❌ WHERE LOWER(email) = 'ivan@mail.ru'                        │
│   ✅ WHERE email ILIKE 'ivan@mail.ru'  (PostgreSQL)             │
│   ✅ Или создай функциональный индекс: INDEX ON LOWER(email)    │
│                                                                  │
│   ❌ WHERE price + 100 > 500                                     │
│   ✅ WHERE price > 400                                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Перенеси вычисления на правую сторону (SARGable — Search ARGument Able)

### Ошибка 5: DISTINCT как костыль для дубликатов

**СИМПТОМ:** "Добавил DISTINCT — дубликаты пропали, но запрос стал медленнее"

```sql
-- ❌ Признак проблемы в JOIN
SELECT DISTINCT u.name, o.total
FROM users u
JOIN orders o ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id;
-- Дубликаты из-за множественных items

-- Вопрос: ПОЧЕМУ дубликаты? DISTINCT — это пластырь, не лечение!
```

```
┌─────────────────────────────────────────────────────────────────┐
│              DISTINCT — ПЛАСТЫРЬ, НЕ ЛЕКАРСТВО                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Дубликаты появляются из-за:                                   │
│   1. Лишнего JOIN (убери ненужные таблицы)                      │
│   2. JOIN по не-уникальному ключу (проверь связи)               │
│   3. Отсутствия GROUP BY (добавь группировку)                   │
│                                                                  │
│   DISTINCT:                                                      │
│   • Сортирует ВСЕ результаты                                    │
│   • Сравнивает ВСЕ колонки                                      │
│   • Затратная операция                                          │
│                                                                  │
│   Вместо DISTINCT попробуй:                                     │
│   • GROUP BY — если нужна агрегация                             │
│   • EXISTS — если проверяешь наличие                            │
│   • DISTINCT ON — если нужна первая строка группы (PostgreSQL)  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Найди причину дубликатов и исправь её, а не маскируй DISTINCT

### Ошибка 6: Подзапрос вместо JOIN

**СИМПТОМ:** "Запрос очень медленный" (correlated subquery)

```sql
-- ❌ МЕДЛЕННО: подзапрос выполняется для КАЖДОЙ строки
SELECT
    u.name,
    (SELECT COUNT(*) FROM orders WHERE user_id = u.id) as order_count
FROM users u;
-- 1000 пользователей = 1000 подзапросов!

-- ✅ БЫСТРО: один JOIN + GROUP BY
SELECT u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;
-- 1 запрос!
```

```
┌─────────────────────────────────────────────────────────────────┐
│           CORRELATED SUBQUERY VS JOIN                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Correlated subquery (N+1 проблема):                           │
│   ┌─────────────────────────────────────────┐                   │
│   │ Для каждого user:                       │                   │
│   │   → Выполни подзапрос                   │ × 1000 раз!       │
│   │   → Верни результат                     │                   │
│   └─────────────────────────────────────────┘                   │
│   Сложность: O(N × M)                                           │
│                                                                  │
│   JOIN (один проход):                                           │
│   ┌─────────────────────────────────────────┐                   │
│   │ 1. Соедини таблицы                      │ 1 раз             │
│   │ 2. Сгруппируй                           │ 1 раз             │
│   │ 3. Верни результат                      │                   │
│   └─────────────────────────────────────────┘                   │
│   Сложность: O(N + M)                                           │
│                                                                  │
│   На 10,000 строках: разница в 1000x!                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**РЕШЕНИЕ:** Преобразуй correlated subquery в JOIN или используй CTE

### Модель 3: Window Functions vs GROUP BY

```
┌─────────────────────────────────────────────────────────────────┐
│           WINDOW FUNCTIONS VS GROUP BY                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   GROUP BY: "Сплющивает" данные                                 │
│   ┌────────────────────┐    ┌────────────────────┐              │
│   │ Отдел  │ Зарплата  │    │ Отдел  │ Сумма    │              │

│   │ IT     │ 5000      │    │ IT     │ 11000    │              │
│   │ IT     │ 6000      │ ──▶│ HR     │ 9000     │              │
│   │ HR     │ 4000      │    └────────────────────┘              │
│   │ HR     │ 5000      │    3 строки → 2 строки                 │
│   └────────────────────┘                                        │
│                                                                  │
│   WINDOW: Добавляет колонку, сохраняя строки                    │
│   ┌────────────────────┐    ┌──────────────────────────────┐    │
│   │ Отдел  │ Зарплата  │    │ Отдел  │ Зарплата │ Сумма   │    │
│   ├────────────────────┤    ├──────────────────────────────┤    │
│   │ IT     │ 5000      │    │ IT     │ 5000     │ 11000   │    │
│   │ IT     │ 6000      │ ──▶│ IT     │ 6000     │ 11000   │    │
│   │ HR     │ 4000      │    │ HR     │ 4000     │ 9000    │    │
│   │ HR     │ 5000      │    │ HR     │ 5000     │ 9000    │    │
│   └────────────────────┘    └──────────────────────────────┘    │
│                             4 строки → 4 строки + агрегат       │
│                                                                  │
│   Когда что использовать:                                       │
│   • GROUP BY: нужен только агрегат (отчёты, итоги)             │
│   • WINDOW: нужен агрегат + детали (ранжирование, % от общего) │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## Терминология

| Термин | Значение |
|--------|----------|
| **DDL** | Data Definition Language — CREATE, ALTER, DROP (структура) |
| **DML** | Data Manipulation Language — SELECT, INSERT, UPDATE, DELETE (данные) |

| **Window Function** | Функция над "окном" строк без группировки |
| **Aggregate Function** | SUM, COUNT, AVG, MAX, MIN — свёртка строк в одну |

## JOIN: объединение таблиц

### Типы JOIN

```
┌───────────────────────────────────────────────────────────────────┐
│                        ТИПЫ JOIN                                  │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│   INNER JOIN          LEFT JOIN           RIGHT JOIN              │
│   ┌─────┬─────┐      ┌─────┬─────┐       ┌─────┬─────┐           │
│   │  A  │█████│ B    │█████│█████│ B     │  A  │█████│█████│     │
│   │     │█████│      │█████│█████│       │     │█████│█████│     │
│   └─────┴─────┘      └─────┴─────┘       └─────┴─────┴─────┘     │
│   Только совпадения  A + совпадения      Совпадения + B          │
│                                                                   │
│   FULL OUTER JOIN    CROSS JOIN                                   │
│   ┌─────┬─────┬─────┐  A × B = все комбинации                    │
│   │█████│█████│█████│  3 строки × 4 строки = 12 строк            │
│   │█████│█████│█████│                                             │
│   └─────┴─────┴─────┘                                             │
│   Все из A и B                                                    │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### Примеры

```sql
-- Таблицы для примеров
-- users: id, name, department_id
-- departments: id, name
-- orders: id, user_id, amount, created_at

-- ✅ INNER JOIN — только пользователи с отделами
SELECT u.name, d.name as department
FROM users u
INNER JOIN departments d ON u.department_id = d.id;

-- ✅ LEFT JOIN — все пользователи, даже без отдела
SELECT u.name, d.name as department
FROM users u
LEFT JOIN departments d ON u.department_id = d.id;
-- Если department_id = NULL → department = NULL

-- ✅ Найти пользователей БЕЗ отдела
SELECT u.name
FROM users u
LEFT JOIN departments d ON u.department_id = d.id
WHERE d.id IS NULL;

-- ✅ Multiple JOINs
SELECT u.name, d.name as department, COUNT(o.id) as order_count
FROM users u
LEFT JOIN departments d ON u.department_id = d.id
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name, d.name;
```

### Anti-join pattern

```sql
-- ❌ Медленно: подзапрос с NOT IN
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders);

-- ✅ Быстро: LEFT JOIN + IS NULL
SELECT u.*
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.id IS NULL;

-- ✅ Альтернатива: NOT EXISTS
SELECT * FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);
```

## Подзапросы (Subqueries)

### Типы подзапросов

```sql
-- 1. Scalar subquery (возвращает одно значение)
SELECT name,
       (SELECT AVG(salary) FROM employees) as avg_salary
FROM employees;

-- 2. Row subquery (возвращает одну строку)
SELECT * FROM employees
WHERE (department_id, salary) = (
    SELECT department_id, MAX(salary)
    FROM employees
    GROUP BY department_id
    LIMIT 1
);

-- 3. Table subquery (возвращает таблицу)
SELECT * FROM (
    SELECT name, salary,
           RANK() OVER (ORDER BY salary DESC) as rank
    FROM employees
) ranked
WHERE rank <= 5;

-- 4. EXISTS subquery (проверка существования)
SELECT * FROM departments d
WHERE EXISTS (
    SELECT 1 FROM employees e
    WHERE e.department_id = d.id
);
```

### Correlated vs Non-correlated

```sql
-- Non-correlated: выполняется один раз
SELECT * FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Correlated: выполняется для каждой строки внешнего запроса
-- ⚠️ Может быть медленным!
SELECT * FROM employees e1
WHERE salary > (
    SELECT AVG(salary) FROM employees e2
    WHERE e2.department_id = e1.department_id  -- ссылка на внешний запрос
);
```

## CTE (Common Table Expressions)

### Базовый синтаксис

```sql
-- ✅ CTE делает запрос читаемым
WITH active_users AS (
    SELECT * FROM users
    WHERE status = 'active'
    AND last_login > CURRENT_DATE - INTERVAL '30 days'
),
user_orders AS (
    SELECT user_id, COUNT(*) as order_count, SUM(amount) as total
    FROM orders
    WHERE created_at > CURRENT_DATE - INTERVAL '30 days'
    GROUP BY user_id
)
SELECT
    u.name,
    u.email,
    COALESCE(o.order_count, 0) as orders,
    COALESCE(o.total, 0) as revenue
FROM active_users u
LEFT JOIN user_orders o ON o.user_id = u.id
ORDER BY revenue DESC;
```

### Recursive CTE

```sql
-- Иерархическая структура (дерево категорий, org chart)
WITH RECURSIVE category_tree AS (
    -- Base case: корневые категории
    SELECT id, name, parent_id, 1 as level,
           name::text as path
    FROM categories
    WHERE parent_id IS NULL

    UNION ALL

    -- Recursive case: дочерние категории
    SELECT c.id, c.name, c.parent_id, ct.level + 1,
           ct.path || ' > ' || c.name
    FROM categories c
    INNER JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree ORDER BY path;

-- Результат:
-- | id | name        | level | path                           |
-- |----|-------------|-------|--------------------------------|
-- | 1  | Electronics | 1     | Electronics                    |
-- | 2  | Phones      | 2     | Electronics > Phones           |
-- | 3  | iPhone      | 3     | Electronics > Phones > iPhone  |
```

## Оконные функции (Window Functions)

### Концепция окна

```
┌──────────────────────────────────────────────────────────────┐
│                    ОКОННАЯ ФУНКЦИЯ                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   Обычная агрегация:        Оконная функция:                │
│   ┌────────────────┐        ┌────────────────────────────┐  │
│   │ name  │ salary │        │ name  │ salary │ avg_dept  │  │
│   ├────────────────┤        ├────────────────────────────┤  │
│   │ Alice │  5000  │        │ Alice │  5000  │   5500    │  │
│   │ Bob   │  6000  │  ──▶   │ Bob   │  6000  │   5500    │  │
│   ├────────────────┤        │ Carol │  7000  │   6500    │  │
│   │ AVG   │  5500  │        │ Dave  │  6000  │   6500    │  │
│   └────────────────┘        └────────────────────────────┘  │
│   GROUP BY схлопывает       OVER() сохраняет строки         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Синтаксис

```sql
function_name() OVER (
    [PARTITION BY column1, column2, ...]  -- группировка окна
    [ORDER BY column3 [ASC|DESC], ...]    -- сортировка в окне
    [ROWS/RANGE BETWEEN ... AND ...]       -- границы окна
)
```

### Основные функции

```sql
-- 1. ROW_NUMBER, RANK, DENSE_RANK
SELECT
    name,
    department,
    salary,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as row_num,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) as rank,
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dense_rank
FROM employees;

-- Разница:
-- salary: 100, 100, 90, 80
-- ROW_NUMBER: 1, 2, 3, 4      (всегда уникальный)
-- RANK:       1, 1, 3, 4      (пропускает после дубликатов)
-- DENSE_RANK: 1, 1, 2, 3      (не пропускает)

-- 2. LAG / LEAD — доступ к соседним строкам
SELECT
    date,
    revenue,
    LAG(revenue, 1) OVER (ORDER BY date) as prev_day,
    LEAD(revenue, 1) OVER (ORDER BY date) as next_day,
    revenue - LAG(revenue, 1) OVER (ORDER BY date) as daily_change
FROM daily_sales;

-- 3. FIRST_VALUE / LAST_VALUE / NTH_VALUE
SELECT
    name,
    salary,
    FIRST_VALUE(name) OVER (ORDER BY salary DESC) as top_earner,
    LAST_VALUE(name) OVER (
        ORDER BY salary DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) as lowest_earner
FROM employees;

-- 4. Кумулятивные агрегаты
SELECT
    date,
    amount,
    SUM(amount) OVER (ORDER BY date) as running_total,
    AVG(amount) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as moving_avg_7d
FROM transactions;
```

### Практические примеры

```sql
-- Топ-3 продукта в каждой категории
WITH ranked AS (
    SELECT
        category_id,
        name,
        sales,
        ROW_NUMBER() OVER (PARTITION BY category_id ORDER BY sales DESC) as rn
    FROM products
)
SELECT * FROM ranked WHERE rn <= 3;

-- Процент от общего
SELECT
    department,
    name,
    salary,
    ROUND(100.0 * salary / SUM(salary) OVER (), 2) as pct_of_total,
    ROUND(100.0 * salary / SUM(salary) OVER (PARTITION BY department), 2) as pct_of_dept
FROM employees;

-- Year-over-year growth
SELECT
    year,
    month,
    revenue,
    LAG(revenue, 12) OVER (ORDER BY year, month) as same_month_last_year,
    ROUND(100.0 * (revenue - LAG(revenue, 12) OVER (ORDER BY year, month))
          / NULLIF(LAG(revenue, 12) OVER (ORDER BY year, month), 0), 2) as yoy_growth
FROM monthly_revenue;
```

## Агрегация: GROUP BY + HAVING

### Основы

```sql
-- GROUP BY группирует строки
SELECT department_id, COUNT(*) as emp_count, AVG(salary) as avg_salary
FROM employees
GROUP BY department_id;

-- HAVING фильтрует после группировки
SELECT department_id, COUNT(*) as emp_count
FROM employees
GROUP BY department_id
HAVING COUNT(*) > 5;  -- только отделы с >5 сотрудниками

-- ⚠️ WHERE vs HAVING
-- WHERE: фильтрует СТРОКИ до группировки
-- HAVING: фильтрует ГРУППЫ после группировки

SELECT department_id, AVG(salary) as avg_salary
FROM employees
WHERE status = 'active'      -- сначала только активных
GROUP BY department_id
HAVING AVG(salary) > 50000;  -- потом только богатые отделы
```

### GROUPING SETS, ROLLUP, CUBE

```sql
-- GROUPING SETS — несколько группировок в одном запросе
SELECT
    COALESCE(department, 'ALL') as department,
    COALESCE(year::text, 'ALL') as year,
    SUM(salary) as total
FROM employees
GROUP BY GROUPING SETS (
    (department, year),  -- по отделу и году
    (department),         -- только по отделу
    (year),               -- только по году
    ()                    -- общий итог
);

-- ROLLUP — иерархическая агрегация
SELECT department, team, SUM(salary)
FROM employees
GROUP BY ROLLUP (department, team);
-- Даёт: (dept, team), (dept), ()

-- CUBE — все комбинации
SELECT department, team, SUM(salary)
FROM employees
GROUP BY CUBE (department, team);
-- Даёт: (dept, team), (dept), (team), ()
```

## Полезные конструкции

### COALESCE и NULLIF

```sql
-- COALESCE — первое не-NULL значение
SELECT COALESCE(nickname, name, 'Anonymous') as display_name
FROM users;

-- NULLIF — возвращает NULL если равны (избегаем деления на 0)
SELECT total / NULLIF(count, 0) as average
FROM stats;
```

### CASE WHEN

```sql
SELECT
    name,
    salary,
    CASE
        WHEN salary >= 100000 THEN 'Senior'
        WHEN salary >= 50000 THEN 'Middle'
        ELSE 'Junior'
    END as level
FROM employees;
```

### DISTINCT ON (PostgreSQL)

```sql
-- Первая запись в каждой группе
SELECT DISTINCT ON (user_id)
    user_id,
    order_id,
    created_at
FROM orders
ORDER BY user_id, created_at DESC;
-- Последний заказ каждого пользователя
```

### LATERAL JOIN

```sql
-- Подзапрос с доступом к внешним колонкам
SELECT u.name, recent_orders.*
FROM users u
CROSS JOIN LATERAL (
    SELECT order_id, amount, created_at
    FROM orders o
    WHERE o.user_id = u.id
    ORDER BY created_at DESC
    LIMIT 3
) recent_orders;
```

## Частые ошибки

### 1. NULL в сравнениях

```sql
-- ❌ Неправильно: NULL = NULL → NULL (не TRUE!)
SELECT * FROM users WHERE status = NULL;

-- ✅ Правильно
SELECT * FROM users WHERE status IS NULL;

-- ⚠️ NOT IN с NULL
SELECT * FROM users WHERE id NOT IN (1, 2, NULL);
-- Вернёт 0 строк! NULL "отравляет" результат

-- ✅ Используй NOT EXISTS или исключи NULL
SELECT * FROM users WHERE id NOT IN (
    SELECT user_id FROM banned WHERE user_id IS NOT NULL
);
```

### 2. Агрегация без GROUP BY

```sql
-- ❌ Ошибка: name не в GROUP BY и не в агрегатной функции
SELECT department_id, name, COUNT(*)
FROM employees
GROUP BY department_id;

-- ✅ Правильно: добавь в GROUP BY или используй агрегат
SELECT department_id, ARRAY_AGG(name), COUNT(*)
FROM employees
GROUP BY department_id;
```

### 3. ORDER BY с DISTINCT или UNION

```sql
-- ❌ DISTINCT + ORDER BY по не-выбранной колонке
SELECT DISTINCT category FROM products ORDER BY created_at;

-- ✅ Добавь в SELECT или используй подзапрос
SELECT DISTINCT ON (category) category, created_at
FROM products
ORDER BY category, created_at DESC;
```

## Проверь себя

> [!question]- Почему порядок выполнения SQL-запроса отличается от порядка его записи?
> SQL — декларативный язык: мы описываем ЧТО хотим получить, а не КАК. Логический порядок выполнения: FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY -> LIMIT. SELECT пишется первым для читаемости, но выполняется почти последним. Это объясняет, почему нельзя использовать алиас из SELECT в WHERE — WHERE выполняется раньше.

> [!question]- Когда использовать LEFT JOIN вместо INNER JOIN и чем отличается результат?
> INNER JOIN возвращает только совпадающие строки из обеих таблиц. LEFT JOIN сохраняет ВСЕ строки из левой таблицы, заполняя NULL для отсутствующих данных правой. Используй LEFT JOIN когда нужны все записи основной таблицы (например, все пользователи, даже без заказов). INNER JOIN теряет записи без совпадений.

> [!question]- В чём принципиальная разница между оконной функцией и GROUP BY?
> GROUP BY схлопывает строки в группы — на выходе одна строка на группу. Оконная функция (OVER) вычисляет агрегат, но сохраняет все исходные строки. Например, SUM с GROUP BY покажет одну строку с итогом на отдел, а SUM с OVER покажет каждого сотрудника с общей суммой его отдела рядом.

> [!question]- Почему SELECT * считается антипаттерном в production-коде?
> SELECT * читает все колонки, включая ненужные, что увеличивает I/O и сетевой трафик. При добавлении новой колонки в таблицу запрос неожиданно начинает возвращать больше данных. Индексы covering scan не работают. В production всегда перечисляй только нужные колонки явно.

## Ключевые карточки

Каков логический порядок выполнения SQL-запроса?
?
FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY -> LIMIT. Написание начинается с SELECT, но выполнение — с FROM.

Чем отличаются WHERE и HAVING?
?
WHERE фильтрует строки ДО группировки, HAVING — ПОСЛЕ GROUP BY. WHERE не может использовать агрегатные функции, HAVING — может (например, HAVING COUNT(*) > 5).

Что такое CTE (Common Table Expression)?
?
Именованный временный результат запроса через WITH ... AS (...). Работает как переменная: вычисляется один раз и используется в основном запросе. Улучшает читаемость вместо вложенных подзапросов.

Какие типы JOIN существуют?
?
INNER JOIN (только совпадения), LEFT/RIGHT JOIN (все из одной таблицы + совпадения), FULL OUTER JOIN (все из обеих), CROSS JOIN (декартово произведение). Самые частые: INNER и LEFT.

Что делает оконная функция ROW_NUMBER()?
?
Присваивает порядковый номер каждой строке внутри партиции (PARTITION BY) в указанном порядке (ORDER BY). Полезна для пагинации, дедупликации (взять первую/последнюю запись из группы).

Что такое SARGable запросы?
?
SARGable (Search ARGument ABLE) — запросы, позволяющие использовать индекс. Функции вокруг колонки (WHERE YEAR(date) = 2024) делают запрос non-SARGable. Правильно: WHERE date >= '2024-01-01' AND date < '2025-01-01'.

Чем отличается UNION от UNION ALL?
?
UNION удаляет дубликаты (выполняет сортировку и сравнение). UNION ALL возвращает все строки, включая дубликаты. UNION ALL быстрее, используй его если дубликаты не проблема.
