# Rate Limiting: защита API от перегрузки

Представь, что ты владеешь маленькой кофейню. Утром приходит 10 клиентов — ты справляешься. Но однажды автобус с туристами высаживает 200 человек, и все хотят кофе СЕЙЧАС. Бариста не успевает, очередь растёт, клиенты злятся, кофемашина перегревается и ломается. **Rate limiting — это турникет на входе**, который говорит: "Извините, мы можем обслужить только 20 человек в минуту. Остальные — подождите или приходите позже."

---

## Теоретические основы

### Формальное определение

> **Rate Limiting** — механизм контроля скорости поступления запросов в систему, ограничивающий число операций, которые сущность (пользователь, IP, API-ключ) может выполнить за единицу времени.

Rate limiting — частный случай более общей концепции **flow control** (управление потоком), которая изучается в теории массового обслуживания (queueing theory) и теории управления.

### Теоретические корни

| Теория | Вклад в Rate Limiting | Автор/Год |
|--------|----------------------|-----------|
| **Queueing Theory** | Модели M/M/1, M/M/c — анализ очередей, throughput, latency | Erlang, 1909; Kendall, 1953 |
| **Token Bucket** | Формальная модель контроля скорости в сетях | Turner, 1986 (Bell Labs) |
| **Leaky Bucket** | Регулирование трафика с постоянной скоростью выхода | Turner, 1986 |
| **Little's Law** | `L = λW` (среднее число в системе = rate × среднее время) | Little, 1961 |
| **Congestion Control** | Additive Increase / Multiplicative Decrease (AIMD) | Chiu & Jain, 1989 |

### Формальная модель Token Bucket

**Token Bucket** — наиболее распространённый алгоритм rate limiting. Формально:

> **Определение:** Token Bucket — тройка `(r, b, tokens(t))`, где:
> - `r` — rate (скорость пополнения токенов, tokens/sec)
> - `b` — burst size (ёмкость корзины)
> - `tokens(t) = min(b, tokens(t₀) + r·(t - t₀))` — количество токенов в момент `t`

**Свойства:**
- Долгосрочная средняя скорость ≤ `r` запросов/сек
- Допускает **burst** до `b` запросов мгновенно
- Сглаживает трафик: `lim(t→∞) throughput(t) = r`

### Leaky Bucket vs Token Bucket

| Свойство | Token Bucket | Leaky Bucket |
|----------|-------------|--------------|
| **Модель** | Ведро с токенами, пополняемое с rate `r` | Ведро с дырой, вытекающее с rate `r` |
| **Burst** | Разрешён (до `b` мгновенно) | Запрещён (выход строго `r` req/sec) |
| **Аналогия** | Билетная касса: копишь билеты, тратишь пачкой | Капельница: постоянная скорость |
| **Формула** | `tokens = min(b, tokens + r·Δt)` | `water = max(0, water - r·Δt) + 1` |
| **Применение** | API rate limiting (Stripe, GitHub) | Traffic shaping в сетях |

### Sliding Window — комбинаторная модель

**Sliding Window Log** хранит timestamp каждого запроса и подсчитывает попадающие в окно `[t - W, t]`:

> `count(t) = |{tᵢ : tᵢ ∈ [t - W, t]}|`

**Sliding Window Counter** (Redis-оптимизированный) аппроксимирует точный подсчёт через взвешенную сумму двух смежных фиксированных окон:

> `count ≈ count_prev × (1 - elapsed/W) + count_curr`

Cloudflare показала, что эта аппроксимация даёт ошибку ≤ `max(count_prev, count_curr) × elapsed/W` — на практике <0.003% при нормальном трафике.

### Distributed Rate Limiting: теоретические ограничения

В распределённой системе **точный** глобальный rate limit невозможен без синхронизации (следствие CAP-теоремы):

- **Strict consistency** (Redis + Lua): точный лимит, но latency синхронизации
- **Eventual consistency** (local counters + sync): минимальная latency, но возможен кратковременный overshoot
- **Практический компромисс**: допустить ~5-10% overshoot ради latency (подход Stripe, Lyft)

## TL;DR (если совсем нет времени)

- **Rate limiting** — ограничение количества запросов за период времени
- **Зачем:** защита от DDoS, справедливое распределение ресурсов, контроль затрат
- **Главные алгоритмы:** Token Bucket (позволяет всплески), Leaky Bucket (равномерный поток)
- **В distributed системах:** используй Redis + Lua скрипты для атомарности
- **Ответ клиенту:** HTTP 429 + заголовок `Retry-After`

## Терминология (объяснение каждого термина)

| Термин | Что это простыми словами | Аналогия из жизни |
|--------|--------------------------|-------------------|
| **Rate Limit** | Максимум запросов за период (например, 100 запросов в минуту) | Лимит на снятие наличных: не больше 50 000 ₽ в день |
| **Throttling** | Замедление запросов вместо полного отклонения | Вместо "Нет мест!" — "Подождите 5 минут" |
| **Quota** | Лимит за длительный период (например, 10 000 запросов в день) | Месячный лимит интернета в тарифе |
| **Burst** | Кратковременный всплеск запросов выше обычной нормы | Толпа покупателей в Чёрную пятницу |
| **Token** | "Жетон" на право сделать один запрос | Талончик в очереди в банке |
| **Bucket** | "Ведро", где хранятся токены | Копилка с монетами |
| **Window** | Временное окно для подсчёта запросов | Период с 10:00 до 10:01 |
| **429** | HTTP код "Too Many Requests" (слишком много запросов) | Табличка "Извините, мы закрыты" |
| **Retry-After** | Заголовок, говорящий "попробуй через X секунд" | "Приходите через 10 минут" |

## Зачем нужен Rate Limiting? (проблема без решения)

### Реальная история: GitHub упал из-за одного клиента

В 2018 году GitHub столкнулся с проблемой: один автоматизированный клиент отправлял миллионы запросов в минуту, пытаясь синхронизировать репозитории. Серверы не справились — GitHub был недоступен для миллионов разработчиков по всему миру.

**Без rate limiting:**

```
Ситуация:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   Клиент A: 1 000 000 запросов/мин    ──────┐                       │
│   Клиент B: 10 запросов/мин           ──────┼───▶  СЕРВЕР  ──▶ 💥   │
│   Клиент C: 50 запросов/мин           ──────┘      (упал)          │
│                                                                     │
│   Результат: Клиент A "съел" все ресурсы                           │
│              Клиенты B и C не получили ничего                      │
│              Сервер перегружен и недоступен                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**С rate limiting:**

```
Ситуация:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   Клиент A: 1 000 000 запросов/мин ──┐                              │
│              │                        │                             │
│              ▼                        │                             │
│         [Rate Limiter]                │                             │
│         "Стоп! Лимит 100/мин"         │                             │
│              │                        │                             │
│              ▼                        ▼                             │
│         100 запросов/мин ────────────────▶  СЕРВЕР  ──▶ ✅          │
│   Клиент B: 10 запросов/мин   ───────────▶  (работает)             │
│   Клиент C: 50 запросов/мин   ───────────▶                         │
│                                                                     │
│   Результат: Все клиенты получают справедливую долю                │
│              Сервер работает стабильно                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пять причин использовать Rate Limiting

| Причина | Проблема без rate limiting | Решение |
|---------|---------------------------|---------|
| **1. Защита от DDoS** | Атакующий отправляет миллионы запросов, сервер падает | Ограничить каждый IP до N запросов/сек |
| **2. Справедливость** | Один клиент потребляет 99% ресурсов | Равные лимиты для всех |
| **3. Контроль затрат** | Дорогие операции (OpenAI API, база данных) выполняются бесконтрольно | Лимиты на дорогие endpoints |
| **4. Защита от брутфорса** | Взломщик пробует 10 000 паролей в секунду | 5 попыток логина в минуту |
| **5. Стабильность** | Внезапный наплыв трафика ломает систему | Отклонять излишек, сохранять стабильность |

### Когда НЕ использовать Rate Limiting

| Ситуация | Почему не нужен | Что делать вместо |
|----------|-----------------|-------------------|
| Внутренние сервисы | Доверенный трафик, известная нагрузка | Circuit Breaker, масштабирование |
| Health checks | Критически важны для мониторинга | Исключить из rate limiting |
| Webhooks от партнёров | Нельзя просить партнёра "подождать" | Очередь + обработка в фоне |
| Статические файлы | CDN справляется лучше | Использовать CDN |

## Алгоритмы Rate Limiting (от простого к сложному)

Существует 5 основных алгоритмов. Каждый решает проблему по-своему. Разберём каждый.

### Алгоритм 1: Fixed Window Counter (Фиксированное окно)

**Аналогия:** Представь ночной клуб с вместимостью 100 человек. Каждый час охранник обнуляет счётчик и впускает следующие 100.

**Как работает:**

```
FIXED WINDOW COUNTER

Лимит: 100 запросов в минуту

Время:     00:00                 01:00                 02:00
           ◄─── Окно 1 ──────────►◄─── Окно 2 ──────────►

           Счётчик: 0 → 1 → 2 → ... → 100 → СТОП!
                                            │
           В 01:00 счётчик обнуляется: 0 → 1 → 2 → ...

Пример:
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  Окно 00:00-01:00          │  Окно 01:00-02:00                   │
│  ┌──────────────────────┐  │  ┌──────────────────────┐           │
│  │ Запросы: ████████░░ │  │  │ Запросы: █████░░░░░ │           │
│  │ Счётчик: 80/100     │  │  │ Счётчик: 50/100     │           │
│  │ Статус: ОК          │  │  │ Статус: ОК          │           │
│  └──────────────────────┘  │  └──────────────────────┘           │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**Проблема "пограничного всплеска" (Boundary Burst):**

```
ПРОБЛЕМА FIXED WINDOW — почему этот алгоритм опасен

Лимит: 100 запросов в минуту

Что может произойти:
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  Время:  00:00        00:59  │  01:00  01:01        02:00        │
│          ◄─── Окно 1 ────────┼────────── Окно 2 ──────►          │
│                              │                                    │
│          0 запросов... 99    │  99 запросов... 0                 │
│                        ↑     │  ↑                                 │
│                        │     │  │                                 │
│                За 2 секунды: 99 + 99 = 198 запросов!              │
│                Это почти 2x от лимита!                            │
│                                                                   │
│  Проблема: Клиент отправил 99 запросов в 00:59                   │
│            и ещё 99 запросов в 01:01                             │
│            Оба раза формально в лимите, но сервер перегружен     │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**Реализация (простая, но с проблемой):**

```python
# Объяснение: Fixed Window — самый простой алгоритм.
# Мы храним счётчик для каждого временного окна.
# Когда окно меняется — счётчик обнуляется.

import time
from typing import Tuple

class FixedWindowLimiter:
    """
    Fixed Window Rate Limiter.

    Принцип работы:
    1. Разбиваем время на окна (например, по 60 секунд)
    2. Считаем запросы в текущем окне
    3. Если превысили лимит — отклоняем
    4. Когда окно меняется — счётчик = 0
    """

    def __init__(self, limit: int, window_seconds: int):
        # limit = максимум запросов (например, 100)
        self.limit = limit

        # window_seconds = размер окна в секундах (например, 60)
        self.window_seconds = window_seconds

        # Словарь: ключ → (номер_окна, счётчик)
        # Ключ — это идентификатор клиента (IP, user_id и т.д.)
        self.counters = {}

    def _get_window(self) -> int:
        """
        Вычисляем номер текущего окна.

        Пример:
        - Сейчас 125 секунд с начала эпохи
        - Размер окна = 60 секунд
        - Номер окна = 125 // 60 = 2
        """
        return int(time.time() // self.window_seconds)

    def is_allowed(self, key: str) -> Tuple[bool, int]:
        """
        Проверяем, можно ли пропустить запрос.

        Возвращает:
        - (True, remaining) — запрос разрешён, осталось remaining попыток
        - (False, 0) — лимит превышен
        """
        current_window = self._get_window()

        # Получаем данные для этого ключа
        if key not in self.counters:
            # Первый запрос — создаём запись
            self.counters[key] = (current_window, 0)

        stored_window, count = self.counters[key]

        # Если окно изменилось — обнуляем счётчик
        if stored_window != current_window:
            count = 0

        # Проверяем лимит
        if count >= self.limit:
            return False, 0

        # Увеличиваем счётчик и сохраняем
        count += 1
        self.counters[key] = (current_window, count)

        return True, self.limit - count


# Пример использования:
limiter = FixedWindowLimiter(limit=100, window_seconds=60)

# Проверяем запрос от пользователя с IP "192.168.1.1"
allowed, remaining = limiter.is_allowed("192.168.1.1")
if allowed:
    print(f"Запрос разрешён. Осталось попыток: {remaining}")
else:
    print("Лимит превышен! Попробуйте через минуту.")
```

**Плюсы и минусы:**

| ✅ Плюсы | ❌ Минусы |
|---------|----------|
| Очень просто реализовать | Проблема пограничного всплеска (2x лимит!) |
| Минимум памяти (1 число на клиента) | Неравномерное распределение |
| Быстро работает | Не подходит для критичных API |

### Алгоритм 2: Sliding Window Log (Скользящее окно с логом)

**Аналогия:** Вместо подсчёта "сколько человек вошло с 10:00 до 11:00", охранник записывает время входа КАЖДОГО посетителя и считает: "Сколько человек вошло за ПОСЛЕДНИЕ 60 минут от текущего момента?"

**Как работает:**

```
SLIDING WINDOW LOG

Лимит: 5 запросов в 60 секунд

Храним timestamp (время) каждого запроса:

Лог запросов: [00:15, 00:23, 00:45, 00:58, 01:02, 01:15]

Сейчас 01:20. Окно = последние 60 секунд = [00:20, 01:20]

Запросы в окне:
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  Время:  00:00   00:15  00:23   00:45  00:58  01:02  01:15  01:20 │
│                   │      │       │      │      │      │      │   │
│                   ▼      ▼       ▼      ▼      ▼      ▼      │   │
│                   ●      ●       ●      ●      ●      ●      │   │
│                   │      │       │      │      │      │      │   │
│          ◄────────┼──────┼───────┼──────┼──────┼──────┼──────┤   │
│          Вне окна │             Внутри окна [00:20 - 01:20]  │   │
│                   │                                          │   │
│  00:15 — УДАЛИТЬ (вне окна)                                  │   │
│  00:23, 00:45, 00:58, 01:02, 01:15 — СЧИТАТЬ                 │   │
│                                                               │   │
│  Итого в окне: 5 запросов. Новый запрос = ОТКЛОНИТЬ          │   │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**Почему это лучше Fixed Window?**

```
РЕШЕНИЕ ПРОБЛЕМЫ ПОГРАНИЧНОГО ВСПЛЕСКА

Fixed Window:
┌────────────────┬────────────────┐
│   Окно 1       │   Окно 2       │
│   99 запросов  │   99 запросов  │
│                │                │
│    ──────99────┼99───────       │  198 запросов за 2 секунды!
└────────────────┴────────────────┘

Sliding Window Log:
┌──────────────────────────────────┐
│                                  │
│     ◄──── 60 секунд ────►        │
│                                  │
│  Любой момент времени:           │
│  Считаем запросы за ПОСЛЕДНИЕ    │
│  60 секунд, не за "с 00:00"      │
│                                  │
│  99 в 00:59 + 1 в 01:00?         │
│  Считаем: за последние 60 сек    │
│  от 01:00 = [00:00, 01:00]       │
│  Уже есть 99 → ОТКЛОНЯЕМ         │
│                                  │
└──────────────────────────────────┘
```

**Реализация:**

```python
# Объяснение: Sliding Window Log точнее Fixed Window,
# но требует хранить timestamp каждого запроса.
# Используй для API с небольшим трафиком, где важна точность.

import time
from collections import defaultdict
from typing import Tuple, List

class SlidingWindowLogLimiter:
    """
    Sliding Window Log Rate Limiter.

    Принцип работы:
    1. Храним timestamp каждого запроса
    2. Удаляем старые записи (за пределами окна)
    3. Считаем оставшиеся записи
    4. Если меньше лимита — пропускаем
    """

    def __init__(self, limit: int, window_seconds: int):
        self.limit = limit
        self.window_seconds = window_seconds

        # Словарь: ключ → список timestamp'ов
        # Например: {"user123": [1704067200.5, 1704067201.3, ...]}
        self.logs: dict[str, List[float]] = defaultdict(list)

    def is_allowed(self, key: str) -> Tuple[bool, int, float]:
        """
        Проверяем, можно ли пропустить запрос.

        Возвращает:
        - (True, remaining, 0) — разрешён
        - (False, 0, retry_after) — отклонён, ждать retry_after секунд
        """
        now = time.time()
        window_start = now - self.window_seconds

        # Шаг 1: Удаляем старые записи
        # Оставляем только те, что в текущем окне
        self.logs[key] = [
            ts for ts in self.logs[key]
            if ts > window_start
        ]

        # Шаг 2: Проверяем лимит
        current_count = len(self.logs[key])

        if current_count >= self.limit:
            # Лимит превышен
            # Вычисляем, когда освободится место
            oldest = self.logs[key][0]  # Самый старый запрос
            retry_after = oldest + self.window_seconds - now
            return False, 0, max(0, retry_after)

        # Шаг 3: Добавляем текущий запрос
        self.logs[key].append(now)

        remaining = self.limit - current_count - 1
        return True, remaining, 0


# Пример использования:
limiter = SlidingWindowLogLimiter(limit=5, window_seconds=60)

for i in range(7):
    allowed, remaining, retry_after = limiter.is_allowed("user123")
    if allowed:
        print(f"Запрос {i+1}: ОК. Осталось: {remaining}")
    else:
        print(f"Запрос {i+1}: ОТКЛОНЁН. Повторите через {retry_after:.1f} сек")
    time.sleep(0.1)

# Вывод:
# Запрос 1: ОК. Осталось: 4
# Запрос 2: ОК. Осталось: 3
# Запрос 3: ОК. Осталось: 2
# Запрос 4: ОК. Осталось: 1
# Запрос 5: ОК. Осталось: 0
# Запрос 6: ОТКЛОНЁН. Повторите через 59.5 сек
# Запрос 7: ОТКЛОНЁН. Повторите через 59.4 сек
```

**Плюсы и минусы:**

| ✅ Плюсы | ❌ Минусы |
|---------|----------|
| Максимально точный | Память растёт с трафиком |
| Нет проблемы пограничного всплеска | При 1M запросов = 1M timestamp'ов |
| Точный Retry-After | Медленнее при высокой нагрузке |

### Алгоритм 3: Sliding Window Counter (Гибридный)

**Аналогия:** Компромисс между точностью и памятью. Вместо хранения каждого timestamp, мы храним только 2 счётчика (текущее и предыдущее окно) и вычисляем приблизительное значение.

**Формула (разбор по шагам):**

```
SLIDING WINDOW COUNTER — формула

Входные данные:
- previous_count = запросы в предыдущем окне (например, 80)
- current_count = запросы в текущем окне (например, 40)
- time_in_window = сколько % текущего окна прошло (например, 70%)

Формула:
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│   estimated_count = previous_count × (1 - time_in_window)        │
│                   + current_count                                 │
│                                                                   │
│   Пример:                                                         │
│   estimated = 80 × (1 - 0.70) + 40                               │
│             = 80 × 0.30 + 40                                      │
│             = 24 + 40                                             │
│             = 64 запроса                                          │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘

Визуализация:
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│   Предыдущее окно          Текущее окно                          │
│   ┌───────────────────┐    ┌───────────────────┐                 │
│   │ 80 запросов       │    │ 40 запросов       │                 │
│   │ ████████████████ │    │ █████████░░░░░░░░ │                 │
│   └───────────────────┘    └───────────────────┘                 │
│                            ▲                 ▲                    │
│                            │     70%        │                    │
│                            │   прошло       │                    │
│                            │                │                    │
│   ◄────────────────────────┼────────────────┤                    │
│   │  30% предыдущего       │ 70% текущего   │                    │
│   │  ещё "в окне"          │ прошло         │                    │
│   │                        │                │                    │
│   │  80 × 0.30 = 24        │ 40             │                    │
│   │                        │                │                    │
│   │  Итого: 24 + 40 = 64 запроса                                 │
│   │                                                               │
│   Скользящее окно = [30% предыдущего] + [текущее]                │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**Реализация:**

```python
# Объяснение: Sliding Window Counter — лучший компромисс.
# Точность близка к Log, память как у Fixed Window.
# Используется в большинстве production-систем.

import time
from typing import Tuple

class SlidingWindowCounterLimiter:
    """
    Sliding Window Counter Rate Limiter.

    Принцип работы:
    1. Храним счётчики для текущего и предыдущего окна
    2. Вычисляем взвешенную сумму
    3. Сравниваем с лимитом
    """

    def __init__(self, limit: int, window_seconds: int):
        self.limit = limit
        self.window_seconds = window_seconds

        # Для каждого ключа храним:
        # {key: (prev_window, prev_count, curr_window, curr_count)}
        self.data: dict[str, tuple] = {}

    def _get_window(self) -> int:
        """Номер текущего окна."""
        return int(time.time() // self.window_seconds)

    def _get_window_position(self) -> float:
        """
        Какая часть текущего окна прошла (от 0.0 до 1.0).

        Пример:
        - Окно = 60 секунд
        - Сейчас 00:45 (45 секунд в окне)
        - Позиция = 45 / 60 = 0.75 (75% окна прошло)
        """
        current_time = time.time()
        window_start = (current_time // self.window_seconds) * self.window_seconds
        elapsed = current_time - window_start
        return elapsed / self.window_seconds

    def is_allowed(self, key: str) -> Tuple[bool, int]:
        """
        Проверяем, можно ли пропустить запрос.
        """
        current_window = self._get_window()
        window_position = self._get_window_position()

        # Получаем или инициализируем данные
        if key not in self.data:
            self.data[key] = (current_window - 1, 0, current_window, 0)

        prev_window, prev_count, curr_window, curr_count = self.data[key]

        # Сдвигаем окна если нужно
        if curr_window != current_window:
            if curr_window == current_window - 1:
                # Предыдущее окно стало current
                prev_window = curr_window
                prev_count = curr_count
            else:
                # Прошло больше одного окна — обнуляем предыдущее
                prev_window = current_window - 1
                prev_count = 0
            curr_window = current_window
            curr_count = 0

        # Вычисляем взвешенную сумму
        # (1 - window_position) — сколько предыдущего окна ещё "в зоне"
        prev_weight = 1 - window_position
        estimated = (prev_count * prev_weight) + curr_count

        # Проверяем лимит
        if estimated >= self.limit:
            return False, 0

        # Увеличиваем счётчик текущего окна
        curr_count += 1
        self.data[key] = (prev_window, prev_count, curr_window, curr_count)

        remaining = int(self.limit - estimated - 1)
        return True, max(0, remaining)


# Пример использования:
limiter = SlidingWindowCounterLimiter(limit=100, window_seconds=60)
allowed, remaining = limiter.is_allowed("user123")
print(f"Разрешено: {allowed}, осталось: {remaining}")
```

**Сравнение трёх оконных алгоритмов:**

| Алгоритм | Память | Точность | Скорость | Рекомендация |
|----------|--------|----------|----------|--------------|
| Fixed Window | O(1) | Низкая | Очень быстро | Только для некритичных API |
| Sliding Log | O(n) | Идеальная | Медленно | Малый трафик, нужна точность |
| Sliding Counter | O(1) | Высокая | Быстро | **Лучший выбор для большинства** |

### Алгоритм 4: Token Bucket (Ведро с токенами)

**Аналогия для 5-летнего:**

Представь, что у тебя есть копилка с монетками. Каждую секунду мама кладёт в неё 1 монетку (но не больше 10 — копилка маленькая). Когда хочешь купить конфету, нужна 1 монетка. Если монетки есть — берёшь и покупаешь. Если копилка пустая — жди, пока мама положит ещё.

**Почему Token Bucket лучше оконных алгоритмов для многих случаев:**

```
ПРЕИМУЩЕСТВО TOKEN BUCKET: разрешает всплески

Ситуация: лимит 10 запросов в секунду

Оконный алгоритм:
┌─────────────────────────────────────────────────────────────────┐
│  Секунда 1: 10 запросов ✅                                       │
│  Секунда 2: 0 запросов (клиент спал)                            │
│  Секунда 3: 15 запросов → только 10 ✅, 5 ❌                     │
│                                                                  │
│  Проблема: клиент "потерял" 10 запросов за секунду 2            │
└─────────────────────────────────────────────────────────────────┘

Token Bucket (rate=10/сек, capacity=20):
┌─────────────────────────────────────────────────────────────────┐
│  Секунда 1: 10 запросов ✅ (было 10 токенов, стало 0)           │
│  Секунда 2: 0 запросов (токены накапливаются: 0 → 10)           │
│  Секунда 3: 15 запросов ✅ (было 10+10=20, стало 5)             │
│                                                                  │
│  Бонус: накопленные токены позволяют "всплеск"                  │
└─────────────────────────────────────────────────────────────────┘
```

**Визуализация работы Token Bucket:**

```
TOKEN BUCKET — как работает

Параметры:
- rate = 10 токенов/секунду (скорость пополнения)
- capacity = 20 токенов (максимум в ведре)

Начальное состояние: ведро полное (20 токенов)

┌────────────────────────────────────────────────────────────────────┐
│                                                                     │
│            ┌─────────────────────┐                                  │
│            │   Генератор токенов │                                  │
│            │   (rate: 10/сек)    │                                  │
│            └──────────┬──────────┘                                  │
│                       │                                             │
│                       │ +10 токенов каждую секунду                  │
│                       │ (но не больше capacity)                     │
│                       ▼                                             │
│            ┌─────────────────────┐                                  │
│            │      ВЕДРО          │                                  │
│            │   ┌───────────┐     │ ← capacity = 20 (максимум)       │
│            │   │ ● ● ● ● ● │     │                                  │
│            │   │ ● ● ● ● ● │     │                                  │
│            │   │ ● ● ● ●   │     │ ← текущие токены = 14            │
│            │   │           │     │                                  │
│            │   └───────────┘     │                                  │
│            └──────────┬──────────┘                                  │
│                       │                                             │
│     Запрос ──────────▶│ Берём 1 токен                              │
│                       │                                             │
│            ┌──────────┴──────────┐                                  │
│            │                     │                                  │
│      токены >= 1?          токены == 0?                             │
│            │                     │                                  │
│            ▼                     ▼                                  │
│     ┌──────────┐          ┌──────────┐                              │
│     │ РАЗРЕШИТЬ│          │ ОТКЛОНИТЬ│                              │
│     │ (токен--)│          │  (429)   │                              │
│     └──────────┘          └──────────┘                              │
│                                                                     │
└────────────────────────────────────────────────────────────────────┘
```

**Математические формулы Token Bucket:**

```
ФОРМУЛЫ TOKEN BUCKET

1. Пополнение токенов:
   ┌────────────────────────────────────────────────────────────────┐
   │                                                                 │
   │   new_tokens = old_tokens + (elapsed_time × rate)              │
   │   tokens = min(new_tokens, capacity)                           │
   │                                                                 │
   │   Пример:                                                       │
   │   - old_tokens = 5                                              │
   │   - elapsed_time = 2 секунды                                    │
   │   - rate = 10 токенов/сек                                       │
   │   - capacity = 20                                               │
   │                                                                 │
   │   new_tokens = 5 + (2 × 10) = 25                                │
   │   tokens = min(25, 20) = 20  ← ограничено вместимостью         │
   │                                                                 │
   └────────────────────────────────────────────────────────────────┘

2. Время ожидания (если токенов не хватает):
   ┌────────────────────────────────────────────────────────────────┐
   │                                                                 │
   │   tokens_needed = requested - current_tokens                   │
   │   wait_time = tokens_needed / rate                             │
   │                                                                 │
   │   Пример:                                                       │
   │   - Хотим 5 токенов, есть 2                                    │
   │   - rate = 10 токенов/сек                                       │
   │                                                                 │
   │   tokens_needed = 5 - 2 = 3                                     │
   │   wait_time = 3 / 10 = 0.3 секунды                             │
   │                                                                 │
   └────────────────────────────────────────────────────────────────┘

3. Максимальная скорость всплеска:
   ┌────────────────────────────────────────────────────────────────┐
   │                                                                 │
   │   max_burst_rate = capacity / 1 секунду                        │
   │                                                                 │
   │   Если capacity = 20 и rate = 10:                              │
   │   - Мгновенный всплеск: до 20 запросов                         │
   │   - Устойчивая скорость: 10 запросов/сек                       │
   │                                                                 │
   └────────────────────────────────────────────────────────────────┘
```

**Полная реализация с подробными комментариями:**

```python
# Token Bucket — самый популярный алгоритм.
# Используется в: Amazon AWS, Stripe, GitHub, Google Cloud.
# Причина: позволяет всплески, при этом ограничивая среднюю скорость.

import time
from dataclasses import dataclass
from typing import Tuple

@dataclass
class TokenBucket:
    """
    Token Bucket Rate Limiter.

    Параметры:
    - rate: сколько токенов добавляется в секунду (средняя скорость)
    - capacity: максимум токенов в ведре (размер всплеска)

    Пример: rate=10, capacity=20
    - Средняя скорость: 10 запросов/сек
    - Максимальный всплеск: 20 запросов мгновенно
    """
    rate: float          # Токенов в секунду
    capacity: float      # Максимум токенов
    tokens: float = None # Текущее количество токенов
    last_update: float = None  # Время последнего обновления

    def __post_init__(self):
        # При создании ведро полное
        if self.tokens is None:
            self.tokens = self.capacity
        if self.last_update is None:
            self.last_update = time.time()

    def _refill(self):
        """
        Пополняем токены на основе прошедшего времени.

        Это "ленивое" пополнение:
        - Не используем таймер
        - Вычисляем, сколько токенов накопилось с прошлого раза
        """
        now = time.time()

        # Сколько секунд прошло с последнего обновления
        elapsed = now - self.last_update

        # Сколько токенов накопилось
        # Формула: новые_токены = прошло_секунд × скорость_пополнения
        new_tokens = elapsed * self.rate

        # Добавляем токены, но не больше вместимости
        self.tokens = min(self.capacity, self.tokens + new_tokens)

        # Обновляем время
        self.last_update = now

    def consume(self, tokens: int = 1) -> Tuple[bool, float]:
        """
        Пытаемся "потратить" токены.

        Параметры:
        - tokens: сколько токенов нужно (обычно 1)

        Возвращает:
        - (True, 0) — запрос разрешён
        - (False, wait_time) — отклонён, ждать wait_time секунд
        """
        # Сначала пополняем
        self._refill()

        # Проверяем, хватает ли токенов
        if self.tokens >= tokens:
            # Токенов хватает — тратим и разрешаем
            self.tokens -= tokens
            return True, 0

        # Токенов не хватает — вычисляем время ожидания
        tokens_needed = tokens - self.tokens
        wait_time = tokens_needed / self.rate

        return False, wait_time

    def get_state(self) -> dict:
        """Получить текущее состояние для отладки/мониторинга."""
        self._refill()
        return {
            "tokens": round(self.tokens, 2),
            "capacity": self.capacity,
            "rate": self.rate,
            "fill_percentage": round(self.tokens / self.capacity * 100, 1)
        }


# ═══════════════════════════════════════════════════════════════════
# ПРИМЕР ИСПОЛЬЗОВАНИЯ
# ═══════════════════════════════════════════════════════════════════

# Создаём лимитер:
# - 10 запросов в секунду (средняя скорость)
# - Максимальный всплеск: 20 запросов
limiter = TokenBucket(rate=10, capacity=20)

# Симулируем запросы
print("=== Token Bucket Demo ===\n")

# Тест 1: Всплеск запросов
print("Тест 1: 25 запросов подряд")
for i in range(25):
    allowed, wait_time = limiter.consume()
    status = "✅" if allowed else f"❌ (ждать {wait_time:.2f}с)"
    print(f"  Запрос {i+1:2d}: {status}")

print(f"\nСостояние: {limiter.get_state()}")

# Ждём пополнения
print("\nЖдём 1 секунду...")
time.sleep(1)

print(f"Состояние после ожидания: {limiter.get_state()}")

# Тест 2: Ещё запросы после ожидания
print("\nТест 2: 5 запросов после ожидания")
for i in range(5):
    allowed, wait_time = limiter.consume()
    status = "✅" if allowed else f"❌ (ждать {wait_time:.2f}с)"
    print(f"  Запрос {i+1}: {status}")
```

### Алгоритм 5: Leaky Bucket (Дырявое ведро)

**Аналогия:** Представь ведро с дыркой внизу. Вода (запросы) наливается сверху с разной скоростью, но вытекает (обрабатывается) всегда одинаково — по капле в секунду.

**Главное отличие от Token Bucket:**

```
TOKEN BUCKET vs LEAKY BUCKET

Token Bucket:                    Leaky Bucket:
┌─────────────────────┐          ┌─────────────────────┐
│                     │          │                     │
│  Запросы: ●●●●●●●●  │          │  Запросы: ●●●●●●●●  │
│           ↓↓↓↓↓↓↓↓  │          │           ↓↓↓↓↓↓↓↓  │
│    ┌───────────┐    │          │    ┌───────────┐    │
│    │ Токены    │    │          │    │ Очередь   │    │
│    │ ● ● ● ●   │    │          │    │ → → → → → │    │
│    └─────┬─────┘    │          │    └─────┬─────┘    │
│          │          │          │          │          │
│          ▼          │          │          ▼          │
│   ПЕРЕМЕННАЯ        │          │   ПОСТОЯННАЯ        │
│   скорость выхода   │          │   скорость выхода   │
│   (до capacity/сек) │          │   (всегда rate/сек) │
│                     │          │                     │
│   Подходит для:     │          │   Подходит для:     │
│   • API клиентов    │          │   • Запись в БД     │
│   • Burst-трафик    │          │   • Внешние API     │
│   • User-facing     │          │   • Стабильность    │
│                     │          │                     │
└─────────────────────┘          └─────────────────────┘
```

**Когда использовать Leaky Bucket:**

| Сценарий | Почему Leaky Bucket |
|----------|---------------------|
| Запись в базу данных | БД не любит всплесков, нужна равномерная нагрузка |
| Вызов внешнего API со строгим лимитом | Stripe API: точно 100 запросов/сек, не больше |
| Отправка email/SMS | Провайдер ограничивает скорость |
| Логирование | Равномерный поток в систему логов |

**Реализация с очередью:**

```python
# Leaky Bucket — когда нужна РАВНОМЕРНАЯ обработка.
# Запросы ставятся в очередь и обрабатываются с постоянной скоростью.

import asyncio
from collections import deque
from dataclasses import dataclass, field
from typing import Any, Callable, Awaitable

@dataclass
class LeakyBucket:
    """
    Leaky Bucket Rate Limiter с очередью.

    Параметры:
    - rate: сколько запросов обрабатывать в секунду
    - capacity: максимальный размер очереди

    Принцип:
    1. Запросы добавляются в очередь
    2. Очередь обрабатывается с постоянной скоростью
    3. Если очередь полная — отклоняем запрос
    """
    rate: float                          # Запросов в секунду
    capacity: int                        # Размер очереди
    queue: deque = field(default_factory=deque)
    processing: bool = False

    # Функция для обработки элементов (переопределяется)
    handler: Callable[[Any], Awaitable[None]] = None

    async def add(self, item: Any) -> bool:
        """
        Добавить элемент в очередь.

        Возвращает:
        - True — добавлен в очередь
        - False — очередь полная, отклонён
        """
        # Проверяем, есть ли место в очереди
        if len(self.queue) >= self.capacity:
            return False

        # Добавляем в очередь
        self.queue.append(item)

        # Запускаем обработку, если ещё не запущена
        if not self.processing:
            asyncio.create_task(self._process())

        return True

    async def _process(self):
        """
        Обрабатывать очередь с постоянной скоростью.

        Интервал между обработками = 1 / rate
        Например, rate=10 → интервал = 0.1 сек
        """
        self.processing = True

        # Интервал между обработками
        interval = 1.0 / self.rate

        while self.queue:
            # Берём элемент из начала очереди
            item = self.queue.popleft()

            # Обрабатываем
            if self.handler:
                await self.handler(item)

            # Ждём перед следующей обработкой
            await asyncio.sleep(interval)

        self.processing = False


# ═══════════════════════════════════════════════════════════════════
# ПРАКТИЧЕСКИЙ ПРИМЕР: Отправка уведомлений
# ═══════════════════════════════════════════════════════════════════

async def send_notification(message: str):
    """Имитация отправки уведомления."""
    print(f"📤 Отправлено: {message}")

async def demo():
    # Создаём Leaky Bucket:
    # - Скорость: 2 уведомления в секунду
    # - Очередь: максимум 10 уведомлений
    bucket = LeakyBucket(rate=2, capacity=10)
    bucket.handler = send_notification

    # Добавляем 5 уведомлений мгновенно
    messages = [
        "Привет!",
        "Ваш заказ принят",
        "Доставка завтра",

        "Новый товар"

    print("Добавляем 5 сообщений в очередь...")

    for msg in messages:
        added = await bucket.add(msg)
        if added:
            print(f"✅ В очереди: {msg}")
        else:
            print(f"❌ Очередь полная: {msg}")

    print(f"\nОчередь: {len(bucket.queue)} сообщений")
    print("Начинаем отправку (2 сообщения/сек)...\n")

    # Ждём, пока очередь опустеет
    while bucket.queue or bucket.processing:
        await asyncio.sleep(0.1)

    print("\n✅ Все сообщения отправлены!")
```

### Redis + Lua: полная реализация с объяснениями

```python
# Distributed Rate Limiter с Redis.
# Использует Lua скрипты для атомарности.
# Подходит для production-систем с несколькими серверами.

import redis
import time
from typing import Tuple

class RedisRateLimiter:
    """
    Distributed Rate Limiter using Redis.

    Почему Redis:
    - Быстрый (in-memory)
    - Поддерживает атомарные операции через Lua
    - Легко масштабируется (Redis Cluster)
    - Автоматическое удаление старых ключей (TTL)
    """

    def __init__(self, redis_client: redis.Redis, key_prefix: str = "rate_limit"):
        self.redis = redis_client
        self.key_prefix = key_prefix

    def _key(self, identifier: str) -> str:
        """Формируем Redis ключ для пользователя."""
        return f"{self.key_prefix}:{identifier}"

    # ═══════════════════════════════════════════════════════════════
    # МЕТОД 1: Sliding Window с Sorted Set
    # ═══════════════════════════════════════════════════════════════

    def check_sliding_window(
        self,
        identifier: str,
        limit: int,
        window_seconds: int
    ) -> Tuple[bool, int, int]:
        """
        Sliding Window Counter using Redis Sorted Set.

        Как работает:
        1. Храним timestamp каждого запроса в Sorted Set
        2. Score = timestamp
        3. Удаляем старые записи (за пределами окна)
        4. Считаем оставшиеся

        Параметры:
        - identifier: ID пользователя (IP, user_id, API key)
        - limit: максимум запросов
        - window_seconds: размер окна в секундах

        Возвращает:
        - (allowed, remaining, reset_after_seconds)
        """
        key = self._key(identifier)
        now = time.time()
        window_start = now - window_seconds

        # Pipeline — группируем команды для одного round-trip
        pipe = self.redis.pipeline()

        # Шаг 1: Удаляем старые записи
        # ZREMRANGEBYSCORE удаляет элементы со score от 0 до window_start
        pipe.zremrangebyscore(key, 0, window_start)

        # Шаг 2: Считаем текущие записи
        # ZCARD возвращает количество элементов в Sorted Set
        pipe.zcard(key)

        # Шаг 3: Добавляем текущий запрос
        # ZADD добавляет элемент с score = now
        # Используем now как и ключ, и score (уникальность)
        pipe.zadd(key, {str(now): now})

        # Шаг 4: Устанавливаем TTL (автоудаление)
        pipe.expire(key, window_seconds)

        # Выполняем все команды
        results = pipe.execute()

        # results[1] = результат ZCARD (количество до добавления)
        current_count = results[1]

        if current_count >= limit:
            # Лимит превышен
            # Находим самый старый запрос для вычисления reset time
            oldest = self.redis.zrange(key, 0, 0, withscores=True)
            if oldest:
                oldest_ts = oldest[0][1]
                reset_after = int(oldest_ts + window_seconds - now) + 1
            else:
                reset_after = window_seconds
            return False, 0, reset_after

        remaining = limit - current_count - 1
        return True, remaining, window_seconds

    # ═══════════════════════════════════════════════════════════════
    # МЕТОД 2: Token Bucket с Lua скриптом
    # ═══════════════════════════════════════════════════════════════

    def check_token_bucket(
        self,
        identifier: str,
        rate: float,
        capacity: float,
        tokens: int = 1
    ) -> Tuple[bool, float, float]:
        """
        Token Bucket using Redis + Lua.

        Lua скрипт выполняется АТОМАРНО — это ключевое!
        Никакой race condition.

        Возвращает:
        - (allowed, tokens_remaining, retry_after_seconds)
        """
        key = self._key(identifier)
        now = time.time()

        # ────────────────────────────────────────────────────────────
        # LUA СКРИПТ — сердце алгоритма
        # ────────────────────────────────────────────────────────────

        lua_script = """
        -- KEYS[1] = Redis ключ для этого пользователя
        -- ARGV[1] = rate (токенов в секунду)
        -- ARGV[2] = capacity (максимум токенов)
        -- ARGV[3] = now (текущее время)
        -- ARGV[4] = requested (сколько токенов нужно)

        local key = KEYS[1]
        local rate = tonumber(ARGV[1])
        local capacity = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])
        local requested = tonumber(ARGV[4])

        -- Получаем текущее состояние bucket
        -- HMGET возвращает несколько полей из Hash
        local bucket = redis.call('HMGET', key, 'tokens', 'last_update')

        -- Если bucket не существует, начинаем с полного ведра
        local tokens = tonumber(bucket[1]) or capacity
        local last_update = tonumber(bucket[2]) or now

        -- ══════════════════════════════════════════════════════════
        -- ПОПОЛНЕНИЕ ТОКЕНОВ
        -- ══════════════════════════════════════════════════════════
        -- Формула: новые_токены = (сейчас - прошлый_раз) × скорость
        -- Но не больше capacity

        local elapsed = now - last_update
        tokens = math.min(capacity, tokens + elapsed * rate)

        -- ══════════════════════════════════════════════════════════
        -- ПРОВЕРКА И РЕШЕНИЕ
        -- ══════════════════════════════════════════════════════════

        local allowed = 0      -- 0 = отклонено, 1 = разрешено
        local retry_after = 0  -- сколько ждать если отклонено

        if tokens >= requested then
            -- Токенов хватает — разрешаем
            tokens = tokens - requested
            allowed = 1
        else
            -- Токенов не хватает — отклоняем
            -- Вычисляем время ожидания:
            -- (сколько_нужно - сколько_есть) / скорость_пополнения
            retry_after = (requested - tokens) / rate
        end

        -- ══════════════════════════════════════════════════════════
        -- СОХРАНЕНИЕ СОСТОЯНИЯ
        -- ══════════════════════════════════════════════════════════

        -- Сохраняем новое состояние в Redis Hash
        redis.call('HMSET', key, 'tokens', tokens, 'last_update', now)

        -- Устанавливаем TTL — автоудаление неактивных ключей
        -- Время жизни = время заполнения пустого ведра + запас
        redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)

        -- Возвращаем результат
        return {allowed, tokens, retry_after}
        """

        # Выполняем Lua скрипт
        result = self.redis.eval(
            lua_script,
            1,              # Количество KEYS
            key,            # KEYS[1]
            rate,           # ARGV[1]
            capacity,       # ARGV[2]
            now,            # ARGV[3]
            tokens          # ARGV[4]
        )

        allowed = result[0] == 1
        tokens_remaining = result[1]
        retry_after = result[2]

        return allowed, tokens_remaining, retry_after


# ═══════════════════════════════════════════════════════════════════
# ИНТЕГРАЦИЯ С FASTAPI
# ═══════════════════════════════════════════════════════════════════

# Пример middleware для FastAPI:
"""
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
import redis
import time

app = FastAPI()
redis_client = redis.Redis(host='localhost', port=6379, decode_responses=True)
```

## Типичные ошибки и как их избежать

На основе реального опыта компаний ([источники исследования](#источники)):

### Ошибка 1: Race Condition

```
❌ НЕПРАВИЛЬНО: GET → проверка → SET

if redis.get("counter") < 100:
    redis.set("counter", redis.get("counter") + 1)
    allow_request()

Проблема: Между GET и SET другой сервер тоже может прочитать 99

✅ ПРАВИЛЬНО: Атомарная операция с Lua

result = redis.eval(lua_script, ...)
if result == "allowed":
    allow_request()
```

### Ошибка 2: Не синхронизированы часы

```
❌ Проблема:
Сервер A: 12:00:00
Сервер B: 12:00:05 (отстаёт на 5 сек)

При Fixed Window сервер B сбросит счётчик позже,
пропустив лишние запросы.

✅ Решение:
1. Используй NTP для синхронизации часов
2. Используй Redis TIME вместо локального времени
3. Используй Sliding Window (менее чувствителен)
```

### Ошибка 3: Sticky Sessions не масштабируются

```
❌ Идея: Направлять пользователя всегда на один сервер

Проблемы:
- Сервер упал → все его пользователи без сервиса
- Неравномерная нагрузка (один сервер перегружен)
- Сложно масштабировать

✅ Решение: Centralized rate limiting (Redis)
```

### Ошибка 4: Нет fallback при отказе Redis

```
❌ Если Redis недоступен:
throw error → Все запросы падают

✅ Правильный fallback:

try:
    allowed = rate_limiter.check(user_id)
except redis.RedisError:
    # Redis недоступен
    # Вариант 1: Пропускаем всех (опасно при атаке)
    # Вариант 2: Local rate limiting (менее точно)
    # Вариант 3: Отклоняем всех (безопасно, но жёстко)

    allowed = local_rate_limiter.check(user_id)  # Fallback
    log.warning("Redis unavailable, using local limiter")
```

## Проверь себя

> [!question]- Твой API обслуживает и бесплатных, и платных пользователей. Бесплатные генерируют 80% трафика, но приносят 5% дохода. Платные жалуются на медленные ответы. Какую стратегию rate limiting ты выберешь и почему одного алгоритма недостаточно?
> Нужна многоуровневая стратегия, как у Stripe. Уровень 1: разные лимиты по tier'ам (free: 10 req/мин, pro: 100 req/мин) через Token Bucket. Уровень 2: Fleet Usage Load Shedding — при перегрузке приоритизируем запросы платных клиентов и отклоняем некритичные запросы бесплатных. Уровень 3: Concurrent Requests Limiter для тяжёлых эндпоинтов. Одного алгоритма недостаточно, потому что rate limiting по количеству запросов не учитывает стоимость запроса (GET списка vs. генерация отчёта) и бизнес-приоритеты.

> [!question]- Почему Token Bucket стал стандартом для публичных API (AWS, Stripe, GitHub), а не Sliding Window Counter, хотя последний точнее?
> Token Bucket позволяет контролируемые burst'ы — кратковременные всплески трафика, которые естественны для реальных клиентов (например, при загрузке страницы браузер одновременно запрашивает несколько ресурсов). Sliding Window Counter строго ограничивает среднюю скорость без возможности всплесков, что ухудшает пользовательский опыт. Token Bucket разделяет два параметра: среднюю скорость (rate) и допустимый всплеск (capacity), давая оператору API гибкость настройки. Точность Sliding Window избыточна, когда цель — не идеальный подсчёт, а защита инфраструктуры и справедливое распределение ресурсов.

> [!question]- Команда реализовала rate limiting через Redis, но при нагрузочном тесте с 3 серверов лимит 100 req/мин пропускает до 300 запросов. Архитектор говорит "используйте Lua скрипты". Объясни, какую именно проблему решают Lua скрипты и почему pipeline из ZREMRANGEBYSCORE + ZCARD + ZADD без Lua тоже может иметь race condition.
> Pipeline группирует команды в один round-trip для снижения latency, но НЕ делает их атомарными — между командами pipeline другой сервер может выполнить свои команды. Между ZCARD (чтение количества) и ZADD (запись) другой сервер может тоже прочитать то же количество и тоже записать — классический TOCTOU race condition. Lua скрипт выполняется Redis'ом атомарно: пока скрипт работает, никакая другая команда не может вмешаться. Это гарантирует, что чтение счётчика и инкремент происходят как единая неделимая операция.

> [!question]- Как концепция rate limiting связана с TCP congestion control (управление перегрузкой в сети)? Какой алгоритм rate limiting концептуально ближе всего к TCP и почему?
> TCP congestion control и rate limiting решают одну и ту же фундаментальную проблему — управление потоком в системе с ограниченной пропускной способностью. Leaky Bucket концептуально ближе всего к TCP: оба обеспечивают равномерный выходной поток независимо от входного. TCP использует скользящее окно (sliding window) для контроля потока, аналогично Sliding Window Counter в rate limiting. А механизм slow start в TCP напоминает Token Bucket с пустым начальным состоянием — пропускная способность нарастает постепенно. Эта связь не случайна: Token Bucket изначально был формализован именно в контексте сетевого traffic shaping.

## Ключевые карточки

Какую проблему решает rate limiting?
?
Rate limiting защищает серверы от перегрузки, обеспечивает справедливое распределение ресурсов между клиентами, предотвращает DDoS-атаки и брутфорс, а также контролирует затраты на дорогие операции.

В чём проблема "пограничного всплеска" (boundary burst) у Fixed Window Counter?
?
Клиент может отправить почти 2x от лимита за короткий период на границе двух окон: например, 99 запросов в конце первого окна и 99 в начале второго — оба раза формально в лимите, но сервер получает 198 запросов за 2 секунды.

Чем Token Bucket отличается от Leaky Bucket?
?
Token Bucket позволяет всплески (burst) — накопленные токены можно потратить мгновенно, скорость выхода переменная. Leaky Bucket обеспечивает постоянную скорость обработки — запросы ставятся в очередь и обрабатываются равномерно. Token Bucket подходит для API клиентов, Leaky Bucket — для записи в БД или вызовов внешних API.

Какая формула у Sliding Window Counter для оценки количества запросов?
?
`estimated = previous_count × (1 - time_in_window) + current_count`, где `time_in_window` — доля прошедшего времени в текущем окне (от 0.0 до 1.0). Это взвешенная сумма предыдущего и текущего окна, дающая приближение к точному скользящему окну при O(1) памяти.

Почему в distributed rate limiting нужны Lua скрипты в Redis, а не обычные GET/SET?
?
Между GET (чтение счётчика) и SET (запись нового значения) другой сервер может выполнить свой GET и тоже получить старое значение — это race condition. Lua скрипт в Redis выполняется атомарно: никакая другая команда не может вмешаться между чтением и записью внутри скрипта.

Какие HTTP заголовки обязательны при rate limiting?
?
При 429: `Retry-After` (обязателен — сколько секунд ждать). При всех ответах: `X-RateLimit-Limit` (максимум запросов), `X-RateLimit-Remaining` (сколько осталось), `X-RateLimit-Reset` (Unix timestamp сброса лимита).

Какой алгоритм rate limiting лучше всего подходит для большинства production-систем и почему?
?
Sliding Window Counter — лучший компромисс: O(1) памяти как у Fixed Window, высокая точность близкая к Sliding Log, быстрая работа. Для API с burst-трафиком — Token Bucket (разделяет среднюю скорость и допустимый всплеск).

Что делать если Redis (центральный rate limiter) недоступен?
?
Нужен fallback: переключение на локальный rate limiter (менее точный, но работающий). Нельзя просто пропускать все запросы (опасно при атаке) или отклонять все (сервис недоступен). Также обязательно логировать событие для мониторинга.
