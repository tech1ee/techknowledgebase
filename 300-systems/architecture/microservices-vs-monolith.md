# Микросервисы vs Монолит: когда что выбирать

> 61% компаний внедрили микросервисы, 42% консолидировали обратно (CNCF 2025). Amazon Prime Video сэкономил 90% затрат, вернувшись к монолиту. 60% команд жалеют о переходе на микросервисы для малых/средних приложений (Gartner 2025).

---

## Теоретические основы

### Формальные определения

> **Монолит** — deployment unit, в котором все компоненты системы компилируются и развёртываются как единый артефакт (один процесс, одна кодовая база).

> **Микросервисы** — архитектурный стиль, при котором система состоит из набора независимо развёртываемых сервисов, каждый из которых реализует одну бизнес-возможность (business capability) и общается с другими через сетевые протоколы.

### Историческая атрибуция

| Концепция | Автор | Год | Вклад |
|-----------|-------|-----|-------|
| **Modular programming** | Parnas, D.L. | 1972 | "On the Criteria to be Used in Decomposing Systems into Modules" — information hiding |
| **SOA** | Gartner | 1996 | Service-Oriented Architecture — предшественник микросервисов |
| **Microservices term** | James Lewis & Fred George | 2011 | Впервые использован на workshop |
| **Microservices article** | Lewis & Fowler | 2014 | Каноническое определение: "independently deployable services organized around business capabilities" |
| **Conway's Law** | Conway, M. | 1967 | "Organizations design systems that mirror their communication structures" |

### Conway's Law и архитектура

> **Conway's Law (1967):** "Any organization that designs a system will produce a design whose structure is a copy of the organization's communication structure."

**Следствие (Inverse Conway Maneuver):** чтобы получить нужную архитектуру (микросервисы), нужно сначала реорганизовать команды. Монолит с 3 разработчиками — правильно. Микросервисы с 3 разработчиками — антипаттерн.

### Формальные trade-offs

| Свойство | Монолит | Микросервисы |
|----------|---------|--------------|
| **Coupling** | In-process (method call, ~ns) | Network (HTTP/gRPC, ~ms) |
| **Consistency** | ACID (одна БД) | Eventual (saga, compensating tx) |
| **Deployment** | Atomic (всё или ничего) | Independent (per service) |
| **Debugging** | Stack trace | Distributed tracing |
| **Сложность** | Code complexity | Operational complexity |
| **Team scaling** | Merge conflicts при >15 devs | Independent team ownership |

> **Закон Мартина Фаулера:** "Don't even consider microservices unless you have a system that's too complex to manage as a monolith." (MonolithFirst pattern)

## Зачем это нужно

### Проблема: Cargo Cult Architecture

Каждый разработчик слышал истории успеха Netflix и Uber. Эти истории вдохновляют — но опускают важную деталь: **Netflix перешёл на микросервисы при миллионах пользователей и сотнях инженеров**.

### Актуальность в 2024-2025

**Индустрия переосмысливает микросервисы:**

| Метрика | Источник | Значение |
|---------|----------|----------|
| Внедрили микросервисы | O'Reilly 2024 | 61% enterprise |
| Вернулись к монолитам | O'Reilly 2024 | 29% компаний |
| Консолидировали сервисы | CNCF 2025 | 42% организаций |
| Жалеют о переходе | Gartner 2025 | 60% для small/medium apps |
| Дополнительные затраты на отладку | DZone 2024 | +35% времени |
| Overhead TCO | FinOps 2025 | +50-100% |

### Ключевой кейс 2024: Amazon Prime Video

В марте 2024 команда Amazon Prime Video опубликовала статью о миграции **обратно к монолиту**:

```
Было: Микросервисы + Step Functions + S3
Стало: Монолит на одном EC2 инстансе

Результат:
- Снижение затрат: 90%
- Упрощение инфраструктуры: значительное
- Масштабируемость: сохранена
```

**Почему это важно:** Это не стартап, это Amazon — компания, которая популяризировала микросервисы. Их вывод: **микросервисы — не универсальное решение**.

### Когда выбор критичен

| Сценарий | Правильный выбор | Почему |
|----------|------------------|--------|
| Стартап, 5 разработчиков | Монолит | Скорость важнее масштаба |
| Команда <10, product-market fit не найден | Монолит | Пивоты требуют гибкости |
| Разные части с разной нагрузкой 10x+ | Микросервисы | Реальная экономия |
| 50+ разработчиков, частые конфликты | Микросервисы | Организационная необходимость |
| Непонятно что выбрать | **Modular Monolith** | Лучшее из двух миров |

## Терминология

| Термин | Значение |
|--------|----------|
| **Монолит** | Единое приложение, весь код в одном деплое |
| **Микросервис** | Независимый сервис с отдельным деплоем |
| **Distributed system** | Система из нескольких компонентов по сети |
| **Service mesh** | Инфраструктура для общения микросервисов |
| **Bounded context** | Границы микросервиса в DDD |
| **Modular monolith** | Монолит с чёткими границами модулей |
| **Eventual consistency** | Согласованность данных со временем |

## История вопроса: почему это вообще дискуссия

В 2009 году Netflix столкнулся с проблемой. Их монолитное приложение не справлялось с ростом. Время деплоя измерялось часами, каждый релиз — это лотерея "сломается или нет", а масштабирование требовало покупки всё более мощных серверов.

Они приняли радикальное решение: разбить монолит на сотни независимых сервисов. Миграция заняла годы. Но результат впечатляет: сегодня у Netflix более 1000 микросервисов, а инженеры деплоят код тысячи раз в день.

И вот тут начинается проблема.

1️⃣ Каждый стартап прочитал эту историю и решил: "Мы тоже так хотим!"

2️⃣ Забыв важную деталь: Netflix перешёл на микросервисы когда у них уже были миллионы пользователей, сотни инженеров и реальные проблемы масштабирования.

3️⃣ А большинство стартапов пытается решить микросервисами проблемы, которых у них ещё нет. И создают новые — которых без микросервисов не было бы.

## Реальные кейсы: кто выиграл, кто проиграл

### Netflix, Uber, Amazon — микросервисы сработали

**Почему у них получилось:**
- Тысячи инженеров в разных командах
- Разные части системы масштабируются по-разному (у Uber платёжный сервис нагружен иначе, чем карты)
- Могут позволить себе инфраструктурные команды
- Проблемы монолита были реальными, а не гипотетическими

**Uber** перешёл на микросервисы из-за конкретных проблем:
- Частые конфликты в коде между командами
- Невозможность масштабировать отдельные функции
- Проблемы с надёжностью — падение одного модуля роняло всё

### Shopify — монолит победил

Shopify — один из крупнейших e-commerce платформ в мире. И они сознательно остаются на монолите.

*Почему?* Потому что их монолит хорошо структурирован. Они инвестировали в модульность внутри монолита, а не в разделение на сервисы.

Результат: простота деплоя, единая кодовая база, меньше накладных расходов на инфраструктуру.

### Friendster — как технический долг убил компанию

Помните Friendster? Это была первая крупная социальная сеть — до MySpace и Facebook.

История их провала поучительна. Они росли слишком быстро, копили технический долг, и в какой-то момент страницы стали грузиться по 20-40 секунд. Пользователи массово ушли на MySpace.

*Важный момент:* проблема Friendster была не в выборе "монолит vs микросервисы". Проблема была в том, что они игнорировали архитектурные проблемы, пока не стало слишком поздно.

### 29% вернулись обратно

По данным O'Reilly за 2024 год, почти треть компаний, внедривших микросервисы, вернулась к монолитной архитектуре.

Типичные причины:
- "Мы недооценили сложность distributed systems"
- "У нас не было инфраструктурной команды"
- "Latency между сервисами убила производительность"
- "Дебаг распределённой системы — это ад"

## Честное сравнение

### Визуально: как это выглядит

```
МОНОЛИТ                              МИКРОСЕРВИСЫ
=========                            ==============

┌─────────────────────┐              ┌───────┐ ┌───────┐ ┌───────┐
│                     │              │ Users │ │Orders │ │Payment│
│   Один большой      │              │Service│ │Service│ │Service│
│   процесс           │              └───┬───┘ └───┬───┘ └───┬───┘
│                     │                  │         │         │
│  ┌─────┐ ┌─────┐   │              ┌───┴─────────┴─────────┴───┐
│  │Users│ │Orders│   │              │       API Gateway         │
│  └─────┘ └─────┘   │              └─────────────┬─────────────┘
│  ┌─────┐ ┌─────┐   │                            │
│  │Pay  │ │Auth │   │              ┌─────────────┴─────────────┐
│  └─────┘ └─────┘   │              │                           │
│                     │              ▼                           ▼
└─────────┬───────────┘         ┌───────────┐            ┌───────────┐
          │                     │  Users DB │            │ Orders DB │
          ▼                     └───────────┘            └───────────┘
    ┌───────────┐
    │  Одна БД  │               + Service Discovery, Load Balancer,
    └───────────┘                 Message Queue, Distributed Tracing...
```

### Монолит

**Плюсы (и почему они реально важны):**

```
✅ Просто начать
   git clone → npm install → npm start
   Через 5 минут работает

✅ Один деплой
   CI/CD: один pipeline, одна кнопка, один артефакт
   Не нужно думать о версионировании между сервисами

✅ Дебаг — стектрейс показывает ВСЁ
   Ошибка в User → Order → Payment? Один лог, одна цепочка

✅ Транзакции работают
   BEGIN → UPDATE users → UPDATE orders → COMMIT
   Атомарно, надёжно, понятно

✅ Не нужна DevOps-команда
   Один сервер, один процесс, справится один человек
```

**Минусы:**

```
❌ Масштабируется ЦЕЛИКОМ
   Нужно больше Orders? Подними весь монолит ×3
   Платишь за Users и Auth, которые не нагружены

❌ Merge conflicts каждый день
   10 разработчиков → все в одном репозитории → война

❌ Билд 15 минут
   Изменил одну строчку? Жди пересборки всего

❌ Падает всё или ничего
   Баг в Payment уронил весь сайт
```

**Когда выбирать:**
- Стартап на ранней стадии (0-15 разработчиков)
- Непонятно, какие части будут нагружены
- Важна скорость разработки, не масштаб
- Нет выделенной DevOps/Platform команды

### Микросервисы

**Плюсы (и почему они переоценены для большинства):**

```
✅ Независимое масштабирование
   Orders нагружен? Подними только Orders ×5
   Платишь только за то, что нужно

✅ Команды работают автономно
   Team A деплоит Users, Team B деплоит Orders
   Нет merge conflicts между командами

✅ Разные технологии
   Users на Python, Orders на Go, ML на Rust
   (Но на практике — это кошмар поддержки)

✅ Fault isolation
   Payment упал? Users и Orders работают
   (Если правильно настроил circuit breakers)
```

**Минусы (и почему они недооценены):**

```
❌ Distributed systems — это СЛОЖНО
   "Сеть надёжна" — первое заблуждение из списка
   Timeout, retry, circuit breaker — это не "потом добавим"

❌ Latency убивает
   Монолит: function call = ~1 наносекунда
   Микросервисы: HTTP call = ~1-10 миллисекунд
   Разница: 1,000,000x медленнее

❌ Дебаг — это ад
   Запрос: Gateway → Auth → Users → Orders → Payment → Notification
   Где ошибка? В каком из 6 логов? Какой request ID?

❌ Инфраструктура как отдельный продукт
   Service Discovery, API Gateway, Load Balancer
   Message Queue, Distributed Tracing, Centralized Logging
   Это 6+ новых систем, каждая требует экспертизы

❌ Транзакции... ну, удачи
   "Списать деньги И создать заказ" — это теперь Saga Pattern
   Eventual consistency вместо ACID
```

**Когда выбирать:**
- 50+ разработчиков в нескольких командах
- РЕАЛЬНЫЕ (не гипотетические!) проблемы масштабирования
- Есть 2-3 человека только на инфраструктуру
- Разные части системы масштабируются ×10 по-разному

## Третий путь: модульный монолит

В 2024-2025 **Modular Monolith** — рекомендуемая стартовая архитектура. Это компромисс: код организован как микросервисы (чёткие границы, отдельные модули), но деплоится как монолит (один артефакт).

```
Модульный монолит:
┌─────────────────────────────────────┐
│            Один деплой              │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐│
│  │ Users   │ │ Orders  │ │Payments ││
│  │ Module  │ │ Module  │ │ Module  ││
│  │ ──────  │ │ ──────  │ │ ──────  ││
│  │ Public  │ │ Public  │ │ Public  ││
│  │   API   │ │   API   │ │   API   ││
│  └────┬────┘ └────┬────┘ └────┬────┘│
│       └──────────┼──────────┘      │
│            Internal Events          │
│                  │                  │
│           ┌──────┴──────┐          │
│           │  Shared DB  │          │
│           │ (schema     │          │
│           │  per module)│          │
│           └─────────────┘          │
└─────────────────────────────────────┘
```

### Преимущества

| Аспект | Монолит | Modular Monolith | Микросервисы |
|--------|---------|------------------|--------------|
| Сложность деплоя | ✅ Низкая | ✅ Низкая | ❌ Высокая |
| Границы модулей | ❌ Нет | ✅ Чёткие | ✅ Чёткие |
| Изоляция команд | ❌ Нет | ⚠️ Частичная | ✅ Полная |
| Путь к микросервисам | ❌ Сложно | ✅ Подготовлен | — |
| Инфраструктура | ✅ Минимальная | ✅ Минимальная | ❌ Сложная |

### Фреймворки и инструменты (2024-2025)

**Java/Kotlin:**
- **Spring Modulith** — официальная поддержка от Spring (рекомендуется)
- **ArchUnit** — проверка архитектурных правил в тестах
- **jMolecules** — аннотации для DDD концепций

**Другие языки:**
- **Light-hybrid-4j** — Java, lightweight framework
- **Modular monolith** паттерны есть в NestJS (Node.js), Django (Python)

```kotlin
// Spring Modulith — пример структуры
app/
├── orders/           // @ApplicationModule
│   ├── OrderService
│   ├── OrderRepository
│   └── api/         // публичный API модуля
├── payments/         // @ApplicationModule
│   ├── PaymentService
│   └── api/
└── shared/          // общие компоненты
```

### Кто использует

| Компания | Примечание |
|----------|------------|
| **Shopify** | Крупнейший e-commerce, сознательно остаётся на монолите |
| **Gusto** | Payroll платформа, modular monolith с путём к микросервисам |
| **Appsmith** | Open-source low-code, modular структура |
| **PlayTech** | Gaming платформа |

### Правила модульного монолита

```
1. Модули общаются только через Public API
   ❌ OrderService вызывает PaymentRepository напрямую
   ✅ OrderService вызывает PaymentService.processPayment()

2. Каждый модуль владеет своими данными
   ❌ Один модуль читает таблицы другого
   ✅ Separate schemas + API для доступа

3. Events для loose coupling
   ❌ OrderService напрямую уведомляет NotificationService
   ✅ OrderService публикует OrderCreatedEvent

4. Архитектурные тесты обязательны
   ✅ ArchUnit проверяет границы модулей в CI
```

## Практическое руководство: как выбирать

### Чеклист "Нужны ли нам микросервисы?"

```
☐ У нас больше 50 разработчиков?
☐ Команды часто блокируют друг друга в коде?
☐ Разные части системы имеют разную нагрузку?
☐ Мы можем выделить инфраструктурную команду (2-3 человека)?
☐ У нас уже есть проблемы масштабирования, а не "могут появиться"?
☐ У нас есть опыт с distributed systems?
```

**Если да на 4+ пункта** — микросервисы могут иметь смысл.
**Если да на 0-2** — оставайтесь на монолите, инвестируйте в его качество.

### Эволюционный подход

1️⃣ **Старт:** Монолит. Всегда. Не спорьте.

2️⃣ **Рост:** Модульный монолит. Чёткие границы, но один деплой.

3️⃣ **Масштаб:** Выделяйте сервисы по мере необходимости. Сначала то, что реально нужно масштабировать отдельно.

*Это называется "Evolutionary Architecture" — архитектура эволюционирует вместе с потребностями, а не предвосхищает их.*

## Типичные ошибки

### "Мы стартап, но сразу на микросервисах"

Почему плохо: Вы тратите 70% времени на инфраструктуру вместо продукта. А потом пивотитесь — и переписываете все сервисы.

### "У Netflix/Uber микросервисы, значит и нам надо"

Почему плохо: Карго-культ. У них были конкретные проблемы, которые они решали. У вас — скорее всего другие.

### "Мы заранее разобьём, чтобы потом было легче"

Почему плохо: Вы не знаете, где будут реальные границы. Границы определяются практикой, а не теорией. Преждевременное разделение создаёт больше проблем, чем решает.

### "Микросервисы = современно, монолит = legacy"

Почему плохо: Это маркетинговый миф. Хорошо написанный монолит лучше плохо написанных микросервисов. Архитектура — это инструмент, а не статус.

## Distributed Monolith: худший из миров

**Distributed Monolith** — антипаттерн, когда вы получили все минусы микросервисов без их плюсов.

### Признаки Distributed Monolith

```
❌ Сервисы нельзя деплоить независимо
   → Изменение в Service A требует редеплоя Service B и C

❌ Общая база данных
   → Все сервисы ходят в одну БД, нет изоляции

❌ Синхронные зависимости везде
   → A вызывает B, B вызывает C, C вызывает A (circular deps)

❌ Shared libraries с бизнес-логикой
   → Изменение shared lib → редеплой всего

❌ Координированные релизы
   → "Мы релизим по пятницам все вместе"
```

### Как это происходит

```
Шаг 1: "Давайте разобьём монолит на сервисы!"
Шаг 2: Разбиваем по техническим слоям (API, Business, Data)
Шаг 3: Сервисы сильно связаны, общая БД
Шаг 4: Distributed Monolith готов 🎉

Правильно: разбивать по bounded contexts (DDD)
```

### Решения

| Проблема | Решение |
|----------|---------|
| Общая БД | Database-per-service + API для доступа к данным |
| Синхронные вызовы | Event-driven architecture, async messaging |
| Circular deps | Анализ boundaries, выделение shared kernel |
| Shared libs | Только utility libs, бизнес-логика в сервисах |

## Actionable: что делать прямо сейчас

**Если вы стартап:**
- Пишите монолит
- Инвестируйте в модульность и тесты
- Не слушайте тех, кто говорит "потом переписывать дороже"

**Если у вас растущий монолит:**
- Выделите модули с чёткими границами
- Используйте внутренние API между модулями
- Документируйте зависимости
- Это подготовит вас к разделению когда оно понадобится

**Если у вас проблемы масштабирования:**
- Определите, какая часть системы реально bottleneck
- Выделите именно её в отдельный сервис
- Не трогайте то, что работает

## Проверь себя

> [!question]- Amazon Prime Video сэкономил 90% затрат, вернувшись к монолиту. Означает ли это, что микросервисы были ошибкой? Какой архитектурный принцип здесь нарушили?
> Нет, микросервисы не были ошибкой изначально — ошибкой было применение микросервисов к задаче, которая не требовала независимого масштабирования частей. Video monitoring — это pipeline (последовательная обработка), а не набор независимых доменов. Нарушен принцип: разбивать по bounded contexts (бизнес-доменам), а не по техническим слоям. Step Functions + S3 между шагами создавали latency и стоимость при отсутствии реальной выгоды от разделения. Вывод: архитектура должна соответствовать характеру задачи, а не тренду.

> [!question]- Ваш монолит (15 разработчиков) начинает "болеть": билд 12 минут, merge conflicts ежедневно, Payment модуль требует ×5 масштабирования. Что делать: сразу микросервисы или модульный монолит?
> Модульный монолит: 1) Выделить Payment, Orders, Users в модули с чёткими публичными API (Spring Modulith / ArchUnit). 2) Разделить schema per module — каждый модуль владеет своими таблицами. 3) Добавить internal events для loose coupling. 4) Только после этого выделить Payment в отдельный сервис, если масштабирование ×5 реально наблюдается. Ключевая ошибка: разбивать "на будущее" — границы модулей определяются практикой. Merge conflicts решаются модульностью, не сервисами.

> [!question]- Как отличить настоящие микросервисы от Distributed Monolith? Назовите 3 ключевых признака антипаттерна и способы их устранения.
> **Distributed Monolith признаки:** 1) Координированные релизы — "деплоим все вместе по пятницам" (fix: независимые контракты через API versioning). 2) Общая БД — сервисы читают чужие таблицы напрямую (fix: database-per-service + API для данных). 3) Circular dependencies — A→B→C→A (fix: event-driven decoupling, выделение shared kernel). Дополнительно: shared libraries с бизнес-логикой = скрытый coupling. Правило: если нельзя деплоить сервис независимо — это не микросервис.

> [!question]- 29% компаний вернулись с микросервисов на монолит (O'Reilly 2024). Какие три главные причины и как их можно было предотвратить?
> 1) **Недооценка distributed systems** — timeout, retry, circuit breaker, eventual consistency — это не "потом добавим", а фундамент. Предотвращение: инвестировать в инфраструктурную команду ДО миграции. 2) **Latency:** function call ~1ns vs HTTP call ~1-10ms (×1M разница). Предотвращение: идентифицировать hot paths и оставить их внутри одного сервиса. 3) **Debugging hell:** запрос через 6 сервисов, где ошибка? Предотвращение: distributed tracing (Jaeger/Zipkin) и centralized logging с самого начала, а не после проблем.

## Ключевые карточки

В чём суть антипаттерна Distributed Monolith?
?
Все минусы микросервисов (сетевые вызовы, сложность) без плюсов (независимый деплой). Признаки: общая БД, синхронные circular deps, координированные релизы, shared libs с бизнес-логикой.

Чеклист: когда микросервисы оправданы? (4+ из 6)
?
1) >50 разработчиков. 2) Команды блокируют друг друга. 3) Разная нагрузка на части ×10+. 4) Есть инфраструктурная команда (2-3 человека). 5) РЕАЛЬНЫЕ проблемы масштабирования. 6) Опыт с distributed systems. Менее 4 — оставайтесь на монолите.

Что такое Evolutionary Architecture?
?
Архитектура эволюционирует вместе с потребностями: Монолит → Модульный монолит → Выделение сервисов по необходимости. Не предвосхищать проблемы, а решать реальные. "Start with monolith, split when it hurts."

Какой overhead дают микросервисы по сравнению с монолитом?
?
Latency: function call ~1ns vs HTTP ~1-10ms (×1M). Debugging: +35% времени (DZone 2024). TCO: +50-100% (FinOps 2025). Инфраструктура: Service Discovery + API Gateway + Load Balancer + Message Queue + Tracing + Logging = 6+ систем.

4 правила модульного монолита?
?
1) Модули общаются только через Public API. 2) Каждый модуль владеет своими данными (separate schemas). 3) Events для loose coupling (не прямые вызовы). 4) Архитектурные тесты (ArchUnit) в CI проверяют границы.

Модульный монолит vs монолит vs микросервисы — сложность деплоя?
?
Монолит: низкая (1 артефакт). Modular Monolith: низкая (1 артефакт, модули внутри). Микросервисы: высокая (N артефактов, service discovery, API gateway). Modular Monolith даёт границы модулей при простоте деплоя.

Почему Shopify остаётся на монолите при миллиардах транзакций?
?
Инвестировали в модульность внутри монолита, а не в разделение на сервисы. Результат: простота деплоя, единая кодовая база, меньше инфраструктурных затрат. Хорошо структурированный монолит лучше плохих микросервисов.
