# API Design: REST, GraphQL, gRPC

> REST — простота и HTTP кэширование. GraphQL — гибкость и точные запросы. gRPC — 5-7x быстрее REST для микросервисов. В 2025 году гибридные стеки — норма: REST для публичных API, gRPC внутри платформы.

---

## Теоретические основы

### Что такое API формально

> **API (Application Programming Interface)** — формальный контракт, определяющий набор операций, их входы, выходы и побочные эффекты, через которые одна программная система взаимодействует с другой.

API — это **абстракция**: клиент знает **что** может сделать, но не знает **как** это реализовано. Это прямое применение принципа **information hiding** (Parnas, 1972).

### Историческая эволюция удалённых API

| Парадигма | Период | Ключевая идея | Автор/Стандарт |
|-----------|--------|---------------|----------------|
| **RPC** | 1976-1990s | Удалённый вызов как локальный | Birrell & Nelson, 1984 |
| **CORBA** | 1991-2000s | Объекты через сеть (IDL) | OMG, 1991 |
| **SOAP/WSDL** | 1998-2010s | XML-сообщения, строгие контракты | W3C, 1998 |
| **REST** | 2000-н.в. | Ресурсы + HTTP глаголы + HATEOAS | Fielding, 2000 (PhD thesis) |
| **GraphQL** | 2015-н.в. | Клиент описывает, что хочет получить | Facebook, 2012 (internal), 2015 (open) |
| **gRPC** | 2015-н.в. | Binary protocol (protobuf) + HTTP/2 | Google, 2015 |
| **tRPC** | 2021-н.в. | End-to-end type safety без schema | Colin McDonnell, 2021 |

### REST: архитектурный стиль Fielding

REST (Representational State Transfer) — **не протокол, а архитектурный стиль**, определённый Roy Fielding в его докторской диссертации (2000). Fielding описал 6 ограничений:

| Ограничение | Суть | Зачем |
|-------------|------|-------|
| **Client-Server** | Разделение интерфейса и хранения | Независимая эволюция |
| **Stateless** | Каждый запрос содержит всю информацию | Масштабируемость |
| **Cacheable** | Ответы помечены как кэшируемые/нет | Производительность |
| **Layered System** | Клиент не знает, с кем говорит (proxy, LB) | Масштабируемость |
| **Uniform Interface** | Единый интерфейс: ресурсы, представления, self-descriptive messages, HATEOAS | Простота и decoupling |
| **Code on Demand** (опц.) | Сервер может отправлять исполняемый код | Расширяемость |

> **HATEOAS** (Hypermedia As The Engine Of Application State) — наименее реализуемое ограничение REST. Fielding неоднократно подчёркивал, что без HATEOAS API не является RESTful, но на практике большинство "REST API" — это HTTP API с JSON, не реализующие HATEOAS.

### Richardson Maturity Model (2008)

Leonard Richardson предложил модель зрелости REST API:

```
Уровень 3:  Hypermedia Controls (HATEOAS)     ← "настоящий" REST
Уровень 2:  HTTP Verbs (GET, POST, PUT, DELETE) ← большинство "REST" API
Уровень 1:  Resources (/users, /orders)
Уровень 0:  The Swamp of POX (единый endpoint)  ← SOAP, XML-RPC
```

### GraphQL: теоретические основы

> **GraphQL** — язык запросов для API и среда выполнения. Клиент описывает **форму** нужных данных (типизированный граф), сервер возвращает данные ровно этой формы.

GraphQL основан на **теории графов**: schema — это типизированный ориентированный граф, query — это **подграф** schema, result — изоморфен структуре query.

**Формальная модель:** GraphQL schema = `(Types, Fields, Resolvers)`, где каждый `Field: Type → Type` — ребро в графе типов.

### gRPC: формальная модель

> **gRPC** использует **Interface Definition Language** (Protocol Buffers) для описания контракта, **HTTP/2** для транспорта, и поддерживает 4 паттерна взаимодействия:

| Паттерн | Описание | Формально |
|---------|----------|-----------|
| **Unary** | Запрос → Ответ | `f: Request → Response` |
| **Server Streaming** | Запрос → Поток ответов | `f: Request → Stream<Response>` |
| **Client Streaming** | Поток запросов → Ответ | `f: Stream<Request> → Response` |
| **Bidirectional** | Поток ↔ Поток | `f: Stream<Request> → Stream<Response>` |

### Принципы проектирования API (Hyrum's Law)

> **Hyrum's Law** (Hyrum Wright, ~2012): "При достаточном количестве пользователей API, не имеет значения, что вы обещаете в контракте: все наблюдаемые поведения системы будут зависимостью для кого-то."

Следствие: **любое изменение** API потенциально ломает кого-то. Отсюда — необходимость версионирования, deprecation policy, и чётких compatibility guarantees.

## Зачем это нужно

### Проблема: Неправильный выбор = переписывание через год

Неправильный выбор API-архитектуры приводит к:
- **Переписыванию** через 1-2 года
- **Performance bottlenecks** на масштабе
- **Сложностям интеграции** для клиентов

### Реальные примеры компаний

**Stripe (REST):**
- Public API для платежей
- Используют REST + JSON для простоты интеграции
- Date-based versioning: `Stripe-Version: 2024-06-20`
- Документация — золотой стандарт индустрии

**GitHub (REST + GraphQL):**
- REST API v3 — для простых операций
- GraphQL API v4 — для сложных запросов с nested data
- Rate limits: 60 req/час без auth, 5,000 с auth

**Netflix (gRPC):**
- gRPC между микросервисами
- REST/GraphQL для клиентских приложений
- Результат: миллисекундные latency внутри платформы

**Twilio (REST):**
- REST API для SMS/Voice
- Дают год предупреждения перед deprecation версии
- POST вместо PUT для обновлений (не strict REST)

### Бенчмарки 2024-2025

| Метрика | REST | GraphQL | gRPC |
|---------|------|---------|------|
| Latency | 250ms | 180ms (complex) | **25ms** |
| Throughput | 20k RPS | 15k RPS | **50k RPS** |
| CPU usage | Baseline | +20% | **-40%** |
| Payload size | 100% (JSON) | 70-100% | **10-30% (Protobuf)** |

### Hyrum's Law — Закон неизбежности

> "При достаточном количестве пользователей API, неважно что вы обещаете в контракте — от всех наблюдаемых поведений системы кто-то будет зависеть."

**Что это значит на практике:**
```
Вы документировали:
  "Поле created_at содержит дату создания"

Вы НЕ документировали:
  "Формат даты: ISO 8601"
  "Timezone: UTC"
  "Поле всегда присутствует"

Но какой-то клиент уже зависит от того, что:
- Формат всегда "2024-01-15T10:30:00Z"
- Timezone всегда UTC
- Поле никогда не null

Вы решили "улучшить" формат → сломали этого клиента.
```

**Следствия:**
- Даже недокументированное поведение становится API
- Breaking changes ломают больше чем ожидалось
- Backwards compatibility важнее новых фич

## Терминология (с аналогиями)

| Термин | Значение | Аналогия |
|--------|----------|----------|
| **API** | Application Programming Interface | Меню ресторана |
| **Endpoint** | URL для запроса | Отдел в магазине (молочный, мясной) |
| **Resource** | Сущность в REST (users, orders) | Товар на полке |
| **HTTP verb** | GET, POST, PUT, DELETE | Действие: взять, положить, заменить, выбросить |
| **Request** | Запрос от клиента | Заказ официанту |
| **Response** | Ответ от сервера | Блюдо на тарелке |
| **Status code** | Код результата (200, 404, 500) | "Готово", "Нет в наличии", "Пожар на кухне" |
| **Payload** | Данные в запросе/ответе | Содержимое заказа |
| **Schema** | Описание структуры данных | Рецепт блюда |
| **OAuth** | Протокол авторизации | Пропускная система |
| **JWT** | JSON Web Token | Пропуск с QR-кодом |
| **Rate limit** | Ограничение запросов | "Не более 3 блюд в час" |

## Три парадигмы: когда какую выбирать

### REST — "Ресурсы и HTTP-глаголы"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              REST                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Аналогия: Библиотека с каталогом                                     │
│                                                                         │
│   • Каждая книга (ресурс) имеет уникальный номер (ID)                  │
│   • Действия: взять (GET), вернуть (DELETE), заказать (POST)           │
│   • Каталог (URL) организован логично: /полка/ряд/книга                │
│                                                                         │
│   GET    /users/123        → Получить пользователя 123                 │
│   POST   /users            → Создать нового пользователя               │
│   PUT    /users/123        → Полностью обновить пользователя 123       │
│   PATCH  /users/123        → Частично обновить пользователя 123        │
│   DELETE /users/123        → Удалить пользователя 123                  │
│                                                                         │
│   ✓ Простота — любой разработчик поймёт                               │
│   ✓ HTTP кэширование из коробки                                        │
│   ✓ Стандарты (OpenAPI, JSON Schema)                                   │
│                                                                         │
│   ✗ Over-fetching: GET /users/123 возвращает ВСЕ поля                  │
│   ✗ Under-fetching: нужно 3 запроса для связанных данных              │
│   ✗ Множество endpoints                                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### GraphQL — "Запроси только то, что нужно"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            GraphQL                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Аналогия: Шведский стол                                              │
│                                                                         │
│   • Вы сами выбираете что положить на тарелку                          │
│   • Берёте только то, что хотите съесть                                │
│   • Один подход к столу = один запрос                                  │
│                                                                         │
│   query {                                                               │
│     user(id: 123) {                                                     │
│       name            ← Только имя                                      │
│       email           ← и email                                         │
│       posts {         ← Вложенные данные                               │
│         title         ← в одном запросе!                                │
│       }                                                                 │
│     }                                                                   │
│   }                                                                     │
│                                                                         │
│   ✓ Точные запросы — никакого over-fetching                           │
│   ✓ Один запрос для связанных данных                                   │
│   ✓ Строгая типизация (schema)                                         │
│                                                                         │
│   ✗ Сложнее кэшировать (один endpoint для всего)                       │
│   ✗ N+1 проблема на сервере (решается DataLoader)                      │
│   ✗ Выше порог входа для команды                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### gRPC — "Binary protocol для микросервисов"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              gRPC                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Аналогия: Внутренняя связь на заводе                                 │
│                                                                         │
│   • Станки "говорят" на специальном языке (Protocol Buffers)           │
│   • Очень быстро, но люди не понимают без "переводчика"                │
│   • Двусторонняя связь (streaming)                                     │
│                                                                         │
│   // users.proto                                                        │
│   service UserService {                                                 │
│     rpc GetUser(UserRequest) returns (User);                           │
│     rpc StreamUsers(Empty) returns (stream User);  // Streaming!       │
│   }                                                                     │
│                                                                         │
│   message User {                                                        │
│     int64 id = 1;                                                       │
│     string name = 2;                                                    │
│   }                                                                     │
│                                                                         │
│   ✓ 5-7x быстрее REST (binary + HTTP/2)                                │
│   ✓ Bidirectional streaming                                             │
│   ✓ Автогенерация клиентов на 10+ языках                               │
│                                                                         │
│   ✗ Не работает напрямую в браузерах (нужен proxy)                     │
│   ✗ Сложнее дебажить (бинарный формат)                                 │
│   ✗ Выше порог входа                                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Матрица выбора

| Сценарий | Рекомендация | Почему |
|----------|--------------|--------|
| Public API, 3rd party devs | **REST** + OpenAPI | Стандарты, кэширование, простота |
| Mobile app с сложным UI | **GraphQL** | Точные запросы, один roundtrip |
| Микросервисы, внутренняя связь | **gRPC** | 5-7x быстрее, streaming, типизация |
| Real-time, bidirectional | **gRPC** / WebSocket | Streaming |
| Event-driven, webhooks | **REST** + AsyncAPI | Простота, стандарты |
| MVP, быстрый старт | **REST** | Минимальный порог входа |

## REST: делаем правильно

### Именование ресурсов

```
❌ Плохо (глаголы в URL):
GET  /getUsers              ← GET уже означает "получить"
GET  /user/get/123          ← Избыточно
POST /createNewUser         ← POST уже означает "создать"
GET  /users/123/getOrders   ← Избыточно

✅ Хорошо (существительные, множественное число):
GET    /users               → Список пользователей
GET    /users/123           → Конкретный пользователь
POST   /users               → Создать пользователя
PUT    /users/123           → Обновить пользователя
DELETE /users/123           → Удалить пользователя
GET    /users/123/orders    → Заказы пользователя

Правила:
────────
1. Существительные во множественном числе: /users, /products, /orders
2. Иерархия через вложенность: /users/123/orders/456
3. HTTP-глагол определяет действие, не URL
4. Не более 2-3 уровней вложенности
5. Используйте kebab-case: /user-profiles, не /userProfiles
```

### HTTP-статусы: шпаргалка

```
│                      HTTP STATUS CODES                                  │

│                                                                         │
```


### Пагинация

```json
// =====================================================================
// OFFSET-BASED (простая, но проблемная на больших данных)
// =====================================================================
//
// Проблема: При offset=100000 БД должна пропустить 100000 записей
// Это ОЧЕНЬ медленно на больших таблицах

GET /users?offset=100&limit=20

// Response
{
  "data": [...],
  "pagination": {
    "total": 1523,
    "offset": 100,
    "limit": 20
  }
}

// =====================================================================
// CURSOR-BASED (быстрая, рекомендуется)
// =====================================================================
//
// Cursor = зашифрованный указатель на последнюю запись
// БД использует индекс, не нужно пропускать записи
//
// Как это работает:
// 1. Клиент запрашивает первую страницу
// 2. Сервер возвращает данные + next_cursor (закодированный ID последней записи)
// 3. Клиент запрашивает следующую страницу с этим cursor
// 4. Сервер декодирует cursor и делает WHERE id > cursor_id LIMIT 20

GET /users?cursor=eyJpZCI6MTAwfQ&limit=20
```

### N+1 проблема (критично!)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       N+1 ПРОБЛЕМА                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Запрос:                                                                │
│  ────────                                                               │
│  query {                                                                │
│    users {          ← Получить всех пользователей                       │
│      name                                                               │
│      posts {        ← Для КАЖДОГО пользователя получить посты          │
│        title                                                            │
│      }                                                                  │
│    }                                                                    │
│  }                                                                      │
│                                                                         │
│  ─────────────────────────────────────────────────────────────────────  │
│                                                                         │
│  ❌ БЕЗ DataLoader (N+1 запросов):                                      │
│                                                                         │
│  SELECT * FROM users                     → 1 запрос (получили 100 users)│
│  SELECT * FROM posts WHERE user_id = 1   → 1 запрос                     │
│  SELECT * FROM posts WHERE user_id = 2   → 1 запрос                     │
│  SELECT * FROM posts WHERE user_id = 3   → 1 запрос                     │
│  ...                                                                    │
│  SELECT * FROM posts WHERE user_id = 100 → 1 запрос                     │
│                                                                         │
│  Итого: 1 + 100 = 101 запрос к БД! 🔥                                  │
│                                                                         │
│  ─────────────────────────────────────────────────────────────────────  │
│                                                                         │
│  ✅ С DataLoader (2 запроса):                                           │
│                                                                         │
│  SELECT * FROM users                                   → 1 запрос       │
│  SELECT * FROM posts WHERE user_id IN (1,2,3,...,100)  → 1 запрос       │
│                                                                         │
│  Итого: 2 запроса! ⚡                                                   │
│                                                                         │
│  ─────────────────────────────────────────────────────────────────────  │
│                                                                         │
│  DataLoader собирает все ID за один "тик" event loop                   │
│  и делает один batch запрос вместо N отдельных.                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

```javascript
// =============================================================
// DATALOADER: Решение N+1 проблемы
// =============================================================

const DataLoader = require('dataloader');

// Создаём loader для постов
const postLoader = new DataLoader(async (userIds) => {
  // userIds = [1, 2, 3, ..., 100] — все ID, собранные за один тик

  // Один запрос вместо 100
  const posts = await db.query(
    'SELECT * FROM posts WHERE user_id IN ($1)',
    [userIds]
  );

  // DataLoader требует вернуть результаты в том же порядке, что и входные ID
  const postsByUserId = {};
  posts.forEach(post => {
    if (!postsByUserId[post.user_id]) {
      postsByUserId[post.user_id] = [];
    }
    postsByUserId[post.user_id].push(post);
  });

  // Возвращаем в порядке userIds
  return userIds.map(id => postsByUserId[id] || []);
});

// Использование в resolver
const resolvers = {
  User: {
    posts: (user) => postLoader.load(user.id)
    // Каждый вызов .load() не делает запрос сразу
    // DataLoader собирает все ID и делает один batch запрос
  }
};
```

### JWT: Почему каждая практика важна

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    JWT SECURITY: ПОЧЕМУ ЭТО ВАЖНО                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  JWT = Header.Payload.Signature (три части через точку)                │
│  eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiIxMjMifQ.signature_here                │
│                                                                         │
│  Структура:                                                             │
│  ──────────                                                             │
│  {                                                                      │
│    "header": {                                                          │
│      "alg": "RS256",  // Алгоритм подписи                              │
│      "typ": "JWT"                                                       │
│    },                                                                   │
│    "payload": {                                                         │
│      "sub": "user_123",       // Subject (кто)                         │
│      "iat": 1700000000,       // Issued At (когда выдан)               │
│      "exp": 1700003600,       // Expiration (когда истекает)           │
│      "scope": "read:users"    // Разрешения                            │
│    },                                                                   │
│    "signature": "..."         // Криптографическая подпись             │
│  }                                                                      │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ПРАКТИКА 1: Короткий срок жизни (15 мин - 1 час)                      │
│  ─────────────────────────────────────────────────                     │
│                                                                         │
│  ПОЧЕМУ: JWT нельзя отозвать после выдачи!                             │
│                                                                         │
│  Сценарий:                                                              │
│  1. Пользователь залогинился, получил JWT на 30 дней                   │
│  2. Пользователь уволился из компании                                   │
│  3. Админ деактивировал аккаунт                                         │
│  4. JWT ВСЁ ЕЩЁ РАБОТАЕТ 30 дней! 🔥                                   │
│                                                                         │
│  Решение:                                                               │
│  • Access Token: 15 минут - 1 час                                       │
│  • Refresh Token: 7-30 дней (хранится в secure httpOnly cookie)        │
│  • При каждом refresh — проверяем актуальность пользователя            │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ПРАКТИКА 2: Используйте RS256, не HS256                               │
│  ───────────────────────────────────────                               │
│                                                                         │
│  HS256 (HMAC):                                                          │
│  • Один секретный ключ для подписи И проверки                          │
│  • Если ключ утёк — можно подделать любой токен                        │
│  • Ключ должен быть на всех серверах → больше точек утечки             │
│                                                                         │
│  RS256 (RSA):                                                           │
│  • Приватный ключ подписывает (только на auth server)                  │
│  • Публичный ключ проверяет (можно раздать всем)                       │
│  • Утечка публичного ключа ≠ возможность подделки                      │
│                                                                         │
│  Атака "Algorithm Confusion":                                           │
│  Злоумышленник меняет header: {"alg": "HS256"}                         │
│  И подписывает публичным ключом как HMAC                               │
│  Если сервер не проверяет алгоритм строго — токен примется!            │
│                                                                         │
│  Защита: ВСЕГДА явно указывайте ожидаемый алгоритм при верификации     │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ПРАКТИКА 3: Не храните sensitive данные в JWT                         │
│  ─────────────────────────────────────────────                         │
│                                                                         │
│  JWT НЕ зашифрован, только подписан!                                   │
│  Любой может декодировать Base64 и увидеть payload.                    │
│                                                                         │
│  ❌ НЕ хранить:                                                         │
│  • Пароли (очевидно)                                                   │
│  • Персональные данные (GDPR нарушение)                                │
│  • Секреты, API ключи                                                  │
│  • Данные кредитных карт                                               │
│                                                                         │
│  ✅ Хранить:                                                            │
│  • User ID (sub claim)                                                 │
│  • Роли / Permissions                                                   │
│  • Tenant ID (для multi-tenant)                                         │
│  • Expiration                                                           │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  УЯЗВИМОСТЬ: "none" Algorithm Attack                                   │
│  ─────────────────────────────────                                     │
│                                                                         │
│  JWT спецификация разрешает алгоритм "none" (без подписи)              │
│                                                                         │
│  Атака:                                                                 │
│  1. Злоумышленник берёт легитимный JWT                                 │
│  2. Меняет header: {"alg": "none"}                                     │
│  3. Меняет payload: {"sub": "admin", "role": "superuser"}              │
│  4. Убирает signature                                                   │
│  5. Если сервер не проверяет — токен принимается                       │
│                                                                         │
│  Защита: Явный whitelist алгоритмов, НИКОГДА не принимать "none"       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

```javascript
// =============================================================
// JWT ВЕРИФИКАЦИЯ: Правильный способ
// =============================================================

const jwt = require('jsonwebtoken');

// ❌ ОПАСНО: Алгоритм берётся из токена
const decoded = jwt.verify(token, publicKey);  // ПЛОХО!

// ✅ БЕЗОПАСНО: Явно указываем разрешённые алгоритмы
const decoded = jwt.verify(token, publicKey, {
  algorithms: ['RS256'],        // Только RS256!
  issuer: 'https://auth.example.com',  // Проверяем issuer
  audience: 'my-api',           // Проверяем audience
  clockTolerance: 30            // 30 сек допуск на расхождение часов
});

// Дополнительные проверки после верификации:
if (decoded.exp < Date.now() / 1000) {
  throw new Error('Token expired');
}

// Проверка, что пользователь ещё активен (не заблокирован)
const user = await db.users.findById(decoded.sub);
if (!user || user.status !== 'active') {
  throw new Error('User not active');
}
```

### Rate Limiting

```javascript
// =============================================================
// RATE LIMITING: Защита от злоупотреблений
// =============================================================

const rateLimit = require('express-rate-limit');

// Общий лимит для всех endpoints
const generalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 минут
  max: 100,                   // 100 запросов за окно

  // Ответ при превышении лимита
  message: {
    error: {
      code: 'RATE_LIMIT_EXCEEDED',
      message: 'Too many requests, please try again later',
      retryAfter: 15 * 60     // Секунд до сброса
    }
  },

  // Стандартные заголовки (RFC 6585)
  standardHeaders: true,     // RateLimit-* headers
  legacyHeaders: false,      // Отключаем X-RateLimit-*

  // Как идентифицировать клиента
  keyGenerator: (req) => {
    // По API ключу если есть, иначе по IP
    return req.headers['x-api-key'] || req.ip;
  }
});

// Строгий лимит для auth endpoints (защита от brute force)
const authLimiter = rateLimit({
  windowMs: 60 * 60 * 1000,  // 1 час
  max: 5,                     // 5 попыток
  message: {
    error: {
      code: 'AUTH_RATE_LIMIT',
      message: 'Too many login attempts, account locked for 1 hour'
    }
  }
});

app.use('/api/', generalLimiter);
app.use('/api/auth/login', authLimiter);

// Response headers:
// RateLimit-Limit: 100
// RateLimit-Remaining: 42
// RateLimit-Reset: 1700000000
```

| Breaking changes | Ломают всех клиентов | Deprecation + новая версия |
| Offset пагинация | Медленно на больших данных | Cursor-based |

## Проверь себя

> [!question]- Компания запускает публичный API для партнёров и одновременно строит внутреннюю микросервисную платформу. Для партнёрского API выбрали GraphQL, для внутренней коммуникации — REST. Оцените это решение: какие проблемы возникнут и какой выбор был бы оптимальным?
> Решение неоптимально в обоих случаях. Для публичного партнёрского API лучше подходит REST: он проще для интеграции сторонними разработчиками, имеет HTTP-кэширование из коробки, низкий порог входа и зрелую экосистему (OpenAPI). GraphQL создаёт лишнюю сложность для партнёров (нужно изучать SDL, query language) и требует дополнительной защиты (depth limiting, cost analysis). Для внутренних микросервисов лучше подходит gRPC: он в 5-7 раз быстрее REST благодаря бинарному протоколу и HTTP/2, поддерживает bidirectional streaming, автогенерацию клиентов и строгую типизацию через Protobuf. REST между микросервисами — это overhead на сериализацию JSON и текстовый протокол. Оптимальный стек: REST для публичного API + gRPC для внутренней платформы.

> [!question]- Почему cursor-based пагинация рекомендуется вместо offset-based, хотя offset проще в реализации? В каких случаях offset-based всё же приемлема?
> Причина — производительность на масштабе. При offset=100000 БД должна физически пропустить 100000 записей (sequential scan), что становится всё медленнее с ростом данных. Cursor-based использует `WHERE id > cursor_id LIMIT N`, что работает через индекс и имеет постоянную скорость O(log N) независимо от позиции. Offset-based приемлема когда: (1) данных мало (до ~10K записей), (2) нужна навигация по номерам страниц (cursor не поддерживает "перейти на страницу 5"), (3) данные статичны и не обновляются часто (при добавлении/удалении записей offset пагинация теряет элементы или дублирует их).

> [!question]- Вы унаследовали API, которое на все ошибки возвращает HTTP 200 со структурой `{"success": false, "error": "Something went wrong"}`. Клиенты уже зависят от этого поведения (Hyrum's Law). Как исправить ситуацию, не сломав существующих клиентов?
> Нужна стратегия постепенной миграции. (1) Ввести версионирование API — новая версия (v2) возвращает правильные HTTP-статусы (400, 401, 404, 500) и консистентный формат ошибок с machine-readable кодом, сообщением, деталями и request_id. (2) Старую версию (v1) оставить как есть — по Hyrum's Law, клиенты могут зависеть не только от `success: false`, но и от формата строки ошибки. (3) Объявить deprecation v1 с конкретной датой отключения (6-12 месяцев). (4) Дать клиентам документацию по миграции с mapping старых ошибок на новые коды. (5) Мониторить использование v1 и уведомлять активных клиентов. Главный урок: Hyrum's Law делает исправление дороже с каждым днём, поэтому правильные статус-коды нужно закладывать с самого начала.

> [!question]- Почему для JWT рекомендуется алгоритм RS256 вместо HS256, если HS256 проще и быстрее? В каком единственном сценарии HS256 может быть приемлем?
> RS256 использует асимметричную криптографию: приватный ключ (подписывает) хранится только на auth-сервере, а публичный ключ (проверяет) можно безопасно раздать всем сервисам. Утечка публичного ключа не позволяет подделать токен. HS256 использует один секретный ключ для подписи и проверки — этот ключ должен быть на всех сервисах, что увеличивает поверхность атаки. Если ключ утёк с любого сервиса — можно подделать любой токен. Дополнительно, HS256 уязвим к Algorithm Confusion атаке: злоумышленник берёт публичный ключ RS256 и подписывает им токен как HMAC. HS256 приемлем только в одном сценарии: монолитное приложение с единственным сервером, где один и тот же процесс и подписывает, и проверяет токены — в этом случае нет необходимости распространять ключ.

## Ключевые карточки

Какую проблему решает GraphQL по сравнению с REST?
?
Over-fetching (REST возвращает все поля) и under-fetching (нужно несколько запросов для связанных данных). GraphQL позволяет клиенту запросить ровно те поля, которые нужны, в одном запросе.

Что такое N+1 проблема в GraphQL и как её решить?
?
При запросе вложенных данных (например, users → posts) GraphQL делает 1 запрос на список пользователей + N запросов на посты каждого пользователя. Решается через DataLoader, который собирает все ID за один тик event loop и делает один batch-запрос (`WHERE user_id IN (...)`).

В чём разница между HTTP-статусами 401 и 403?
?
401 Unauthorized — клиент не аутентифицирован («Я не знаю, кто вы»). 403 Forbidden — клиент аутентифицирован, но не имеет прав на данный ресурс («Я знаю, кто вы, но вам сюда нельзя»).

Почему JWT нужен короткий срок жизни (15 мин — 1 час)?
?
JWT нельзя отозвать после выдачи — он валиден до истечения срока. Если JWT выдан на 30 дней и аккаунт деактивирован, токен продолжит работать. Решение: короткий access token (15 мин) + refresh token (7-30 дней), при каждом refresh проверяется актуальность пользователя.

Когда выбирать gRPC вместо REST?
?
Для внутренней коммуникации между микросервисами, где важны производительность (gRPC в 5-7 раз быстрее REST), бинарный формат (Protobuf — 10-30% от размера JSON), bidirectional streaming и строгая типизация. gRPC не подходит для публичных API, так как не работает напрямую в браузерах.

Чем cursor-based пагинация лучше offset-based?
?
Offset при больших значениях заставляет БД пропускать тысячи записей (медленно). Cursor использует `WHERE id > cursor_id LIMIT N` через индекс — постоянная скорость независимо от позиции в данных.

Что такое Hyrum's Law и как она влияет на API-дизайн?
?
«При достаточном количестве пользователей все наблюдаемые поведения системы станут зависимостью для кого-то.» Это означает, что даже недокументированное поведение API (формат даты, порядок полей) — это фактический контракт. Breaking changes ломают больше, чем ожидается.

Какие три стратегии версионирования API существуют и какая рекомендуется?
?
URL path (`/v1/users`) — рекомендуется для публичных API, очевидна и удобна для кэширования. Header (`Stripe-Version: 2024-06-20`) — URL остаётся чистым, используют Stripe и GitHub. Query parameter (`?version=2`) — не рекомендуется, портит кэширование.
