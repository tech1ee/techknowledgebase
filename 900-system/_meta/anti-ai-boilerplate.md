# Запрет на AI-шаблоны и заполнители

> Этот файл — жёсткий фильтр на лексику и конструкции, которые делают текст похожим на «сгенерированную статью без автора». Применяется ко всему vault. Если материал содержит хотя бы один запрещённый маркер — он отправляется на переработку.

Правила сформулированы после сравнения эталонных deep-dive из [[big-o-complexity]], [[solid-principles]], [[bytecode-virtual-machines]] с типичной продукцией LLM без редактуры. Разница между текстом, который учит, и текстом, который заполняет пространство — именно в этих маркерах.

---

## Почему это важно отдельным документом

Стандарты [[depth-standard]] и [[cognitive-science-rules]] говорят про структуру и научную базу контента. Но даже при соблюдении структуры материал можно завалить лексикой: «модный, мощный, гибкий фреймворк», «стоит отметить», «в заключение». Такой текст формально правильный, но не учит — он создаёт иллюзию работы, не закрывая непонимание.

Этот файл отвечает за фильтр на уровне слов, а не на уровне скелета. Задача одна: чтобы каждое предложение удаляло конкретное непонимание у читателя, а не занимало строку.

---

## Позитивный критерий

**Правило удаления:** если абзац можно удалить без потери смысла, его не должно быть.

**Правило замены:** если два предложения передают одну мысль, остаётся одно.

**Правило конкретики:** каждое заявление либо опирается на источник (ссылка/цитата/цифра), либо снимает конкретное заблуждение читателя. Абзацы «о важности X вообще» запрещены.

Если фраза из материала могла бы стоять в любом другом материале на любую тему — она заполнитель и подлежит удалению.

## Запрещённые конструкции на английском

Ниже — полный чёрный список. При нахождении любой из этих фраз материал не публикуется, пока они не будут переписаны.

### Вступительные штампы

- 🚫 "In this article we will explore..."
- 🚫 "In this section we'll dive into..."
- 🚫 "Let's dive in / Let's take a look / Let's explore"
- 🚫 "In the world of [X]" / "In today's fast-paced world"
- 🚫 "When it comes to..." как открытие абзаца
- 🚫 "It's important to note that..." / "It is worth noting that..."
- 🚫 "Keep in mind that..." как вводное к очевидному
- 🚫 "By the way" / "Interestingly" / "Fun fact" как лексические костыли

### Заключительные штампы

- 🚫 "In conclusion..." / "To sum up..." / "In summary, we've seen..."
- 🚫 "Hopefully this helps!" / "I hope you found this useful"
- 🚫 "Thanks for reading!" / "Happy coding!"

### Маркетинговые прилагательные без содержания

- 🚫 "X is a modern, powerful, flexible framework that..."
- 🚫 "Cutting-edge / state-of-the-art / revolutionary / game-changing"
- 🚫 "Seamlessly / effortlessly / elegantly" без конкретной метрики
- 🚫 "Leverage / utilize" вместо "use"
- 🚫 "Robust and scalable solution"
- 🚫 "Best-in-class"

### Бесполезные связки

- 🚫 "As we can see..." / "As you can see..." / "As mentioned above..."
- 🚫 "As we'll see later..." / "We'll come back to this"
- 🚫 "Without further ado"
- 🚫 "Needless to say"
- 🚫 "At the end of the day"

## Запрещённые конструкции на русском

### Вступительные штампы

- 🚫 «Давайте разберёмся»
- 🚫 «Давайте рассмотрим»
- 🚫 «Итак, начнём»
- 🚫 «В этой статье мы рассмотрим...»
- 🚫 «В данном разделе мы изучим...»
- 🚫 «Стоит отметить, что...»
- 🚫 «Необходимо понимать, что...»
- 🚫 «Важно помнить, что...» как вводное к очевидному
- 🚫 «Как ни странно» / «Любопытно» / «Кстати»

### Заключительные штампы

- 🚫 «В заключение хочется сказать...»
- 🚫 «Подводя итог, можно сделать вывод...»
- 🚫 «Таким образом, мы рассмотрели...»
- 🚫 «Надеюсь, эта статья была полезна»

### Маркетинговая вода

- 🚫 «В современном мире / в эпоху цифровой трансформации»
- 🚫 «Мощный, гибкий, современный инструмент»
- 🚫 «Позволяет разработчикам легко и просто...»
- 🚫 «Революционный подход» / «Прорыв в индустрии»
- 🚫 «Не имеющий аналогов»

### Бесполезные связки

- 🚫 «Как мы уже упоминали...»
- 🚫 «Как мы увидим дальше...»
- 🚫 «Как было сказано выше...»
- 🚫 «Стоит отметить, что...» (повторно — частая болезнь)
- 🚫 «В любом случае» / «Так или иначе»

## Структурные антипаттерны

Кроме отдельных фраз, запрещены целые конструкции.

### Антипаттерн 1 — Парафраз заголовка

```markdown
🚫 ## Преимущества Kotlin

Kotlin имеет множество преимуществ, которые делают его хорошим выбором.
Среди преимуществ Kotlin можно выделить следующие:
...
```

Это повтор заголовка другими словами. После H2 должна сразу идти содержательная мысль: или тезис, или конкретный факт, или пример.

```markdown
✅ ## Преимущества Kotlin

Null safety на уровне типов сокращает NPE в production в 5–10 раз —
JetBrains собрала эту статистику на своих продуктах после миграции
с Java (2018).
```

### Антипаттерн 2 — Вводный абзац «ни о чём»

Пять строк про то, что «тема важная и популярная», перед первым содержательным утверждением. Если читатель откроет файл случайно, в первом абзаце должен быть инсайт, который заставит читать дальше. Не «тема Y набирает популярность», а «Y в 2024 заменил X по трём причинам: ...».

### Антипаттерн 3 — Списки-повторы

```markdown
🚫
Преимущества:
- Быстро
- Надёжно
- Удобно
- Эффективно
- Современно
```

Пункты без раскрытия — шум. Либо три пункта с объяснением каждого (пример/цифра/ссылка), либо удалить список совсем.

### Антипаттерн 4 — «Объяснение в объяснении»

Абзац, который поясняет предыдущий абзац, но добавляет ноль новой информации. Признак: если удалить второй абзац, первый читается так же. Оставляется только один.

### Антипаттерн 5 — Вечный «почему это важно»

Абстрактные размышления о том, почему X важен, без привязки к конкретной проблеме читателя. Важность доказывается одним конкретным сценарием с цифрой, а не тремя абзацами про «современный мир».

### Антипаттерн 6 — Код без текста вокруг него

Блок кода идёт подряд за заголовком без абзаца до и без разбора после. Нарушение [[depth-standard#Правило «Код как иллюстрация»]]. Код всегда обёрнут поясняющими абзацами.

### Антипаттерн 7 — Цепочка восклицательных перечислений

Признак: три подряд предложения с «это действительно важно», «это критически необходимо», «это существенно влияет». Усиления без обоснования = заполнитель.

### Антипаттерн 8 — Маркированный перевод документации

Если материал — буквальный пересказ официальной документации пункт-в-пункт, он не нужен. Ценность vault в синтезе, сравнении, исторической и архитектурной интерпретации, а не в дублировании Khronos или developer.android.com. Документацию читают по ссылке из `primary_sources`.

## Проверка: как отлавливать

Автоматический grep перед коммитом:

```bash
# Английские маркеры
grep -iE '(it.?s worth noting|let.?s dive|in the world of|in conclusion|keep in mind|cutting-edge|state-of-the-art)' file.md

# Русские маркеры
grep -iE '(стоит отметить|давайте разберёмся|в заключение|в современном мире|необходимо понимать)' file.md

# Любое из них → файл не готов к публикации
```

Финальная проверка — чтение вслух. Если при чтении возникает ощущение «этот абзац — общие слова», абзац удаляется.

## Хорошие примеры из vault

Эталоны, у которых нет ни одного маркера:

- [[big-o-complexity]] — вводит тему через конкретную цифру (3333× разница) в первом абзаце.
- [[solid-principles]] — начинается с года и имени, а не с «В мире программирования».
- [[bytecode-virtual-machines]] — исторический нарратив с датами вместо «современное состояние».

При написании нового материала открывай один из них рядом и смотри, как сделано открытие и связки. Копировать структуру — нормально; копировать фразы — нет.
