# Команда не доставляет результат

Диагностическое руководство для инженерных лидеров: определи корневую причину и получи план действий.

---

## Быстрая диагностика

```
Команда не доставляет
├── Процессные проблемы?
│   ├── Нет чёткого процесса → Раздел 1
│   └── Процесс есть, но не работает → Раздел 1
├── Проблемы коммуникации?
│   ├── Конфликты в команде → Раздел 2
│   └── Нет обратной связи → Раздел 2
├── Технический долг?
│   ├── Код невозможно менять → Раздел 3
│   └── Каждый фикс ломает другое → Раздел 3
├── Проблемы с людьми?
│   ├── Не хватает людей → Раздел 4
│   └── Люди уходят → Раздел 4
└── Проблемы лидерства?
    ├── Нечёткие приоритеты → Раздел 5
    └── Микроменеджмент → Раздел 5
```

---

## Раздел 1: Процессные проблемы

### Симптомы
- Задачи не имеют чётких сроков и ответственных
- Спринты срываются регулярно
- Нет понимания что в работе прямо сейчас
- Релизы болезненные и непредсказуемые

### Материалы
- [[development-process]] — выстраивание процесса разработки
- [[agile-practices]] — Agile-практики и их адаптация
- [[engineering-metrics]] — метрики для понимания здоровья процесса
- [[engineering-practices]] — инженерные практики (code review, CI/CD)
- [[ci-cd-pipelines]] — автоматизация релизов

### Действия
1. Установить прозрачность: доска задач, еженедельный статус
2. Определить Definition of Done
3. Ввести ретроспективы
4. Измерить cycle time — сколько задача идёт от начала до продакшена

---

## Раздел 2: Проблемы коммуникации

### Симптомы
- Конфликты между разработчиками
- Люди не делятся информацией
- Решения принимаются без обсуждения
- Обратная связь не даётся или даётся неконструктивно

### Материалы
- [[giving-feedback]] — как давать обратную связь
- [[receiving-feedback]] — как принимать обратную связь
- [[conflict-resolution]] — разрешение конфликтов
- [[async-communication]] — асинхронная коммуникация
- [[remote-team-communication]] — коммуникация удалённых команд
- [[saying-no]] — как говорить нет
- [[delivering-bad-news]] — как доносить плохие новости

### Действия
1. Ввести регулярные 1-1 с каждым членом команды
2. Установить нормы коммуникации (RFC, ADR)
3. Создать безопасную среду для обсуждений
4. Модерировать конфликты до их эскалации

---

## Раздел 3: Технический долг

### Симптомы
- Новые фичи занимают в 3x больше времени чем раньше
- Каждое изменение ломает что-то неожиданное
- Разработчики боятся трогать определённые части кода
- Тесты отсутствуют или постоянно красные

### Материалы
- [[technical-debt]] — управление техническим долгом
- [[tech-debt-management]] — стратегии погашения долга
- [[refactoring-catalog]] — техники рефакторинга
- [[testing-fundamentals]] — стратегии тестирования
- [[clean-code]] — принципы чистого кода

### Действия
1. Инвентаризация долга — составить список проблемных зон
2. Выделить 20% спринта на погашение долга
3. Правило бойскаута: оставляй код чище чем нашёл
4. Начать писать тесты для нового кода

## Раздел 4: Проблемы с командой

### Симптомы
- Команда слишком маленькая для объёма работы
- Высокая текучка — люди уходят
- Новые люди долго входят в проект
- Ключевые знания у одного человека (bus factor = 1)

### Материалы
- [[hiring-engineers]] — найм инженеров
- [[interview-process-design]] — дизайн процесса интервью
- [[onboarding]] — онбординг новых сотрудников
- [[motivation]] — мотивация команды
- [[building-engineering-team]] — построение команды
- [[team-culture]] — командная культура
- [[team-dynamics]] — динамика команды

### Действия
1. Провести stay-интервью — узнать почему люди остаются/уходят
2. Улучшить онбординг — документация, менторство
3. Распределить знания — парное программирование, ротация
4. Если нужно нанимать — [[sourcing-candidates]]

## Раздел 5: Проблемы лидерства

### Симптомы
- Приоритеты меняются каждую неделю
- Руководитель контролирует каждый PR
- Нет чёткого технического направления
- Команда не чувствует автономию

### Материалы
- [[delegation]] — делегирование
- [[performance-management]] — управление производительностью
- [[one-on-one-meetings]] — проведение 1-1 встреч
- [[em-fundamentals]] — основы инженерного менеджмента
- [[okrs-kpis]] — цели и метрики
- [[technical-vision]] — техническое видение
- [[stakeholder-management]] — управление стейкхолдерами

### Действия
1. Зафиксировать приоритеты на квартал — [[okrs-kpis]]
2. Делегировать решения ближе к исполнителям
3. Установить регулярный ритм 1-1 встреч
4. Создать техническое видение и донести его до команды
